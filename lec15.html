1. The Big Picture: Why This Lecture Matters

This lecture sits at a crucial crossroads in your programming languages course. You've been learning Scheme — a language built around a beautifully simple core: lambdas (anonymous functions) and function calls. The philosophy of Scheme is that everything complex can be explained in terms of these simple building blocks.

Today's lecture extends that philosophy in three directions:

First, we finally see why continuations (introduced in the previous lecture) are useful. Last time, you learned the mechanism — how call/cc captures the current state of execution. Now you'll see the purpose: using them to jump out of deep computations, and even to build a cooperative threading system from scratch.

Second, we learn that continuations aren't magic. Through Continuation Passing Style (CPS), we can achieve the same power using nothing but lambdas and function calls — the core primitives you already know. This is the Scheme philosophy in action: every fancy feature can be reduced to the basics.

Third, we shift gears entirely into memory management — the nuts and bolts of how programming languages actually store all the data structures, return addresses, and variables your programs create. This sets the stage for understanding why different languages make different tradeoffs.

How it all connects

Continuations → Green Threads → Memory Management forms a beautiful arc. Continuations require the runtime to keep old activation records alive (rather than throwing them away when functions return). That's a memory management challenge! So understanding how continuations work leads naturally to asking how do we manage all this memory?

2. Quick Scheme Foundations Recap

Before diving in, let's make sure you're solid on the Scheme features this lecture uses heavily. If you already know these, this section will be quick. If anything is fuzzy, this is your chance to lock it in.

Functions (Lambdas)

In Scheme, you create an anonymous function with lambda. A function is a first-class value — you can pass it around, store it in a variable, or call it like any other value.

;; A function that takes x and returns x + 1
(lambda (x) (+ x 1))

;; Give it a name
(define add1 (lambda (x) (+ x 1)))

;; Shorthand: define with parameter list
(define (add1 x) (+ x 1))

;; Call it
(add1 5)  ; → 6

Lists and their operations

Scheme's fundamental data structure is the linked list. You manipulate lists with three core operations:

;; car — get the first element
(car '(10 20 30))  ; → 10

;; cdr — get everything except the first element
(cdr '(10 20 30))  ; → (20 30)

;; null? — check if a list is empty
(null? '())        ; → #t (true)
(null? '(1))       ; → #f (false)

Named let (Internal Recursive Functions)

This is used extensively in this lecture. A named let defines a local recursive function and immediately calls it:

;; Count from 0 to 4
(let loop ((i 0))
  (when (< i 5)
    (display i)
    (loop (+ i 1))))

Here's how to read it: let loop ((i 0)) means "define a function called loop that takes one parameter i, and immediately call it with i = 0." Inside the body, loop can call itself recursively.

cond (Multi-way If)

cond is Scheme's if-then-else-if chain. It's cleaner than nesting if statements:

(cond
  ((< x 0)  "negative")   ; if x < 0, return "negative"
  ((= x 0)  "zero")       ; else if x = 0, return "zero"
  (else      "positive"))  ; else return "positive"

set! (Mutation / Assignment)

set! (pronounced "set bang") is Scheme's assignment operator. It destructively changes a variable's value — this is not functional programming!

(define x 10)
(set! x 20)  ; x is now 20

begin (Sequencing)

begin evaluates multiple expressions in order and returns the value of the last one:

(begin
  (display "hello")  ; side effect, value thrown away
  (display "world")  ; side effect, value thrown away
  42)                ; → returns 42

3. What is call/cc?

Now let's talk about the star of the show: call/cc, which stands for "call with current continuation."

To understand call/cc, you first need to understand what a continuation is. Think of it this way:

Real-world analogy

Imagine you're reading a book. You're on page 147, halfway through a sentence. Your continuation is everything you're about to do next: finish the sentence, read the rest of the page, read all remaining pages, close the book, put it on the shelf. It's the "rest of the computation" that follows from where you currently are.

Now imagine you could take a photograph of your reading state — the page, the line, the word you're on — and tuck it in your pocket. You go do something completely different. Later, you pull out the photo, and snap — you're back on page 147, continuing right where you left off.

That photograph is a continuation object. And call/cc is the camera that takes the photo.

The Formal Mechanics

Here's exactly what call/cc does, step by step:

Step 1: call/cc captures the current continuation — a snapshot of "what will happen next after this call/cc expression finishes."

Step 2: It wraps that continuation up into a special function. We'll call it k.

Step 3: It takes the function you gave it (the argument to call/cc) and calls that function, passing k as an argument.

Step 4: If the code never calls k, then call/cc just returns normally — whatever the function returns becomes the value of the call/cc expression.

Step 5: If the code does call k with some value (like (k 42)), then execution immediately jumps back to where the call/cc was, and the call/cc expression evaluates to that value (42). All the intermediate computation is abandoned.

;; Basic pattern
(call/cc
  (lambda (k)          ; k = the continuation
    ;; ... do stuff ...
    ;; if we call (k value), jump out!
    ;; if we never call k, return normally
    ))
Key insight

Calling a continuation is like a "non-local goto" — it doesn't just return from the current function. It can jump all the way back to wherever the continuation was created, no matter how deeply nested the code currently is. This is far more powerful than a regular return statement.

Check your understanding

What does this expression evaluate to?

(+ 1 (call/cc (lambda (k) (+ 2 (k 10)))))
  • 11
  • 13
  • 10
  • 3
When (k 10) is called, execution jumps back to the call/cc, which "returns" 10. The (+ 2 ...) around the (k 10) is never completed — it gets abandoned. So the expression becomes (+ 1 10) = 11. The + 2 is just skipped!

4. Example: Efficient Product of a List

Now let's see call/cc solve a real (if somewhat artificial) problem. The professor uses this as a gentle first example before the more impressive green threads example.

The Problem

We have a list of integers, like (3 5 0 7 2 4), and we want to compute the product of all integers in the list. The twist: we're told that multiplication is expensive (imagine these are huge numbers with thousands of digits — multiplying them takes a long time). So we want to avoid multiplying when we don't have to.

If we spot a zero anywhere in the list, the entire product is zero. So we'd love to bail out immediately without doing any multiplications at all.

The Naive Approach (Without call/cc)

In C++ or Python, you might do two passes: first scan for zeros, then multiply. That works but scans the list twice. Or you could multiply as you go and check for zero each time, but you'd still do all the multiplications up to the zero.

The call/cc Approach: One Pass, Zero Multiplications if Zero Found

Here's the code — read the comments carefully:

(define (prod ls)
  (call/cc                        ; Step 1: capture continuation
    (lambda (k)                  ; k = "return from prod"
      (let pr ((items ls))      ; Step 2: recursive helper
        (cond
          ((null? items) 1)      ; empty list → product is 1
          ((= (car items) 0)     ; found a zero!
           (k 0))                 ; → JUMP OUT, return 0 immediately
          (else                    ; normal case:
           (* (car items)         ; multiply first element
              (pr (cdr items))))  ; × product of the rest
          )))))

Walking Through the Code

Let's trace through (prod '(3 5 0 7)):

Why is this interesting?

The key observation is that pr is not tail-recursive. When pr calls itself in the else branch, there's still work to do afterward — the multiplication (* (car items) ...). So normally, all those pending multiplications would be sitting on the stack, waiting.

But (k 0) bypasses all of that. It's a "non-local goto" — it doesn't just return from the current pr call; it jumps all the way out to the top level. The pending multiplications on the stack are simply abandoned.

Connection to other languages

This is similar to try/catch in C++ or Python. You could wrap the multiplication loop in a try, and throw when you find a zero. The catch at the top level would return zero. Continuations are more general than exceptions (as we're about to see), but for this example, they serve a similar purpose.

Check your understanding

With no zeros, the second cond branch is never triggered, so k is never called. pr recurses to the end of the list, returns 1, then unwinds the recursion: 1 × 5 = 5, 5 × 4 = 20, 20 × 3 = 60, 60 × 2 = 120. The lambda given to call/cc returns 120, and call/cc returns that value normally.

5. Green Threads: The Real Power of Continuations

The product example was a warm-up. Now we see something truly powerful: continuations that outlive the function that created them. This is what makes Scheme's continuations more powerful than try/catch.

What Are Green Threads?

Definition

Green threads are threads managed entirely by user-level code (not the operating system). Only one thread runs at a time on a single CPU core. Threads voluntarily give up control to let other threads run. This is called cooperative multithreading.

Real-world analogy

Think of a group of friends sharing one microphone at karaoke. Only one person can sing at a time. When they finish a verse, they voluntarily hand the mic to the next person. Nobody is forced to give up the mic — it's cooperative. If someone hogs it (infinite loop), nobody else gets to sing.

Compare this to OS threads (preemptive multithreading), where a bouncer (the OS scheduler) can yank the mic away from you after a set time.

Advantages of green threads: No race conditions! Since only one thread runs at a time, two threads can never access the same memory simultaneously. Programming is much simpler. Node.js is a famous example of this model.

Disadvantages: You can't use multiple CPU cores. And if one thread infinite-loops, everything freezes.

Building Green Threads from Scratch

We're going to build an entire cooperative threading system using nothing but call/cc and lists. No operating system help. Here are the four components:

Component 1: The Thread List

(define thr-list '())

This is just a list. It holds all the threads that want to run but aren't currently running. Think of it as a waiting queue.

Component 2: Creating a Thread — thr

(define (thr thunk)
  (set! thr-list (append thr-list (list thunk))))

What's a thunk? It's a function that takes no arguments and is executed purely for its side effects (like printing). The term comes from compiler theory. A thunk is a piece of code you want to run later.

thr takes a thunk and appends it to the end of the thread list. It doesn't run the thunk — it just registers it as "I want to run."

Component 3: Starting Execution — start

(define (start)
  (let ((thunk (car thr-list)))   ; grab first thread
    (set! thr-list (cdr thr-list))  ; remove it from list
    (thunk)))                        ; RUN it!

start takes the first thread off the waiting list and runs it. At this point, we've left our "kernel" — we're now executing whatever code that thread contains. We won't come back until the thunk finishes or yields.

Component 4: Yielding Control — yield

This is the magic part. When a running thread wants to let someone else run, it calls yield:

(define (yield)
  (call/cc
    (lambda (k)
      (thr (lambda () (k 42)))  ; save our position, add to queue
      (start))))

Let's break this down piece by piece:

1. call/cc creates a continuation k that represents "where to resume this thread after yield returns."

2. (thr (lambda () (k 42))) — This is crucial. We create a new thunk (a zero-argument function) that, when called, will invoke the continuation k with the value 42. This thunk gets appended to the end of the thread list. The (lambda () ...) wrapper is necessary because k requires an argument (continuations always need a "return value"), but thunks take no arguments. So we wrap it.

3. (start) — Now we call start, which grabs the next thread from the front of the list and starts running it. Our current thread is no longer running — it's been saved as a continuation in the queue.

Why (lambda () (k 42))?

The professor pointed out a subtle bug: if you wrote (thr k) instead of (thr (lambda () (k 42))), it wouldn't work! k is a continuation that needs an argument. But thr expects a thunk (zero-argument function). The wrapper (lambda () (k 42)) converts a one-argument continuation into a zero-argument thunk. The value 42 is arbitrary — it becomes the return value of yield, but we throw it away anyway.

Putting It All Together: The "hi" Example

(begin
  (thr (lambda () (let f () (display "h") (yield) (f))))
  (thr (lambda () (let f () (display "i") (yield) (f))))
  (thr (lambda () (let f () (newline) (yield) (f))))
  (start))

Three threads are created, each in an infinite loop. The first prints "h", the second prints "i", the third prints a newline. Each one calls yield after its action to let the next thread go.

Key insight: Continuations survive the return of yield! When yield was called, it "returned" (via start) into a completely different thread. But the continuation KH still holds the state of Thread-H. Later, when KH is invoked, Thread-H picks up exactly where it left off. The activation records are frozen by the garbage collector, not reclaimed.

This is what makes scheme continuations more powerful than try/catch:

6. How Continuations Work Internally

To really understand what's happening, let's peek inside the Scheme interpreter.

Activation Records as a Linked List

When your program runs, the Scheme interpreter maintains a chain of activation records (stack frames). Each one contains an instruction pointer (IP) — what code to execute — and an environment pointer (EP) — a pointer to the next activation record (the caller's frame).

This chain is your continuation. It's the list of "things left to do":

A continuation is a linked list of activation records

What call/cc Does at This Level

When call/cc executes, it creates a reference (a pointer) to the current IP/EP pair. This reference keeps those activation records alive — even if the function that created them returns. In a language with garbage collection (like Scheme), the garbage collector says: "Someone still has a pointer to this data, so I can't reclaim it."

When you later invoke the continuation (k value), the interpreter restores the IP and EP to those saved values. It's as if you teleported back in time to that exact execution state.

Why this matters for memory management

In C, when a function returns, its stack frame is gone — deallocated. You can't go back to it. But in Scheme with continuations, the "stack frames" (activation records) might need to live on the heap, kept alive by the garbage collector. This is one reason why Scheme needs garbage collection and C doesn't (for this purpose). We'll explore this more in the memory management section.

C's Failed Attempt: ucontext.h

C tried to have something similar with getcontext and setcontext from <ucontext.h>. These save and restore the CPU registers (instruction pointer, etc.). But there's a critical problem: C uses stack-based storage. If the function that called getcontext has returned, its stack frame has been overwritten. Calling setcontext to restore that context points you at garbage.

These functions were so dangerous and rarely used that they were removed from POSIX. Scheme's garbage collector solves this problem — but C's manual memory management can't.

7. Continuation Passing Style (CPS)

Here's a profound idea: you don't need call/cc at all. You can get the same power using only lambdas and function calls — the most basic features of Scheme. The technique is called Continuation Passing Style (CPS).

The Core Idea

In CPS, every function takes one extra argument: its continuation. Instead of returning a value, a function calls its continuation with the value.

Real-world analogy

Normal style: You ask your assistant to compute something. They compute it and hand you the answer. You then decide what to do with it.

CPS: You ask your assistant to compute something, and you hand them an envelope with instructions for what to do with the answer. They compute it, then follow the instructions in the envelope. They never "return" to you — they just pass the result forward to whatever the envelope says.

Here's a simple comparison:

;; Normal style: function returns a value
(define (add a b)
  (+ a b))

;; CPS style: function calls continuation with the value
(define (add-cps a b k)
  (k (+ a b)))

;; Using normal style:
(display (add 3 4))            ; prints 7

;; Using CPS:
(add-cps 3 4 display)           ; prints 7

In the CPS version, add-cps never "returns" in the normal sense. It calls k (the continuation) with the result, and k decides what happens next. If k is display, the result gets printed.

8. CPS Product Example

Let's rewrite our product function in CPS — no call/cc needed:

(define (prod ls break)     ; break = top-level continuation
  (let pr ((ls ls)           ; list to process
           (k break))         ; k = current continuation
    (cond
      ((null? ls) (k 1))      ; empty → call continuation with 1
      ((zero? (car ls))      ; found zero!
       (break 0))             ; call TOP-LEVEL continuation with 0
      (#t                      ; normal case
       (pr (cdr ls)           ; recurse with rest of list
           (lambda (v)        ; new continuation:
             (k (* v (car ls))))))  ; multiply, pass to our k
      )))

Notice the differences from the call/cc version:

Extra arguments: prod takes break (the top-level continuation). pr takes k (the current continuation — initially set to break).

No returns: Instead of returning 1, we call (k 1). Instead of returning the multiplication result, we call (k (* v (car ls))).

The "break" escape: When we find zero, we call (break 0) — the top-level continuation. This is equivalent to (k 0) in the call/cc version. It "returns" 0 to whoever called prod, skipping all pending multiplications.

The lambda continuation: In the normal case, instead of doing (* (car ls) (pr (cdr ls))), we call pr with a new continuation: (lambda (v) (k (* v (car ls)))). This says "when you get the product of the rest of the list (v), multiply it by my first element, then pass the result to my continuation." This is how the multiplication gets "deferred" — it's captured in the continuation, not waiting on the stack.

Key difference from call/cc version

In the call/cc version, the pending multiplications pile up on the call stack. In the CPS version, there are no pending operations on the stack. The "what to do next" information is entirely in the continuation argument k. Each recursive call to pr is actually tail-recursive — there's nothing left to do after it. The continuations replace the stack.

10. Proper Tail Recursion

Scheme makes a language-level guarantee that other languages don't: if a function call is in tail position, the compiler must optimize it so that it doesn't consume extra stack space. This is called proper tail recursion, and it's mandatory in Scheme.

What is Tail Position?

A function call is in tail position if it's the very last thing a function does — there's no more work after it returns. When a call is in tail position, there's no reason to keep the caller's stack frame around; the caller has nothing left to do except return whatever the callee returns.

;; Tail-recursive: the recursive call is the LAST thing
(define (factorial-tail n acc)
  (if (= n 0)
      acc
      (factorial-tail (- n 1) (* n acc))))

;; NOT tail-recursive: multiplication happens AFTER the recursive call
(define (factorial n)
  (if (= n 0)
      1
      (* n (factorial (- n 1)))))

In factorial-tail, the recursive call is the last operation — the function just returns whatever the recursive call returns. The compiler can reuse the current stack frame. This is equivalent to a loop and uses O(1) stack space.

In plain factorial, after the recursive call returns, we still need to multiply by n. So the stack frame must be kept alive. This uses O(n) stack space.

11. Where Exactly is Tail Position?

Scheme formally defines which positions are "proper tail positions." Here's the map — the boxed/checked items are tail positions:

ConstructWhat's in Tail Position?What's NOT?
(lambda (...) e1 e2 ... en)The last expression en ✓All earlier expressions
(if test then else)Both then and else ✓The test expression
(cond (c1 e1) (c2 e2) ... (else en))Each result expression ✓The condition expressions
(and e1 e2 ... en)Only the last expression ✓All earlier expressions (might short-circuit)
(or e1 e2 ... en)Only the last expression ✓All earlier expressions (might short-circuit)
(begin e1 e2 ... en)The last expression ✓All earlier expressions
(let (...) e1 e2 ... en)The last expression ✓All earlier expressions + bindings

The pattern is consistent: the last expression (or the branches you commit to) are in tail position. Expressions that still have work to do afterward — like conditions that might branch, or earlier expressions in a sequence — are not.

Why "proper"?

"Proper" tail recursion means the standard formally defines where tail optimization is required. A smart compiler might optimize other cases too (like recognizing that (if (p x) #t #t) is equivalent to (p x)), but that's optional. The "proper" positions are the ones the programmer can rely on. If it's in the list above, the compiler must optimize it.

Check your understanding

In this code, is the call to g in tail position?

(define (f x)
  (+ 1 (g x)))
  • Yes — it's the last function call
  • No — there's still the addition of 1 to do after g returns
Even though (g x) appears last syntactically, the + still needs to add 1 to g's result. So f can't throw away its stack frame before calling g — it needs to stick around to do the addition. The call to g is NOT in tail position.

12. What Goes in Memory?

Now we shift to a completely different topic: memory management. This is about the practical reality of how programming languages work on real hardware with finite RAM.

Before discussing how to manage memory, let's catalog what needs to live in memory:

1. Contents of variables. Even in functional languages, once you have enough variables, their values need to be stored somewhere in RAM.

2. Return addresses (saved instruction pointers). When function A calls function B, we need to remember where in A to resume after B finishes.

3. Saved environment pointers. The pointer to the caller's activation record, so we can restore the calling environment.

4. Temps. Values the compiler invents names for. For example, in (f (g 1) (h 2) (i 3)), after calling g(1), the result needs to be stored somewhere while we call h(2). These temporary storage slots are "temps."

5. Code. Instruction pointers point to actual machine instructions, which are also stored in memory.

6. Constants. String literals like "hello world" in C are stored in memory. They're not variables, but they need space.

7. I/O buffers. When you call getc(f) in C, the file's input stream is buffered in memory. These buffers need space.

8. Heap objects. Anything created with malloc (C), new (C++/Java), or the equivalent. These are objects not tied to any particular variable or stack frame.

13. Static Allocation: The Fortran 58 Approach

The very first successful programming language, Fortran (circa 1958), used the simplest memory management possible: everything was statically allocated.

What "Static Allocation" Means

Every single piece of memory — every variable, every return address, every array, every constant — had a fixed location determined at compile time. The compiler and linker assigned exact addresses before the program ever ran.

If you declared an array of 100 integers, 100 words of memory were reserved at a specific address. The return address for function foo was always stored at the same memory location, no matter how many times foo was called.

Extremely simple to implementNo recursion possible
Predictable memory usage — you know exactly how much RAM the program needs before runningOver-allocation is common (arrays sized for worst case)
No stack overflow possibleEasy to attack (predictable memory layout)
Very fast — no allocation/deallocation overhead at runtimeNo shared libraries (each executable needs its own copy of library code)
Why no recursion?

This is the critical flaw. If function f has a single static slot for its return address, what happens when f calls itself? The inner call to f overwrites the return address that the outer call needs to get back to its caller. When the inner call finishes and tries to return, it has the wrong return address. Same problem for local variables and parameters — there's only one copy of each, shared by all calls to the function.

Fortran 58 dealt with this by saying: if your function calls itself (directly or indirectly), behavior is undefined. Recursion was simply illegal.

Check your understanding

Why can Fortran 58 guarantee that a program will never run out of memory at runtime?

  • Because all memory is allocated at compile time, so you know the total before running
  • Because Fortran programs are small
  • Because the OS provides unlimited memory
  • Because Fortran has a garbage collector
With static allocation, the total memory a program will ever use is determined at compile/link time. If you have enough RAM to load the program, it will never ask for more. There's no dynamic allocation to fail. This is a significant advantage for embedded systems and safety-critical applications.

14. Stack Allocation: Algol 60

Just two years after Fortran, Algol 60 introduced the stack — and with it, recursion.

How the Stack Solves Recursion

Instead of having one fixed slot for each function's variables, Algol 60 pushes a new activation record (stack frame) onto the stack every time a function is called. Each call gets its own copy of local variables, parameters, and return address. When the function returns, its frame is popped off.

Now f can call itself 1000 times, and each call gets its own separate storage. Recursion works!

Algol 60 Also Added: Variable-Size Locals

Algol 60 allowed arrays whose size was determined at runtime. Something like (in C-like syntax):

void f(int n) {
    int a[n];  // array of size n, on the stack
}

This reduces over-allocation: you allocate exactly the space you need at runtime.

The Downsides

Stack allocation introduces new problems: stack overflow (recursive calls can exhaust the stack), the runtime overhead of pushing/popping frames (though modern CPUs make this very fast), and potential for negative-size array attacks (where the stack shrinks instead of grows, corrupting the caller's data).

15. C's Compromise (1972)

C took the Algol 60 approach and stripped it down for safety and speed:

Keep: Stack-based allocation with recursion.

Remove: Variable-length local arrays. In (original) C, all local variables have sizes known at compile time. This means the compiler knows exactly how big each stack frame will be — it can allocate it with a single instruction (subtract a constant from the stack pointer).

Add: malloc / free for heap allocation. This gives programmers flexible, any-order allocation when they need it, while keeping stack allocation fast and predictable.

16. Activation Records in Detail

An activation record (frame) for a single function call contains:

Local variables (including parameters) — the values the function is working with.

Return address — the instruction pointer (IP) indicating where to resume in the caller after this function finishes.

[Optional] Saved environment pointer — a pointer to the caller's frame. On some systems, this can be computed from the stack pointer rather than stored explicitly.

Temps — compiler-generated slots for intermediate values.

In practice, much of this is cached in CPU registers for the currently executing function. The activation record may have slots reserved, but they contain stale values while the real values live in registers. The values get "spilled" to the activation record only when the CPU runs out of registers or when calling another function.

17. Nested Functions: Static Chain vs. Dynamic Chain

This section covers a subtle but important topic: what happens when functions are defined inside other functions, and the inner function accesses variables from the outer function.

The Problem

C and C++ don't allow nested function definitions (ignoring GCC extensions). But many languages do — Algol 60, Pascal, Scheme, Python, JavaScript. Consider:

int f(int x) {
    int g(int y) {
        return x + y;  // accesses x from f!
    }
    return g(10);
}

When g is executing, accessing y is easy — it's in g's own activation record. But x is in f's activation record. Where is that?

If f is recursive, there might be multiple activation records for f on the stack. Which one has the right x?

Two Chains Through the Stack

The solution involves two separate linked lists threaded through the activation records:

Dynamic chain — each frame points to the frame of its caller (who called me?). This is the standard call stack, present in every language. It represents the runtime call sequence.

Static chain — each frame points to the frame of its definer (the function that contains my definition in the source code). This represents the compile-time nesting structure.

The static chain may skip activation records that the dynamic chain visits

The static chain's length at any point is bounded by the nesting depth in the source code — no matter how deep the recursion goes, the static chain stays short.

Function Pointers in Nested Languages

In C, a function pointer is just an instruction address — one pointer. But in languages with nested functions, a function value is a pair: (instruction pointer, environment pointer). The IP says what code to run. The EP says which activation record to use for accessing the definer's variables. This pair is sometimes called a closure.

Full circle: closures and continuations

A closure is an (IP, EP) pair. A continuation is also an (IP, EP) pair — just pointing to a different place. They're the same data structure! This is why Scheme can implement continuations with the same machinery it uses for closures. And it's why C, which only has bare function pointers (no EP), can't safely support either closures or continuations. Everything in this lecture connects: continuations, closures, activation records, and memory management are all facets of the same underlying mechanism.

Final review

Why don't C and C++ need a static chain?

  • Because C/C++ don't allow nested function definitions (in the standard)
  • Because C/C++ use garbage collection
  • Because C/C++ functions are always tail-recursive
  • Because the static chain is the same as the dynamic chain in C
The static chain exists to let inner functions access variables from their enclosing (defining) function. Since C and C++ don't allow functions to be defined inside other functions (standard C, at least), there's never a need to "look up" a definer's frame. Every function is defined at the top level, so a function pointer is just a bare code address.

Summary: What You Learned

Continuations in action: call/cc captures "where we are" in execution. Calling a continuation jumps back to that point. Used for non-local exits (product example) and for implementing green threads (continuations that outlive their creating function).

Green threads: Cooperative multithreading built from scratch using call/cc, a list, and four small functions (thr-list, thr, start, yield).

CPS: Every function takes an extra continuation argument and calls it instead of returning. Achieves the same power as call/cc but with just lambdas.

Proper tail recursion: Scheme mandates tail-call optimization in well-defined positions. Tail calls reuse stack frames; non-tail calls grow the stack.

Memory management: Programs need memory for variables, return addresses, temps, code, constants, I/O buffers, and heap objects. Historical progression: Fortran's static allocation (simple, no recursion) → Algol 60's stack (recursion, but stack overflow) → C's compromise (fixed-size stack frames + heap).

Activation records: Contain locals, return address, saved EP, temps. Nested functions require a static chain (pointer to definer's frame) in addition to the dynamic chain (pointer to caller's frame).
