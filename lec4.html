Lecture 4

Syntax & Grammar

How do we rigorously define what counts as a valid program — and why does it matter?

Chapter 1

Why Grammars Exist: The Big Picture

Imagine you sit down and write a program in Python. You type it into your text editor, hit "Run," and Python tells you either "okay, I'll run this" or "SyntaxError: invalid syntax." How does Python know whether your program is valid or not?

Somewhere, someone had to write down the rules for what counts as a valid Python program. Those rules are what we call a grammar. And this lecture is all about how those rules work, what problems they can cause, and how the real world uses them.

🔗 Why This Matters

Grammars are the foundation of every programming language, every data format (HTML, JSON, XML), and every protocol (email, HTTP). If you ever design your own language, notation, or configuration file format, you'll need to write a grammar. Even if you don't, understanding grammars helps you understand why languages work the way they do — including weird rules you may have encountered in C, Python, or JavaScript.

The Fortran Problem: No Formal Rules

The very first successful programming language was Fortran, created by IBM in the 1950s. Fortran didn't have a formal grammar. Its "specification" was essentially: whatever IBM's compiler accepts is Fortran, and whatever it rejects is not.

IBM did write some manuals, but they were vague — lots of English descriptions and examples, nothing precise. This created a serious problem: if you were a competing company and you wanted to build your own Fortran compiler, you had no reliable reference. You'd have to reverse-engineer IBM's behavior, and IBM could change their compiler anytime (which they did) to break your work.

This was actually strategic for IBM — by not writing down the exact rules, they kept competitors at a disadvantage. It's one of the reasons IBM became the dominant computer company of that era.

The Algol Solution: A Formal Specification

The second major programming language, Algol (short for "algorithmic language"), was partly developed by IBM's competitors. One of its key goals was to have a formal specification for its syntax — precise, mathematical rules that anyone could read and independently implement a compiler from.

This meant that if Company A and Company B both read the Algol spec and both wrote compilers, their compilers would agree on which programs are valid. No more guessing, no more reverse-engineering.

Definition

A grammar is a formal specification for syntax. It acts as a contract between two groups: users of a language (people writing programs) and implementers (people writing compilers/interpreters). Users agree to follow the grammar when writing code, and implementers agree that any code following the grammar must be accepted and work correctly.

The technology Algol used for this specification was called a context-free grammar (CFG). This is the main subject of today's lecture. Before we dive into the formal definition though, let's build up the intuition piece by piece.

Chapter 2

What Is a Grammar, Intuitively?

Think of a grammar as a set of rules that tells you how to build valid sentences in a language. Just like English has rules (a sentence needs a subject and a predicate), programming languages have rules (a while loop needs a condition and a body).

But there's a crucial difference: English grammar rules are fuzzy, full of exceptions, and impossible to capture completely in a formal system. Programming language rules are designed to be precise and unambiguous (at least, that's the goal — as we'll see, achieving this is harder than it sounds).

Grammars Aren't Just for Programming Languages

This idea of having a formal grammar applies far beyond programming languages. Every time your web browser loads a page, it receives a string of bytes that must conform to the HTML grammar. HTML is a data language, not a programming language, but it still needs a grammar so that web servers and browsers agree on what valid HTML looks like.

The same goes for email (there's a grammar for email messages), XML, JSON, URLs — really any structured format where different pieces of software need to agree on what's valid.

🔗
Connecting to the big picture: A grammar is like a specification or contract. This is the same idea from software engineering — specs help different teams work independently while staying compatible. Grammars are specs for languages.
Chapter 3

Context-Free Grammars: The Formal Definition

Now let's get precise. A context-free grammar (CFG) has four components. I'm going to explain each one carefully, because these are the building blocks for everything else in this lecture.

Definition — Context-Free Grammar

A context-free grammar (CFG) consists of four things:

1. A finite set of terminals (also called tokens)
2. A finite set of non-terminals (disjoint from the terminals)
3. A start symbol (one of the non-terminals)
4. A finite set of production rules

Let's break each of these down.

Terminals (Tokens)

Terminals are the actual symbols that appear in the final program — the things a programmer types and that appear on screen. They're called "terminals" because they're the end points of the grammar; you can't break them down any further.

In a programming language like C, the terminals include:

Reserved words like if, else, while, return, break. Symbols like +, *, (, ), {, }, ;. Abstract tokens like ID (standing for any identifier, like myVar or count) and NUM (standing for any integer constant, like 42 or 0).

A language like C might have around five dozen different terminals. That's the complete vocabulary — every valid C program is built from these tokens.

Non-Terminals

Non-terminals are abstract categories that don't appear directly in your program — they exist in the grammar to describe structure. Think of them as labels for the different kinds of "building blocks" your language has.

For English, if you tried to write a grammar, your non-terminals might be things like "sentence," "subject," "predicate," "noun phrase," "adverbial phrase." For a programming language, non-terminals might include Program, Statement, Expression, FunctionDeclaration, and so on.

Non-terminals are "in your head" in the sense that the programmer never types them. They're the grammar designer's way of describing the different structural components of the language.

Key Idea

Choosing good non-terminal names matters. You want them to correspond to natural, meaningful units of your language — things like "expression" or "statement" — so that the grammar is readable and makes intuitive sense.

Start Symbol

The start symbol is a special non-terminal that represents the "whole thing" you're trying to describe. For a programming language, it's typically called Program, because you want to describe what a complete program looks like. For English, it might be Sentence.

You need at least one non-terminal (the start symbol), so the set of non-terminals must be non-empty.

Production Rules

This is where the interesting stuff happens. Production rules (or just "rules") describe how to build the structure of your language. Each rule has this form:

NonTerminal → sequence of terminals and non-terminals

The left-hand side is always a single non-terminal. The right-hand side is a sequence of zero or more symbols — each symbol can be either a terminal or a non-terminal. The right-hand side can even be empty (the empty string), meaning "this non-terminal can expand to nothing."

Here's an example of a rule: S → T ( means "the non-terminal S can be replaced by the non-terminal T followed by the terminal (." Another: T → ε (where ε means "empty") means "the non-terminal T can expand to nothing at all."

A Simple Example Grammar

Let's look at a tiny grammar for arithmetic expressions. It has only one non-terminal (E for "expression"), which is also the start symbol:

Grammar: Simple Expressions E → ID E → NUM E → E + E E → E * E E → ( E )

Reading these rules in English:

Rule 1: An expression can be an identifier (a variable name like x or foo).
Rule 2: An expression can be a number (like 42).
Rule 3: An expression can be two expressions with a + between them.
Rule 4: An expression can be two expressions with a * between them.
Rule 5: An expression can be another expression wrapped in parentheses.

This grammar generates things like x, 42, x + 3, x + 3 * (y + 2), and infinitely many other arithmetic expressions. Pretty powerful for just five rules!

🔗 Why "Context-Free"?

The name "context-free" comes from the fact that each rule has exactly one non-terminal on the left-hand side. This means you can always apply a rule to replace a non-terminal regardless of what's around it (its "context"). There are more powerful kinds of grammars (context-sensitive grammars) where the left side can have multiple symbols, meaning you can only apply the rule in certain contexts. But context-free grammars hit the sweet spot for programming languages: powerful enough to describe their syntax, simple enough to work with efficiently.

Context-free grammars were originally developed for natural languages (like English), but they turned out to be too simple for that purpose — English is far too messy and exception-filled. However, since programming languages are deliberately designed to be simpler and more regular than English, CFGs turned out to be a perfect fit.

Chapter 4

Parse Trees: Proving a Program Is Valid

So you have a grammar and you have a string of tokens (your program). How do you know if the program is valid according to the grammar? You build a parse tree.

Definition

A parse tree is a tree diagram that proves a sequence of tokens belongs to a language. The root is the start symbol, the leaves (read left to right) give you the original tokens, and each internal node corresponds to exactly one production rule from the grammar.

How a Parse Tree Works

A parse tree has three types of things in it:

The root is the start symbol (e.g., E for expression or Program for a whole program).

Internal nodes are non-terminals. Each internal node, together with its children, corresponds to one production rule. For example, if an internal node is labeled E and has three children E, +, E, that corresponds to the rule E → E + E.

Leaves are terminals (tokens). When you read all the leaves from left to right, you get back the original token sequence — your program.

Sometimes an internal node can have no children at all. This corresponds to a rule where the right-hand side is empty (like T → ε). The node exists, but it contributes nothing to the final token sequence.

Example: Parsing x + 3 * (y + 2)

Let's use our simple grammar to parse the expression x + 3 * (y + 2). We need to build a tree where the root is E and the leaves, read left to right, give us x + 3 * ( y + 2 ).

In this tree, every internal node (blue circle) corresponds to a grammar rule. The root E used the rule E → E + E. The right child used E → E * E. The bottom-right used E → ( E ). And so on. Reading the yellow leaves left to right gives us back: x + 3 * ( y + 2 ).

This tree is essentially a proof that the token sequence x + 3 * (y + 2) is a valid expression according to our grammar. It's a very easy proof to check — just verify that each node matches a rule.

Chapter 5

The Danger: Ambiguous Grammars

Here's where things get tricky — and this is the central problem of the entire lecture.

Look at our simple expression grammar again. We just built one parse tree for x + 3 * (y + 2). But can we build a different parse tree for the exact same expression?

Yes, we can. And that's a problem.

Definition

A grammar is ambiguous if there exists some string of tokens for which you can build two (or more) different parse trees. Each different parse tree represents a different interpretation of the program.

Why Ambiguity Is Dangerous

In mathematics, having two different proofs of the same theorem is fine — even cool. But in programming languages, two different parse trees for the same code means two different interpretations of what the program does. One compiler might execute the program one way, another compiler might execute it differently, and the programmer gets confused or gets wrong results.

The Two Parse Trees for x + 3 * (y + 2)

With our simple grammar, there are two ways to parse x + 3 * (y + 2):

Parse Tree 1 (correct): The root E splits into E + E, where the right E splits into E * E. This means: first compute 3 * (y + 2), then add x. This respects the standard mathematical precedence where multiplication happens before addition.

Parse Tree 2 (incorrect): The root E splits into E * E, where the left E splits into E + E. This means: first compute x + 3, then multiply by (y + 2). This is wrong — it violates the precedence rule we all learned in math class.

The grammar allows both trees because it has no concept of "multiplication before addition." As far as the grammar is concerned, E → E + E and E → E * E are equally valid rules that can be applied at any level.

Warning

Ambiguity is especially dangerous because it can sneak up on you. You might think your grammar is fine and only discover the ambiguity later. The opposite problem — a grammar that rejects valid programs — is less dangerous because users will immediately complain, and you'll fix it. Ambiguity is silent until it causes wrong behavior.

In fact, the problem of determining whether an arbitrary context-free grammar is ambiguous is undecidable — meaning there's no algorithm that can always tell you yes or no. You can sometimes prove specific grammars are unambiguous, but there's no general method. This is why we spend so much time learning to spot and fix ambiguity by hand.

🔗
Two types of ambiguity we'll fix: The ambiguity in our grammar comes from two sources that we need to handle separately. First, precedence ambiguity: should + or * be grouped first? Second, associativity ambiguity: in a + b + c, should we group (a+b)+c or a+(b+c)? Let's tackle them one at a time.
Chapter 6

Fixing Precedence Ambiguity

The Problem

Consider E + E * E. Should this be (E + E) * E or E + (E * E)? We know from math that multiplication has higher precedence than addition, so the answer is E + (E * E). But our grammar doesn't know this.

The Fix: Separate Non-Terminals for Each Precedence Level

The idea is beautifully simple: instead of one non-terminal E that handles everything, we create multiple non-terminals — one for each precedence level. Operators with higher precedence get "deeper" (more nested) non-terminals.

Here's the key insight: deeper non-terminals = higher precedence. Things that are deeper in the tree get grouped together first, which means they get evaluated first.

Grammar: Fixed Precedence (* E handles + at the top level *) E → E + E E → F (* F handles * at a deeper level — so * binds tighter *) F → F * F F → ID F → NUM F → ( E )

Here's what changed: we introduced a new non-terminal F (for "factor"). The rule E → E + E only works at the E level. The rule F → F * F only works at the F level. And crucially, + takes Es as operands (which can be sums or factors), but * takes Fs as operands (which can only be factors — not bare sums).

This means if you try to build the "wrong" parse tree (where + is deeper than *), you'll get stuck — there's no rule that lets you multiply two expressions that contain un-parenthesized additions.

▶ Step-Through: Why the Bad Parse Fails
1 / 5

Goal: Try to parse x + 3 * (y + 2) with the wrong grouping: (x + 3) * (y + 2).

Let's see if our new grammar allows this incorrect interpretation. We'll try to build the parse tree bottom-up and see where we get stuck.

Step 1: The tokens x and 3 are each an F (via F → ID and F → NUM).

So far so good. Every identifier and number can become a factor (F). This part always works.

Step 2: Each F can become an E (via E → F). Now we want to combine x and 3 with + using E → E + E.

We successfully build: E(x) + E(3) → E(x + 3). This gives us an E for the sub-expression x + 3. No problems yet.

Step 3: Now we need to multiply (x + 3) by (y + 2). That means we need a rule like ? * ?. The only multiplication rule is F → F * F. So we need both sides to be F.

Here's the problem. We have E(x + 3) on the left side. But F → F * F requires both operands to be Fs, not Es. Is there a way to turn our E into an F?

Step 4: STUCK! We have an E but need an F. There's no rule F → E. The only way to get an E inside an F is via F → ( E ) — but that requires actual parentheses in the token stream, which aren't there!

The bad parse tree cannot be built. Our grammar successfully prevents the wrong precedence. The only way to parse this expression is the correct way: x + (3 * (y + 2)), where the * is grouped at the deeper F level.

The grammar went from 5 rules to 6, and from 1 non-terminal to 2. That's the cost: more complexity. But the payoff is that the wrong parse tree is now impossible.

Chapter 7

Fixing Associativity Ambiguity

The Problem

Even with our precedence fix, the grammar is still ambiguous! Consider x * y * z. Since both uses of * are at the same precedence level, the grammar allows two trees:

Left-associative: (x * y) * z — do the left multiplication first.
Right-associative: x * (y * z) — do the right multiplication first.

Why Associativity Matters

You might think "multiplication is associative in math, so it doesn't matter." But in computers, it does matter:

Integer overflow: If x * y overflows but y * z doesn't, the order changes the result. For example, with 32-bit integers, (1000000 * 1000000) * 1 overflows, but 1000000 * (1000000 * 1) also overflows the same way — bad example. But (a + b) + c vs a + (b + c) can differ if a + b overflows.

Floating-point rounding: (0.1 + 0.2) + 0.3 and 0.1 + (0.2 + 0.3) can give different results due to rounding.

Non-associative operators: For division, (a / b) / c is very different from a / (b / c). Same for subtraction.

The Fix: Left Recursion for Left Associativity

The trick is elegant: if you want an operator to be left-associative, put the recursive non-terminal on the left side of the rule. If you want right-associativity, put it on the right.

Grammar: Fixed Precedence AND Associativity (* Left side recurses through E → forces left associativity for + *) (* Groups as: ((a + b) + c) + d *) E → E + T E → T (* Left side recurses through T → forces left associativity for * *) (* Groups as: ((a * b) * c) * d *) T → T * F T → F (* Atoms: the base cases *) F → ID F → NUM F → ( E )

Notice the asymmetry: in E → E + T, the left side of + is E (which can contain more additions), but the right side is T (which can only be a single term, not a bare addition). This forces additions to group to the left.

Similarly, T → T * F forces multiplications to group to the left.

🔍 Detailed Example: Why x * y * z can only be (x * y) * z ▶

Let's try to build the wrong tree: x * (y * z).

Starting from the leaves: y is an F, z is an F. We use T → T * F to get T(y * z). Wait — for this rule, the left side of * must be a T, and y can become a T via T → F. So we get T(y * z).

Now, x is an F, and we need T(x) * T(y * z). But the rule is T → T * F — the right side of * must be an F, not a T. And T(y * z) is a T, not an F. There's no rule F → T either. We're stuck!

The correct tree works: x is F, y is F. T(x) * F(y) → T(x * y). Then T(x * y) * F(z) → T((x * y) * z). Left-associative!

Chapter 8

The Complete Unambiguous Expression Grammar

Let's put it all together and really understand the final grammar:

Final Unambiguous Grammar E → E + T (* addition: left-associative, low precedence *) E → T (* an expression can just be a term *) T → T * F (* multiplication: left-associative, high precedence *) T → F (* a term can just be a factor *) F → ID (* a factor can be a variable *) F → NUM (* a factor can be a number *) F → ( E ) (* a factor can be a parenthesized expression *)

The three levels form a hierarchy: E (expressions with +) → T (terms with *) → F (atomic factors). Each level can "drop down" to the next (E → T, T → F), and the deepest level (F) can jump all the way back to the top via parentheses (F → ( E )). This creates a beautiful cycle where parentheses let you override precedence.

Key Pattern

Whenever you need to add a new precedence level, you add a new non-terminal between the existing ones. Want to add an exponentiation operator ^ with even higher precedence than *? Add a non-terminal between F and the atoms. Want to add a low-precedence comparison operator? Add one above E. The pattern scales naturally.

Chapter 9

Real-World Example: The C Statement Grammar

Now let's leave our toy expression grammar and look at something real. Here's (part of) the grammar for statements in C. The C standard writes grammar rules in a slightly different notation: the left-hand side, followed by a colon, followed by indented alternatives.

C Statement Grammar (subset) statement : ; (* empty statement *) break ; (* break *) continue ; (* continue *) expr ; (* expression statement *) return expr ; (* return with value *) return ; (* return without value *) goto ID ; (* goto *) { statements } (* compound/block *) while ( expr ) statement (* while loop *) for ( expropt ; expropt ; expropt ) statement (* for loop *) switch ( expr ) statement (* switch *) if ( expr ) statement (* if without else *) if ( expr ) statement else statement (* if with else *) do statement while ( expr ) ; (* do-while loop *)

A few important things to notice:

The simplest statement is just a semicolon — the empty statement. It does nothing, but it's syntactically valid.

Assignment is not a special statement in C. In C, x = 5 is an expression, not a statement. The "assignment statement" x = 5; is actually an expression followed by a semicolon — it's just the general expr ; rule. This is different from many other languages (like Python) where assignment is a statement.

No semicolons after compound statements. Look at the while, for, if, etc. — none of them end with a semicolon. Why? Because the sub-statement inside them already provides its own semicolon. If you write while (x > 0) x--;, the semicolon belongs to the x--; expression statement, not to the while loop itself.

Parentheses in return are optional. The rule says return expr ; — the expression can be anything, including a parenthesized expression. So return (x+1); is valid, but those parentheses belong to the expression, not to the return statement. Some C programmers always put parentheses around return values as a style choice, but it's not required by the grammar.

Chapter 10

Why Do while and if Need Parentheses?

Look at the while rule: while ( expr ) statement. Those parentheses around the expression aren't optional — they're required by the grammar. But why? Wouldn't while expr statement be cleaner and simpler?

Let's imagine a simplified C (call it "EZ-C") where we drop those parentheses. The while rule becomes: while expr statement.

The Ambiguity Proof

To prove this is ambiguous, we need to find one program that can be parsed in two different ways. Consider this code in our parenthesis-free language:

while i(5) ;

This can be parsed in two ways:

Interpretation 1: The expression is i(5) (calling function i with argument 5), and the body is ; (the empty statement). This means: keep calling i(5) as long as it returns a truthy value, doing nothing in the loop body.

Interpretation 2: The expression is just i (checking whether i is truthy), and the body is (5); (evaluating 5 and throwing it away). This means: as long as i is truthy, evaluate 5 and discard it.

Without the required parentheses, the parser can't tell where the expression ends and the statement begins. The parentheses while (expr) create an unambiguous boundary.

Key Lesson

When designing syntax, parentheses (or other delimiters) often serve as disambiguation markers. They tell the parser exactly where one part ends and another begins. Removing them can introduce ambiguity even if the resulting code "looks obvious" to a human reader.

But do-while Doesn't Need Them?

Interestingly, the parentheses in do statement while ( expr ) ; are technically not needed for disambiguation. Even without them (do statement while expr ;), the grammar would still be unambiguous. Why? Because the semicolon at the end forces the parser to consume the entire expression — there's no way to "split" the expression early.

So why are the parentheses there? Consistency. The designers of C figured that if while requires parentheses around its expression, then do-while should too, even though it's not strictly necessary. Otherwise, programmers would get confused and start omitting parentheses where they're actually needed.

Chapter 11

The Dangling Else Problem

This is one of the most famous ambiguities in programming language design, and it affects C, C++, Java, and many other languages.

The Problem

Look at the two if rules in our grammar:

statement → if ( expr ) statement statement → if ( expr ) statement else statement

Now consider this code:

if (a < 0) if (b < 1) x = y; else z = w;

Which if does the else belong to? There are two valid interpretations:

Interpretation 1: else matches inner if (CORRECT) if (a < 0) if (b < 1) x = y; else ← attached to inner if z = w;
Interpretation 2: else matches outer if (WRONG) if (a < 0) if (b < 1) x = y; else ← attached to outer if z = w;

Both are valid parse trees according to the grammar. The grammar is ambiguous!

The universally agreed convention is: an else always matches the closest unmatched if. So Interpretation 1 is correct. But the grammar itself doesn't enforce this.

The Fix: Introducing "Matched Statements"

The fix follows the same pattern we've used before: introduce a new non-terminal that restricts what's allowed in certain positions.

The problem is specifically: in an if-else statement, the "then" part (between the if condition and the else) must not be an if-without-else. If it were, the else could ambiguously attach to either the inner or outer if.

So we create a new non-terminal stmu (short for "matched statement" or "statement-unambiguous") that includes every kind of statement except a bare if-without-else:

Fixed Grammar (dangling else resolved) statement → stmu (* any matched statement *) statement → if ( expr ) statement (* if without else *) statement → if ( expr ) stmu else statement (* if-else: then-part must be matched! *) stmu → (* every other kind of statement: ; | break; | expr; | { ... } | while ... | etc. *) stmu → if ( expr ) stmu else stmu (* a fully matched if-else is also matched *)

The key change: in the if-else rule, the then-part uses stmu instead of statement. This means the then-part cannot be a bare if-without-else. If there's a nested if before the else, it must have its own else — ensuring the outer else is unambiguously attached to the outer if.

Real-World Note

Interestingly, the actual C standard doesn't use this fix. Instead, they write the ambiguous grammar and add a footnote that says essentially: "Yes, we know this is ambiguous. Resolve it by matching each else with the closest unmatched if." So they cheat — they use an informal rule instead of fixing the grammar formally. For precedence and associativity, they do write the unambiguous grammar, but for the dangling else, they take the shortcut.

✎ Check Your Understanding

The dangling else problem is caused by which grammar rules being combined?

Correct! The ambiguity arises because having both if-without-else and if-else as options means that when an else appears after nested ifs, it's unclear which if it belongs to. The fix is to restrict the "then" part of an if-else to only allow fully-matched statements.
Not quite. The dangling else specifically comes from having both if (expr) statement (no else) and if (expr) statement else statement (with else) in the grammar. When you nest them, the else could attach to either if.
Chapter 12

Abstract Syntax Trees vs. Concrete Syntax Trees

Now here's a twist. After spending all this effort making grammars unambiguous, the real world often does something surprising: it uses ambiguous grammars on purpose.

Wait, what? Let me explain.

Definition

A concrete syntax tree (CST), also called a parse tree, mirrors the grammar exactly. Every rule application, every intermediate non-terminal, every parenthesis and keyword appears in the tree. It's the full, faithful representation of the grammar.

Definition

An abstract syntax tree (AST) strips away unnecessary syntactic detail and keeps only the semantic structure — what the program means. Parentheses, intermediate non-terminals, and other "scaffolding" are removed.

Example: x + y

The CST has 6 internal nodes (E, E, T, T, F, F) plus 3 leaves. The AST has just 1 internal node (+) and 2 leaves. The AST captures everything we need to understand the expression, while the CST captures every detail of the grammar derivation.

Why This Matters in Practice

In real compilers, people often work with ASTs rather than CSTs for several reasons: they're smaller (less memory), simpler (easier to work with), and they correspond to how programmers think about their code.

Because of this, many language specifications are written with intentionally ambiguous grammars (which produce nice, clean ASTs) plus side rules that say how to resolve the ambiguity ("use these precedence and associativity rules"). This is extremely common in practice.

Chapter 13

BNF vs. EBNF: Grammar Notations

One annoying thing about grammars is that everyone uses a slightly different notation. But they all describe the same underlying concept. Let's learn the two most important ones.

Definition

BNF (Backus-Naur Form) is the original, minimal notation for context-free grammars. It supports only three things: concatenation (symbols in a row), alternatives (multiple rules for the same non-terminal), and recursion. Nothing else — no shortcuts, no repetition operators.

BNF Example: Defining "digits" (* A single digit *) <digit> ::= 0 | 1 | 2 | ... | 9 (* One or more digits — must use recursion *) <digits> ::= <digit> <digits> ::= <digit> <digits>

In BNF, non-terminals are enclosed in angle brackets < >, the symbol ::= means "is defined as," and | means "or" (alternative). To express "one or more," you have to use recursion — there's no + operator.

Definition

EBNF (Extended BNF) adds convenience operators borrowed from regular expressions. It can express everything BNF can, but more concisely. Any EBNF grammar can be mechanically translated to plain BNF.

Here are the EBNF extensions:

EBNF Syntax Meaning BNF Equivalent
x* or {x} Zero or more x's Recursive rules (A → ε | x A)
x+ One or more x's Recursive rules (A → x | x A)
x? or [x] Optional: x or nothing Two rules (A → x | ε)
( ... ) Grouping (for nesting) Introduce a new non-terminal
x | y x or y (alternative) Same — BNF has this too

EBNF is what most real-world grammar specifications use because it's less verbose. But it's important to understand that EBNF doesn't add any power — anything you can write in EBNF can be converted to plain BNF. EBNF is just syntactic sugar.

Key Principle

If you see an "EBNF" notation and you can't figure out how to translate it to plain BNF, something's wrong — either with the notation or your understanding. Legitimate EBNF should always have a straightforward BNF translation. This is the litmus test.

Chapter 14

Real-World Grammar: RFC 5322 (Email)

Let's look at a real grammar specification that you interact with every day: the grammar for email messages, defined in Internet RFC 5322.

Every email you send or receive is a string of bytes that must conform to this specification. If the bytes don't match the grammar, the recipient's email server might reject your message as spam.

The Message-ID Grammar

Every email has a unique Message-ID header. Here's what one looks like:

Message-ID: <eggert.$9321.fsf@cs.ucla.edu>

And here's the grammar that defines what's valid inside that field:

RFC 5322 EBNF (email Message-ID) msg-id = "<" dot-atom-text "@" id-right ">" id-right = dot-atom-text / no-fold-literal (* / means OR *) no-fold-literal = "[" *dtext "]" (* * = zero or more *) dtext = %d33-90 / %d94-126 (* ASCII ranges *) dot-atom-text = 1*atext *("." 1*atext) (* 1* = one or more *) atext = ALPHA / DIGIT / "!" / "#" / "$" / ... / "~"

This notation has a few quirks worth understanding:

/ means "or" — not a division sign, not a token. It's a meta-symbol. The reason they use / instead of | is historical: when John Postel designed this notation at UCLA in the 1970s, the terminals they had didn't have vertical bar characters on their keyboards! So / became the convention.

* before something means "zero or more" — similar to regex, but the * goes in front. 1* means "one or more."

Parentheses ( ) are for grouping — they're not tokens in the email syntax; they're meta-syntax for the grammar notation itself.

%d33-90 means decimal ASCII characters 33 through 90 — a range of characters. For example, ASCII 33 is ! and ASCII 126 is ~.

Tokens are in double quotes: "<", "@", ">", ".", etc.

Security Implications

The lecture made an interesting point about grammars and security: a grammar tells you what's valid, but it doesn't tell you what's safe. A malicious actor could craft a message-ID that perfectly conforms to this grammar but encodes a bash script or sensitive information. Since spaces aren't in the allowed character set, you can't embed every bash program — but you can still encode plenty of data.

For example, a compromised email client could leak secrets by encoding them into message-IDs. The messages would pass every grammar check perfectly, but they'd be carrying hidden information. This is a reminder that grammars are specifications for form, not intent.

Chapter 15

Real-World Grammar: XML 1.1

The last real-world grammar we'll look at is for XML 1.1, the data interchange format that competes with JSON (and is the basis for HTML). This uses yet another EBNF notation — the one used by W3C web standards.

Simplified XML 1.1 Grammar element ::= EmptyElemTag | STag content ETag content ::= CharData? ((element | Reference) CharData?)* EmptyElemTag ::= '<' Name (S Attribute)* S? '/>' STag ::= '<' Name (S Attribute)* S? '>' ETag ::= '</' Name S? '>' Attribute ::= Name Eq AttValue Name ::= NameStartChar (NameChar)*

This notation uses ::= for the definition arrow, | for alternatives, ? for optional, * for zero or more, + for one or more, parentheses for grouping, and single quotes for tokens.

Reading the first rule: an element is either an EmptyElemTag (a self-closing tag like <br/>) or a STag (start tag) followed by content followed by an ETag (end tag) — the familiar <div>...</div> pattern.

Transferable Skill

The point of looking at these different notations (C standard, RFC 5322, XML W3C) isn't to memorize each one. It's to develop the ability to read any grammar notation you encounter. Every notation is slightly different, but they all express the same underlying concepts: terminals, non-terminals, alternatives, repetition, and optionality. Once you understand these concepts, you can read any grammar.

✎ Final Check

In the XML grammar above, what does (S Attribute)* S? mean?

That's right! The * after the parenthesized group means "zero or more" repetitions of the whole group (S followed by Attribute). Then S? means the final whitespace is optional. So a tag can have no attributes, one, or many — each preceded by whitespace — with optional trailing whitespace before the closing >.
Not quite. Let's break it down: (S Attribute)* means zero or more occurrences of the group (whitespace + attribute). The * applies to the whole parenthesized group. Then S? means an optional trailing whitespace. So you can have zero attributes, one, or many.
Summary

The Big Picture

Here's how everything in this lecture connects:

The Full Story

Grammars are formal specifications that define what's valid in a language — whether it's a programming language, a data format, or a protocol.

Context-free grammars (CFGs) are the specific kind of grammar used for programming languages, consisting of terminals, non-terminals, a start symbol, and production rules.

Parse trees are proofs that a token sequence is valid — each internal node corresponds to one grammar rule.

Ambiguity is the central danger: when the same token sequence has multiple parse trees, it has multiple interpretations. Ambiguity comes from precedence (which operator binds tighter), associativity (which direction to group), and structural issues like the dangling else.

We fix ambiguity by adding non-terminals that restrict what can appear where. Deeper non-terminals = higher precedence. Left recursion = left associativity.

In practice, people often use abstract syntax trees (ASTs) with intentionally ambiguous grammars plus side rules, because the resulting trees are simpler.

Grammars are written in various notations (BNF, EBNF, and countless variants), but they all express the same underlying concepts.
