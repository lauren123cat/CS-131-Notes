Part I

Parameter Passing

When you call a function, you give it some data to work with. How that data gets from the caller to the function — and what the function can do with it — is one of the most important design decisions in any programming language.

Why Should I Care?

Function calls are the primary mechanism by which software gets glued together. Think about any program you've ever written — it's full of function calls. Because we do them so often, two things must be true:

1. They have to be fast. If every function call is even a tiny bit slower than it needs to be, that cost multiplies across millions of calls.

2. They have to be clear. When you call a function, you should know exactly what happens to the data you pass in. Does the function get its own private copy? Can it modify your original data? If you don't know the answer, you can't reason about your program.

This lecture covers the many different ways programming languages handle parameter passing — each with different tradeoffs between speed, memory usage, clarity, and power. Understanding these tradeoffs is essential to understanding why different languages work the way they do.

Arguments vs. Parameters: Getting the Words Right

Before we dive in, let's nail down two terms that people often mix up. We need to be precise, because the distinction matters.

Consider this code:

// The CALLER — the code that calls the function
f(x + 1)     // (x + 1) is the ARGUMENT

// The CALLEE — the function being called
def f(y):     // y is the PARAMETER
    ...
Key Distinction

Argument = what the caller passes in. It's the expression at the call site: x + 1.

Parameter = what the callee (the function) receives. It's the variable name in the function definition: y.

Think of it this way: Arguments are what you Actually pass. Parameters are the Placeholders waiting to receive them.

Throughout this lecture, we'll be consistent: "argument" is from the caller's perspective, "parameter" is from the callee's perspective.

Correspondence Issues: Which Argument Goes Where?

When you call a function with multiple arguments, the language needs a rule for which argument matches up with which parameter. This is called the correspondence issue. There are three common approaches:

1. Positional Correspondence (Most Common)

This is the one you already know intuitively. Arguments are listed left to right. Parameters are listed left to right. The first argument goes to the first parameter, the second to the second, and so on.

def f(x, y, z):
    ...

f(1, 2, 3)   // x=1, y=2, z=3 — simple and obvious

This is by far the most common approach and appears in virtually every language.

2. Keyword Correspondence

Some languages (like Python) let you specify which parameter each argument should go to by name. This means the order doesn't matter:

def f(x, y):
    print(x, y)

f(y=9, x=7)   // x=7, y=9 — order doesn't matter!

Even though the arguments are "backwards" compared to the function definition, Python matches them up by name. This is especially useful when a function has many optional parameters — you can skip ones you don't care about and name the ones you do.

3. Varying Number of Arguments

Some functions can accept a different number of arguments each time they're called. You've seen this in Python with *args, in C with ... (variadic functions like printf), and in Scheme:

;; Scheme example
(f 3 9 12)

;; f is defined as: (lambda (x . r) ...)
;; x gets bound to 3
;; r gets bound to the list (9 12)

The first argument binds to x normally. All remaining arguments get collected into a list called r. This lets one function handle many different use cases.

The Bigger Question

Correspondence is about which argument goes where. But the far more important question — and the rest of this lecture — is about how the argument value gets passed. Does the callee get a copy? A reference? Something else entirely? This determines the meaning of your function call.

Call by Value

This is the most common parameter passing mechanism, used in C, Java, Python (for immutable types), OCaml, Scheme, and many more languages. Let's understand it from scratch.

How It Works

The process is straightforward:

Step 1 / 5

The Key Rule

Because the callee works with a copy of each value, anything the callee does to its parameters does not affect the caller. The caller's original data is safe and untouched.

void increment(int n) {
    n = n + 1;      // modifies the LOCAL copy
}

int main() {
    int x = 5;
    increment(x);
    printf("%d", x);  // still prints 5!
}

Pros

Simple and well-understood — the most common convention, you already know it

Safe — the callee can't mess with the caller's data, keeping code decoupled

Easy to reason about — each function works with its own private data

Cons

Expensive for large data — copying a billion-element array is wasteful in both time and memory

Sometimes you WANT the callee to modify the caller's data — e.g., a function that reads from a file into a buffer

The efficiency problem is the big one. Passing an integer? The copy is basically free — it's just one register operation. Passing a massive struct or an array of a million elements? Now you're copying megabytes of data, allocating memory, and burning CPU time for a copy that might not even be necessary.

Call by Reference

Call by reference was invented to fix the two big problems with call by value: (1) it's too expensive for large objects, and (2) sometimes you want the callee to modify the caller's data.

The Basic Idea

Instead of copying the value, the caller passes the address (a pointer) to the original value. The callee then works directly with the original data through that pointer.

C++ Syntax vs. What Really Happens

C++ has a nice syntax for call by reference using the & symbol. But underneath, it's really just passing pointers — it's "syntactic sugar" that makes the code cleaner:

C++ Nice Syntax
What Really Happens (C equivalent)
int f(int &n) {   // n is a reference
    n++;              // looks like normal variable access
    return n * 2;
}

int main() {
    int i = 27;
    int result = f(i);  // just pass i, looks simple
    // i is now 28! f modified it.
}
// The COMPILER converts the & reference into pointers:

int f(int *n) {    // pointer to int
    (*n)++;           // dereference to access
    return (*n) * 2;  // dereference again
}

int main() {
    int i = 27;
    int result = f(&i);  // pass ADDRESS of i
    // i is now 28
}

The C++ reference syntax (int &n) is what the professor calls a "polite fiction." Under the hood, it's pointers. But the notation is more compact and less error-prone — no manual * dereferencing scattered everywhere.

Why It Helps with Large Objects

If you have a struct with thousands of fields, call by value would copy the entire thing. Call by reference just passes one pointer (8 bytes on a 64-bit machine), regardless of how big the struct is. The callee accesses the original directly through that pointer.

struct BigData {
    int values[1000000];  // ~4 MB of data
};

// Call by value: copies 4 MB every time. Slow!
void process_val(struct BigData s) { ... }

// Call by reference: passes 8 bytes. Fast!
void process_ref(struct BigData &s) { ... }

Pros and Cons

Pros

Fast for large objects — only an 8-byte pointer is passed

Callee can modify caller's data — useful for output parameters

Cons

Tighter coupling — callee can "mess with the caller's brain"

Slower for small objects — forces the value into memory for address-taking

Aliasing problems — the big one (see below)

Partial Fix: const

You can partially address the coupling problem by declaring a reference as const:

void f(const int &n) {
    // n++; ← ERROR! Can't modify a const reference
    printf("%d", n);  // reading is fine
}

This gives you the efficiency of passing a reference (no copy) while promising not to modify the data. It's a looser coupling — but it doesn't solve the aliasing problem.

The Aliasing Problem

This is the most important problem with call by reference, and it's subtle enough that it has caused real-world disasters.

What Is Aliasing?

Aliasing occurs when two different names refer to the same underlying object or memory location. It's like someone going by both "delegate" and "Frank Templeton" — two names, one person. If something changes about that person, it changes under both names.

Why Compilers Hate Aliasing

Compilers love to optimize code. One very common optimization is to "cache" a value in a register. If a compiler sees x = 0 followed by print(x), it wants to just emit code that prints 0 directly — no need to load x from memory.

But with aliasing, the compiler cannot do this optimization in general. After y = 1, the value of x might also have changed (if x and y are aliases). So the compiler must always reload from memory, just in case.

This is one of the reasons C++ can be slower than C in practice — C++ has call by reference, which introduces aliasing possibilities that force the compiler to be more conservative with optimizations.

The Rocket That Crashed Because of Call by Reference

In an early space program, flight software was written in a variant of Fortran that used call by reference. A function was called like this:

CALL F(3.14159)

The Fortran compiler passed the address of the constant 3.14159 to the function. Inside the function, the code modified its parameter: X = X + 1. Because of call by reference, this changed the value of the constant π itself in memory.

The next time the program used π, it was 4.14159 instead of 3.14159. Orbit calculations went wrong. The rocket had to be destroyed.

This is the danger of call by reference: it can modify things you never intended to be modified, including constants.

Call by Result (Ada's Innovation)

Motivation

The Ada programming language was designed for highly reliable software — the kind that controls airplane flaps or missile systems. The designers wanted to avoid the aliasing problems of call by reference while still allowing functions to return multiple values efficiently. Their solution: call by result.

The Core Idea

Call by result says: a function can have special "output parameters" that start off uninitialized. The function's job is to assign values to them. Only when the function returns are those values copied back to the caller's variables.

// Using C-like syntax for Ada's concept
int f(int n, int result o, int result p) {
    // o and p start UNINITIALIZED
    // It's f's job to set them
    o = 37;
    p = 28;
    return 12;
}

// Caller:
int a = 100, b, c;
int d = f(a, b, c);
// After return: d = 12, b = 37, c = 28

How It Differs from Call by Reference

This looks similar to call by reference, but there are critical differences:

PropertyCall by ReferenceCall by Result
When does the caller see changes?Immediately, as the callee modifiesOnly at the point of return
Can callee read the original value?YesNo — parameter starts uninitialized
Aliasing possible?Yes — a major problemNo — passing the same variable twice is prohibited
Callee works with...The original data directlyIts own local copy

Real-World Example: The read() System Call

In C/Linux, the read() function reads data from a file into a buffer:

ssize_t read(int fd, void *buf, size_t count);

The buf parameter is passed by reference (pointer). But conceptually, read() should never look at what's already in buf — it only writes into it. If C had call by result, buf would be declared as an output parameter, making the intent crystal clear and enabling more compiler optimizations.

Ada's Anti-Aliasing Rules

Ada prohibits passing the same variable as two different output parameters. The compiler will reject code like f(a, x, x) where x appears twice in output positions. This prevents aliasing and enables aggressive optimization.

In fact, Ada compilers are allowed to secretly implement call by value-result using call by reference under the hood — because the anti-aliasing rules guarantee the behavior will be identical. They get the clarity of value semantics with the speed of reference passing.

Macro Calls: A Completely Different Kind of "Call"

Everything we've discussed so far involves passing values at runtime. Macro calls are fundamentally different: the arguments are pieces of your program's source code, not runtime values. Macros operate at compile time.

Macros vs. Functions

Function call: Arguments are evaluated → values are passed → function runs with those values.

Macro call: Arguments are not evaluated. They are chunks of text (in C) or syntax trees (in Scheme). The macro system substitutes them into a template to produce new code, which is then compiled.

You've seen macros in two forms:

// C preprocessor macros — arguments are token strings
#define SQUARE(x) ((x) * (x))
SQUARE(3 + 1)  →  ((3 + 1) * (3 + 1))

;; Scheme define-syntax — arguments are syntax trees
;; (and a b) expands to (if a b #f)

Recursion in Macros

What happens if a macro expands to code that calls itself?

C/C++ Approach
Scheme Approach

C/C++ prevents infinite expansion by a simple rule: if macro M is being expanded and the expansion contains another call to M, that inner call is not expanded — it's left as plain text.

#define f(x, y)  f(y, x)

f(1, 2)
// Expands to: f(2, 1)
// The inner f(2,1) is NOT expanded again
// (just reverses arguments once)

This also handles "ping-pong" recursion: if f expands to something calling g, and g expands to something calling f, the second call to f is not expanded. The rule checks the entire chain of nested macro expansions.

Guarantee: Every C program can be compiled in finite time. Macros never loop.

Scheme allows recursive macros. You've used this — the and macro was defined in terms of itself! Scheme macros can be genuinely recursive.

This means:

⚠️ A Scheme program can fail to compile if a macro expansion loops infinitely. The compiler will eventually overflow its stack or hit a time limit.

This is a deliberate tradeoff: recursive macros are more powerful, but they can cause compilation to hang.

Critical Point

When using any language with macros, you must know whether you're calling a macro or a function, because they behave completely differently. A macro substitutes text; a function passes values. Confusing the two leads to bugs.

Call by Name

Call by name is a more exotic parameter passing mechanism. The relationship between call by name and call by reference is analogous to the relationship between functions and pointers:

The Analogy

Call by reference: The caller passes a pointer to each argument. The callee dereferences it to get the value.

Call by name: The caller passes a tiny function (called a "thunk") for each argument. The callee calls it to get the value.

How It Works

When the caller passes an argument using call by name, the argument expression is not evaluated yet. Instead, the compiler wraps it in a small, anonymous function (a thunk) that takes no arguments and, when called, evaluates the expression in the caller's environment.

The Integration Example

Call by name enables some elegant patterns that other calling conventions can't express directly:

// Imagine this keyword: "name" means call-by-name
double integrate(double name expression,
                double low, double high) {
    double sum = 0;
    for (x = low; x <= high; x += 0.001) {
        sum += expression;  // each access CALLS the thunk
    }
    return sum * 0.001;
}

// Caller: (x is a shared global variable)
result = integrate(x*x + 2*x + 1, 0, 10);

What really happens under the hood:

// Compiler wraps the expression in a thunk:
result = integrate(
    () => x*x + 2*x + 1,  // a function, not a value!
    0, 10
);

// Inside integrate, "expression" really means:
sum += expression();  // CALL the thunk each iteration

Each time through the loop, x has a different value (set by the loop). Each time the thunk is called, it re-evaluates x*x + 2*x + 1 with the current x. This naturally computes the integral!

The Downside: Expense

Every time the callee accesses a call-by-name parameter, it makes a function call. That's much slower than reading a register or a memory location. And if the argument expression is itself expensive, that cost is paid every single time.

Call by Need (Lazy Evaluation)

Call by need is call by name with one crucial optimization:

The Rule

Call by need = Call by name + caching. The thunk is called at most once. The first time the callee needs the parameter's value, the thunk runs. Its result is cached. Every subsequent access just uses the cached value.

This means our integration example stops working with call by need — the expression would only be evaluated once (with the initial value of x), and the cached result would be reused for every iteration. But most programs don't need that pattern, and the efficiency gain from caching is huge.

Lazy Evaluation: Call by Need on Steroids

Languages like Haskell take call by need and apply it everywhere. This philosophy is called lazy evaluation: don't compute anything until you absolutely need its value for something observable (like printing output).

-- Haskell: define ALL prime numbers (infinite list!)
primes = sieve [2..]
  where sieve (p:xs) = p : sieve [x | x <- xs, x `mod` p /= 0]

-- Only compute as many as we need:
take 3 primes   -- returns [2, 3, 5]
-- The rest of the infinite list is never computed!

Lazy vs. Eager: A Correctness Win

Lazy evaluation isn't just about efficiency — it can make programs correct that would crash under eager evaluation:

void printAverage(int n, double avg) {
    if (n == 0) print("*");
    else print(avg);
}

// Caller:
printAverage(n, sum / n);  // what if n = 0?

Eager evaluation (C, Python, etc.)

The caller evaluates sum / n before calling the function. If n = 0, the program crashes with a division-by-zero error — even though the function would never have used that value!

Lazy evaluation (Haskell)

The caller wraps sum / n in a thunk but doesn't evaluate it. Inside the function, when n = 0, we take the if branch and never need avg. The thunk is never called. No crash!

Formal Result

It can be formally proven that under reasonable assumptions, lazy evaluation will produce a correct result in every case where eager evaluation does — plus some additional cases where eager evaluation crashes or loops. The converse is not true. This is a genuine theoretical advantage.

Connection to Currying

Call by name uses the same fundamental mechanism as currying! Currying creates a function with one argument "on the fly." Call by name creates a function with zero arguments on the fly (a thunk). Same technology, different application.

Call by Unification (Prolog)

Prolog uses a completely different calling convention that doesn't fit any of the categories above. When you call a Prolog predicate, the arguments are unified with the parameters.

This means parameter passing in Prolog is bidirectional: unification can bind variables in both the caller and the callee. This is what makes Prolog so different — the caller and callee both contribute to finding a solution, rather than one just passing data to the other.

Part II

Cost Models

A cost model is your mental picture of how much it costs to do things in a programming language — in time, memory, energy, and more. Without a cost model, you can't write efficient programs.

Why This Connects to Parameter Passing

We just spent an hour on parameter passing. Why? Because function calls are the most common operation in most programs. Understanding the cost of each calling convention — how many copies, how many pointer dereferences, how many function calls — is a cost model question. Parameter passing and cost models are deeply intertwined.

Cost models have two levels, just like algorithm analysis:

LevelWhat It MeasuresExample
Big-O (asymptotic)How does cost scale with problem size?Accessing element i in a linked list is O(i)
AbsoluteWhat's the actual cost in CPU cycles, bytes, etc.?A function call costs ~10 instructions of overhead

Types of Costs

When you're building software, there are many costs to consider. They fall into two broad categories:

Runtime Costs (Production)

Time — CPU seconds, wall-clock time

Space — RAM usage, cache usage

Energy — battery life (laptops, phones)

Power — heat dissipation, cooling requirements (data centers measure in gigawatts!)

Network — throughput and latency (how fast and how responsive)

API/Token costs — if using external services like AI APIs, each call costs money

Development Costs

Cognitive load — how hard is the code to understand and maintain?

Portability — how much work to move the code to another platform?

Development time — how long does it take to write and debug?

Energy vs. Power

These are often confused but they're different physical quantities. Energy is measured in joules (or kilowatt-hours) — "how much total fuel did we burn?" Power is measured in watts — "how fast are we burning fuel right now?" If your laptop overheats, that's a power problem. If your battery dies, that's an energy problem.

Data Structure Costs: Lisp Lists vs. Python Lists

Different languages implement the same concept — "a list" — very differently, and this has huge cost implications. Let's compare two approaches you've seen in this class:

Lisp/Scheme Lists
Python Lists

A Lisp list is a linked list of cons cells. Each cell holds a value and a pointer to the next cell:

[val|→]→[val|→]→[val|→]→[val|→]→[val|/]

Each box is a "cons cell" with two parts:
  car (the value) and cdr (pointer to next)

Access element i: O(i) — must walk through i cells

Prepend an element: O(1) — just create a new cons cell pointing to the old head

Structure sharing: Multiple lists can share tails. If list B is list A with one element prepended, they share all of A's cells. Efficient, but creates aliasing.

A Python list is a dynamic array — a contiguous block of memory with a header:

[header: size=5, capacity=8] → [v₀|v₁|v₂|v₃|v₄|__|__|__]

header stores current size and allocated capacity
elements are stored contiguously in memory

Access element i: O(1) — direct indexing: address = base + i × element_size

Prepend an element: O(n) — must shift all existing elements right

Append an element: O(1) amortized — if there's room in the capacity, just add it

OperationLisp ListPython List
Access i-th elementO(i)O(1)
Prepend (add to front)O(1)O(n)
Append (add to end)O(n)O(1) amortized
Structure sharingNaturalRequires custom code
The Tradeoff

Structure sharing (aliasing) enables efficiency but introduces the same aliasing problems we discussed with call by reference. When two lists share data, modifying one can unexpectedly modify the other.

The Anatomy of a Function Call

Since function calls are so common, let's understand exactly what happens during one. Most of this is pure overhead — bureaucratic paperwork that doesn't do any useful work but is necessary for the machinery to function.

Step 1 / 10
Hidden Parameters

The return address is really just another parameter — a hidden one that every function receives. Similarly, in object-oriented languages, the self/this pointer is another hidden parameter. And in languages with closures, the environment pointer is yet another hidden parameter. All of these add to the cost.

Tail Call Optimization

We've seen tail call optimization before in the context of saving stack space. But it also saves CPU time by eliminating much of the overhead listed above.

When It Applies

If the very last thing a function does is call another function and return whatever that function returns, many of the overhead steps can be skipped:

• No new frame allocation — reuse the current frame

• No return address copy — the callee inherits the existing one

• Some register saves can be skipped

LanguageTail Call OptimizationWhy
SchemeRequired by the language specRecursion is the primary looping mechanism
C / C++Optional (compiler may or may not do it)Nice optimization but not guaranteed
JavaProhibitedJava allows creating backtrace objects at runtime — tail call optimization would destroy stack frames that backtraces need

Tail Calls in Prolog

Prolog has a similar optimization, but it's more restricted. Consider:

p(X) :- q(X), r(X).   % Can we optimize the call to r(X)?
p(X) :- s(X).          % Can we optimize the call to s(X)?

For r(X) in the first clause: not directly, because if r(X) fails, Prolog might need to backtrack and try the second clause. The system hasn't "committed" to this clause yet.

But if we add a cut before r(X): p(X) :- q(X), !, r(X). — now Prolog has committed. The tail call optimization applies.

For s(X) in the last clause: yes, because there are no more clauses to backtrack to. The optimization applies automatically.

Prolog Efficiency: First-Argument Indexing

When Prolog tries to match a goal against clauses, you might think it scans every clause in your program. It doesn't — it uses indexing to jump directly to the relevant clauses.

Two Levels of Indexing

Level 1: Predicate name + arity. A call to p(A, B) only looks at clauses for p/2 (predicate p with 2 arguments). This is implemented via a hash table.

Level 2: First argument value. If the first argument is an atom or number, Prolog hashes on it and only examines clauses whose first argument matches.

p(a, 1).     % clause 1
p(b, 2).     % clause 2
p(c, 3).     % clause 3
p(d, 4).     % clause 4

?- p(b, X).  % Prolog hashes on 'b' → jumps directly to clause 2
             % Never examines clauses 1, 3, or 4!
When Indexing Doesn't Help

First-argument indexing only works when the first argument at the call site is a simple value (atom or number). If it's a logical variable or a complex structure, Prolog gives up on hashing and falls back to scanning clauses sequentially.

Prolog programming tip: Put the most "discriminating" argument first in your predicates. If you have many facts that differ in their first argument, indexing makes lookups nearly O(1).

Array Access Costs: The Hidden Math

Accessing an array element seems simple — a[i] — but the compiler has to do real work behind the scenes, especially if the language has bounds checking and non-zero-based arrays.

The Full Cost

Consider an array of tax rates for years 2001–2026, stored as doubles (8 bytes each), with the array starting at memory address α:

// To compute the address of a[i]:

// Step 1: Bounds checking
if (i < 2001) error();    // compare + branch
if (i > 2026) error();    // compare + branch

// Step 2: Address calculation
address = α + (i - 2001) * 8;  // subtract + multiply + add

That's 2 compares, 2 branches, 1 subtract, 1 multiply, and 1 add — just to look up one element!

Optimization: One Branch Instead of Two

The compiler can reduce the two bounds checks to just one using an unsigned comparison trick:

// Instead of two checks:
if (i < 2001) error();
if (i > 2026) error();

// Use ONE unsigned comparison:
if ((unsigned)(i - 2001) > 25) error();

Why does this work? If i is less than 2001, then i - 2001 is negative. When treated as an unsigned number, a negative value becomes a very large positive number — definitely greater than 25. If i is greater than 2026, then i - 2001 is greater than 25. Either way, the single check catches both out-of-bounds conditions!

Why This Matters

Array access happens millions of times in typical programs. Saving one branch instruction per access can make a measurable performance difference. This is the kind of low-level optimization that compilers do — and understanding it helps you appreciate why languages make the design choices they do (e.g., zero-based arrays in C eliminate the subtraction step entirely).

  • Java allows creating backtrace objects at runtime, which need all stack frames to be preserved
  • Java uses call by reference, which is incompatible with tail calls
