Part I — The Big Picture

Three Ways to Program a Computer

Before we dive into Prolog, let's zoom out and understand where logic programming fits among the different ways humans talk to computers. This context will help you see why logic programming exists and what it gives up — and gains — compared to what you already know.

Throughout this course, you've been learning that there isn't just one way to write a program. There are fundamentally different paradigms — different philosophies about what a program is and how you should build one. The three major paradigms are:

ParadigmCore UnitGlueWhat You Give UpLanguages
Imperative Statements Sequencing
(do this, then that)		 — C, C++, Java, Python, JavaScript
Functional Functions Function calls
(compose functions)		 Assignment statements (side effects) OCaml, Haskell, Scheme
Logic Predicates Logical connectives
(AND, OR, NOT)		 Functions and assignment and sequencing Prolog

Think of these paradigms as a ladder of increasing abstraction. Each step up the ladder, you give up some low-level control, but in exchange you get to think at a higher level:

Imperative programming is the most familiar. You write step-by-step instructions: "do this, then do that." You have full control — you can assign variables, loop, branch. Languages like C, Java, and Python work this way.

Functional programming (which you studied with OCaml) takes one thing away: assignment statements. Once a value is bound, it can't be changed. This feels like programming with one arm tied behind your back. But it turns out that giving up mutation makes programs easier to reason about, and functional style has become popular even in imperative languages like JavaScript.

Logic programming goes even further. It takes away functions, assignment, and sequencing. You don't write step-by-step instructions at all. Instead, you describe what you want, and let the system figure out how to get it. This feels like tying both arms and legs behind your back — and yet, surprisingly, you can still get real work done.

Key Insight

Each paradigm is defined not just by what it has, but by what it gives up. Functional programming gives up assignment. Logic programming gives up functions entirely. The more you give up, the more freedom the system has to find solutions on its own.

What is Logic Programming?

In logic programming, there are no functions in the traditional sense. You cannot write a function that takes inputs and returns an output. Instead, you write predicates.

Definition — Predicate

A predicate is like a function that can only answer "yes" or "no" (true or false). It doesn't return a value — it either succeeds or fails. Predicates are connected to each other using logical connectives like AND, OR, and NOT.

Instead of calling a function and getting a return value, in Prolog you ask a query — you pose a question to the system. The system then searches for a way to make your query true, and if it finds one, it tells you the answer.

Analogy: Imagine you have a very smart librarian. In imperative programming, you tell the librarian exactly which shelf to go to, which book to grab, and which page to open. In functional programming, you describe a recipe for finding the book. In logic programming, you just say "I want a book about sorting algorithms published after 2020" and the librarian figures out how to find it for you.

This approach is sometimes compared to modern "vibe coding" with AI — you describe what you want, and the system generates the solution. Except with Prolog, there's no AI magic; the system uses precise logical inference to derive answers from the rules you've given it.

Declarative vs. Imperative Thinking

Logic programming emphasizes declarative programming. Let's break down exactly what that means, because it's a fundamental shift in how you think about code.

Definition — Declarative Programming

Declarative programming means you describe what the answer should look like, without specifying how to compute it. You declare the properties of the solution and let the system figure out the steps.

You've actually already encountered declarative thinking, even if you didn't realize it. HTML is declarative — you say "I want a heading here and a paragraph there," not "move the cursor 20 pixels down, draw characters in bold font at size 24..." SQL databases are declarative — you say SELECT * FROM users WHERE age > 18, not "loop through each row, check the age column..."

Prolog takes this idea and applies it to general-purpose programming. Instead of writing a sorting algorithm (bubble sort, quicksort, etc.), you can describe the properties of a sorted list and let Prolog figure out how to produce one.

Logic vs. Control: The Big Separation

This is one of the most important concepts in this lecture. In Prolog, every program can be split into two independent parts:

The Algorithm Equation

Algorithm = Logic + Control

Logic is the formal specification of what the answer should be. It's the constraints: "we want answers that look like this." Whether the answer is correct or not depends entirely on the logic. All your correctness concerns live here.

Control is advice to the Prolog interpreter about how to search for the answer efficiently. Ideally, control is just about performance improvement. It doesn't change what answers you get — it only changes how fast you get them.

Why does this separation matter? It's a software engineering discipline. You break one big problem into two independent sub-problems. If you screw up the control, your program gets slower but still gives correct answers. If you screw up the logic, your program gives wrong answers — which is much worse. This separation lets you focus on getting the logic right first, then optimize performance separately.

The logic is more important than the control, because incorrect logic means wrong answers no matter how fast you run. But you still need good control, because bad control can make a correct program unacceptably slow (as we'll see shortly).

Part II — Learning Prolog Through Sorting

Building a Sort Predicate from Scratch

Instead of teaching Prolog syntax in the abstract, we're going to learn by doing something concrete: building a sorting predicate. Along the way, you'll absorb the core ideas of how Prolog thinks.

In every programming language you've used before, sorting means writing an algorithm — bubble sort, quicksort, merge sort. You tell the computer how to sort step by step. In Prolog, we take a completely different approach: we describe what it means for a list to be sorted, and let Prolog figure out the rest.

Predicates Instead of Functions

Since Prolog has no functions, we write a predicate instead. Our predicate will be called sort and it takes two arguments:

Prolog — The Sort Predicate (Goal)
/* sort(L, S) is true when S is the sorted version of L */
sort(L, S).

Notice something important: in a functional language, a sort function takes one argument (the list) and returns the sorted list. In Prolog, since predicates don't return values, we use a second argument to represent what would have been the return value. This is a common pattern: an n-argument function becomes an (n+1)-argument predicate, where the extra argument holds the "result."

Here's how you'd use this predicate as a query:

Prolog — Using the Sort Predicate
/* Ask: "Is [-2, 3, 5] the sorted version of [3, 5, -2]?" */
?- sort([3, 5, -2], [-2, 3, 5]).
yes

/* Ask: "What is the sorted version of [3, 5, -2]?" */
?- sort([3, 5, -2], X).
X = [-2, 3, 5]

Look at what just happened. In the first query, we gave both arguments and asked "is this true?" — Prolog said yes. In the second query, we used a variable X (capital letters are variables in Prolog) and asked Prolog to figure out what X should be. Prolog found the answer on its own.

This ability to "run things backwards" — to give the system a partial question and have it fill in the blanks — is one of the most powerful features of Prolog.

Defining Sort: What Properties Must Hold?

Now we need to actually define what sort(L, S) means. In a CS algorithms class, you'd just assume everyone knows what "sorted" means and jump to an algorithm. In Prolog, we have to spell out the exact specification — what properties must S have relative to L?

Let's think about this carefully. What makes S a "sorted version of L"?

Property 1: S and L must be permutations of each other. This means every element in L appears in S the exact same number of times, and vice versa. This also guarantees they're the same length. (Note: we initially thought we'd need a "same length" requirement, but "permutation" already implies that — it's a stronger condition that subsumes it.)

Property 2: S must be in non-decreasing order. Each element is less than or equal to the next one. If there are duplicates, they sit next to each other.

That's it. Those two properties completely define what it means for S to be the sorted version of L. Here's the Prolog code:

Prolog — Sort Definition
sort(L, S) :-
    perm(L, S),    /* S is a permutation of L  (AND) */
    sorted(S).     /* S is in non-decreasing order     */

Let's decode this syntax piece by piece:

The :- symbol means "if" (it's read right-to-left: "sort(L,S) is true if..."). The comma , at the top level means "AND." So this reads: "For all L and S, sort(L, S) is true if perm(L, S) is true and sorted(S) is true."

We haven't told Prolog how to sort. We've only told it what it means for something to be sorted. Now we need to define the helper predicates sorted and perm.

Helper: The sorted Predicate

The sorted predicate checks whether a list is in non-decreasing order. Let's build it up from the simplest cases to the complex ones.

Base Cases

An empty list is sorted. There are no elements to be out of order, so this is trivially true.

Prolog
sorted([]).   /* The empty list is sorted */

A single-element list is sorted. One element can't be out of order with itself.

Prolog
sorted([_]).  /* Any single-element list is sorted */

The _ is a special variable called an anonymous variable or wildcard. It means "I don't care what this value is." Every _ you write is a different variable. We use it when a variable appears only once in a clause and we don't need to reference it again. (You could also write sorted([X]). — it means the same thing, but the convention is to use _ when you don't need the name.)

The Recursive Case (Two or More Elements)

For a list with two or more elements, we need to check that each consecutive pair is in order. Here's the Prolog pattern for "a list with at least two elements":

Prolog — List Pattern
[X, Y | L]   /* X is first, Y is second, L is the rest */

This is Prolog's way of pattern matching on a list. The | (vertical bar) separates the explicitly listed elements from the "rest" of the list. So [X, Y | L] matches any list of length ≥ 2, binding X to the first element, Y to the second, and L to everything else (which could be empty).

If you've used OCaml, this is similar to writing x :: y :: l as a pattern.

A Subtle Bug

Your first instinct might be to write: sorted([X, Y | L]) :- X =< Y, sorted(L).

This is wrong! It checks that X ≤ Y, then checks the tail of the tail (L, which skips Y). This would say that [1, 3, 2, 4] is sorted — because 1 ≤ 3, and then [2, 4] is sorted. But clearly 3 > 2, so it's not sorted!

The fix: recurse on [Y | L] instead of L, so you re-check the second element against the third.

Here is the correct version:

Prolog — Sorted (Recursive Case)
sorted([X, Y | L]) :-
    X =< Y,              /* first ≤ second */
    sorted([Y | L]).    /* [second | rest] is also sorted */

The =< operator is Prolog's "less than or equal to." (Yes, it's written backwards from what you might expect. This is because Prolog was originally designed for natural language parsing, and the arrow symbols -> and <- were reserved for that purpose.)

By recursing on [Y | L] instead of just L, we ensure that Y gets compared against the next element in the next recursive call. This way, every consecutive pair gets checked.

The Complete Sorted Predicate

Prolog — Complete sorted/1 Predicate
sorted([]).                        /* empty list: sorted */
sorted([_]).                       /* single element: sorted */
sorted([X, Y | L]) :-
    X =< Y,
    sorted([Y | L]).
Check Your Understanding

What would ?- sorted([5, 3, 8]). return?

A yes
B no
C An error
Correct! Prolog matches [5, 3, 8] against the pattern [X, Y | L], binding X=5, Y=3, L=[8]. It then checks 5 =< 3, which is false. So the predicate fails and Prolog says no.
Not quite. Prolog checks if 5 ≤ 3 (the first two elements). Since 5 > 3, the condition fails, and Prolog answers no — the list is not sorted.

Helper: The perm Predicate

Now for the harder part: defining what it means for one list to be a permutation of another. We want perm(L, R) to be true when R contains exactly the same elements as L, just possibly in a different order.

Building Up Intuitively

Let's start with the simplest case and work our way up:

The empty list: The only permutation of [] is [].

Prolog
perm([], []).  /* base case */

The general case: Here's where the thinking gets interesting. We need a recursive definition. Take a non-empty list [X | L] (where X is the first element and L is the rest). What does a permutation R of this list look like?

R must contain X somewhere, and everything else in R must be a permutation of L. So we can think of building R in two steps:

  1. Take L (everything except X), and recursively find a permutation P of it.
  2. Insert X somewhere into P to form R.

"Inserting X somewhere into P" means: split P into two pieces (call them P1 and P2), then concatenate P1 + [X] + P2 to get R. We can express this split-and-insert using an append predicate:

Prolog — Permutation (Recursive Case)
perm([X | L], R) :-
    perm(L, P),                /* P is a permutation of L  */
    append(P1, P2, P),        /* split P into P1 and P2    */
    append(P1, [X | P2], R). /* R = P1 + [X] + P2        */

Let's trace through the logic carefully with a picture:

Visual Walkthrough: How perm works
Step 1 of 4

Start with the original list [X | L]:

We have a list where X is the head and L is the tail. For example, if our list is [3, 1, 2], then X = 3 and L = [1, 2].

X = 3 L = [1, 2]
Step 2 of 4

Recursively permute L to get P:

perm(L, P) — find some permutation P of L. For example, P might be [2, 1].

L = [1, 2] → permute → P = [2, 1]
Step 3 of 4

Split P into two pieces P1 and P2:

append(P1, P2, P) — find P1 and P2 such that P1 ++ P2 = P. One possibility: P1 = [2], P2 = [1].

P = [2, 1] → split → P1 = [2] P2 = [1]
Step 4 of 4

Insert X between P1 and P2 to form R:

append(P1, [X | P2], R) — concatenate P1 + [X | P2] to get R.

[2] ++ [3] ++ [1] = R = [2, 3, 1]

And [2, 3, 1] is indeed a permutation of [3, 1, 2]! By trying different splits and different recursive permutations, Prolog can generate all permutations.

The Complete Perm Predicate

Prolog — Complete perm/2 Predicate
perm([], []).                         /* base case */
perm([X | L], R) :-
    perm(L, P),
    append(P1, P2, P),
    append(P1, [X | P2], R).
Note on Variable Scope

Every variable in Prolog (anything starting with a capital letter) has its scope limited to the single clause it appears in. So the X and L in the base case are completely independent from the X and L in the recursive case. Each clause is like a separate "function call" with its own local variables. You don't need to declare them — just use them.

Helper: The append Predicate

We used append inside perm, so we need to define it. append(A, B, C) is true when C is the result of concatenating lists A and B.

Prolog — Complete append/3 Predicate
/* Base case: appending [] to any list gives that list */
append([], L, L).

/* Recursive case: if the first list starts with X... */
append([X | L], M, [X | LM]) :-
    append(L, M, LM).

Let's read each clause:

Base case: Appending the empty list [] to any list L gives you L. Notice that the second and third arguments are the same variable L — this forces them to be equal.

Recursive case: If the first list starts with X (followed by L), and the second list is M, then the result must start with X (followed by LM), where LM is the result of appending L and M.

Think of it this way: to concatenate [1, 2, 3] with [4, 5], you take the 1 off the front, concatenate [2, 3] with [4, 5] (which gives [2, 3, 4, 5]), and stick the 1 back on front: [1, 2, 3, 4, 5].

The Complete Specification

With sort, sorted, perm, and append, we have now fully specified what sorting means — all the way down to the bare-bones level. The only built-in we relied on is =< (numeric less-than-or-equal). We never said "use bubble sort" or "use quicksort." We just described the properties of a sorted list, and Prolog can derive the answer.

Running Predicates "Backwards"

One of the most striking features of Prolog is that predicates can often be used in multiple directions. Let's see this with append:

Prolog — append Used Multiple Ways
/* "Forward" — concatenate two known lists */
?- append([1, 2], [3, 4], Z).
Z = [1, 2, 3, 4]

/* "Backward" — find all ways to split a list into two parts */
?- append(X, Y, [3, 4, 7]).
X = [],       Y = [3, 4, 7]  ;
X = [3],     Y = [4, 7]     ;
X = [3, 4], Y = [7]        ;
X = [3, 4, 7], Y = []     ;
no

/* Completely unconstrained */
?- append(X, Y, Z).
X = [],  Y = Z                  ;
X = [_29], Z = [_29 | Y]     ;
... (infinite answers) ...

In the second example, we gave Prolog the result and asked it to find all possible inputs. It found four ways to split [3, 4, 7] into two pieces. In a functional language, you'd have to write a completely separate function to do this. In Prolog, the same predicate works in both directions.

The semicolon ; is what you type to ask Prolog for the next answer. Prolog will pause after each answer and wait — if you want more, you type ; and it backtracks to find another solution. When there are no more solutions, it prints no.

This "running backwards" capability is exactly what perm relies on. When perm calls append(P1, P2, P), it knows P but not P1 or P2 — it's running append backwards to find all ways to split P.

The Control Problem: Why Our Sort is Terrible

Our sort predicate is logically correct — it will always give the right answer. But it's catastrophically slow. Here's why:

Prolog evaluates goals left to right. So when processing sort(L, S) :- perm(L, S), sorted(S)., it first calls perm, which generates a permutation, and then checks if that permutation is sorted.

If it's not sorted (and it usually won't be), Prolog backtracks and perm generates another permutation. And another. And another. Until it happens to stumble upon the one that's sorted.

Performance Disaster

For a list of n elements, there are n! (n factorial) permutations. That means this sorting algorithm is O(n!) — far worse than even the slowest sorting algorithms you learned in your algorithms class. Bubble sort and insertion sort are O(n²), which is vastly better than n!. For a list of just 20 elements, n! ≈ 2.4 × 10¹⁸ — that's 2.4 quintillion permutations to check.

This is the control problem, not a logic problem. The logic is right — S really does need to be a permutation of L that's sorted. The issue is that Prolog's strategy of "generate all permutations, then check" is horrifically inefficient.

Can We Fix It by Reordering?

Since the comma means "AND" (which is commutative), you might think: "What if I put the second append first in the perm clause, to constrain R before generating permutations?" Let's see what happens:

Prolog — Reordered perm (WORSE!)
perm([X | L], R) :-
    /* Moving this first makes control WORSE! */
    append(P1, [X | P2], R),
    perm(L, P),
    append(P1, P2, P).

Logically, this is identical — AND is commutative and associative. But Prolog evaluates left to right, and now the first thing it does is append(P1, [X | P2], R) where P1, P2, and R are all unknown. This generates an infinite number of solutions, making things even worse.

Key Insight — Logic vs. Control Revisited

Reordering clauses does not change the logic (the set of correct answers stays the same), but it does change the control (how efficiently Prolog finds those answers). Prolog evaluates subgoals in the exact order you write them. If you put things in the wrong order, you can make your program infinitely slower — or even make it loop forever — even though the logic is correct.

Think about what Prolog knows at each step. If sorted(S) runs first, S hasn't been bound by perm yet — it's still unknown. Prolog would struggle because it's trying to check if an unknown list is sorted.
Part III — Prolog Syntax

Terms: The Building Blocks of Everything

Now that you've seen Prolog in action, let's formally learn the syntax. Everything in Prolog is built out of terms. There are exactly four kinds of terms, and that's all there is to the core language.

What's remarkable about Prolog is how simple its syntax is at its core. Every piece of data, every predicate call, every rule — they're all just terms. Here are the four types:

1. Atoms

Definition — Atom

An atom is the simplest kind of term. It's atomic — meaning it can't be broken into smaller pieces. Each atom equals only itself and nothing else. Atoms are like unique labels or symbols.

The syntax for atoms comes in two forms:

Unquoted atoms start with a lowercase letter, followed by any mix of letters, digits, and underscores:

Examples of Atoms
abc          /* a simple atom */
a9_x         /* letters, digits, underscores OK */
hello_world  /* must start with lowercase */
x            /* single character works */

Quoted atoms are wrapped in single quotes and can contain any characters, including spaces and special symbols:

Quoted Atoms
'Hello World'   /* spaces are fine inside quotes */
'o''clock'      /* double the quote to include an apostrophe */
'+'             /* operators are atoms too! */
':-'            /* the "if" symbol is an atom */

Atoms don't have "values" — they just are. The atom hello is simply hello. It doesn't store a number or a string. It's just a unique symbol that equals only itself.

2. Numbers

Numbers in Prolog work like numbers in most languages — you have integers and floating-point values. The syntax is straightforward: 42, -7, 3.14, -0.5, etc. Nothing surprising here.

3. Logical Variables

Definition — Logical Variable

A logical variable is a term that represents an unknown value. It starts with a capital letter or an underscore, followed by any mix of letters, digits, and underscores. Examples: X, Hello, _hello, H9, _.

Logical variables are fundamentally different from variables in languages like Java or Python, and understanding this difference is crucial:

PropertyJava / Python VariableProlog Logical Variable
Starts as Undefined (error to use) Unbound (perfectly legal to use)
Getting a value Assignment: x = 5 Binding through matching
Changing value Re-assign anytime: x = 10 Cannot change! Only unbinds on backtracking
Scope Block / function scope Single clause (the thing ending in .)

The lifecycle of a Prolog variable looks like this:

  1. Unbound: The variable starts as a question mark — "I don't know my value yet." This is totally fine; it's not an error.
  2. Bound: As computation succeeds, the variable gets bound to a value (some term). Once bound, it stays bound to that value — there are no assignment statements.
  3. Unbound again (on failure): If Prolog backtracks, the variable becomes unbound again, freeing it to be bound to a different value in the next attempt.

This bind → unbind cycle is how Prolog explores different possibilities. In the sort example, the variable S gets bound to a permutation, then if sorted(S) fails, S gets unbound and perm binds it to a different permutation.

The Underscore Variable

The special variable _ (single underscore) is the anonymous variable. Every _ in your code is a different, independent variable. Use it when you need a variable to make the pattern work but don't care about its value. For example, sorted([_]). means "any singleton list is sorted" — we don't need to name the element.

4. Structures

Definition — Structure

A structure is a compound term that represents a data structure. It consists of a functor (which must be an atom) followed by one or more arguments in parentheses: f(t1, t2, ..., tn). Each argument is itself a term. You must have at least one argument (n ≥ 1).

A structure looks like a function call, but it's not a function call. It's a constructor — all it does is build a piece of data in memory. When you write foo(bar, 42, X), Prolog creates a little chunk of storage containing the atom foo and pointers to bar, 42, and X. That's it. No computation happens.

You can think of structures as nodes in a tree. The functor is the node label, and the arguments are the children. Any Prolog term can be drawn as a tree.

The arity of a structure is how many arguments it has. We write this as functor/arity. For example, append/3 means the append predicate with 3 arguments. Importantly, append/2 and append/3 are completely independent — they don't collide at all, just like two different predicates with different names.

Correct! Hello starts with a capital letter, so it's a logical variable. hello (lowercase) is an atom. 42 is a number. 'Hello' (quoted) is an atom — the quotes override the capital letter.

Syntactic Sugar: Making Prolog Readable

The four term types (atoms, numbers, variables, structures) are the complete core syntax of Prolog. Everything else is syntactic sugar — convenient shorthand that gets translated into the core forms internally.

Operators

Prolog has built-in operators (both binary and unary) that are just syntactic sugar for structures:

Operators are Structures in Disguise
/* What you write          What Prolog sees internally */
3 + 4 * 5           ≡    '+'(3, '*'(4, 5))
2 * -3              ≡    '*'(2, '-'(3))

/* The "if" operator */
a :- b              ≡    ':-'(a, b)

/* AND and OR */
a , b               ≡    ','(a, b)
a ; b               ≡    ';'(a, b)

The key insight: operators like +, *, :-, ,, and ; don't do anything by themselves. They just build structures. Writing 3 + 4 doesn't add 3 and 4 — it builds the data structure '+'(3, 4).

List Syntax

The list syntax you've been seeing is also syntactic sugar. Internally, lists are built from a special structure called '.' (dot), which works like cons in OCaml:

List Syntax is Structure Sugar
/* What you write          What it really is */
[]                  ≡    '[]'
[a, b, c]           ≡    '.'(a, '.'(b, '.'(c, '[]')))
[X, Y | L]          ≡    '.'(X, '.'(Y, L))

So a list is really a chain of ./2 structures ending in '[]'. The bracket notation is just syntactic sugar that makes lists readable. The | (vertical bar) separates explicitly named elements from the tail of the list.

Arithmetic: The is Operator

Since 3 + 4 just builds a structure and doesn't do math, Prolog needs a special mechanism for actual computation. That mechanism is the built-in is operator:

Prolog — Arithmetic with is
N = 19,
M is N * 12.
/* Internally: is builds '*'(19, 12), evaluates it, binds M = 228 */

Here's what happens step by step: N * 12 builds the data structure '*'(19, 12) (since N is already bound to 19). Then is takes that data structure, walks through the tree, does the actual arithmetic in machine code, and binds M to the result (228).

Critical Restriction

The right side of is must be a ground term — a term with no unbound logical variables. It can be a complex expression with structures, atoms, and numbers, but every variable in it must already have a value. This is because is is implemented in machine code (C), which can't handle unknowns. If you try M is X * 12 when X is unbound, you'll get an error.

This also means you can't run is backwards. You can't say "M is 228, what values of N make N * 12 = 228?" — unlike append, is only works in one direction.

Definition — Ground Term

A ground term is any term that contains no logical variables anywhere. It can be an atom, a number, or a structure whose arguments are all ground terms. For example, f(a, 3, g(b)) is ground, but f(a, X, g(b)) is not (because of X).

Part IV — How Prolog Runs

Facts, Rules, and Queries

A Prolog program is a collection of clauses. Each clause is a term, but there are three distinct types depending on their form and purpose.

1. Facts — Unconditional Truths

A fact is a clause that ends with a period and contains no :- (turnstile). It's unconditionally true — no conditions needed. When Prolog encounters a fact during its search, it can immediately succeed.

Examples of Facts
append([], L, L).     /* true for any L */
perm([], []).            /* true unconditionally */
sorted([]).              /* the empty list is sorted */
sorted([_]).             /* any singleton is sorted */

Facts can contain variables (like the first example, which is true for any list L), or they can be entirely ground (like parent(tom, bob).).

2. Rules — Conditional Truths

A rule has a turnstile :- in it. The part before :- is called the head, and the part after is called the body. A rule means: "the head is true if all the goals in the body are true."

Anatomy of a Rule
/*   HEAD                    BODY                          */
sorted([X, Y | L])  :-   X =< Y, sorted([Y | L]).

/* Read as: sorted([X, Y | L]) is true IF X ≤ Y AND sorted([Y | L]) */

When Prolog matches a goal to the head of a rule, it doesn't immediately succeed. Instead, it creates subgoals from the body and tries to prove them. Rules make Prolog recurse.

Technically, a fact is just a rule with no body (or equivalently, with the body true). You could write sorted([]) :- true. and it would mean exactly the same as sorted([]).

3. Queries (Goals) — Questions for Prolog

A query (also called a goal) is what you type at the Prolog prompt. It has a body but no head — it starts with ?- and tells Prolog "prove this."

Examples of Queries
?- sorted([1, 2, 3]).
?- append(X, Y, [1, 2, 3]).
?- sort([3, 1, 2], S).

Facts and rules are passive — Prolog stores them in its database without doing anything. The only thing that triggers computation is a query. When you type a query, Prolog springs into action and starts searching.

Clause TypeHas Head?Has Body?Meaning
FactYesNoUnconditionally true
RuleYesYesTrue if body is provable
QueryNoYes"Please prove this"

A predicate is the ordered collection of all facts and rules that share the same functor name and arity. The order matters because Prolog tries them top-to-bottom.

How Prolog Searches: Backward Chaining & Depth-First Search

Understanding how Prolog actually runs your code is essential for writing efficient programs. There are two key ideas:

Backward Chaining

Prolog is completely goal-oriented. It starts with the query you give it and works backwards toward facts. Here's the process:

  1. Start with the goal from the query.
  2. Scan through the database (facts and rules), in order, looking for a clause whose head matches the goal.
  3. If it matches a fact: success! The goal is proved.
  4. If it matches a rule: the body of the rule becomes a set of new subgoals. Go to step 2 for each subgoal.
  5. If nothing matches: the goal fails. Prolog backtracks and tries the next matching clause.

Depth-First Search

When there are multiple matching clauses and multiple subgoals, Prolog explores them as a tree using depth-first search:

The search tree: Imagine your goal at the root. Each matching clause creates a branch. Each subgoal within a clause creates a child node. Prolog always goes deep first (explores the first branch all the way down), before backing up and trying the next branch. It searches left to right across the database, and depth first down the tree.

This means Prolog tries the first matching clause first, and within that clause, it evaluates subgoals left to right. If a subgoal fails, it backtracks to the most recent choice point and tries the next alternative.

This is why the order of clauses in your database matters (it determines which branch Prolog tries first), and why the order of subgoals in a rule matters (it determines what gets evaluated first).

Special Predicates: true, fail, loop, repeat

These simple built-in predicates illustrate important aspects of how Prolog works:

true

Prolog
true.   /* It's a fact. It always succeeds. */

true is a fact with no arguments. Whenever Prolog needs to prove true, it immediately succeeds. Simple.

fail

Prolog
/* The source code for fail is... nothing! */
/* There are no clauses for fail at all. */

There are zero clauses that mention fail. When Prolog tries to prove fail, it searches the entire database, finds nothing, and fails. That's the whole trick — fail is defined by having no definition.

fail is useful for forcing backtracking. A common pattern is to use it with side effects to iterate through all solutions:

Prolog — Using fail to Print All Solutions
?- append(X, Y, [3, 4, 7]), write(X), nl, fail.
/* Prints each X, then fail forces backtracking to get next answer */
/* Output: [], [3], [3,4], [3,4,7] — then "no" */

The query can never succeed (because fail is always false), but along the way it uses write and nl (newline) as side effects to print all possible values of X.

Software Engineering Note

Because fail works by having no clauses, a typo like fale would also fail silently. To catch such bugs, most Prolog interpreters print a warning if you call a predicate that has no clauses — unless it's the built-in fail. So fail is the one and only predicate that's allowed to have zero clauses without a warning.

loop — An Infinite Loop

Prolog
loop :- loop.   /* To prove loop, you must prove loop... */

Calling loop causes Prolog to try to prove loop, which requires proving loop, which requires proving loop... forever. It never succeeds, never fails, just runs infinitely. Logically, this clause tells you nothing (X is true if X is true — you already knew that). Procedurally, it's an infinite loop.

repeat — Infinite Backtracking Success

Prolog
repeat.                  /* repeat is true (fact) */
repeat :- repeat.        /* repeat is true if repeat is true (rule) */

repeat is logically equivalent to loop — both say "repeat is true." But procedurally it behaves very differently:

When you call repeat, Prolog matches the first clause (the fact) and immediately succeeds. If something later causes backtracking to repeat, Prolog tries the second clause, which calls repeat recursively — and that also succeeds via the first clause. So repeat succeeds an infinite number of times.

This makes repeat useful for creating loops:

Prolog — Infinite Newlines
?- repeat, nl, fail.
/* repeat succeeds → nl prints a newline → fail backtracks
   → nl can't redo → backtracks to repeat → repeat succeeds again
   → nl prints again → fail backtracks → ... forever */

rep — The Useless Variant

Prolog
rep :- rep.   /* rule first */
rep.            /* fact second */

This is logically identical to repeat, but useless in practice. Why? Because Prolog tries clauses top to bottom. It hits the rule first, which says "to prove rep, prove rep." It recurses into the rule again, and again — going deeper and deeper (depth-first!) and never reaching the fact on the second line.

The search tree has the fact (success) at every level, but always branching off to the right. Since Prolog searches depth-first (down before right), it keeps going down forever and never finds any of the successes.

Lesson

repeat and rep teach a critical lesson about Prolog: the order of clauses matters. Facts should generally come before recursive rules, so Prolog finds the base case before diving into recursion. This is the same principle as making sure your base case comes first in any recursive function.

The fact lets it succeed immediately. On backtracking, the rule recurses, but immediately hits the fact again — so it succeeds infinitely many times.

Putting It All Together

Let's step back and see how everything connects:

The Big Picture Map ▶

Programming paradigms → Logic programming is one of three major paradigms. It gives up functions and assignment in exchange for declarative power.

Declarative thinking → Instead of writing step-by-step algorithms, you describe what the answer looks like. This connects to the Algorithm = Logic + Control equation.

Logic vs. Control → Logic determines correctness; control determines efficiency. They should be independent. This was demonstrated by the sort example (correct logic, terrible control at O(n!)).

Predicates → The core unit of Prolog. They succeed or fail, and can often be "run backwards." This is what makes sort work — append running backwards lets perm generate permutations.

Terms → Everything in Prolog is a term (atoms, numbers, variables, structures). Even operators and lists are just syntactic sugar for structures.

Execution model → Prolog uses backward chaining (goal-oriented) with depth-first search. This is why clause order matters, why repeat works but rep doesn't, and why reordering subgoals can make programs slower or cause infinite loops.

Complete Sort Code — All Predicates Together ▶
Prolog — Complete Sorting Program
/* ─── sort/2 ─── */
sort(L, S) :-
    perm(L, S),
    sorted(S).

/* ─── sorted/1 ─── */
sorted([]).
sorted([_]).
sorted([X, Y | L]) :-
    X =< Y,
    sorted([Y | L]).

/* ─── perm/2 ─── */
perm([], []).
perm([X | L], R) :-
    perm(L, P),
    append(P1, P2, P),
    append(P1, [X | P2], R).

/* ─── append/3 ─── */
append([], L, L).
append([X | L], M, [X | LM]) :-
    append(L, M, LM).
