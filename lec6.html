How programming languages decide what data is, what you can do with it, and how they catch your mistakes — before or after you run the program.

Why This Matters

Every piece of data in every program you'll ever write has a type. The rules a language uses for types determine whether your programs crash at 3 AM in production or get caught by the compiler before you even ship. This lecture ties together ideas from low-level hardware (IEEE floating point), language design (OCaml, C, C++, Java, Python), and the theory behind making software more reliable. By the end, you'll understand the deep design decisions behind types — and why different languages made different trade-offs.

Part 1 — FoundationsWhat Is a Type?

Before we can talk about anything else, we need to be crystal clear about what a "type" actually is in programming.

Definition — Type

A type is a set of values, where all elements in the set share a common representation in memory and support the same collection of operations.

Let's unpack that piece by piece, because every word matters.

"A set of values" — When you say a variable has type int, you're saying that variable can hold any value from the set of integers the machine can represent (like -2,147,483,648 through 2,147,483,647 for a 32-bit int). It cannot hold the string "hello" or the floating-point number 3.14. The type restricts which values are allowed.

"Common representation in memory" — All integers are stored using the same number of bits, in the same format (e.g., two's complement). All floats are stored using the IEEE 754 format. The representation is consistent across all values of that type.

"Same collection of operations" — If something is an int, you can add it, subtract it, multiply it, compare it, etc. If something is a string, you can concatenate it, take its length, access characters by index. The type tells you what you're allowed to do with a value.

Example — Type as a Set

If you write int n; in C, you're telling the compiler: "n can be any value from the set of 32-bit integers. I should be able to do arithmetic on it. It must never contain the string 'hello' or the value 3.14."

The compiler uses this promise to (a) allocate the right amount of memory, (b) generate the right machine instructions for operations, and (c) catch mistakes if you accidentally use n in a way that doesn't make sense for integers.

Primitive vs. Constructed Types

Types come in two flavors:

Definition — Primitive Type

A primitive type is a type built into the language that you cannot define yourself. The language just has it. Examples: int, float, bool, char.

Definition — Constructed Type

A constructed type is a type that you build using primitive types (or other constructed types). Examples: arrays, structs, lists, tuples, classes.

Example

In OCaml, int is primitive. But type intpair = int * int is a constructed type — you've built a new type (a pair of two integers) out of the primitive type int.

An important subtlety: different compilers for the same language might implement primitive types differently. A C compiler on a 64-bit machine might make long 8 bytes, while a 32-bit compiler might make it 4 bytes. The language definition specifies the rules, but leaves room for different implementations — this is what allows the same language to work efficiently on different hardware.

Type Annotations in OCaml

Connecting to the Bigger Picture

We've just learned what types are. Now let's see how a specific language — OCaml — handles types. OCaml is special because it can figure out your types automatically, but it also lets you write them down explicitly. This leads to an interesting design question: should a programmer have to write types, or should the language infer them?

In OCaml, when you define a function, you don't have to write down what types the arguments are. OCaml has a feature called type inference — it looks at how you use a variable and figures out what type it must be.

Example — Type Inference in Action

Suppose you write a function that reverses a list:

let rec reverse l = ...  (* OCaml figures out: l must be 'a list *)

OCaml sees that you're pattern matching on a list, calling list operations, etc. It concludes: "l has to be a list. And since you never do anything specific to the elements (like adding them), the elements can be of any type." So it infers the type 'a list — a list of "anything."

The 'a (pronounced "alpha" or "tick-a") is a type variable. It's a placeholder that means "any type." We'll explore this more deeply when we get to parametric polymorphism.

But OCaml also lets you explicitly annotate types:

(* Annotating with the general type — same as what OCaml infers *)
let rec reverse (l: 'a list) = ...

(* Annotating with a MORE SPECIFIC type *)
let rec reverse (l: int list) = ...

The first annotation just restates what OCaml already knows. The second one is more interesting — it restricts the function. Now reverse only works on lists of integers, even though the algorithm works on lists of anything.

Watch Out

If you annotate a function with a type that's more specific than necessary (like int list when the function could work on 'a list), you've made your function less reusable. It could work on any kind of list, but your annotation has locked it to only integers. As a general rule: let OCaml infer the most general type, unless you have a specific reason to restrict it.

So when should you annotate types?

Once programs get large, annotations serve as documentation for the human reader. You might forget what a function expects, and the annotation reminds you. It's like leaving a note for yourself. But in small programs, OCaml's inference is usually enough.

However, there are cases where OCaml can't infer types — specifically when you're defining your own custom types. That's what we'll learn next.

Part 2 — Building Your Own TypesDiscriminated Unions

Why Are We Learning This?

We've seen primitive types (int, float) and the idea of constructed types. Now we get to the real power of a language like OCaml: defining your own types from scratch. This is how you model real-world data — like a list that's either empty or has elements, a value that might exist or might not, or a tree that has branches and leaves. The mechanism for this is called a discriminated union (also called a "variant" or "tagged union" or "algebraic data type" depending on the language).

Let's build up to this concept step by step. Imagine you want to create your own version of a list — not OCaml's built-in list, but your own.

A list is one of two things:

1. Empty — there's nothing in it, or

2. Non-empty — it has a value, followed by the rest of the list.

In OCaml, you express this directly:

type 'a mylist =
  | Empty                          (* case 1: nothing here *)
  | Nonempty of 'a * 'a mylist    (* case 2: a value AND more list *)

Let's break down every piece of this syntax:

type 'a mylist = — We're defining a new type called mylist. The 'a means it's parameterized — it works for any type. An int mylist is a list of integers. A string mylist is a list of strings.

| Empty — This is one possible "shape" the data can take. Empty is called a constructor. It takes no arguments — it just is. It's a value of type 'a mylist that represents an empty list.

| Nonempty of 'a * 'a mylist — This is another constructor. Unlike Empty, this one takes an argument: a pair consisting of a value of type 'a (the head of the list) and another 'a mylist (the rest of the list). So Nonempty(5, Nonempty(3, Empty)) represents the list [5, 3].

Definition — Discriminated Union

A discriminated union (also called a "tagged union" or "variant") is a type that can have multiple different "shapes." Each shape has a tag (like Empty or Nonempty) that identifies which shape it is.

The key word is discriminated — at runtime, the system can always tell which variant you have by reading the tag. This is fundamentally different from C/C++ unions (which we'll see next), where there's NO tag and the programmer has to remember which shape the data is in.

The tag is an implicit marker stored alongside the data. When you write Nonempty(5, Empty), the runtime stores something like:

tag: Nonempty
value: 5
next: →Empty
A Nonempty node stores a tag, a value, and a pointer to the rest of the list

Because of the tag, the system always knows what it's looking at. When you write a match expression, OCaml reads the tag and goes to the right case:

match mylist with
| Empty -> "list is empty"
| Nonempty(head, tail) -> "first element is " ^ string_of_int head

And here's the crucial safety property: OCaml forces you to handle every case. If you forget to handle Empty, the compiler warns you. This is what makes discriminated unions so much safer than C-style unions.

Unions in C/C++ — The Dangerous Kind

Now let's see the alternative approach that C and C++ take, so you can appreciate why discriminated unions are better.

In C/C++, a union looks like this:

union {
    int i;
    float f;      // allocates enough memory for the largest member
};

A C union stores all its members at the same memory location. The size of the union is the size of its largest member. So if int is 4 bytes and float is 4 bytes, the union is 4 bytes — and the integer and the float literally share the same 4 bytes of memory.

int i — at offset 0
float f — at offset 0
Both members occupy the exact same memory. There's NO tag to tell them apart.

The critical difference: there is no tag. Nothing in memory tells you whether this 4-byte block currently holds an integer or a float. The only way to know is to track in your head (or in your code) which member you last stored. If you store an int and then read it as a float, you get undefined behavior — the program can crash, produce garbage, or appear to work but be subtly wrong.

Why This Is Dangerous

Accessing the wrong member of a C union is exactly like reading an uninitialized variable. The bits are there, but they don't mean what you think they mean. This is one of the most common sources of bugs in C/C++ programs, especially when unions are passed between functions and the "which member is active" information gets lost.

The Option Type — Goodbye, Null Pointers

Connecting to the Bigger Picture

We've seen how discriminated unions let you define data with multiple shapes. Now let's look at one of the most important uses of this idea: solving the infamous null pointer problem. This connects our abstract discussion of types to real-world reliability.

In languages like C, C++, Java, and Python, there's a special value that means "nothing is here":

In C/C++, it's NULL (or nullptr). In Java, it's null. In Python, it's None.

The problem? If you have a pointer or reference, and someone passes you null, and you try to use it (call a method, access a field), your program crashes at runtime. This is so common that Tony Hoare, the inventor of null references, famously called it his "billion-dollar mistake."

OCaml's solution is the option type:

type 'a option =
  | None            (* nothing here — like null *)
  | Some of 'a     (* a value exists, and it's wrapped inside Some *)

At first glance, this looks the same as Python's None. But there's a crucial difference: the compiler enforces that you handle both cases.

Suppose you have a value of type int option. You cannot just add 1 to it:

let x : int option = Some 42

(* THIS DOES NOT COMPILE: *)
let y = x + 1   (* Error! x is int option, not int *)

(* You MUST pattern match to extract the value: *)
let y = match x with
  | None   -> 0           (* handle the "nothing" case *)
  | Some n -> n + 1       (* only here can you use the value *)

The compiler refuses to let you use the value without first checking whether it exists. This turns what would be a runtime crash in other languages into a compile-time error in OCaml. You physically cannot forget to handle the null case — the program won't compile until you do.

Key Insight

The same logic applies to lists. In Python, if you accidentally walk past the end of a list, you get a runtime error. In OCaml, because lists are discriminated unions ([] or head :: tail), the compiler forces you to handle the empty case. As long as you use match expressions (and not unsafe helper functions), you cannot fall off the end of a list — it's a compile-time guarantee.

Compile-Time vs. Runtime Error Catching

This gives us a spectrum of how languages handle "bad things happening":

ApproachWhat HappensExample Language
Compile-time checkingThe compiler prevents the error before you run the programOCaml (match on option/list)
Runtime errorThe program crashes when the bad thing happensPython (accessing None), C (null pointer)
Special valuesReturn a sentinel value (like NaN, infinity) instead of crashingIEEE floating point

Each approach has trade-offs. Compile-time checking is the safest — bugs are caught before the code ever runs. But it can make programs longer and more tedious to write (you have to handle every case). Runtime errors are simpler to code around, but they can slip through testing. Special values let computation continue but can silently propagate garbage through your program.

We're about to see all three approaches in action through the lens of floating-point numbers.

Part 3 — Floating Point Deep DiveWhy Study Float as a Type?

Why Are We Learning This?

You might wonder: why is a programming languages class talking about floating-point hardware? Here's why: float is a type. And it's a type with some of the trickiest design decisions in all of computing. By studying how float handles its edge cases (zero, overflow, undefined operations), we learn lessons about how to design any type — including types you'll invent yourself.

The design choices made for float (use special values like infinity and NaN instead of crashing) are the exact same kinds of choices you'll face when designing your own data structures. Should popping an empty stack return a special value? Crash? Be caught at compile time? Float's designers faced these questions decades ago, and their answers affect every program running on every computer today.

Float is also an abstract type — languages don't let you directly manipulate the individual bits. You interact with it through operations (add, multiply, compare), and the representation is hidden. This is the same abstraction principle that applies to any well-designed type.

Sign (s) — 1 bit. If it's 0, the number is positive. If it's 1, the number is negative. Simple.

Exponent (e) — 8 bits. This determines the scale of the number — how big or small it is. But it's not used directly. The actual exponent used in the formula is e - 127. This "bias" of 127 means that an exponent field of 127 means 2⁰ = 1, an exponent field of 128 means 2¹ = 2, and so on. This lets us represent both very large and very small numbers.

Fraction (f) — 23 bits. This is the precision — the significant digits of the number. It represents the fractional part after a binary point.

For normal numbers (when 0 < e < 255), the formula is:

value = ±2(e−127) × 1.fbinary
The "1." before f is the hidden one bit — it's implied, not stored

The 1.f means: take the binary string f, put a "1." in front of it, and interpret that as a binary number. For example, if f = 01000...0, then 1.f = 1.01 in binary = 1.25 in decimal.

The Hidden One Bit

Here's a clever trick. Since we're representing normalized numbers (where the leading digit is non-zero), and since in binary the only non-zero digit is 1, the leading digit is always 1. So why waste a bit storing it? We don't. We just assume it's there.

This gives us an extra bit of precision for free. Instead of 23 bits of precision, we effectively get 24 bits (the 23 stored bits plus the hidden 1). For 32-bit floats, that's roughly 3% more precision — and in machine learning, where 16-bit and even 8-bit floats are common, this extra bit matters even more.

The Problem

But the hidden one bit creates a huge problem: if every number is of the form 1.something, then every number is non-zero. We can't represent zero! We can get close (make the exponent as small as possible), but 1.000...0 × 2-127 is still not zero. This is the most important floating-point number, and our representation can't handle it.

This leads us to the first "special case" in the float design — and the broader lesson that when you design a type, you'll often need special cases for important edge values.

Representing Zero (and ±0.0)

To represent zero, IEEE 754 carves out a special case: when e = 0, the rules change.

When e = 0 and f = 0: the value is zero. Since the sign bit can still be 0 or 1, we get two zeros: +0.0 and -0.0.

Key Design Decision

Having two zeros is mathematically weird — there's only one zero in real mathematics. This was actually controversial when the standard was being designed in the 1970s-80s. Professor Kahan at UC Berkeley (who led the IEEE 754 effort) argued for keeping -0.0 because it preserves useful information about the direction from which a computation approached zero.

The practical result: +0.0 == -0.0 evaluates to true (they compare as equal), but they have different bit patterns and can behave differently in certain operations. This means two values with different bit representations compare as equal — remember this, because NaN will give us the exact opposite problem.

How can you tell them apart? Division:

1.0 / +0.0 = +infinity
1.0 / -0.0 = -infinity

The sign of zero matters when you divide by it — it determines whether you get positive or negative infinity. This was Kahan's key argument for why -0.0 should exist.

Denormalized Numbers

When e = 0 but f ≠ 0, we get denormalized numbers (also called "subnormal" numbers). The formula changes:

value = ±2−126 × 0.fbinary
Notice: the hidden bit is now 0, not 1. And the exponent is fixed at -126.

Two critical changes: (1) the hidden bit becomes 0 instead of 1, and (2) the exponent is fixed at -126 (not -127, as you might expect from e-127 = 0-127 = -127 — this is a deliberate choice to avoid a gap in representable numbers).

Why do denormalized numbers exist? They solve a subtle but devastating bug called catastrophic cancellation underflow:

The Problem They Solve

Without denormalized numbers, the smallest positive representable number would be 2-126. Anything smaller gets rounded to 0.

Now imagine two numbers that are very close together: a = 2-126 and b = 2-126 - 2-150. These are different numbers (a ≠ b). But without denormalized numbers, both would be rounded to 2-126, and their difference a - b would be 0.

This means a != b but a - b == 0 — a direct violation of basic arithmetic that breaks many algorithms. Denormalized numbers prevent this by allowing numbers smaller than 2-126 to be represented, so the difference between close-but-different numbers is always non-zero.

Overflow & Infinity

The largest representable normal float has e = 254 (not 255 — that's reserved) and all fraction bits set to 1, giving roughly 3.4 × 1038. What happens when a computation exceeds this?

In older systems and with integers: the behavior is undefined. The program might crash, produce garbage, or (with signed integer overflow in C) literally anything can happen.

In IEEE 754 floating point: you get infinity, which is the special case where e = 255 and f = 0.

// Overflow produces infinity, not a crash
3.4e38 * 2.0  →  +infinity

// Division by zero also produces infinity
1.0 / 0.0    →  +infinity
-1.0 / 0.0   →  -infinity

// Infinity arithmetic is defined
infinity + 42  →  infinity
infinity * -1  →  -infinity

The sign of infinity follows the same rule as normal arithmetic — the sign of the result is the XOR of the operand signs.

Connection to Type Design

This is the "special values" approach from our three strategies. Instead of crashing (runtime error) or preventing overflow at compile time (which would be too restrictive for most programs), IEEE 754 uses a special value that propagates through computations. The program keeps running, and you can check for infinity later if you want.

NaN — Not a Number

But infinity can't handle everything. What about:

infinity - infinity  →  ???  (could be any number!)
0.0 / 0.0          →  ???  (undefined in math)
sqrt(-1.0)         →  ???  (no real answer)

For these, IEEE 754 has one more special case: e = 255 and f ≠ 0. This is NaN (Not a Number).

Definition — NaN

NaN represents a computation that has gone so far off the rails that the result isn't even a number anymore. It's a floating-point value, but it is not a number. Once a NaN appears, it "infects" all further arithmetic: NaN + x = NaN, NaN * x = NaN, and so on.

There are actually many different NaN values — since f can be any non-zero 23-bit value and the sign can be 0 or 1, there are roughly 224 - 2 different NaN bit patterns. The standard doesn't specify exactly which NaN you get from a given operation — that's up to the hardware.

Real-world consequence: Weather maps sometimes display "NaN" where a sensor stopped working and the software divided zero by zero (or did some other undefined computation). The NaN propagated through the entire weather calculation pipeline and showed up in the final display. Wouldn't it be nice if the compiler caught this before the program ever ran?

NaN & Broken Comparisons

Here's where NaN gets truly weird, and where it reveals deep problems for type systems.

If x is NaN, what should x < 1 return? The "correct" answer is neither true nor false — the comparison itself is meaningless, because NaN isn't on the number line. But booleans only have two values! The designers couldn't return a "third value" without breaking every if-statement ever written.

Their solution: all comparisons involving NaN return false.

// If x is NaN:
x < 1    →  false
x > 1    →  false
x == 1   →  false
x == x   →  false   // NaN is not equal to ITSELF!

Read that last line again. A NaN value, compared to itself — the exact same bit pattern in the exact same register — returns false. This means:

Two Broken Rules of Normal Logic

1. ±0.0: Two different bit patterns compare as equal (+0.0 == -0.0 is true).

2. NaN: Two identical bit patterns compare as not equal (NaN == NaN is false).

This also means that !(x < y) is not the same as x >= y when NaN is involved:

// If x is NaN:
x < y        →  false
!(x < y)     →  !(false)  →  true   // suggests x >= y

x >= y       →  false          // but this is also false!

// So: !(x < y) ≠ (x >= y)  — basic logic is broken!

You can actually exploit this weirdness to detect NaN:

bool isNaN(float x) {
    return x != x;  // only true if x is NaN
}

Three Strategies for Handling Bad Values

Let's zoom out and see what we've learned about handling errors in types:

StrategyHow It WorksProsCons
Runtime error (crash)When bad things happen, the program stops immediatelyCan't do any damage with garbage dataProgram crashes — maybe at 3 AM in production
Special values (NaN, infinity)Return a sentinel value and keep goingProgram doesn't crash; computations continueGarbage can propagate silently; broken comparison logic
Compile-time checkingThe compiler prevents the error before the program runsSafest — bugs caught before deploymentPrograms get longer/harder to write; impractical for floats

Why doesn't float use compile-time checking? Because it would require you to prove at compile time that no overflow, underflow, or undefined operation is possible — which would roughly double program size and make float arithmetic painfully verbose. The users of floating point (scientists, ML engineers) don't want that. They want to throw numbers together and deal with problems later. Many machine learning algorithms actually don't care about rounding errors because the numbers are approximate anyway.

Hardware actually gives you a choice: every x86 and ARM chip has a special bit you can turn on that switches from "special values" mode to "crash on error" mode. Almost nobody turns it on — but it's there.

When you design your own types, you'll face the same three choices. The right answer depends on your users and your use case.

Part 4 — Type EquivalenceAbstract vs. Concrete Types

Why Are We Learning This?

We've been designing and exploring individual types. Now we ask a bigger question: when you have two types, are they the same? This sounds simple, but the answer has profound implications for how languages work — and different languages answer it differently.

First, a quick distinction. A concrete type is one where you can see and manipulate the internal representation. A abstract type is one where the internal structure is hidden — you interact with it only through defined operations.

float is an abstract type in most languages. You can't peek at the individual bits (sign, exponent, fraction) directly. You just use operations like +, -, *, /. In fact, different hardware might arrange the bits in a different order, and your program would still work, because you never look at the bits directly.

Structural Equivalence

Definition — Structural Equivalence

Two types are structurally equivalent if they have the same structure — they're built the same way from the same primitives. It doesn't matter what they're named; if they're constructed identically, they're the same type.

Example
(* OCaml — both are really just ints underneath *)
type A = int
type B = int

Under structural equivalence, A and B are the same type — because they're both built from int in the same way. You could assign a value of type A to a variable of type B with no problem.

Advantage: Flexible. You don't need to worry about naming conflicts.

Disadvantage: Semantically dangerous. Just because two types have the same structure doesn't mean they represent the same concept.

Name Equivalence

Definition — Name Equivalence

Two types are name-equivalent if and only if they have the same name. Different names = different types, even if they're structured identically.

Example
// C++ — two structs with identical internals
struct X { int data; X* next; };
struct Y { int data; Y* next; };

Under name equivalence (which C++ uses for structs), X and Y are different types — even though they have exactly the same fields in the same order. You cannot pass a struct X to a function expecting struct Y.

Advantage: Safe. Prevents accidentally mixing up types that happen to have the same structure but mean different things.

Disadvantage: Restrictive. Sometimes you want two identically-structured types to be interchangeable.

Structural EquivalenceName Equivalence
How comparedBy internal structure / shapeBy the name of the type
FlexibilityMore flexibleMore restrictive
SafetyLess safe — easy to mix up semanticsVery safe — prevents semantic bugs
Example languagesOCaml, ML, TypeScriptC, C++, Java

C/C++: Uses Both!

Here's the twist: C and C++ actually use both forms of equivalence in different situations.

Name equivalence for struct and class:

struct X { int data; X* next; };
struct Y { int data; Y* next; };
// X and Y are DIFFERENT types (name equivalence)

Structural equivalence for typedef:

typedef int pid_t;
typedef int uid_t;
// pid_t and uid_t are the SAME type (structural equivalence)
// Both are just int underneath

pid_t x;
uid_t y;
x = y + 1;  // This compiles! Even though PIDs and UIDs are different concepts

Why the inconsistency? Convenience. If typedef used name equivalence, you couldn't add 1 to a uid_t to iterate through user IDs. C's designers decided that typedef is just an alias — a shorthand name for an existing type — not a new type.

The result: pid_t and uid_t are semantically different things (process IDs and user IDs), but C treats them as the same type. You can accidentally assign a PID to a UID variable, and the compiler won't complain. This is the trade-off C made: convenience over safety.

Part 5 — SubtypesWhat Is a Subtype?

Connecting to the Bigger Picture

Type equivalence asks "are these two types the same?" But there's a more general and more important question: "is type T a kind of type U?" This is the subtype question, and it's the foundation of how inheritance works in object-oriented programming — and much more.

Definition — Subtype

Type T is a subtype of type U (written T ⊆ U) if every value of type T can safely be used wherever a value of type U is expected. In other words: every T value is also a valid U value.

If T is a subtype of U, and U is a subtype of T, then T and U are equivalent types. So subtyping is a generalization of type equivalence.

Subtyping is fundamentally about safe substitution: can I take a value of type T and use it anywhere type U is expected, without anything breaking? Let's see why this is trickier than it sounds.

Example: int vs. long

Is int a subtype of long? Every int value fits inside a long (a long is at least as big as an int). So intuitively, yes — every int value is a valid long value.

But now: is int[] (an array of ints) a subtype of long[] (an array of longs)?

Intuitively, it seems like it should be. But in C-like languages, the answer is no, and here's why:

Arrays in C are represented as a pointer to the first element. To access element N, you do: base_address + N × element_size. For int[], each element is 4 bytes, so you jump by 4 bytes. For long[], each element is 8 bytes, so you jump by 8 bytes.

If you passed an int[] to a function expecting long[], the function would try to read 8 bytes per element from an array where elements are only 4 bytes wide — it would read garbage. The internal representation is physically incompatible.

Key Insight

Subtyping is about safe substitution, not just value ranges. Even though every int value fits in a long, arrays of int and arrays of long have different memory layouts and cannot be safely substituted.

Example: Class Inheritance

The most familiar form of subtyping is class inheritance in object-oriented programming:

class C extends D {
    // C has everything D has, plus more
}

Here, C is a subtype of D. Every C object is also a valid D object. If a function expects a D, you can pass it a C — because C has all of D's methods (and more).

Notice the pattern:

D — supertype (parent)
has method m()
↑ extends
C — subtype (child)
has methods m(), n(), o(), p()
Subtypes are MORE SPECIFIC and have MORE operations than their supertypes

This is counterintuitive: the subtype has a superset of operations. D is vague and generic (few methods). C is specific and powerful (many methods). Since C can do everything D can do (and more), a C value is always safe to use where a D is expected.

Example: const Pointers in C

This is where subtyping gets tricky and interesting. Consider these two types:

char*          // pointer to char — can read AND write through the pointer
char const*    // pointer to const char — can only READ through the pointer

First, let's be precise about what char const* means. It does not mean "a pointer to a character that can never change." It means "I promise not to modify the character through this pointer." The character itself might be modified by other means:

char c = 'x';
char const* p = &c;    // p promises not to modify c through p

char d = *p;            // reading through p is fine
// *p = 'y';            // ILLEGAL — can't write through a const pointer

c = 'y';                // perfectly legal! c can be modified directly
char e = *p;            // now e is 'y' — the value changed, even though p is const

Now: is char* a subtype of char const*, or the other way around?

Apply the subtyping rule from class inheritance: the subtype has more capabilities. char* can do everything char const* can do (read) plus more (write). So char* is more powerful, more specific — it's the subtype.

char* ⊆ char const*

This means you can assign a char* to a char const* variable (you're giving up write capability — safe), but NOT the other way around (you'd be gaining write capability that wasn't promised — unsafe).

Why is char* a subtype of char const*?

Example: volatile

char volatile* is a pointer that tells the compiler: "the value at this memory address can change at any time, for reasons you can't see in the code — don't optimize away reads or writes."

This is used for hardware device registers (where the hardware can change the value) and multithreaded programs (where another thread can change the value).

The subtyping question: volatile places restrictions on what the compiler can do (no caching reads, no reordering writes). Since it adds restrictions, the same logic applies as with const:

char* ⊆ char volatile*

A regular char* doesn't promise volatile semantics, so it can be used anywhere volatile is expected (the compiler will just be more cautious). But a char volatile* shouldn't be used where a regular char* is expected, because optimizations that assume the value doesn't change would be unsafe.

Part 6 — PolymorphismWhat Is Polymorphism?

Why Are We Learning This?

We've spent the whole lecture building up from "what is a type" to how types relate to each other (equivalence, subtypes). Now we ask: what happens when a single function or operator can work with multiple types? This is polymorphism — literally "many forms" from Greek — and it's one of the most powerful ideas in programming language design. It's the reason you can use + on both integers and floats, and the reason Java generics and OCaml's 'a list work.

Definition — Polymorphism

A function or operator is polymorphic if it can accept arguments of different types and/or return different types, depending on context. The same symbol or name has "many forms."

There are three main kinds of polymorphism, each with different mechanisms and trade-offs:

Overloading (ad hoc polymorphism)Same name, different implementations for different types
CoercionImplicit type conversion to make things "just work"
Parametric polymorphismOne implementation that works for ALL types (generics)

Overloading (Ad Hoc Polymorphism)

Definition — Overloading

Overloading means a single function name or operator symbol has multiple definitions that are distinguished by the types of their arguments. The compiler picks the right one based on what types you pass.

The simplest example: the + operator.

// The + operator is "overloaded" — it means different things
// for different types:

3 + 5           // integer addition (uses ADD instruction)
3.0 + 5.0       // floating-point addition (uses FADD instruction)
"hi" + "there"  // string concatenation (calls a function)

Even though all three use +, they're doing completely different things at the machine level — different instructions, different hardware units. The compiler looks at the types of the operands and generates the appropriate code.

You can also overload functions explicitly:

// C++ — two different functions with the same name
int square(int x)    { return x * x; }
double square(double x) { return x * x; }

square(5);     // calls the int version
square(5.0);   // calls the double version

How is overloading implemented? At the machine level, the compiler uses name mangling — it takes the function name and appends the argument types to create a unique name. So square(int) might become square$int and square(double) becomes square$double internally. These are actually two completely separate functions that just share a name in the source code.

This has been around since the very first programming languages. Fortran (1957) had overloaded arithmetic operators and even overloaded math functions like cos(x), which called different implementations depending on whether x was single or double precision.

Overloading with Objects
// C++ — method overloading with objects
obj1.method(x);
obj2.method(y, z);

// Even if obj1 and obj2 are the same type, these call
// different method implementations because the argument
// lists are different.

Coercion — Implicit Type Conversion

Definition — Coercion

Coercion (also called implicit conversion) is when the language automatically converts a value from one type to another to make an operation work. You don't write a cast — the language does it for you.

Coercion exists to simplify overloading. Without coercion, if you have 3 types (int, float, bool) and the + operator, you'd need 9 different implementations (every combination of two types). With coercion, you only need 2 (int+int and float+float), and the language automatically converts operands to match.

// int + float — what happens?
int a = 5;
float b = 3.14;
float c = a + b;

// The compiler implicitly converts a to float:
// float c = ((float) a) + b;
// Result: c = 8.14 (floating-point addition)
Coercion Can Lose Information!

Here's a subtle and dangerous case:

int i = 0x7FFFFFFF;  // 2,147,483,647 (max 32-bit int)
float f = i;         // implicit conversion: int → float

The int has 31 bits of meaningful data. But a 32-bit float only has 24 bits of precision (23 fraction bits + 1 hidden bit). So the conversion loses information — f won't exactly equal i. It will be rounded to 2,147,483,648 (231).

Now if you use f in a computation, you're working with a slightly wrong number, and the compiler didn't warn you. This is the fundamental problem with coercion: it trades safety for convenience.

Another classic coercion trap:

unsigned int a = 5;
if (a > -1) { ... }  // Always TRUE? No — always FALSE!

// Why: -1 is coerced to unsigned, becoming 4,294,967,295 (max uint)
// So this is really: if (5 > 4294967295) — which is false

OCaml avoids all of this by not doing implicit coercion. If you want to add an int and a float, you must explicitly convert: float_of_int a +. b. OCaml even uses different operators for int addition (+) and float addition (+.). This is more verbose, but eliminates an entire class of bugs.

Parametric Polymorphism

Connecting to the Bigger Picture

Overloading and coercion are called "ad hoc polymorphism" — they were invented early and kind of bolted on. Now we get to the elegant version: parametric polymorphism, where a single piece of code works for any type. This is what makes OCaml's 'a list work, and what Java generics (List<T>) are based on. ML was one of the first languages to get this right.

Definition — Parametric Polymorphism

Parametric polymorphism lets you write code using type variables (type parameters) instead of concrete types. The same code works for any type that the variable is instantiated with — and you get type safety guaranteed by the compiler.

In OCaml, when you write:

let rec reverse l = match l with
  | [] -> []
  | h :: t -> (reverse t) @ [h]

OCaml infers the type as 'a list -> 'a list. The 'a is a type variable — this function works for int list, string list, bool list list, anything. And it's one implementation, not a different function for each type.

Why does this work? Because the code doesn't depend on what 'a is. It only uses list operations (pattern matching, append). It never adds, compares, or does anything type-specific to the elements. This is the key insight: if the code doesn't depend on properties of the concrete type, it can be generic.

This gives us three things overloading and coercion don't:

Reusability — one function works for all types.

Type safety — the compiler guarantees correct usage at compile time.

No information loss — no coercion means no rounding, no truncation.

Old Java — Before Parametric Polymorphism

To really appreciate parametric polymorphism, let's see what life was like without it.

In old Java (before version 5, released in 2004), every object was ultimately of type Object. If you wanted a collection that could hold strings, you had to use Collection (which held Objects) and manually cast:

// Old Java — remove all single-character strings from a collection
for (Iterator i = c.iterator(); i.hasNext();) {
    // i.next() returns Object — we MUST cast to String
    if (((String) i.next()).length() == 1)
        i.remove();
}

The problems:

1. No compile-time safety. The compiler doesn't know that the collection contains strings. If someone accidentally adds an Integer to the collection, the code compiles fine — but crashes at runtime when the (String) cast fails.

2. Runtime overhead. Every cast has to be checked at runtime (is this really a String?). That costs CPU time.

3. You only find bugs when the code runs. If the bad element happens to be at position 10,000 in the collection, you won't discover the bug until the loop gets there — maybe days after deployment.

New Java — With Parametric Polymorphism (Generics)

Java 5 added generics — parametric polymorphism. Now you can say what type a collection contains:

// New Java — the collection is declared as holding Strings
for (Iterator<String> i = c.iterator(); i.hasNext();) {
    // i.next() returns String — no cast needed!
    if (i.next().length() == 1)
        i.remove();
}

The improvements:

1. Compile-time safety. If someone tries to add an Integer to a Collection<String>, the compiler catches it immediately — before the program ever runs.

2. No casts. i.next() returns a String directly, not an Object that needs casting. Less code, fewer chances for mistakes.

3. No runtime overhead. The compiler already verified the types. No runtime checks needed.

Old Java (pre-generics)New Java (with generics)
Collection typeCollection (holds Object)Collection<String>
Getting elementsReturns Object; manual castReturns String directly
Type errorsDiscovered at runtime (crash)Caught at compile time
PerformanceRuntime cast checking overheadNo overhead — types verified at compile time

The Recurring Theme

Throughout this entire lecture, we've seen the same theme over and over: catching errors at compile time is safer than catching them at runtime. OCaml's discriminated unions catch null pointer bugs at compile time. OCaml's type inference catches type mismatches at compile time. Java's generics catch collection-type bugs at compile time. Even the float discussion was about whether we could catch floating-point errors at compile time (we concluded it's too impractical for floats, so we use special values instead). The design of a type system is fundamentally about deciding where on the spectrum from "total compile-time safety" to "anything goes at runtime" your language should sit.
