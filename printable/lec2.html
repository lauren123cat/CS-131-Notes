Language Design &
Functional Programming

Why do programming languages look the way they do? What tradeoffs do designers wrestle with? And what is functional programming — the paradigm that gives up assignment statements and gains enormous power in return?

Core takeaway: Language features aren't random or arbitrary. They're the result of hardware constraints, safety tradeoffs, backward compatibility pressures, compiler optimization strategies, and how human brains think about computation.
Part 1

Why Are There So Many Programming Languages?

A professor once asked: "Why can't there just be one programming language? Wouldn't that make life simpler?" This deceptively simple question has a deep answer — and understanding it is the foundation for everything else we'll learn.

Think about the question for a moment. If one language could do everything, you'd learn it once and be done. No need for a course on programming languages at all. But the reality is that there are hundreds of programming languages in active use, and new ones keep being created. Why?

The answer is that every programming language is a set of tradeoffs. A language that's excellent at one thing is often mediocre or even terrible at another. These tradeoffs stem from design goals — the priorities that the language's creators had in mind. Since different people have different priorities, they create different languages.

Analogy: Tools in a Workshop

Think of programming languages like tools. A hammer is great for nails but terrible for screws. A screwdriver handles screws perfectly but can't drive nails. You could build a multi-tool that sort of does everything — but it wouldn't do any single thing as well as a dedicated tool. Languages are the same: each one is optimized for certain jobs.

Language Design Goals

Let's walk through the major goals that language designers care about. These goals often compete with each other — optimizing for one frequently means sacrificing another. Understanding these tradeoffs is essential for choosing the right language for a project.

Goal 1: Performance

Performance means how fast your programs run and how efficiently they use resources like memory. Some applications — video games, operating systems, real-time control systems — need to squeeze every last bit of speed out of the hardware.

Languages like C and C++ are designed with performance as a top priority. They give you fine-grained control over memory, let you work close to the hardware, and their compilers can produce extremely optimized machine code. But this control comes at a cost: these languages are among the hardest to learn and use correctly.

Goal 2: Understandability

Understandability means how easy it is for a human to read, write, and reason about code. This matters enormously because code is read far more often than it's written — by you later, by your teammates, by people who maintain the code after you've moved on.

There's an interesting sub-question here: should a language give you one way to write something, or many ways?

Example: Perl vs Python

Perl was designed by a linguist, and its motto was "there's more than one way to do it." Just like English lets you say "if X, call Y" or "call Y if X," Perl lets you write the same logic multiple ways:

# Both do the same thing in Perl:
if ($x) { call_y(); }
call_y() if $x;

Python was originally created as a reaction to Perl's approach. The Python philosophy said: "There should be one — and preferably only one — obvious way to do it." Having one way means consistency and easier readability. Having many ways means flexibility but makes unfamiliar code harder to read.

Ironically, as Python has grown over the decades, it has gradually gained many alternative ways to express things — falling into what the professor calls "the Perl trap."

Connection: The Sapir-Whorf Hypothesis

In the previous lecture, the class discussed the Sapir-Whorf hypothesis from linguistics: the language you speak shapes how you think. This applies to programming too! The language you program in affects how you think about problems. This is exactly why we study multiple language paradigms — they teach us different ways to think about computation.

Goal 3: Compatibility with Existing Practice

No language exists in a vacuum. Real software interacts with libraries, operating systems, hardware, and other software — often written in different languages. A new language needs to be compatible with the existing ecosystem, or nobody will adopt it.

At the lowest level, most systems have some kind of C interface — C is like the "common tongue" that almost every other language can talk to. For example, C++ has a feature called extern C that lets C++ code expose functions in a way that C code can call. This compatibility exists at two levels:

Key Concepts

API compatibility (Application Programming Interface): source-level compatibility. Your old source code still compiles with the new version of the language.

ABI compatibility (Application Binary Interface): binary-level compatibility. Programs and libraries that were already compiled still link and run correctly, even with newer code.

Why this matters

If a language breaks compatibility when it updates, companies can lose the ability to run important programs, and developers have to rewrite their code every time an update occurs. This is enormously expensive. Languages that break backward compatibility (like Python 2 → Python 3) cause years of migration pain.

Goal 4: Safety

Safety means: do your programs behave predictably? When something goes wrong, does the system tell you about it in a controlled way — or does it silently misbehave in dangerous ways?

This isn't just about your laptop showing a "blue screen of death." Safety failures can cause real-world disasters — like a car whose brakes don't work, or a medical device that gives the wrong dosage. This is why safety is a huge deal in modern language design.

Undefined Behavior: The Dark Side of C/C++

C and C++ are the classic examples of unsafe languages. They give you enormous power, but that power includes many ways to "shoot yourself in the foot." The key concept here is undefined behavior (UB).

Definition: Undefined Behavior (UB)

Undefined behavior occurs when the language standard says: "if your program does this, we're not going to define what happens." The program can do anything — crash, produce wrong results, delete your files, or even appear to work correctly. The compiler is free to generate whatever code it likes.

Here's a concrete example that surprises most people, even experienced C/C++ programmers:

Example: Integer Overflow is Undefined

What happens when you take the maximum integer value and add 1?

// What does this do in C/C++?
int x = INT_MAX;  // Maximum int value: 2,147,483,647
x = x + 1;        // What is x now?

If you took a computer architecture class, you might have been taught that this "wraps around" to INT_MIN (the most negative integer). And on most hardware, that is what the machine does. But the C and C++ language standards say this is undefined behavior. The compiler can assume it never happens — and use that assumption to optimize your code in surprising ways.

This isn't just theoretical. Modern compilers like GCC and Clang actively exploit undefined behavior to optimize code. Here's a striking example from the lecture:

Example: How UB Changes Your Program's Meaning
// What does this loop do?
for (int i = x - 3; i <= INT_MAX; i++)
    j++;

If you believe integer overflow wraps around (as hardware actually does), this loop runs forever — because i would eventually overflow past INT_MAX and wrap to a negative number, which is still ≤ INT_MAX, so the loop never ends.

But GCC, with optimization enabled, compiles this to the equivalent of j += 334 (or whatever the mathematically correct count is). Why? Because the standard says signed overflow is undefined, so the compiler assumes it never happens. If overflow can't happen, then i will eventually exceed INT_MAX, and the loop terminates after a finite number of iterations.

The ironic thing is: the compiler's version gives the mathematically "correct" answer — the one a naive programmer would expect. But it does so by reasoning that your loop can't possibly run forever.

Check Your Understanding

In C/C++, what happens when signed integer overflow occurs?

Correct! The C/C++ standard says signed integer overflow is undefined behavior. While the hardware typically wraps around, the compiler is free to assume overflow never occurs and optimize accordingly. This is a key reason C/C++ are considered "unsafe" — programmers can be surprised by compiler optimizations that exploit UB.
Not quite. While the hardware typically does wrap around, the C/C++ language standard says signed integer overflow is undefined behavior. The compiler can assume it never happens and produce surprising optimizations. This is fundamentally different from "always wraps around" or "always crashes."

Goal 5: Privacy & Security

Safety and security often go hand in hand. Here's a fascinating example of how they intersect in C:

Example: memset vs memset_explicit

C has a function called memset that fills a block of memory with a value (usually zeros). There's also a newer function called memset_explicit that does the same thing. Why do we need two functions that do the same thing?

The difference is that memset can be optimized away by the compiler. If the compiler notices that nobody reads the memory after the memset call, it might skip the zeroing entirely — it's "dead code" from the optimizer's perspective.

// Scenario: you just finished using a password
char password[256];
// ... use the password for authentication ...

memset(password, 0, sizeof(password));  // Try to clear it
return;  // Function ends — nobody reads password after this
// Compiler thinks: "Nobody reads password, so I'll skip the memset!"
// Password remains in memory! An attacker can find it.

memset_explicit tells the compiler: "No, really zero this memory, no matter what. Don't optimize this away." This exists for exactly the security scenario above — you want to erase sensitive data from memory even if the compiler thinks nobody will read it, because an attacker might read it through a bug in another part of your program.

memset_explicit(password, 0, sizeof(password));  // MUST zero this memory!
return;  // Password is guaranteed to be erased

This function was added to the C23 standard specifically for this security use case.

Connection: Safety ↔ Security ↔ Optimization

Notice how security, safety, and performance are all tangled together. The compiler optimizes away the memset for performance (why do work nobody needs?), but this creates a security vulnerability. memset_explicit sacrifices a tiny bit of performance to gain security. This is the essence of language design tradeoffs.

More Design Goals

The class identified several additional goals. Each one represents a dimension along which languages make different choices:

Modularity & Abstraction — The ability to break large programs into manageable components. You want to write a 10-million-line program and have it work. Some languages (like Java, with its package system and interfaces) are designed for this scale. Others (like the original Python) weren't really designed for programs that large.

Scalability to Large Programs — Related to modularity, this is about how well a language handles massive codebases. Features like strong type systems, module systems, and visibility controls all help here.

Portability — Can the same code run on different machines and operating systems? C has portability problems: the size of an int is "at least 32 bits" but could be 64. Microsoft keeps it at 32 bits while most other platforms use 64. This means C code that assumes a specific int size can break when moved to a different platform.

Concurrency & Parallelism — Parallelism means different parts of your program can run at the same time. Concurrency means those parts can coordinate without stepping on each other's toes. Threading is a common approach to concurrency, but it's notoriously hard to get right — threads can interfere with each other in subtle, hard-to-reproduce ways.

Debuggability & Robustness — Debuggability: how easy is it to find and fix bugs? Robustness: do programs degrade gracefully with unusual input, or do they crash? Interestingly, these goals connect to safety in a subtle way: if a language prevents certain bugs from ever occurring (like null pointer dereferences), you don't need to debug those bugs — they can't happen.

Ease of Implementation — How hard is it to build a compiler or interpreter for the language? This matters enormously when a language is getting started. JavaScript's early success partly came from being easy to implement — browser makers could add it quickly. C's initial compilers were crude by today's standards, but they were easy to write.

Goal 6: Orthogonality

This is a design goal that's less commonly discussed but deeply important. The name comes from mathematics.

Definition: Orthogonality

In mathematics, orthogonal vectors are at right angles to each other — choosing a value along one axis doesn't constrain your choices along the other axes. In language design, orthogonality means that the language's features don't interfere with each other. If feature A works with X, then it should also work with Y and Z. You can combine features freely without running into unexpected restrictions.

Analogy: Think of a 3D Space

Imagine picking a point in 3D space. You choose an X coordinate, a Y coordinate, and a Z coordinate. Your choice of X doesn't limit your options for Y or Z — you can independently pick any value for each. An orthogonal language works the same way: choosing one feature doesn't restrict what you can do with other features.

The best way to understand orthogonality is through a counter-example — a case where a language is not orthogonal:

Example: C's Non-Orthogonal Return Types

In C and C++, a function can return a value of any type — except arrays. You can return an int, a double, a pointer, a struct (even a massive 400,000-byte struct!) — but you cannot return an array.

// These are all fine:
int f1() { return 42; }
double f2() { return 3.14; }

struct Big { char data[400000]; };
struct Big f3() { /* ... */ }  // This is fine too!

// But this is NOT allowed:
int[10] f4() { /* ... */ }  // ERROR! Can't return an array

This is non-orthogonal: you've chosen to write a function (one axis), and you want to choose what type to return (another axis), but certain return types are forbidden. Your choice along one dimension restricts your choices along the other.

Why can't C functions return arrays? (Historical deep-dive)

It's a combination of historical accidents and design decisions:

1. The original calling convention: When C was first designed on the PDP-11, functions returned their value in a CPU register. Ints, pointers, and doubles fit in a register. Arrays don't. So the designers simply said: you can't return an array.

2. Structs were added later: When structs were added to C, they came up with a new calling convention where the caller provides the address of where the return value should go. This works fine for structs.

3. Array-to-pointer decay: In C, an array name automatically "decays" into a pointer to its first element. This implicit conversion (e.g., int a[1000]; int *p = a;) would make it confusing to combine with function returns — would you be returning the array or a pointer to it?

In contrast, Python is orthogonal in this regard: a function can return any type of value, including lists (Python's equivalent of arrays). No restrictions, no special cases.

Check Your Understanding

Division by zero is related to orthogonality because:

Correct! It's a case of non-orthogonality: you've chosen the division operator (one axis) and an operand value (another axis), but certain operand values (zero) are forbidden. Interestingly, floating-point is more orthogonal here — dividing by zero gives infinity and the program keeps running.
Not quite. Division by zero is related to orthogonality: you can divide any two numbers except when the second is zero. This restriction makes the division feature non-orthogonal. Floating-point partially fixes this by returning infinity instead of erroring.

Goal 7: Mutability / Extensibility

Languages aren't frozen in time. They need to evolve — adding new features, fixing design mistakes, improving performance. But critically, old programs should still work. This is the tension between extensibility and backward compatibility.

Example: The Many Meanings of "static" in C/C++

The keyword static in C/C++ is a poster child for how backward compatibility can distort a language. It has at least three completely different meanings depending on context:

/* Meaning 1: Inside a function — persistent local variable */
void counter() {
    static int x = 0;  // Created once, keeps its value between calls
    x++;
    printf("%d\n", x);   // Prints 1, 2, 3, ... on successive calls
}

/* Meaning 2: At file scope — limits visibility to this file */
static int secret = 42;  // Only this .c file can see this variable

/* Meaning 3: In function parameters — minimum array size */
int f(int a[static 2]) // "a" points to at least 2 elements

Meaning 1 is about lifetime (how long the variable exists). Meaning 2 is about scope (who can see it). Meaning 3 is about minimum size guarantees. These are completely unrelated concepts crammed into a single keyword!

Why? Because introducing a new keyword (like atleast for meaning 3) would break any existing program that uses atleast as a variable or function name. So the designers reused static to maintain backward compatibility — sacrificing understandability in the process.

The Big Picture: Goals Compete

The static example perfectly illustrates how design goals compete. Extensibility (adding new features) conflicts with compatibility (not breaking old code), which then undermines understandability (having one keyword mean three things is confusing). Language designers constantly navigate these tradeoffs.

Part 2

Hardware Shapes Languages

Languages don't exist in a vacuum. They're designed for specific machines, and the constraints of those machines fundamentally shape the language. Understanding this history explains why languages look and act the way they do.

Programming languages evolve much like human languages do — they change over time in response to their environment. A Fortran programmer from the 1950s would barely recognize a modern Fortran program. The same goes for C and C++ — they've evolved enormously from their origins.

Let's look at three specific examples of how hardware shaped language design:

Late 1950s / 1960s
BASIC on the General Electric GE 225

The GE 225 mainframe cost $1 million. It was designed by Arnold Spielberg (Steven Spielberg's father!) as a skunkworks project at General Electric. Its specs:

Integer addition time40 microseconds
Integer division time500 microseconds
RAM~40 KB
Simultaneous users10

BASIC was designed under these brutal constraints. With only 40 KB of RAM shared among 10 users, identifiers could only be 1 or 2 characters long — there simply wasn't memory for longer names. Everything was interpreted and slow. But it was "good enough to teach programming" — which was the goal.

1970s
C on the PDP-11

The PDP-11 was the machine C was born on. Its specs:

Integer addition time4 microseconds (10x faster than GE 225)
RAM16 KB (smaller than GE 225!)
Memory cycle time1.2 microseconds

Here's a critical detail: the memory cycle time (1.2 μs) was about 3 times faster than the CPU's addition time (4 μs). Memory was faster than the CPU! This explains a core feature of C:

Why C Loves Pointers

The *p dereference operator — taking a pointer and following it to the data — was cheap on the PDP-11 because memory access was faster than arithmetic. C's syntax was designed to encourage dereferencing because it was a fast operation. Expressions like *p++ were fast, while arithmetic like a + b * c was relatively slow. C's syntax was optimized for this reality.

1995
JavaScript on the Pentium

JavaScript was developed roughly in 1995. The typical target machine:

Integer addition time~0.015 microseconds (~1000x faster than GE 225)
Integer division time~0.06 microseconds
RAM~4 MB (100x more than GE 225)

With 1000x more speed and 100x more memory, JavaScript could afford features that would have been unthinkable for BASIC or C — garbage collection, dynamic typing, flexible data structures — because the hardware was powerful enough to absorb the overhead.

The Modern Memory Problem

Here's where things get really interesting — and where this history connects to today.

Remember how on the PDP-11, memory was faster than the CPU? On modern machines, the relationship has completely flipped. If you access RAM and the data isn't in the cache, it takes roughly 100 to 200 times longer than a simple addition.

The Irony

C was designed for a machine where memory access was cheap and fast. We're still teaching and using C and C++ today, on machines where memory access is relatively extremely slow. The language's syntax still encourages heavy pointer dereferencing, but on modern hardware, that's often the most expensive thing you can do. The language was designed for "kind of the wrong machine."

Check Your Understanding

On the PDP-11 (where C was designed), memory access was about ___ compared to CPU addition. On modern machines, uncached memory access is about ___.

Correct! This dramatic reversal — from memory being faster than CPU to memory being 100-200x slower — is one of the most important shifts in computing history. Modern machines get decent performance only because of aggressive caching.
Not quite. On the PDP-11, memory was about 3x faster than the CPU. On modern machines, uncached RAM is roughly 100-200x slower than a CPU addition. This massive reversal fundamentally changes which operations are "cheap" and "expensive."
Part 3

The 3D Design Space of Languages

Beyond the design goals (where we want to go), there's the design space (what tools and approaches we have to get there). Think of this as a multi-dimensional map of all possible languages.

While the full design space is enormous and multi-dimensional, we can simplify it into three main axes that capture the most important distinctions.

Axis 1: Paradigm — How You Structure Computation

Imperative (Procedural) Languages

These are the languages you're most familiar with. A program is like a cooking recipe: a sequence of instructions that the computer executes one at a time.

Definition: Imperative Language

In an imperative language, you write commands (statements) that tell the computer what to do, step by step. The fundamental way to connect pieces of your program is sequencing — do this, then do that, then do the next thing.

// The semicolon is the "sequence operator"
C1; C2; C3;
// Execute C1, then C2, then C3 — strictly in order

You have loops, conditionals, and recursion, but at the bottom level, everything is sequential execution of commands. Examples: C, C++, Java, Python.

Functional Languages

In a functional language, the fundamental unit isn't commands — it's functions. You write your program as a collection of functions, and you run it by calling a function with arguments.

Definition: Functional Language

The fundamental way to connect pieces of your program is nested function calls:

F1( F2(x), F3(y) )
// Compute F2(x), compute F3(y), feed both results into F1
// We don't care if F2 or F3 runs first!
// They could even run in parallel — doesn't matter

The crucial insight: functional programming takes away two features you're used to — the semicolon (sequencing) and assignment statements (changing variables). Once a variable has a value, it can never change. If you want a new value, you call a new function.

Analogy: The Functional Restriction

Think of functional programming as starting with your imperative language and then tying one arm behind your back. You can no longer write x = x + 1 (that changes a variable). You can no longer write for loops (those need changing variables). Instead, you do everything through function calls and recursion. It sounds limiting — and it is — but it gives you surprising new powers, especially for parallelism.

An important nuance: most real languages support multiple paradigms. OCaml has assignment statements — it's just that in this class, you'll be working within its functional subset and not using the imperative features. Even C++ has a functional subset. The paradigm classification describes the language's primary design philosophy.

Logic Languages

Logic languages are the most different from what you're used to. There are no commands, no functions, no assignment statements. Instead, you write predicates — statements about the world that are either true or false.

Definition: Logic Language

You write your program as a logical expression describing what you want, and the system finds values that make your description true.

P1(x) AND P2(y) AND NOT P3(z)
// "Find me values of x, y, z that make this expression true"
// Similar in spirit to a database query language

You describe the specification of what you want. The system solves it. The main example is Prolog.

Check Your Understanding

In a functional language, which of the following is NOT allowed (in the pure functional subset)?

Correct! The defining restriction of functional programming is that once a variable has a value, you cannot change it. You can create new variables (by calling functions — their parameters are new variables), and you must use recursion heavily (since you can't have traditional loops without changing a counter variable).
Not quite. The key restriction is that you cannot change an existing variable's value. Things like c = a + b; c = c + 1; are forbidden because the second line changes c. You can still call functions, define new variables through function parameters, and use recursion (in fact, recursion replaces loops).

Axis 2: Execution — Compiled vs Interpreted

How Your Code Gets Executed

Definition: Compiled Language

A compiled language is one where the source code is translated into machine code before the program runs. The compiler reads your entire program, checks for errors, optimizes it, and produces an executable binary. The process is: write → compile → run.

Think of it like translating a book into another language once — then anyone can read the translated version forever, without needing the translator.

Definition: Interpreted Language

An interpreted language is one where an interpreter reads and executes your code as the program runs. There's no separate compilation step — you just write and run. The process is: write → run.

Think of it like having a simultaneous translator at a meeting — they translate each sentence as it's spoken, which is slower but more flexible.

PropertyCompiledInterpreted
When is code translated?Before runningWhile running
Binary executable?Yes — after compilationNo
Runtime speedFasterSlower
Error detectionAt compile time (before running)At runtime (while running)
Changing codeHarder — must recompileEasier — just edit and rerun

These two axes combine. You can have imperative compiled languages (C, C++), imperative interpreted languages (Python, Perl), functional compiled languages (OCaml), functional interpreted languages (Lisp), and so on.

Axis 3: Object-Oriented vs Not

A third dimension is whether a language is object-oriented (OOP). In OOP, programs are organized around "objects" that bundle data together with the functions that operate on that data.

Example: OOP vs Non-OOP

In an object-oriented language (like Java or C++), you define a class that encapsulates data and behavior:

// Object-oriented: data and functions live together
class BankAccount {
    int balance;
    void deposit(int amount) { balance += amount; }
    void withdraw(int amount) { balance -= amount; }
};
// Only these functions can touch 'balance' — encapsulation!

In a non-OOP language (like C), you use separate structs and functions:

// Non-OOP: data and functions are separate
struct BankAccount { int balance; };

// Anyone can write a function that modifies the struct!
void deposit(struct BankAccount* acc, int amount) {
    acc->balance += amount;
}

The OOP version provides encapsulation — only the class's own methods can access the data, which prevents bugs and enforces rules. The compiler can verify these access rules statically.

Part 4

Why Functional Programming?

Functional programming might seem like tying one arm behind your back — giving up assignment statements and loops. So why would anyone do it? There are two powerful motivations: clarity and parallelizability.

Motivation 1: Clarity

Mathematicians have been working with functions (in the modern sense) since at least the 17th century — that's roughly 300-400 years of experience reasoning about functions. Mathematical functions are extremely well understood. Functional programming builds on this centuries of experience rather than reinventing the wheel.

Consider the first thing taught in introductory programming:

Example: The Mathematician's Objection
int i = 1;
i = i + 1;  // In CSS 31, this is "increment i"

Show this to a mathematician and they'll say: "This is nonsense! There's no integer i such that i = i + 1. This is an unsolvable equation."

And yet, this is literally the first thing we teach beginning programmers. The = sign doesn't mean mathematical equality — it means "take the current value of i, add 1, and store the result back into i." Functional programming says: let's stop confusing people. Let's use = to mean what mathematicians have always meant by it.

Motivation 2: Escaping the Von Neumann Bottleneck

Definition: The Von Neumann Bottleneck

John von Neumann, one of the greatest mathematicians of the 20th century, designed the basic architecture used by virtually all modern computers: a CPU connected to RAM through a bus. The CPU can only ask for one word at a time and get one word back. This is the bottleneck — no matter how many processors you have, they all have to go through this narrow channel to access memory.

The problem with imperative programming is that variables can change at any time. If two parts of your program use the same variable i, and one part changes i, the other part needs to know about it. This means the two parts can't easily run in parallel — they need to constantly communicate about shared state.

Example: Why Mutable Variables Kill Parallelism
// Imperative code with mutable variable i
// ... some code using i ...
i = 5;
// ... more code using i ...
i = 10;
// ... even more code using i ...

You want to split this across two processors. But Processor 1 might change i, and Processor 2 needs to see the update. Now you need communication between processors, and communication is expensive.

In a functional language, i can never change. If Processor 1 and Processor 2 both see i, they know it has the same value — forever. No communication needed. They can run in parallel without worrying about each other.

Mathematical Functions: The Foundation

In mathematics, a function is a mapping from a domain (input) to a range (output). If you think of it as a set of ordered pairs (input, output), the key rule is: no two pairs have the same input with different outputs. Once you know the input, the output is determined.

A partial function is one that isn't defined for all possible inputs. For example, the function f(x) = 1/x is a partial function from integers to reals — it's not defined when x is 0.

Higher-Order Functions (Functional Forms)

Definition: Higher-Order Function (Functional Form)

A higher-order function (also called a functional form) is a function where at least one of its arguments is itself a function, or it returns a function. This is like a "function of functions."

Example: Integration as a Higher-Order Function

The definite integral from calculus is a familiar higher-order function:

// The integral ∫₀¹⁰⁰ f(x) dx can be thought of as:
integral(f, 0, 100)
// Three arguments: f (a FUNCTION), 0 (a number), 100 (a number)
// The first argument is a function, not a number!

The integral takes a function f and two numbers, and returns a number (the area under the curve). It operates on a function — it's a function about functions.

Why this matters for the course

In OCaml, almost every function you write for homework will be a higher-order function. This is normal in functional programming — OCaml programmers use them constantly and don't think twice about it. In C++, higher-order functions are technically possible but painful to use (partly because of memory management complications). OCaml makes them natural and easy.

Referential Transparency

Definition: Referential Transparency

Referential transparency means that when you look at an identifier (variable name) in your code, you can easily determine its value by looking at where it was defined. The value is "transparent" to you — you don't have to trace through the program's execution to figure out what it might be at any given point.

In an imperative language, the variable i might have different values every time you encounter it in the code. In a functional language, since you can't change variables, every occurrence of i in the same scope has the same value. The compiler can use this for optimization, and — perhaps more importantly — the programmer can use this to read and understand code more easily.

Since functional programs don't modify variables, they have no "side effects" — calling a function doesn't change any state outside the function. The only thing a function does is compute and return a value.

Part 5

OCaml: Your First Functional Language

Now we get hands-on. OCaml is the functional language you'll be using for homework. Let's learn it from the ground up, starting with its core properties and building up to defining and calling functions.

Key Properties of OCaml

Before writing any code, let's understand what makes OCaml distinctive:

1. Static Type Checking — Before your program runs, OCaml figures out the type of every expression and makes sure everything is consistent. If there's a type mismatch, the program doesn't run at all. This catches bugs early and prevents surprise crashes at runtime.

2. Type Inference — Even though OCaml checks all types statically, you almost never have to write types yourself. OCaml figures them out automatically from how you use things. In C++, you write int x = 5; and must declare the type. In OCaml, you write let x = 5 and OCaml infers that x is an int. This idea was so popular that C++ eventually adopted it with the auto keyword — stolen directly from ML (OCaml's ancestor).

3. Automatic Memory Management (Garbage Collection) — You never call malloc, free, new, or delete. You can allocate objects freely, and the system will reclaim memory automatically when it's no longer needed. This eliminates two of the most common bug categories in C/C++: use-after-free (using memory that's been freed) and memory leaks (forgetting to free memory).

4. Good Support for Higher-Order Functions — Functions that take other functions as arguments, or return functions, are natural and easy in OCaml. This is the bread and butter of functional programming.

If/Else Expressions

In OCaml, there are no "statements" — everything is an expression that produces a value. Even if/else is an expression that returns a value, not a statement that controls flow.

Example: Basic If/Else
(* This is an expression that evaluates to a string *)
# (if 10 < 11 then "a" else "b");;
- : string = "a"

(* Since 10 IS less than 11, the expression evaluates to "a" *)
(* OCaml tells us the TYPE (string) and the VALUE ("a") *)

The parentheses are optional — they don't change the meaning. You can always put parentheses around an expression in OCaml.

Because OCaml is statically type-checked, the then and else branches must have the same type:

Type Error Example
(* This produces an ERROR! *)
# (if 10 < 11 then "a" else 2);;
(* ERROR: "then" branch is string, "else" branch is int *)
(* Even though the else branch would never execute, OCaml rejects this *)

This is an important principle: OCaml is conservative. It rejects programs that might have type errors, even if a particular execution path would never trigger the error. This is the price of static type safety.

Check Your Understanding

What will OCaml do with this expression: if true then 5 else 10?

Correct! Both branches have type int, so the types match. The condition is true, so the then branch is taken and the result is 5. OCaml reports: - : int = 5.
Not quite. In OCaml, if IS an expression (not a statement) that returns a value. Both branches return int, so the types match. The result is 5 with type int.

Tuples

A tuple is a collection of values that can have different types. Think of it as a fixed-size container where each slot can hold a different kind of value.

Example: Creating a Tuple
(* A 3-tuple containing an int, a string, and a bool *)
# (1, "ok", true);;
- : int * string * bool = (1, "ok", true)

(* The * symbol means "cross product" — like mathematical notation *)
(* int * string * bool = "an int AND a string AND a bool" *)

Internally, a tuple is stored as a block of memory with slots for each element, accessed through a pointer. Since OCaml is functional, the contents of a tuple can never change — if you want a tuple with different values, you build a new one.

Lists

A list is a collection of values that must all be the same type (homogeneous). Unlike tuples, lists can be any length.

Example: Lists
(* A list of integers — this works *)
# [1; 19; 3-27];;
- : int list = [1; 19; -24]

(* A mixed list — this DOESN'T work *)
# [1; "a"; true];;
(* ERROR! All elements must be the same type *)
Why lists are homogeneous but tuples are heterogeneous

It comes down to the type system. A list type is written as int list — the type specifies what kind of elements it contains. Since there's only one element type, all elements must match it. But because you only specify one type, the list can have any length.

A tuple type is written as int * string * bool — each position has its own type. This lets you mix types, but the number of elements is fixed by the type.

It's a tradeoff: lists give you flexible length but require uniform types. Tuples give you flexible types but require fixed length.

List Operators

OCaml gives you two operators for building lists:

The cons operator :: — Adds an element to the front of a list:

(* value :: list = new list with value at the front *)
# 1 :: [2; 5];;
- : int list = [1; 2; 5]

(* The left side must be an element, the right side must be a list *)
(* Their types must be compatible *)

The concatenation operator @ — Joins two lists together:

(* list @ list = concatenated list *)
# [1; 7] @ [2; 5];;
- : int list = [1; 7; 2; 5]

(* Both sides must be lists of the same type *)
Check Your Understanding

What is the result of 3 :: [1; 2] @ [4; 5]?

Correct! The @ operator has lower precedence than ::, but actually :: is right-associative and this expression is parsed as 3 :: ([1; 2] @ [4; 5]), which first concatenates to get [1; 2; 4; 5], then prepends 3 to get [3; 1; 2; 4; 5]. The result is the same either way in this case.
The result is actually [3; 1; 2; 4; 5]. The @ concatenation produces [1; 2; 4; 5], and then :: prepends 3 to the front.

Defining Functions

Now we get to the heart of OCaml. Since this is a functional language, functions are the most important concept.

Example: Your First OCaml Function
(* Define a function f that takes x and returns x + 1 *)
# let f x = x + 1;;
val f : int -> int = <fun>

(* Let's break this down: *)
(*   "let" = "I'm defining something"          *)
(*   "f" = the name of the function             *)
(*   "x" = the single argument                  *)
(*   "= x + 1" = the body (what it returns)     *)

(* OCaml responds: *)
(*   "val f" = "f is a value" (functions are values!) *)
(*   "int -> int" = "takes an int, returns an int"    *)
(*   "" = "this is compiled function code"        *)

Notice: we never wrote any types! OCaml saw x + 1, knew that + works on integers, and inferred that x must be an int and the result must be an int. That's type inference in action.

The -> symbol in the type int -> int is a type constructor that means "function from … to …". It's OCaml's way of writing the mathematical arrow ↦. Just like * in int * string means "product of" (a tuple), -> means "function from … to …".

Example: A Function with Different Input/Output Types
(* A function that checks if x is negative *)
# let g x = x < 0;;
val g : int -> bool = <fun>

(* int -> bool = takes an int, returns a boolean *)

You can also bind non-function values with let:

(* This is NOT a function — just a named value *)
# let foo = 37 * 37;;
val foo : int = 1369

Anonymous Functions

Just like you can write an integer without giving it a name (37 * 37 is a perfectly valid expression), you can write a function without giving it a name:

Key Insight: Named Functions Are Syntactic Sugar
(* These two lines are EXACTLY equivalent: *)
# let f x = x + 1;;        (* The shorthand version *)
# let f = fun x -> x + 1;; (* What's really happening *)

(* "fun x -> x + 1" is an anonymous function: *)
(*   "fun" = "here's a function"              *)
(*   "x" = its argument                       *)
(*   "->" = "maps to" (produces)               *)
(*   "x + 1" = the result                     *)
(* Then "let f = ..." gives that function the name f *)

The first form (let f x = x + 1) is syntactic sugar — a sweeter, easier-to-read version — for the second form. Under the hood, OCaml always creates an anonymous function and then binds it to a name.

Calling Functions

In OCaml, every function takes exactly one argument. No more, no less. The syntax for calling a function is the simplest possible:

Function Call Syntax
(* To call function f with argument 12: *)
f 12

(* That's it! No parentheses. No commas. *)
(* Just: function_name  argument *)

(* You CAN add parentheses if you want — same meaning: *)
f(12)     (* works, but not idiomatic OCaml *)
(f)(12)   (* also works, also not idiomatic *)
(f) 12    (* also works *)

The parenthesized versions all work because parenthesizing a subexpression doesn't change its meaning. But idiomatic OCaml skips the parentheses — just write f 12.

Why so simple?

This is a functional programming language. Functions are the most important thing. You call them constantly. So OCaml gives you the absolute simplest syntax possible for the thing you do most often. It's the same design philosophy as C putting * in front of pointers — make the most common operation as easy to write as possible.

Functions with Multiple Arguments

OCaml functions take exactly one argument. So how do you write a function that needs two inputs? There are two approaches:

Approach 1: Use a Tuple

Pack both arguments into a single tuple, and the function takes that one tuple as its single argument:

Example: Multi-Argument via Tuple
(* cons takes a TUPLE of (element, list) as its one argument *)
# let cons(h, t) = h :: t;;
val cons : 'a * 'a list -> 'a list = <fun>

(* Type: 'a * 'a list -> 'a list *)
(*   'a = some type (polymorphic — works with any type) *)
(*   'a * 'a list = a tuple of (element, list)           *)
(*   -> 'a list = returns a list                         *)
(*   Note: * has higher precedence than ->                *)
(*   So it's really: ('a * 'a list) -> 'a list            *)

(* Call it with a tuple: *)
# cons(3, [5; 7]);;
- : int list = [3; 5; 7]

Pattern Matching

Before we look at the second approach to multiple arguments, we need to understand pattern matching — one of OCaml's most powerful features.

Definition: Match Expression

A match expression looks at a value, checks its structure against a series of patterns, and evaluates the expression associated with the first matching pattern.

match expression with
| pattern1 -> result1
| pattern2 -> result2
| pattern3 -> result3

Semantics: evaluate the expression. Try each pattern from top to bottom. When you find the first pattern that matches, bind the pattern's variables to the corresponding parts of the value, then evaluate and return the associated result.

Here's how the cons function actually works under the hood — the shorthand let cons(h, t) = h :: t is syntactic sugar for:

Example: Pattern Matching Under the Hood
let cons = fun x ->
    match x with
    | (h, t) -> h :: t

(* Step by step: *)
(* 1. "fun x ->" creates a function with one argument x *)
(* 2. "match x with" says "look at x's structure" *)
(* 3. "| (h, t)" is a pattern that matches a 2-tuple *)
(*    - h is bound to the first element *)
(*    - t is bound to the second element *)
(* 4. "-> h :: t" is evaluated with those bindings *)

A pattern describes the "shape" of a value: (h, t) matches a pair, [] matches an empty list, x :: xs matches a non-empty list (head x, tail xs).

This is important because OCaml doesn't use array indexing, null checks, or long if/elif/else chains the way imperative languages do. Instead, you use match to decompose data structures, and the compiler can verify that you've handled all possible cases.

Approach 2: Currying (The OCaml Way)

The tuple approach works, but it's not the idiomatic OCaml style. The preferred approach is currying.

Definition: Currying

Currying is a technique where a function that conceptually takes multiple arguments is written as a chain of functions, each taking one argument and returning the next function in the chain.

Example: Curried cons (The "OCaml Way")
(* The idiomatic OCaml version — no parentheses, no commas *)
# let ocons h t = h :: t;;
val ocons : 'a -> 'a list -> 'a list = <fun>

(* Compare the types: *)
(*   Tuple version:   'a * 'a list -> 'a list   (one function, one tuple arg) *)
(*   Curried version: 'a -> 'a list -> 'a list  (chain of one-arg functions) *)

(* The -> arrow is RIGHT-ASSOCIATIVE, so this means: *)
(*   'a -> ('a list -> 'a list) *)
(* "takes an 'a, returns a FUNCTION that takes an 'a list *)
(*  and returns an 'a list" *)

Let's unpack this carefully. Under the hood, the curried version expands to:

(* What "let ocons h t = h :: t" really means: *)
let ocons = fun h -> fun t -> h :: t

(* Step by step: *)
(* 1. "fun h ->" creates a function that takes h         *)
(* 2. It returns ANOTHER function: "fun t -> h :: t"     *)
(* 3. That inner function takes t, and uses BOTH h and t *)
(*    to compute h :: t                                   *)

When you call ocons 3 [5; 7], here's what happens:

Example: Calling a Curried Function
# ocons 3 [5; 7];;

(* Step 1: "ocons 3" *)
(*   Calls ocons with h = 3 *)
(*   Returns a NEW function: fun t -> 3 :: t *)
(*   This new function "remembers" that h is 3 *)

(* Step 2: Apply that returned function to [5; 7] *)
(*   t = [5; 7] *)
(*   Computes 3 :: [5; 7] *)

(* Result: *)
- : int list = [3; 5; 7]
Why currying is preferred over tuples

Currying gives you something tuples don't: partial application. With currying, you can supply just some of the arguments and get back a new, specialized function. For example, ocons 0 gives you a function that prepends 0 to any list. This is extremely useful in functional programming, where you frequently want to create specialized versions of general functions.

With the tuple version, you must always provide all arguments at once — cons(0, some_list). You can't partially apply it.

Check Your Understanding

Given let add x y = x + y, what is the type of add 5?

Correct! add has type int -> int -> int, which means int -> (int -> int). Applying it to 5 gives you a function of type int -> int — specifically, a function that takes a number and adds 5 to it. This is partial application in action!
Not quite. Because of currying, add has type int -> (int -> int). Giving it one argument (add 5) returns a function of type int -> int — one that adds 5 to whatever you give it. This is called partial application.
Final Review

Which of the following best describes what "referential transparency" means?

Correct! Referential transparency means a variable's value is always the same within its scope. You don't need to trace through program execution to figure out what value a variable holds — just look at where it was defined. This is possible because functional languages don't allow you to change variables after they're defined.
Not quite. Referential transparency means that every occurrence of a variable in a scope has the same value, and you can determine that value just by looking at where the variable was defined. This is a direct consequence of variables being immutable in functional languages.

Everything in this lecture connects: hardware constraints (Part 2) drove language design goals (Part 1), which led to different positions in the design space (Part 3). Functional programming (Part 4) is one paradigm in that space, motivated by the goals of clarity and parallelism. OCaml (Part 5) is a specific language that makes functional programming practical — with static type checking, type inference, garbage collection, and first-class support for higher-order functions.

In the coming assignments, you'll use OCaml's functional subset — no assignment statements, no loops. Just functions, recursion, and pattern matching.
