1. The Big Picture: Why Type Parametrization?

Imagine you've written a function that finds the first element of a list of strings. It works great! But now you need the same function for a list of integers. And a list of threads. And a list of anything else you might invent tomorrow.

You could copy-paste your function and change "String" to "Integer" every time. But that's terrible — you'd have dozens of near-identical functions, and a bug fix in one wouldn't automatically fix the others.

Definition
Type parametrization is writing the same code so it works with different types, for reusability. Instead of writing headOfStringList, headOfIntList, etc., you write one piece of code that says "give me a list of some type T, and I'll give you back a T."
Why does this matter?
Type parametrization is the foundation of reusable data structures and algorithms. Without it, every library would need to be rewritten for every type. The collections in Java (List, Map, Set), the containers in C++ (vector, map), and the polymorphic types in OCaml ('a list) all rely on this idea.

There are two major approaches to achieving type parametrization, and they differ in a fundamental way: when the code gets compiled and how much the compiler knows about the type at compile time.

Those two approaches are templates and generics. Let's learn each one from scratch.

2. Templates — The "Recompile Per Type" Approach

Definition
A template is a blueprint for code with an unspecified, fill-in-the-blank type T. The code doesn't actually become real machine code until you fill in what T is. Used in languages like C++ and Ada.

Think of a template like a form letter with a blank: "Dear ______, thank you for your purchase." The letter doesn't exist as a real, sendable letter until you fill in the name. Similarly, a template doesn't exist as executable code until you fill in the type.

How Templates Work, Step by Step

Step 1: You write the template. At this point, no machine code is generated. It's just a blueprint sitting there.

C++ — Writing a Template 
template <typename T>
T head(List<T> lst) {
    // return the first element of the list
    return lst.first();
}

At this point, the compiler has not compiled this function. It doesn't know what T is, so it can't generate actual machine code yet. The template is just sitting in the compiler's memory, waiting.

Step 2: You instantiate the template. Somewhere in your code, you use the template with a specific type:

C++ — Instantiating the Template 
List<String> myList = ...;
String first = head(myList);  // T = String

NOW the compiler goes back to the template, mentally replaces every T with String, and compiles the result. It's as if you had written:

What the Compiler Effectively Generates 
String head(List<String> lst) {
    return lst.first();
}

Step 3: A different instantiation causes a SECOND compilation. If somewhere else you write:

C++ — Another Instantiation 
List<Thread> threadList = ...;
Thread firstThread = head(threadList);  // T = Thread

The compiler goes back to the same template, replaces T with Thread, and compiles it again. You now have two separate copies of the machine code — one for String, one for Thread.

Key Insight
Templates are instantiated per type. Each instantiation can behave differently, generate different machine code, or even fail to compile. A template that works perfectly for String might not compile at all for Thread — and that's allowed!

Why This Matters: Compile-Time Constants

Because the compiler knows the exact type when it compiles a template, it can do powerful optimizations. Consider:

C++ — sizeof Becomes a Constant 
template <typename T>
void process() {
    int size = sizeof(T);  // What value does this have?
}

Before instantiation, sizeof(T) is unknown — T doesn't exist yet. But the moment you say process<String>(), the compiler knows T = String, so sizeof(T) becomes a compile-time constant (say, 32 bytes). The compiler can bake this number directly into the machine code, making the program faster.

If you later call process<int>(), the compiler generates a different version where sizeof(T) is 4. Each version is optimized for its specific type.

3. Generics — The "Compile Once" Approach

Definition
Generics are type-parametrized code that is compiled once and then reused for all types. The code must be written so that it works no matter what type is plugged in. Used in languages like Java, C#, and OCaml.

With generics, the attitude is completely different from templates. Instead of "wait and see what type comes in, then compile," generics say: "compile it right now, and make sure it will work for every possible type."

How Generics Work

When you write generic code in Java, the compiler compiles it immediately and once:

Java — A Generic Method 
public <T> T head(List<T> lst) {
    return lst.get(0);
}

This gets compiled once. When someone calls head(myStringList) or head(myThreadList), they're calling the same compiled code. No recompilation happens.

Analogy — "Every Object Smells the Same"
Your professor used this memorable phrase: in a generic system, every object "smells the same." This means that from the compiler's perspective, it doesn't matter what the type is — every object is just a pointer (an address), typically 8 bytes on a modern machine. A String, a Thread, a Rectangle — they're all just pointers of the same size. The generic code manipulates these opaque pointers without caring what's behind them.

The Catch: No Compile-Time Constants

Because generics are compiled once before knowing the type, the compiler can't do type-specific optimizations:

Generics — sizeof(T) is NOT a Constant 
sizeof(T)   // NOT a constant expression!
              // Because T could be anything, and we compile once.
              // We don't know T at compile time → can't use this.

In a generic system, sizeof(T) depends on what type T ends up being for a particular call, but the code is already compiled. So sizeof(T) cannot be a compile-time constant and can't be used in contexts that require one.

Key Insight
Generic code makes sure objects "smell the same" — the code does not depend on arbitrary characteristics like the size of T or specific behaviors unique to T. Everything must be type-safe in general. This is the tradeoff: you lose per-type optimization but gain the guarantee that once the generic code compiles, it works for every type.

4. Templates vs. Generics: Head-to-Head

FeatureTemplates (C++, Ada)Generics (Java, C#, OCaml)
Compilation Recompiled per type. Each instantiation generates its own machine code. Compiled once. One copy of machine code is reused for all types.
When errors are found At instantiation time. You might not discover a bug in a template until you try to use it with a specific type. At definition time. Once the generic code compiles successfully, you know it's valid for all types.
Performance Can be optimized per type (e.g., sizeof(T) becomes a constant). Potentially faster. Uniform performance. All types go through the same code. "All smell the same."
Code size Larger — one copy of code per type used ("code bloat"). Smaller — one copy of code total.
Complexity for programmer More complex (have to understand that code is compiled twice, error messages can be cryptic). Simpler rules, but type system can still get complex (as we'll see with wildcards).
Check Your Understanding
You write a template in C++ that calls x.sort() on a parameter of type T. You instantiate it with int (which has no .sort() method). When will you discover the error?
Correct! With templates, the code isn't compiled until you instantiate it with a specific type. The compiler only discovers that int has no .sort() method at the moment you try to use the template with int. This is a key difference from generics, where the error would be caught when you define the generic code (the compiler would demand that T supports .sort() up front).
Not quite. With templates, code is only compiled when instantiated. The template itself is just a blueprint — the compiler doesn't try to compile it until you fill in T. So the error appears at instantiation time, when T = int and the compiler discovers int has no .sort().

5. Subtypes Meet Generics: The Invariance Problem

This section connects type parametrization to the concept of subtypes from the previous lecture. You need to understand that in Java, String is a subtype of Object (every String IS an Object). Java uses single inheritance — each class extends exactly one parent. Object is at the very top.

Here's the puzzle: String is a subtype of Object. So intuitively, a List<String> should be a subtype of List<Object>, right? After all, a list of strings is a list of objects…

Wrong. And the reason is subtle but critical. Let's walk through the professor's example to see why.

Step-by-Step: Why List<String> ≠ subtype of List<Object>

Step 1: We create a list of Strings and give it a name.

List<String> ls = ...;   // a list containing Strings

So far so good. ls is a list that's supposed to contain only Strings.

Step 2: Now suppose Java allowed us to assign a List<String> to a List<Object> variable. (Spoiler: it doesn't, but let's pretend.)

List<Object> lo = ls;   // lo and ls point to THE SAME LIST!

This is key: lo and ls are two names for the same list object in memory. They are not copies. Changing one changes the other.

Step 3: Since lo is a List<Object>, its .add() method accepts any Object. Thread is an Object. So we can do this:

lo.add(new Thread());   // ✓ Thread is an Object, lo is List<Object>

But remember — lo and ls are the same list. We just put a Thread into what's supposed to be a list of Strings!

Step 4: Now someone reads from ls, expecting to get a String:

String s = ls.get(0);   // We expect a String...
                           // ...but we get a Thread! 💥 CRASH!

The type system promised that ls.get() returns a String. But the actual object is a Thread. This is a runtime type error — exactly what Java's type system is designed to prevent!

The Verdict: Java catches this at the very first problematic step:

List<Object> lo = ls;   // ✗ COMPILE ERROR!
                           // List<String> is NOT a subtype of List<Object>

The compiler refuses this assignment. By preventing line 2, lines 3 and 4 can never happen. The type system stays sound.

Definition
Invariance: Even if A is a subtype of B, Generic<A> is NOT a subtype of Generic<B>. This guarantees that no invalid elements ever get inserted, no runtime type corruption occurs, and the compiler can enforce strong compile-time safety.
The Deep Reason
Remember from last lecture: if X is a subtype of Y, then you can do a superset of operations on X compared to Y. But List<Object> has an operation — .add(new Thread()) — that is NOT valid on List<String>. Since List<String> can't support all operations of List<Object>, it can't be its subtype.

But this creates a practical problem: how do we write generic code that works on lists of any type?

6. Wildcards — The ? Escape Hatch

Because of invariance, this innocent-looking function doesn't work the way you'd expect:

Java — Broken: Too Restrictive 
void printAll(Collection<Object> c) {
    for (Object o : c)
        System.out.println(o);
}

This function seems like it should accept a collection of anything — after all, every object in Java is an Object. But because of invariance, you cannot pass a Collection<String> or Collection<Thread> to this function. You can only pass a Collection<Object> exactly. That's useless!

Definition
A wildcard ? is a generic, unknown type that tells the compiler "I don't care exactly what the type is." It promises the compiler that ? is one fixed element type that is unknown at compile time, but can be read safely.

The fix is to use ? instead of Object:

Java — Fixed with Wildcard 
void printAll(Collection<?> c) {
    for (Object o : c)     // ✓ Every element is at least an Object
        System.out.println(o); // ✓ println accepts any Object
}

Now you can pass Collection<String>, Collection<Thread>, Collection<anything> to this function. The ? says "this collection holds some type, I don't know which, and I don't care."

The Tradeoff: Read-Only Access

Here's the crucial restriction: with a wildcard ?, you can read from the collection (you'll always get at least an Object), but you cannot add elements to it (except null). Why? Because the compiler doesn't know what type the collection holds, so it can't verify that your addition is safe.

Java — Wildcards Prevent the Original Problem 
List<String> ls = ...;
List<?> lo = ls;          // ✓ This is allowed now!

lo.add(new Thread());     // ✗ COMPILE ERROR!
                            // Can't add to List<?> — type unknown

The wildcard makes the reference more restrictive (can't write), which means it can't break type safety. The original problem from Section 5 is impossible now.

Rule of Thumb
Use wildcards when you want to consume (read) values from a collection, but not produce (write) them.
Check Your Understanding
Given Collection<?> c, which of the following is allowed?
Correct! With Collection<?>, you can read from the collection — every element is guaranteed to be at least an Object, so iterating and printing works. But you cannot add non-null elements because the compiler doesn't know what type the collection actually holds. (Adding null alone is technically allowed, but adding 42 after is not.)
Not quite. With Collection<?>, the compiler doesn't know the actual element type, so it won't let you add anything (except null). But reading is fine — every element is at least an Object, so iterating and printing is the correct answer.

7. Bounded Wildcards & Bounded Generics

Plain wildcards still aren't enough for everything. Consider:

Java — Still Broken with Plain Wildcard 
void printShapes(Collection<Shape> c) {
    for (Shape s : c)
        printShape(s);
}

// We CANNOT pass Collection<Rectangle> here!
// Because Collection<Rectangle> is not a subtype of Collection<Shape>
Collection<Rectangle> cr = ...;
printShapes(cr);  // ✗ COMPILE ERROR

If we change Shape to ?, we'd accept anything — but then we can't treat elements as Shape inside the loop (they'd just be Object). We need something in between.

Definition
A bounded wildcard restricts the wildcard ? to be a subtype of a specified upper bound. The syntax ? extends Shape means "some unknown type that is Shape or any subtype of Shape (like Rectangle, Ellipse, Square)."
Java — Fixed with Bounded Wildcard 
void printShapes(Collection<? extends Shape> c) {
    for (Shape s : c)    // ✓ We know ? is at least a Shape
        printShape(s);    // ✓ So we can treat each element as Shape
}

Collection<Rectangle> cr = ...;
printShapes(cr);  // ✓ Rectangle extends Shape, so this works!

Named Type Parameters: When ? Isn't Enough

What if you need two parameters to match the same type? A wildcard can't do that — each ? is independent.

Java — BAD: Two Independent Wildcards 
void convert(?[] a, Collection<?> c) {
    for (int i = 0; i < a.length; i++)
        c.add(a[i]);  // ✗ The two ?'s could be different types!
}

The first ? might be String and the second might be Integer. The compiler can't guarantee safety. The fix is to give the type a name:

Java — GOOD: Named Type Parameter 
<T> void convert(T[] a, Collection<T> c) {
    for (int i = 0; i < a.length; i++)
        c.add(a[i]);  // ✓ T is the SAME type in both places
}

T is stricter than ?: all occurrences of T must be the same type. The wildcard ? is more flexible — each ? could represent a different type.

The Full Solution: Bounded Generics with super

But even the named T has a limitation. What if we want to convert a Rectangle[] into a Collection<Shape>? With <T> void convert(T[] a, Collection<T> c), T would have to be both Rectangle and Shape — that's impossible.

The solution uses the super keyword — the opposite of extends:

Java — Bounded Generics with super 
<T> void convert(T[] a, Collection<? super T> c) {
    for (int i = 0; i < a.length; i++)
        c.add(a[i]);  // ✓ The collection accepts T or any supertype of T
}

// Now this works:
Rectangle[] rects = ...;
Collection<Shape> shapes = ...;
convert(rects, shapes);  // ✓ T = Rectangle, Shape is a supertype of Rectangle

? super T means "some type that is T or a supertype of T." Since Shape is a supertype of Rectangle, the collection can safely accept Rectangle objects.

? is Like Underscore in OCaml
The professor made an important connection: ? in Java is like _ in OCaml. Both mean "there's a value/type here, but I'm not going to name it because I'll never reference it elsewhere." You could always replace ? with a named type parameter T, but ? signals to the reader "this type doesn't matter for understanding the code." Only give names to things when you might need them later.
Check Your Understanding
You have <T> void process(Collection<? extends T> src, Collection<T> dest). Which call is valid?
Correct! With T = Shape, the source is Collection<? extends Shape> which accepts Collection<Rectangle> (since Rectangle extends Shape), and the destination is Collection<Shape>. Reading Rectangles as Shapes to put into a Shape collection is safe.
Think about the direction: ? extends T in the source means you're reading subtypes of T. You need to read from the source (which could contain subtypes) and write to the destination (which holds T). The first option works with T = Shape.

8. Duck Typing — The Opposite Philosophy

Everything we've learned about generics, wildcards, and bounded types represents one end of a spectrum: maximum compile-time safety. Duck typing sits at the opposite end: maximum flexibility. Understanding both extremes helps you see the tradeoffs every language designer faces.
Definition
Behavioral polymorphism is based on what methods an object supports, rather than what an object is declared to be. Some languages, like Python, take this approach — prioritizing flexibility over safety, relying on runtime behaviors.
Definition
Duck typing is typically dynamic type checking that doesn't concern itself with object types — it only cares about behavior. "If it walks like a duck, quacks like a duck, and swims like a duck — it's a duck."

In duck typing, you don't ask "is this object a Duck?" Instead, you call quack(), waddle(), swim(), fly(), lay_eggs() — and if all of those work, it's a duck. If they don't, you get a runtime error.

This means you could have a Mouse class that you've given wings and a quack method to, and it would pass the duck test. In Java, this would be impossible unless Mouse formally implemented a Duck interface.

FeatureDuck Typing (Python)Generics (Java)
Type checkingRuntime — errors found lateCompile-time — errors found early
Based onBehavior (what methods exist)Types (formal declarations)
FlexibilityVery flexibleLess flexible, more constrained
SafetyLess safe — crashes at runtimeSafer — caught at compile time
PerformanceSlower (runtime checks)Faster (no runtime type dispatch)
Large codebasesNeeds lots of testing to compensateType system provides safety guarantees
The Tradeoff
All that complexity of bounded wildcards and bounded generics exists to buy compile-time type safety — the guarantee that an entire class of errors (wrong type) is impossible. Duck typing gives up that guarantee in exchange for simplicity and flexibility. Neither is universally "better" — it depends on whether you value reliability or simplicity more for your particular project.

9. Names (Identifiers)

We're shifting gears from types to names. But they're deeply connected: types exist to describe what operations are valid on named things. Names are how we refer to values, functions, types, and everything else in a program.
Definition
A name (or identifier) is a symbol you write in code to refer to things — variables, functions, types, modules, labels, etc.

Every language has rules about names: which characters are allowed, whether names are case-sensitive, and which names are reserved words (keywords like if, while, return that you can't use as variable names).

Why Do These Rules Exist?
These rules exist for compilers and parsers. The compiler needs to unambiguously recognize the structure of your source code. If while could be both a keyword and a variable name, the parser wouldn't know if while(x) is a loop or a function call.

A Security Concern: Confusables

With Unicode, some characters look identical on screen but are different bytes. The classic example: the Cyrillic letter "о" looks exactly like the Latin letter "o." An attacker could write a variable name that looks normal but is actually a different identifier, hiding malicious code in a publicly available library. This is a real security concern with open-source code — the code looks right but does something unexpected.

10. Bindings — What Gets Bound, and When

Definition
A binding is an association between a name and a value. Programming languages fundamentally concern what gets bound and when.

Consider this seemingly simple line of C code:

C 
int i = 19;

You might think: "The name i is bound to the value 19. Done." But there's much more going on. The name i is actually bound to multiple things at once:

What Does int i = 19; Actually Bind?

Binding 1: Name → Value

The obvious one: i is bound to the value 19. When you write i in an expression, you get 19.

i          // → 19 (the value)

But this binding is mutable — some other code could change i to 20 later. This value exists only at runtime.

Binding 2: Name → Address

The variable i lives somewhere in memory. It has an address — a specific location where its bytes are stored.

&i         // → 0x7F9C1300 (the memory address)

This is why i = 19 is NOT the same as just writing 19 everywhere. The name i refers to a location that contains 19. You can store new values at that location.

Binding 3: Name → Size

The variable i takes up a certain amount of memory.

sizeof(i)  // → 4 (bytes, on most modern systems)

This is independent of the value and the address — it's about how many bytes the machine uses to represent an int.

Binding 4: Name → Type

The name i is bound to the type int.

typeof(i)  // → int

This tells the compiler what operations are valid on i (addition, subtraction, etc.), how to interpret its bytes, and whether the code is type-correct. In C/C++, the type has no runtime representation — it exists only in the compiler's understanding.

So a single declaration like int i = 19; creates at least four different bindings: to a value, an address, a size, and a type. Understanding what gets bound is essential for understanding any language.

11. Binding Time — When Associations Are Created

Definition
Binding time is when the association between a name and a value gets created (or destroyed). Different elements of a binding are created at different times.

Let's trace when each binding of int i = 19; is created:

ElementWhen is it bound?Explanation
Value (i = 19) Runtime / execution The actual value stored in i exists only when the program is running. Some other function might change it to 20 later.
Address (&i) Link time (static) or block entry time (local) For a global/static variable, the linker decides where it lives in memory. For a local variable inside a function, the address is determined when the block {} is entered.
Size (sizeof) ABI definition time Not decided by the compiler or language spec, but by the ABI — a contract defined by hardware/OS engineers. For x86-64, they decided int = 4 bytes back in the 1980s.
Type (typeof) Compile time The compiler decides what type i has, what operations are valid on it, and whether the code is type-correct.
Definition
The Application Binary Interface (ABI) is a contract that defines how compiled code interacts with machine code and other compiled code. It specifies the size of primitive types, memory layout, alignment rules, stack layout, calling conventions, and more. The ABI ensures that code compiled by different compilers on the same platform can link and work together.
ABI vs. Compiler vs. Language
Think of it this way: the language spec says "there exists a type called int." The ABI says "on this platform, an int is 4 bytes, stored in two's complement, aligned to 4-byte boundaries." The compiler says "this particular variable i is an int." Each operates at a different level and a different time.
Check Your Understanding
You have a local variable int x = 42; inside a function. The function is called 1000 times. How many times is x bound to a memory address?
Correct! Since x is a local (automatic) variable, it lives on the stack. Each time the function is called, a new stack frame is created, and x gets a (potentially different) address. The address binding happens at block entry time, which occurs with every function call. The type binding (int) happens once at compile time, and the size binding (4 bytes) was determined even earlier by the ABI.
Think about where local variables live — on the stack. Each function call creates a new stack frame at a potentially different position. So the address of a local variable is determined each time the block is entered — 1000 calls means 1000 address bindings.

12. Explicit vs. Implicit / Static vs. Dynamic Binding

Explicit vs. Implicit Binding

Definition
Explicit binding: The programmer writes down the binding directly. There is nothing left for the system to infer. Example: int i = 19 — the programmer specifies the type, name, and value.
Definition
Implicit binding: The programmer does NOT write down the binding; the language infers it from context. Example: In FORTRAN, variable names starting with I–N are automatically integers; names starting with A–H or O–Z are automatically floats.

Modern implicit binding is much more sophisticated: OCaml and languages with type inference use implicit binding where the compiler deduces the type from how the variable is used. This is still statically typed and safe — unlike the old FORTRAN approach, which was error-prone because the compiler didn't enforce type checking.

Static vs. Dynamic Binding

Definition
Dynamic binding: You cannot determine the binding until the program is running. The binding depends on a runtime state. Example: the value of i is unknown until it gets its value during execution.
Definition
Static binding: The binding is known before the program runs — at compile time, link time, or load time. Example: the type of i in a statically-typed language is known by the compiler before execution.

13. Namespaces — Organizing Bindings

Names and bindings bring up a natural question: what happens when two things have the same name? Namespaces are the answer — they're the system that prevents (or allows) name collisions.
Definition
A namespace is a context in which names must be unique. Two names can be identical as long as they live in different namespaces.

You already use namespaces intuitively. Your contact list might have two friends named "Alex" — one from work, one from school. "Work" and "School" are namespaces that let you have the same name without confusion.

14. Primitive Namespaces (The C/C++ Jungle)

Definition
Primitive namespaces: A language has multiple, separate name tables that represent different things. The same spelling f can refer to many different entities, as long as they all live in different namespaces.

C and C++ have a surprisingly large number of primitive namespaces. Your professor demonstrated this by writing a single function where the identifier f is used in many different namespaces simultaneously without any collisions. Let's walk through each one:

How Many Namespaces Can f Live In?

Namespace 1: Function names

int f() {   // f is a function name

The name f here identifies the function. This lives in the namespace for function/variable identifiers at file scope.

Namespace 2: Struct tags

    struct f { int f; };  // f is a struct tag

The f after struct is a struct tag. Struct tags live in their own namespace, completely separate from function names. No collision!

Namespace 3: Struct members

    struct f { int f; };  // f is a member of struct f

The f inside the struct is a member name. Each struct has its own private namespace for members. So this f doesn't collide with the struct tag f or the function f.

Namespace 4: Local variables

    struct f f;  // f is a local variable of type struct f

Here f is a local variable whose type is struct f. Local variables live in yet another namespace from struct tags and members.

Namespace 5: Enum tags

    enum f { zero };  // f is an enum tag

Enum tags have their own namespace (separate from struct tags). But note: enum values (like zero) share a namespace with local variables — so we can't call the value f.

Namespace 6: Members of a different struct

    struct g { int f; };  // f is a member of struct g

Even though there's already a struct f with a member f, this struct g has its own private member namespace. So another member called f is fine.

Namespace 7: Preprocessor headers (file names)

    #include <f>  // f is a filename / preprocessor header

This f refers to a file in the filesystem. File names are in a completely different "namespace" — the filesystem.

Namespace 8: Labels

    goto f;
    f: ;  // f is a label (for goto)

Labels have their own namespace. Note: labels in C are scoped to the entire function, not just the enclosing block. This was designed to allow loop-and-a-half patterns with goto.

Namespace 9: Preprocessor macros

    #define f g  // f is a macro name

Macro names live in their own namespace. Warning: the preprocessor runs before the compiler. If this #define appeared at the top of the function, every subsequent f would be replaced with g before the compiler even sees the code!

Important: Macro Pitfall
#define macros are processed by the preprocessor, which runs before compilation. The preprocessor doesn't know about blocks, functions, or scoping — it just does text substitution. So #define f g inside a function still affects all subsequent code in the file, not just the function.
Definition
Macro names (#define f g): Tells the preprocessor "every time you see f (after this line), replace it with g." Macros have their own namespace and are processed before compilation.

15. Declaration vs. Definition

Definition
A declaration is a promise that tells the compiler: "this name exists somewhere, and here's its type/signature." It does NOT allocate memory or provide implementation. The keyword extern in C means "this will be defined elsewhere — don't allocate space here."
Definition
A definition is the actual thing — it allocates memory for variables and provides the implementation code for functions.
C — Declaration vs. Definition 
// DECLARATIONS (promises — no memory allocated, no code generated)
extern int xi;          // "there is an int called xi somewhere"
extern int f(float);    // "there is a function f(float)→int somewhere"

// DEFINITIONS (the real thing — memory allocated, code generated)
int xi = 10;            // actually allocate memory and store 10
int f(float x) {        // actually define what f does
    return x + xi;
}

Why Separate Them? Separate Compilation

Definition
Separate compilation: Languages like C use the distinction between declaration and definition to compile programs piece by piece. The linker connects declarations with definitions.

In practice, declarations go in a header file (foo.h) and definitions go in an implementation file (foo.c):

foo.h — The Interface (Declarations Only) 
extern int xi;
int f(float);
foo.c — The Implementation (Definitions) 
int xi = 10;
int f(float x) { return x + xi; }
main.c — Uses the Interface 
#include "foo.h"
int main() {
    return f(2.0);  // Knows about f from the declaration
}
Three Reasons for Separating Declaration from Definition
1. Fast builds: If you change foo.c, only foo.c needs to be recompiled. main.c doesn't change.

2. Abstraction / information hiding: Users see what a function does (from the header), but not how (the implementation). This lets you change the implementation without affecting users.

3. Modularity: You can replace one implementation with another (e.g., swap database backends) as long as it conforms to the same interface. Prevents collisions during collaboration.

16. Labeled Namespaces & Access Control

Definition
A labeled namespace is a named container that holds a group of declarations. Unlike primitive namespaces, it lets the language control who is allowed to see what inside it (through classes, packages, modules, etc.).

In Java, this is done with access modifier keywords:

Keyword Same package (not subclass) Subclass (different package) Everyone else
public✓✓✓
protected✓✓✗
(default) — no keyword✓✗✗
private✗✗✗
Definitions
public: Visible to anyone, anywhere. Used for external interfaces/APIs.

protected: Visible to subclasses (through inheritance) and classes in the same package. Controls extension.

(default / package-private): Visible only within the same package. No keyword needed. Common for internal helper functions.

private: Visible only within the declaring class. The strongest form of hiding.

Packages group related classes together, define who can see what, and draw boundaries between different parts of the system:

Java — Package Structure Example 
// The world sees graphics.Shape and graphics.Rectangle
// The world CANNOT touch Renderer or Utils

graphics/
├── Shape.java        (public)
├── Rectangle.java    (public)
├── Renderer.java     (package-private — internal)
└── Utils.java        (package-private — internal)

17. Explicit Namespace Creation (OCaml Modules)

Definition
Explicit namespace creation: The programmer manually creates and controls namespaces, including their contents and visibility. Namespaces are not implicit (tied to a class or file) but constructed deliberately. This allows separating interface from implementation with fine-grained control.

OCaml provides module for creating namespaces and module type for defining interfaces (what's visible to the outside world):

OCaml — Defining an Interface (Module Type) 
module type A =
sig
  (* This is the interface — only these are visible outside *)
  val f : int -> int     (* any module of type A must have this function *)
end
OCaml — Creating a Namespace (Module) 
module N : A = struct
  (* Implementation — hidden details *)
  let x = 3            (* internal, NOT in interface A *)
  let f y = x + y      (* visible, because interface A requires it *)
end
OCaml — Accessing the Module from Outside 
N.f 4    (* ✓ Valid — f is defined in the interface A *)
         (* Returns 7 (because x=3, f y = 3 + 4) *)

N.x      (* ✗ Error! x is NOT visible — not in the interface *)

This approach gives the programmer complete control: the module type defines exactly what's visible (the interface), and the module provides the implementation. Anything not listed in the interface is hidden, even though it exists internally.

Connecting It All Together
The progression across this lecture has been: we want to write reusable code (type parametrization) → but we need to keep it type-safe (generics, wildcards, bounded types) → to organize all these names and types, we need namespaces → and to build reliable systems from pieces, we need declarations vs. definitions and access control. Each concept builds on the last to help us write large, safe, modular programs.
Final Comprehension Check
In OCaml, module N is defined with type A. Inside N, there's let secret = 42 which is NOT listed in A. What happens when someone writes N.secret from outside the module?
Correct! The module type A acts as a filter — only names listed in the interface are visible to the outside world. Since secret isn't in A, the compiler rejects any attempt to access N.secret at compile time. This is the power of explicit namespace creation: fine-grained control over what's hidden and what's exposed.
Not quite. When a module is given a type signature (like N : A), only the names listed in that signature are accessible from outside. secret isn't in A, so the compiler produces a compile error. The value exists internally but is invisible externally.
