Programming Languages — Lecture 14

Scheme Core vs Extensions

How the simplest useful programming language builds powerful features from a tiny core — and why that matters.

1. The Big Picture

Before we dive into any code, let's understand why we're studying Scheme in a programming languages course.

Core Philosophy: Scheme tries to be the simplest possible programming language that's still useful. It strips away everything that isn't needed, leaving a tiny core that can be extended into a full language.

Think of it like this: back in the 1930s, Alan Turing designed the Turing machine — the simplest possible computer — so he could prove theorems about what computers can and can't do. But actually programming a Turing machine is a nightmare. It's worse than writing raw machine code.

Scheme tries to solve a harder problem: strip a programming language down to its minimum without being a disaster to use. The designers wanted a language that's theoretically clean and practically useful.

Analogy: Imagine building a house. You could build it from atoms (like a Turing machine), but that's impractical. You could use a kit with 10,000 specialized parts (like C++ or Java). Scheme is like having a small box of Lego bricks — very few types of pieces, but you can build anything if you're clever about combining them.

Core vs. Extensions

This is the central theme of today's lecture. Scheme has two layers:

Core
A tiny set of built-in primitives that can't be defined in terms of anything simpler: lambda, if, cons, car, cdr, and a few others.
Extensions
Everything else — and, or, let, list — is syntactic sugar that can be defined using the core. We'll prove this today.

For example, Scheme's + function can take any number of arguments: (+ a b c). The designers could have limited it to two arguments and made you write (+ a (+ b c)). But they decided that was too painful — there are already enough parentheses in Scheme. So they made + with many arguments an extension of a simpler two-argument core +.

Throughout this lecture, we'll see things we thought were fundamental to the language that are actually just convenient extensions built on top of something simpler. That's a powerful idea — it means the language is more understandable and more flexible than it first appears.

2. Building with Pairs & Lists

Before you can do anything interesting in Scheme, you need to understand how data is built. The fundamental building block is the pair (also called a "cons cell").

cons — Building a Pair

The function cons takes two values and glues them together into a pair:

; cons takes two values and makes a pair
(cons 1 2)   ; → (1 . 2)  — a pair with 1 on the left, 2 on the right

You can visualize a pair as a box with two slots:

car and cdr — Taking a Pair Apart

If cons builds a pair, then car and cdr take it apart:

; car returns the left (first) element of a pair
(car (cons 1 2))   ; → 1

; cdr returns the right (second) element of a pair
(cdr (cons 1 2))   ; → 2

The names car and cdr are historical — they come from the original hardware registers on the IBM 704 computer in the 1950s. They've stuck around ever since. Think of car as "Contents of Address Register" (first element) and cdr as "Contents of Decrement Register" (rest). But most people just memorize: car = first, cdr = rest.

Lists Are Chains of Pairs

A list in Scheme is just a chain of pairs where each pair's cdr points to the next pair, and the last pair's cdr is the empty list '():

; The list (1 2 3) is really:
(cons 1 (cons 2 (cons 3 '())))

list — A Convenient Shorthand

Writing all those nested cons calls is tedious. So Scheme provides the list function:

; list takes any number of arguments and builds a list
(list 1 2 3)      ; → (1 2 3)
(list "hi" 42)    ; → ("hi" 42)
(list)             ; → ()  — the empty list

Here's the beautiful thing: list is not part of the core language. It's an extension! If it weren't built in, you could define it yourself in one line:

; list is just "package up all my arguments as a list"
(define (list . args) args)

This works because the . args syntax collects all arguments into a list (we'll cover that in the variadic functions section). So list just takes its arguments — already packaged as a list — and returns them. That's it!

Check Your Understanding

What does (car (cdr (list 10 20 30))) return?

A 10
B 20
C 30
D (20 30)
Correct! (list 10 20 30) gives (10 20 30). Then cdr gives (20 30) — the list without its first element. Then car gives 20 — the first element of that.
Not quite. Work through it step by step: (list 10 20 30) → (10 20 30). Then (cdr ...) → (20 30). Then (car ...) → 20. The answer is B.

3. Quoting vs. list

This is a subtle but important distinction. Both quoting and list can produce lists, but they behave differently.

Quoting: "Don't Evaluate, Just Give Me the Data"

When you put a ' (quote) in front of something, you're telling Scheme: "Don't try to run this as code. Just treat it as raw data."

; Quoting: treat everything literally as symbols/numbers
'(a b 19)   ; → (a b 19)
            ; a and b are SYMBOLS, not variables
            ; They are NOT evaluated

list: "Evaluate First, Then Package"

The list function evaluates its arguments first, then puts the results into a list:

; Suppose a = 27 and b = (x y z)
(list a b 19)   ; → (27 (x y z) 19)
               ; a and b are VARIABLES — they get evaluated!
Key Difference

Quoting prevents evaluation — symbols stay as symbols. list evaluates its arguments first — variables get looked up and replaced with their values.

Numbers, booleans, and strings are self-evaluating: quoting 19 and just writing 19 give you the same result. The Scheme interpreter sees 19 and says "that's just 19" — no error, no confusion.

Quasiquoting: The Best of Both Worlds

What if you want mostly literal data, but you want to evaluate a few pieces? That's what quasiquoting does. You use a backtick ` instead of a regular quote, and then use , (comma) to "unquote" — to say "evaluate this part":

; Quasiquote with , to evaluate specific parts
; Suppose a = 27, b = (x y z)
`(,a ,b 19)   ; → (27 (x y z) 19)
              ; Same result as (list a b 19)!

So (list a b 19) and `(,a ,b 19) mean the same thing. The quasiquote version reads like a template: "give me a list that looks like this, but fill in a and b with their current values."

Check Your Understanding

If x is bound to the value 5, what does '(x 10) return?

A (5 10)
B Error — x is not defined
C (x 10)
D '(5 10)
Correct! The quote ' prevents evaluation. So x stays as the symbol x, not the value 5. To get (5 10) you'd need (list x 10) or `(,x 10).
Remember: the quote ' means "don't evaluate anything." So x is treated as a literal symbol, not a variable. The answer is C: (x 10).

4. Four Kinds of Equality

In English, we say two things are "equal" and don't think twice. But in programming, "equal" can mean very different things. Scheme makes this explicit by giving you four different equality tests. Understanding which one to use is critical.

Function What It Compares Speed When to Use
eq? Object identity — are A and B literally the same object in memory? (pointer comparison) O(1) Checking if two variables point to the exact same thing
eqv? Value comparison — do A and B have the same value? (one level deep) O(n) Comparing strings, large integers — checks contents, not address
equal? Recursive/structural — walks through lists and sublists comparing everything O(n) Comparing complex data structures
= Numeric — are A and B the same number? (regardless of type) O(n) Comparing numbers across types (int, float, etc.)

Detailed Walkthrough

eq? — Same Object?

Think of eq? as asking: "Do these two things occupy the same memory address?" If you're familiar with C, it's like comparing pointers. It's blazingly fast — just one machine instruction — but it's very strict.

; Two variables pointing to the SAME list object
(define x '(1 2 3))
(define y x)       ; y points to the same object as x
(eq? x y)          ; → #t  (they're the same object)

; Two DIFFERENT list objects with the same contents
(define a '(1 2 3))
(define b '(1 2 3))
(eq? a b)          ; → #f  (different objects, same contents)

eqv? — Same Value?

eqv? goes one step further: it actually looks at the value being pointed to. For strings, it compares the characters. For large integers, it compares the digits. But it only looks one level deep — it won't recurse into nested lists.

(eqv? "hello" "hello")   ; → #t  (same string content)
(eq?  "hello" "hello")   ; → #f (could be: different string objects!)

equal? — Same Structure?

equal? is the most thorough: it walks through entire data structures recursively. Two lists are equal? if all their elements are eqv?, including nested lists.

(equal? '(1 (2 3)) '(1 (2 3)))   ; → #t  (same structure and values)
(eqv?   '(1 (2 3)) '(1 (2 3)))   ; → #f  (different objects, doesn't recurse)

= — Same Number?

= only works on numbers, but it crosses type boundaries:

(= 0 0.0)     ; → #t  (numerically equal, different types!)
(eqv? 0 0.0) ; → #f  (integer ≠ float as objects)
Tricky Edge Case — NaN

Can you find a value where (eq? a b) is #t but (= a b) is #f?

Answer: NaN (Not a Number). If a holds NaN, then (eq? a a) is #t (it's the same object), but (= a a) is #f because by the IEEE floating-point standard, NaN is not equal to itself numerically. This is a famous quirk of floating-point arithmetic.

Check Your Understanding

Which equality function should you use to check if two separately-constructed lists (1 2 3) have the same contents?

A eq?
B equal?
C eqv?
D =
Correct! equal? recursively compares data structures. eq? would return #f because they're different objects. eqv? doesn't recurse into lists. = only works on numbers.
Think about what each function does. eq? checks identity (same object), not contents. eqv? doesn't recurse. = is numbers only. The answer is B: equal? — it recursively walks the structure.

5. Type Predicates

In Scheme, for every type, there's a built-in function that tests whether a value is of that type. These functions are called predicates, and by convention they all end with ?:

(null? '())      ; → #t  (it's the empty list)
(null? '(1 2))   ; → #f  (not empty)

(pair? '(1 2))   ; → #t  (it's a cons pair)
(pair? 42)       ; → #f  (not a pair)

(integer? 42)    ; → #t
(integer? 3.14)  ; → #f

(string? "hi")   ; → #t
(boolean? #t)    ; → #t

An important detail: in Scheme, there is exactly one empty list. Every '() in your program refers to the same object. That's why (eq? '() '()) returns #t — they're literally the same object. Many Scheme implementations represent the empty list as the bit pattern of all zeros internally.

6. define and set!

At the top level, define binds a name to a value:

(define pi 3.14159)
; From here on, pi = 3.14159

(define greeting "hello")
; greeting = "hello"

The right-hand side is evaluated first (it can be any complex expression), then the result is stored in the name. This is not a macro — it actually runs the code and saves the result.

set! — Destructive Assignment (Use with Caution)

You can change a variable's value later with set!:

(define x 10)
(set! x 20)   ; x is now 20
Warning: Using set! means you've left the world of functional programming and entered imperative programming. There's a strong school of thought in Scheme that says this is a bad idea — your programs become harder to reason about when variables can change. In fact, the Racket dialect of Scheme doesn't allow set! on variables defined with define by default!

7. Lambda Expressions

A lambda creates a nameless function (also called an "anonymous function" or a "closure"). This is one of the most important concepts in Scheme — and in programming languages in general.

; A nameless function that takes argument f and calls (f 27)
(lambda (f) (f 27))

By itself, this function just floats in space with no name. To use it, you can either call it immediately or bind it to a name:

; Give it a name using define
(define d (lambda (f) (f 27)))

; Now call it: pass the + function as the argument
(d +)   ; → 27  because (+ 27) = 27
(d -)   ; → -27 because (- 27) = -27

Syntactic Sugar: The Short Form

Defining a function and immediately naming it is so common that Scheme has a shorthand:

; Long form:
(define d (lambda (f) (f 27)))

; Short form (syntactic sugar — means exactly the same thing):
(define (d f) (f 27))

The short form puts the function name and parameters in the same parentheses. This pattern should look familiar if you've used OCaml's let d f = f 27 — same idea, different syntax.

Core vs Extension: lambda is part of the core language — it's a special form, not a regular function. The shorthand (define (name args) body) is syntactic sugar (an extension) that expands to (define name (lambda (args) body)).

8. Variadic Functions

A variadic function accepts a varying number of arguments. You've already seen one: + can take any number of arguments. Here's how to define your own:

At Least One Argument (dot notation)

; format is required; args collects everything else as a list
(define (printf format . args) ...)

; When you call:
(printf "%s %d %x" "abc" 29 372)
; format → "%s %d %x"
; args   → ("abc" 29 372)  ← a list of the remaining arguments

The . (dot) before args is the key. Everything before the dot is a required parameter. The identifier after the dot collects all remaining arguments into a list.

Zero or More Arguments

What if you want a function that takes zero or more arguments? You can't write (lambda (. args) ...) because (. isn't valid syntax. Instead, drop the parentheses entirely:

; Takes zero or more arguments — all collected as a list
(define noargs (lambda args ...))
; Shorthand:
(define (noargs . args) ...)

The list Function Defined

Now we can see how list works — it's the simplest possible variadic function:

; list takes zero or more args and returns them as a list
(define (list . args) args)

; When you call (list 1 2 3):
; args = (1 2 3)
; return args → (1 2 3). That's it!
Check Your Understanding

Given (define (f x . rest) rest), what does (f 1 2 3 4) return?

A 1
B (1 2 3 4)
C (2 3 4)
D Error
Correct! x gets bound to 1 (the first argument). rest collects all remaining arguments: (2 3 4). The function returns rest, so the answer is (2 3 4).
x captures the first argument (1), and rest collects the remaining arguments as a list: (2 3 4). The function returns rest, so the answer is C: (2 3 4).

9. Special Forms: if, begin, and, or

Now we hit something crucial. Some things in Scheme look like function calls, but they are not. They're called special forms, and they have special evaluation rules that functions can't replicate.

if — Three-Way Conditional

(if condition then-expr else-expr)

; If condition is anything other than #f → evaluate then-expr
; If condition is #f → evaluate else-expr
; Exactly ONE of the two branches runs. Never both.

Why if Can't Be a Function

This is a critical insight. Scheme evaluates function arguments eagerly (call-by-value) — all arguments are evaluated before the function runs. But if must not evaluate both branches! Consider:

(if (= x 0) 1 (/ 10 x))
; If x is 0, returns 1 — safe!
; The division (/ 10 x) is NEVER evaluated

If if were a regular function, it would evaluate (/ 10 0) before deciding which branch to take — and crash with a division-by-zero error. This is why if must be a special form in the core language.

Definitions using if
(define (null? x) (eq? x '()))
; Simpler: doesn't even need if!

(define (not x) (if x #f #t))
; Also possible: (eq? x #f)

(define (abs x) (if (< x 0) (- x) x))
; This one truly needs if — can't simplify further

begin — Sequential Evaluation

(begin expr1 expr2 ... exprN)
; Evaluate left to right, return the value of exprN

and — Short-Circuit Conjunction

(and expr1 expr2 ... exprN)
; Evaluate left to right
; If ANY expression returns #f → immediately return #f
; If ALL are truthy → return the value of the LAST one

or — Short-Circuit Disjunction

(or expr1 expr2 ... exprN)
; Evaluate left to right
; The FIRST truthy value → return it immediately
; If ALL are #f → return #f

This is more powerful than C/C++'s ||, which only returns true or false. Scheme's or returns the actual value, which enables a useful pattern for defaults:

(or (getenv "PATH") "/bin:/usr/bin")
; If PATH is set → returns its value
; If PATH is #f (not set) → returns the default string
Check Your Understanding

What does (and 5 "hello" #f 42) return?

A #f
B 42
C "hello"
D 5
Correct! and evaluates left to right. 5 is truthy, "hello" is truthy, #f is false → immediately return #f. The 42 is never even evaluated.
and evaluates left to right: 5 (truthy, continue), "hello" (truthy, continue), #f (false! stop here). It immediately returns #f. The 42 is never reached.

10. Recursion & Tail Calls

Scheme doesn't have traditional loops like for or while. Instead, it uses recursion to repeat operations. This means getting recursion right — especially making it efficient — is absolutely critical.

First Attempt: Naive Reverse

(define (rev ls)
  (if (null? ls)
      '()
      (append (rev (cdr ls))
              (list (car ls)))))

This works, but it has two problems:

Problem 1
Not tail recursive. The last operation is append, not rev. So each recursive call adds a frame to the stack. A list of length N builds up N stack frames → can blow the stack.
Problem 2
O(n²) time. Each append is O(n) because it must walk the entire first list. We call append n times, so the total work is 1 + 2 + ... + n = O(n²).

The Fix: Accumulator Pattern

The solution is the same one we used in OCaml and Prolog: define a helper function that's more general than reverse. Instead of just reversing, it reverses and appends:

; revapp reverses L and prepends the result to A
; (revapp '(1 2 3) '(4 5)) → (3 2 1 4 5)

The trick is that cons is O(1), so moving elements one-at-a-time from L onto A is efficient. Each step takes constant time, we do it N times → O(n) total. And it's tail recursive because the recursive call is the last thing we do.

Scheme guarantees tail-call optimization. This is part of the language specification. If a recursive call is in tail position, the implementation must reuse the current stack frame. So tail-recursive code uses O(1) stack space, just like a loop.

11. Named let

We want our helper function to be private — invisible to the rest of the program. Scheme provides named let for exactly this: defining a local recursive function and immediately calling it.

(define (rev ls)
  (let revapp ((l ls) (a '()))
    (if (null? l)
        a
        (revapp (cdr l) (cons (car l) a)))))

Let's break this down piece by piece:

Anatomy of Named let

(let revapp ((l ls) (a '())) body...)

revapp — the name of the internal recursive function.
(l ls) — parameter l, initially bound to ls (the input list).
(a '()) — parameter a (accumulator), initially the empty list.
body — the code that runs, and can call revapp recursively.

The textual order matches the execution order: you first see the initial values (ls and '()), then you see what happens with them. This is nicer than the OCaml equivalent, where you'd define the helper function first and call it last — writing the code in the opposite order of execution.

Compare: OCaml equivalent ▸ 
let rev ls =
  let rec revapp l a =
    match l with
    | [] -> a
    | h::t -> revapp t (h::a)
  in revapp ls []

Notice how in OCaml you first define revapp (lines 2-5), and then call it (line 6). You write what executes last first. Named let in Scheme avoids this by putting the initial call first (the parameter bindings) and the recursive body after.

Key Properties of This Solution

O(n) time — cons, car, and cdr are all O(1). We do them n times.

Tail recursive — when revapp calls itself, that's the very last thing it does. Scheme will optimize this into a loop.

Private — revapp is invisible outside rev. Nobody else can call it.

Check Your Understanding

Named let is assumed to be recursive by default (no rec keyword needed). Why?

A Because Scheme doesn't support non-recursive functions
B Because it would be a syntax error without it
C Because recursion is so common that requiring 'rec' would be unnecessary boilerplate
D Because named let always uses letrec internally
Correct! The named let pattern is almost always used for recursion. Making it recursive by default saves typing and follows Scheme's philosophy of convenience. If the function doesn't actually call itself, no harm done — the code still works.
Scheme does support non-recursive functions (plain let). The real reason is that named let is so commonly used for recursion that requiring rec would be pointless boilerplate. The answer is C.

12. Macros with define-syntax

This is where Scheme's "core vs extensions" philosophy gets really powerful. Using define-syntax, you can invent new keywords — literally extending the language. It's like macros in C, but much more powerful because they can be recursive and are hygienic (we'll explain that shortly).

Binding Constructs in Scheme

Identifiers in Scheme can play two roles: they can be variables (bound to values via lambda, let, define) or keywords for special forms (bound via define-syntax, let-syntax). The keyword-binding forms let you extend the language with new syntax.

Example: Defining and

and is built into Scheme, but it doesn't have to be. Here's how you'd define it from scratch:

(define-syntax and
  (syntax-rules ()
    ((and)            #t)              ; zero args → true
    ((and x)          x)               ; one arg → just return it
    ((and x1 x2 ...)                    ; two+ args →
     (if x1 (and x2 ...) #f)))        ;   if x1 then (and rest) else #f

Think of syntax-rules as pattern matching (like OCaml's match). The compiler tries each pattern top-to-bottom, and the first match wins. The ... is a special identifier meaning "zero or more trailing things."

Expansion Example

; Source code:
(and (= x 0) "foo")

; Step 1: matches (and x1 x2 ...) with x1=(= x 0), x2...="foo"
→ (if (= x 0) (and "foo") #f)

; Step 2: inner (and "foo") matches (and x) pattern
→ (if (= x 0) "foo" #f)

The compiler recursively expands macros until only core forms remain. The final code uses only if — a core primitive.

Example: Defining or (The Tricky One)

You might think or is just as simple as and, but there's a subtle efficiency issue.

Naive Attempt (Bad)

(define-syntax or
  (syntax-rules ()
    ((or) #f)
    ((or x) x)
    ((or x1 x2 ...)
     (if x1 x1 (or x2 ...))))  ; ← Problem! x1 appears TWICE

Why is this bad? Consider (or (getenv "PATH") "/bin"). This expands to:

(if (getenv "PATH") (getenv "PATH") "/bin")
; getenv is called TWICE! Wasteful and possibly wrong
; (what if it has side effects?)

Correct Version (Save to a Variable)

(define-syntax or
  (syntax-rules ()
    ((or) #f)
    ((or x) x)
    ((or x1 x2 ...)
     (let ((v x1))               ; evaluate x1 ONCE, save in v
       (if v v (or x2 ...)))))  ; reuse v — no double evaluation

Now (or (getenv "PATH") "/bin") expands to:

(let ((v (getenv "PATH")))
  (if v v "/bin"))
; getenv called exactly once. Correct and efficient.

But this creates a new problem: what if the caller already has a variable named v? That's the problem of capture, which leads us to hygienic macros...

13. Macros: let in Terms of Lambda

Here's one of the most satisfying demonstrations of the core-vs-extensions idea: let is just syntactic sugar for a lambda call.

Ordinary let

; What you write:
(let ((x (* a a))
      (y (* b b)))
  (- x (* 2 y)))

; What it really means (macro-expanded):
((lambda (x y) (- x (* 2 y)))
 (* a a)
 (* b b))

The let creates variables and binds values to them. Under the hood, it becomes a lambda (the function) applied to the values (the arguments). The lambda's parameters are the variable names; the arguments are the initial values.

Why let is Nicer

In the lambda version, the evaluation order is: first evaluate (* a a) and (* b b) (the arguments), then execute the body. But textually, the body appears before the arguments.

With let, you read top-to-bottom in evaluation order: first the bindings, then the body. Textual order = execution order. Much easier to read.

Named let → letrec

Named let is similarly just sugar for letrec (which allows recursive bindings):

; Named let:
(let revapp ((l ls) (a '()))
  (if (null? l) a
      (revapp (cdr l) (cons (car l) a))))

; Expands to:
((letrec ((revapp (lambda (l a)
           (if (null? l) a
               (revapp (cdr l) (cons (car l) a))))))
   revapp)
 ls '())

The letrec defines revapp as a recursive function, then the expression returns that function, which is immediately called with the initial arguments ls and '().

14. Hygienic Macros

Scheme solves a fundamental problem that plagues C/C++ macros: variable capture.

The Problem

Remember our or macro introduces a local variable v. What if the caller also has a variable named v?

(let ((u (getenv "USER"))
      (v (getenv "VERBOSE")))
  (or u v))

; Naive macro expansion of (or u v):
(let ((v u))        ; OOPS! This v shadows the caller's v!
  (if v v v))       ; All three v's refer to the SAME variable
                    ; We've lost access to (getenv "VERBOSE")!

In C/C++, this exact problem happens constantly with macros — it's called name collision or capture. C programmers work around it with ugly naming conventions (like __MACRO_INTERNAL_v), but that's fragile.

Scheme's Solution: Hygienic Macros

Scheme determines the scope of all variables before macro expansion happens. When the macro introduces a v, Scheme knows it's a different v from the caller's v. Conceptually, it renames them:

; Scheme internally distinguishes the two v's:
(let ((v₁ u))       ; v₁ = macro's internal variable
  (if v₁ v₁ v₂))   ; v₂ = caller's variable (getenv "VERBOSE")
                    ; Correct! No confusion.

This is called hygienic macro expansion — "hygienic" meaning "clean." The macro can't accidentally mess up the caller's variables, and vice versa. It works because Scheme resolves scope before expanding macros, not after.

Why this matters: Hygienic macros make it safe to extend the language with new syntax. You can write complex macros without worrying about name clashes. This is a significant advantage over C/C++ preprocessor macros, which do purely textual substitution and have no understanding of scope.

15. Continuations

Now we're entering the most mind-bending part of the lecture. Continuations are Scheme's way of exposing something that exists in every programming language's runtime — but that most languages hide from you.

Definition: A continuation is a representation of what a program will do in the future — its "bucket list" of remaining work. It captures the entire future of the computation from a given point.

What the Interpreter Needs to Know

At any point during execution, the Scheme interpreter needs exactly two things to keep running:

IP
Instruction Pointer — what code to execute next. Points to the next machine instruction or bytecode.
EP
Environment Pointer — the context in which that code runs. Points to the activation record with local variables, temporaries, and a pointer to the caller's environment.

Together, the IP/EP pair tells the interpreter everything it needs: what to do and what context to do it in.

Activation Records (Stack Frames)

Each function call creates an activation record containing:

Inside an Activation Record

Local variables — the values of x, y, etc. in this function call

Temporaries — unnamed intermediate values the compiler needs

Caller's EP — pointer to the calling function's activation record

Return address — the IP to jump back to when this function returns

These records form a chain: each one points to its caller's record, all the way back to main. This entire chain — all the IP/EP pairs — represents the continuation: everything the program still plans to do.

Analogy: Imagine you're reading a book, and you use bookmarks. Your current bookmark is where you are now (IP). The stack of notes you've written about context for each chapter is your environment (EP). If someone asked "what will you do next?", the answer is: finish this page, then continue from each bookmark in order. That chain of bookmarks is your continuation.

Scheme vs. C/C++

In C/C++, a function is represented by just an instruction pointer (a function pointer). But in Scheme and OCaml, a function (closure) needs two pointers: the code (IP) and the environment in which it was created (EP). This is because closures can capture variables from their enclosing scope.

Also unlike C/C++, Scheme's activation records may live on the heap (not just the stack) and may need garbage collection. This is essential for continuations to work correctly, since a continuation might outlive the function call that created it.

16. call/cc

Now for the big reveal: how Scheme actually lets you use continuations.

(call-with-current-continuation P)
(call/cc P)   ; shorter alias — same thing

Here's a fun fact about how controversial this feature is: people who love continuations use the short name call/cc. People who think continuations are dangerous and too low-level use the long name call-with-current-continuation — partly because it's annoying to type, which discourages casual use!

What call/cc Does (Step by Step)

Step 1
Create a continuation k that captures the future of the computation from this point. It's just an IP/EP pair — very cheap to create.
Step 2
Call (P k) — invoke the function P, passing the continuation as its argument. Return whatever P returns.

Creating a continuation is cheap — it's just storing two pointers. No copying of the entire stack or anything expensive.

Using a Continuation

The power comes from what you can do with k. If you call (k v) from anywhere:

; Calling a continuation:
(k v)
; 1. Immediately abandons whatever you're currently doing
; 2. Restores BOTH the IP and EP from when k was created
; 3. The call/cc expression "returns" with value v

This is like a super-powered goto. A regular goto in C only changes the instruction pointer. Calling a continuation changes both the instruction pointer and the environment pointer — it teleports you back to a completely different execution context and makes it look like call/cc just returned with the value v.

Key Insight

A continuation is not a copy of the future — it's a pointer to the future. If parts of the computation modify variables between when k was created and when it's invoked, those changes will be visible. The continuation says "resume from here," but "here" might have changed.

Why continuations matter: They're a building block for implementing exceptions, coroutines, backtracking, cooperative multitasking, and more. Every language has continuations internally — Scheme is unique in making them accessible to programmers. This gives you power that Java, C++, and Python don't offer directly, but it also gives you enough rope to hang yourself — which is why they're controversial.

Recap: The Core Language

Here's what we've established as Scheme's core — the primitives that can't be reduced further:

Core PrimitiveWhy It Must Be Core
lambdaCreates functions — the fundamental building block
ifConditional — can't be a function (must skip one branch)
cons, car, cdrBuild and access pairs — the basic data structure
defineTop-level bindings
call/ccContinuation access — can't be written in Scheme itself

Everything else — and, or, let, named let, list, not, null? — can be defined in terms of these core primitives using define-syntax. That's the beauty of Scheme's design.
