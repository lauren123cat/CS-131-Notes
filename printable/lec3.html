Part I

Currying — One Argument at a Time

Why Does Currying Exist?

Before we dive into pattern matching and all the fun stuff, we need to understand a foundational idea in OCaml (and in functional programming generally): currying.

Here is the core principle: in OCaml, every function takes exactly one argument. That's it. Not two, not three — just one. This might sound restrictive, but it's actually a deliberate design choice rooted in mathematics.

The idea is named after Haskell Curry, a logician working in the 1940s. He was writing academic papers full of functions, and every time a function had multiple arguments, he needed subscripts for them. Typesetting subscripts was painful and error-prone. So he had a clever realization: if every function only takes one argument, you never need subscripts at all. And if you need a function that "seems" to take two arguments? You just make the first function return another function that takes the second argument.

Key Concept — Currying

A curried function is a function that appears to take multiple arguments, but actually takes them one at a time. It takes the first argument and returns a new function. That new function takes the second argument and produces the result. You lose no power — it's just a different way to organize things.

Example: The cons Function

Let's see this in action. In the previous lecture, we saw the cons function, which prepends an element to the front of a list:

let cons x y = x :: y

This looks like cons takes two arguments (x and y), but here's what's really happening under the hood:

1

You call cons with a value, say "a". This doesn't produce a list yet — it produces a new function that is waiting for a list to prepend "a" onto.

2

You then pass that returned function a list, like ["bc"; "def"]. Now it builds the result: ["a"; "bc"; "def"].

The type OCaml infers for cons tells the whole story:

'a -> 'a list -> 'a list

Read this as: "give me a value of any type 'a, and I will give you back a function. That function takes a list of 'a and returns a list of 'a." The implied parentheses are:

'a -> ('a list -> 'a list)

The arrow -> is right-associative, so these parentheses are invisible — but they're always there. This is what makes it a curried function: it's a chain of single-argument functions.

Why This Matters Later

Currying isn't just a curiosity. Later in this lecture, you'll see that it lets us do something powerful: partially apply a function to create a new, specialized function. For example, we'll define rev (list reverse) simply by calling apprev [] — giving apprev its first argument and getting back a perfectly good reverse function. No extra code needed.

This is currying in action: we got back a function, not a final result.
Part II

Pattern Matching — Taking Data Apart

Now we arrive at one of OCaml's most powerful features: pattern matching. Think of a pattern as a template that you slide over a piece of data to see if it fits. If it fits, any named holes in the template get filled in with the corresponding pieces of the data.

Pattern matching is how you inspect and decompose data structures in OCaml. It replaces the need for accessor functions like "get the first element" or "get the length" and makes your code safer and more readable in the process.

The car Problem — Why Pattern Matching Matters

In Lisp-family languages, there's a classic function called car that returns the first element (the "head") of a list. Let's try to write it in OCaml:

let car (x::y) = x

This looks reasonable: we have a list whose head is x and whose tail is y, and we return x. OCaml accepts this code — but it gives us a warning.

Compiler Warning — Partial Match

OCaml warns us that our pattern doesn't cover all possible inputs. The pattern x::y matches any list of length 1 or more, but it does not match the empty list []. If someone calls car [], the program crashes at runtime with a "match failure."

This is a partial function — it only works on some inputs. And in OCaml, partial functions are considered dangerous. The whole philosophy of OCaml is: if your code compiles without warnings, it should not crash at runtime.

What's Actually Happening? Syntactic Sugar

That shorthand let car (x::y) = x is actually syntactic sugar (a convenient shortcut) for a longer form using a match expression:

let car = fun l -> match l with
  | x::y -> x

Reading this: "car is a function that takes a list l, and then tries to match l against the pattern x::y. If it matches (the list is non-empty), bind x to the head and y to the tail, then return x."

The problem is clear in this expanded form: there's only one pattern branch. If l is [], no branch matches, and the program dies.

Match Expressions — The Full Story

A match expression is the core mechanism. Its general form is:

match expression with
  | pattern₁ -> result₁
  | pattern₂ -> result₂
  | pattern₃ -> result₃

OCaml evaluates the expression, then tries each pattern from top to bottom. The first pattern that matches "wins," and the corresponding result is evaluated and returned. If no pattern matches, you get a runtime crash.

The first | (vertical bar) at the top is optional — it's just there to make the patterns line up nicely. Many OCaml programmers include it for readability.

Fixing car — but with a Trade-off

We can eliminate the warning by handling the empty list:

let car = fun l -> match l with
  | x::y -> x
  | [] -> 0

No more warning! But there's a problem: this function's type is now:

int list -> int

It's no longer generic — it only works on integer lists! Why? Because we wrote 0 on the right-hand side. OCaml sees that 0 is an int, and since all branches of a match must return the same type, it infers that x must also be an int. Therefore the list must be an int list.

Type Inference in Action

OCaml doesn't require you to write type annotations. It infers all types automatically. But it's strict: every branch of a match must produce the same type. If even one branch introduces a concrete type like int, the whole function becomes specialized. The original car (without the [] branch) had type 'a list -> 'a — polymorphic, working on any list type.

The Pattern Catalog

Let's now systematically learn every kind of pattern you can write. Think of these as the building blocks — they can all be nested inside each other recursively.

1. Constants

A constant pattern matches only itself. Examples: 0, 42, [], true, "hello".

match x with
  | 0 -> "zero"
  | 1 -> "one"
  | _ -> "something else"

2. Variables

A variable like x matches anything — but as a side-effect, it binds (assigns) the matched value to that name. Think of it as a local variable that exists only inside that match branch.

match my_list with
  | x::y -> x      (* x = head, y = tail *)
  | x -> 0          (* x = the whole list — a different x! *)
Scope Matters

Each branch of a match expression is its own scope. The x in the first branch and the x in the second branch are completely different variables. They just happen to share a name. Each is local to its own branch.

3. Underscore (Wildcard)

The underscore _ matches anything, just like a variable — but it discards the value. It's OCaml's way of saying "I don't care what this is, and I'm not going to use it."

let car = fun l -> match l with
  | x::_ -> x     (* we don't need the tail *)
  | _ -> 0         (* we don't need the empty list itself *)
Style Rule

It is considered good style in OCaml to use _ whenever you don't plan to use a variable. This convention originated in ML and has since spread to many other languages (Python, Rust, Swift, etc.).

4. List Cons Pattern: P1 :: P2

This matches any non-empty list where the head matches P1 and the tail matches P2. Since :: is right-associative, you can chain them:

x :: y :: z     (* means: x :: (y :: z) *)

This matches a list of length ≥ 2, where x is the first element, y is the second element, and z is the rest of the list (which could be empty).

5. List Literal Pattern: [P1; P2; P3]

This matches a list of exactly that many elements. For example, [h] matches a list of exactly length 1 (a singleton). This is rarely used because most list processing is recursive — you usually only care about "empty vs. non-empty."

6. Tuple Pattern: (P1, P2, P3)

Matches a tuple of exactly that many elements. Unlike lists, tuples are heterogeneous (elements can have different types), so P1 could match an int, P2 a string, etc. But you must always match all elements — there's no way to match "the first element of a tuple" without specifying placeholders for the rest.

match my_triple with
  | (a, _, _) -> a     (* only want the first element *)

You must know the tuple's size (the number of commas) to write a pattern for it.

What is NOT a Pattern

You cannot write P1 P2 as a pattern (two patterns side by side, like a function call). Patterns only match data — they don't match programs or function applications. You can only use patterns to take data apart, never to put things together or analyze code.

The function Keyword Shorthand

If your function takes one argument and immediately matches it, there's a shorter way to write it. Instead of:

let f = fun l -> match l with
  | ... -> ...

You can write:

let f = function
  | ... -> ...

The function keyword means: "I'm a function, and all I do is pattern-match on my argument." The argument doesn't even need a name.

Don't Use hd or tl — Use Pattern Matching

OCaml's standard library has functions List.hd (head) and List.tl (tail) that extract the first element or the rest of a list. Don't use them.

These are "selector functions" — they work on non-empty lists but crash on empty ones. That crash is a runtime error, exactly the kind of thing OCaml is designed to prevent.

Instead, use pattern matching. When you match a list against h::t and [], the compiler guarantees you've handled all cases. If you accidentally miss a case, the compiler warns you at compile time — long before your program ever runs.

The Big Idea

One of the core reasons we use OCaml in this course is to write programs where if it compiles without warnings, it won't crash. Pattern matching is the primary tool that makes this possible. Every match expression with exhaustive patterns is a promise: "no matter what input you give me, I have an answer."

Part III

Recursion in OCaml

Why You Need let rec

In most languages (C, Python, Java), you can call a function from within its own body without any special syntax — that's just recursion. But in OCaml, you have to explicitly tell the compiler that a function is recursive by writing let rec instead of just let.

Why? Because OCaml has a strict philosophical rule: you should never define something in terms of itself, because that usually leads to nonsense. Consider this in C:

Cint x = x;          // undefined behavior — using uninitialized variable
int x = x + 1;      // even worse — no possible value works

Self-referential definitions are philosophically dangerous. But recursive functions are the one safe exception: when you write let rec f = ..., you're not using f's value right now. You're just noting that later, when f gets called, it should be able to call itself. The actual execution is delayed, which is what makes it safe.

The rec keyword is your way of telling the compiler: "I know this is self-referential, and I'm a competent programmer who understands recursion. Trust me."

Example: Finding the Last Element

Let's build a function that finds the last element of an integer list, returning -1 if the list is empty (a cheat, but fine for now):

let rec lastint = function
  | []    -> -1
  | [h]   -> h
  | _::t  -> lastint t

Let's walk through each branch:

1

[] — the empty list. No last element exists, so we return -1.

2

[h] — a singleton list (length exactly 1). The only element is the last element, so return h.

3

_::t — a list of length ≥ 1. But we already caught length-1 lists in branch 2! So this branch only fires for lists of length ≥ 2. We discard the head (don't need it), and recurse on the tail t. Since t has length ≥ 1, the recursion makes progress toward the base case.

Notice a subtle and clever thing about the third branch: the pattern _::t would normally match any non-empty list (including singletons). But because OCaml tries patterns top to bottom, singletons are already caught by [h]. So by the time we reach _::t, the list must have length ≥ 2, which means t has length ≥ 1.

Expand: The desugared (long) version

Remember, function is shorthand. The long version is:

let rec lastint = fun l -> match l with
  | []    -> -1
  | [h]   -> h
  | _::t  -> lastint t

They mean exactly the same thing.

Reversing a List — The Naive (Slow) Way

Here's a classic exercise: reverse a list. A first attempt:

let rec rev l = match l with
  | []    -> []
  | h::t  -> (rev t) @ h

The professor initially wrote this and OCaml reported a surprising type:

val rev : 'a list list -> 'a list

That's not what we wanted! We wanted 'a list -> 'a list, not something that takes a list of lists. What's the bug?

Debugging Without Running — The Power of Types

This was a pivotal moment. The professor realized: I can debug my code without running it, just by looking at the type. If the type is wrong, the function is wrong — guaranteed. This is a massive advantage over languages like C++, where you might only discover the bug after running a specific test case.

The bug: @ (the append operator) requires both arguments to be lists. Writing (rev t) @ h means h must be a list. Since h is the head of the input list, the input list must be a list of lists — hence the type 'a list list -> 'a list.

The fix: wrap h in a singleton list:

let rec rev l = match l with
  | []    -> []
  | h::t  -> (rev t) @ [h]

Now the type is 'a list -> 'a list. Correct! But this function is O(n²). Why? The @ operator must copy the entire left-hand list to build the result. Since rev t returns a list of length roughly n-1, and we do this n times, the total work is roughly n + (n-1) + (n-2) + ... + 1 = n²/2.

Reversing a List — The Smart (Fast) Way

The key insight: solve a harder problem more efficiently, then use it to solve the original.

Instead of just reversing a list, define a more general function apprev that reverses a list and appends something to the end:

(* apprev a l  =  (reverse of l) appended with a *)
let rec apprev a = function
  | []    -> a
  | h::t  -> apprev (h::a) t

Let's trace through apprev [] [3; 5; 9]:

1

apprev [] [3; 5; 9] → list is non-empty, h=3, t=[5;9] → call apprev [3] [5; 9]

2

apprev [3] [5; 9] → h=5, t=[9] → call apprev [5; 3] [9]

3

apprev [5; 3] [9] → h=9, t=[] → call apprev [9; 5; 3] []

4

apprev [9; 5; 3] [] → list is empty → return [9; 5; 3] ✓

Each step does O(1) work (just a :: cons), and there are n steps. Total: O(n). Now we define rev using currying:

let rev = apprev []

This is a beautiful example of partial application: we call apprev with its first argument ([]) and get back a function — that function is our rev. Its type is 'a list -> 'a list, exactly what we want.

val apprev : 'a list -> 'a list -> 'a list
val rev    : 'a list -> 'a list

Closures — Why Local Storage Doesn't Disappear

When we write let rev = apprev [], the function apprev is called with a = [] and it returns. In C or C++, when a function returns, its local variables are destroyed — the stack frame is popped.

But here, the function returned… and yet later, when someone calls rev [3; 5; 9], the code inside apprev runs and needs to know that a is []. How does it remember?

Closures

In OCaml (and any language that supports currying / first-class functions), a returned function can "capture" variables from its enclosing scope. These captured variables persist even after the outer function returns. This combination of a function plus its captured environment is called a closure.

The practical upshot: in OCaml, local storage does not automatically go away when a function returns. If a returned function might still need those locals, they stick around. The garbage collector will eventually clean them up — but only once nobody needs them anymore.

This is fundamentally different from C/C++, where the rule is simple: function returns → stack pops → locals are gone. OCaml's rule is more nuanced: the "stack" is really a more complex data structure that keeps values alive as long as they're needed. This is slightly slower than a raw stack, but it enables the powerful higher-order functional programming we just used.

Part IV

Syntax — Form vs. Meaning

Syntax vs. Semantics

When we talk about a programming language (or a natural language), we divide the discussion into two pieces:

Syntax (Form)Semantics (Meaning)
What does the sentence/code look like? What's its structure?What does the sentence/code mean? What does it do?
The "easy" part (mechanically checkable)The "hard" part (requires understanding intent)

This distinction seems clean, but in practice it gets fuzzy. Let's see three famous examples that illustrate the boundary.

Example 1: Perfect Syntax, Bogus Meaning

Noam Chomsky's famous 1950s sentence: "Colorless green ideas sleep furiously." The grammar is perfect — adjective, adjective, noun, verb, adverb — all in the right places. But the meaning is nonsensical: ideas have no color, they don't sleep, and sleeping can't be furious. This is form without meaning.

Example 2: Questionable Syntax, Clear Meaning

Consider: "Ireland has leprechauns galore." Every English speaker understands this — Ireland is overrun with leprechauns. But what is "galore" doing there? It's in the position of an adverb, yet it's an adjective. And you can't say "Ireland has galore leprechauns" — it must come after the noun. (We borrowed it from Irish, and brought a bit of Irish grammar along.) This is meaning without clean form.

Example 3: Ambiguity in Both

"Time flies." This could mean "time passes quickly" (noun + verb) or "go measure how fast the flies move" (imperative verb + noun/direct object). Both interpretations are syntactically and semantically valid. This is ambiguity — and it's the thing programming languages must avoid at all costs.

Ambiguity is the Enemy

In natural language, ambiguity can be a feature — diplomats and poets exploit it deliberately. In programming languages, it's a disaster. You never want your program to mean different things on different machines. A major goal of language syntax design is to eliminate ambiguity.

Goals When Designing Syntax

Programming languages are designed (unlike natural languages, which evolve organically). When designing the syntax, the language creator balances several competing goals:

1. Familiarity

Use syntax people already know. Most people write 3 + 4 * 5 (infix notation). PostScript uses postfix: 3 4 5 * +. Lisp uses prefix: (+ 3 (* 4 5)). PostScript and Lisp fans argue their notation is superior — no precedence rules needed! — but most people find infix familiar and comfortable.

2. Simplicity

PostScript and Lisp are arguably simpler: you never have to memorize that * binds tighter than +. The cost is unfamiliarity. Simplicity and familiarity often compete.

3. Intuitiveness

A practical tip from the legendary programmer Val Shaw (credited with inventing goto-free programming): never use >. Always write comparisons so that smaller values appear on the left: 0 <= i and i < n, not i >= 0. This way, the textual order on the page mirrors the number line — making code easier to read intuitively.

4. Unambiguity

A program should have exactly one interpretation. Period.

5. Redundancy (Error Detection)

Redundancy seems wasteful, but it catches mistakes. In infix notation, if you forget a closing parenthesis, the compiler catches it. In PostScript's postfix notation, almost any sequence of symbols is a valid program — it's very hard to make a syntactically invalid program, but very easy to make a wrong one. Lisp, with parentheses everywhere, catches even more errors. Some redundancy is a feature, not a bug.

6. Writeability & Readability

You want code to be fast to write and easy to read. These goals often conflict. APL is famously writeable — you can implement Conway's Game of Life in one line of symbols — but nearly impossible to read six months later. Python's indentation rules make code more readable at the cost of some writeability constraints.

7. Extensibility

Can the syntax evolve? The original Bourne shell author (Steve Bourne) loved Algol 68's syntax so much that he used C preprocessor macros to redefine C's syntax: #define IF if(, #define THEN ){, etc. His successors hated it and eventually rewrote the shell from scratch (giving us Bash). Extensibility is powerful but dangerous.

Part V

Tokenization — The First Step of Compilation

We've been writing programs and thinking about their meaning. But how does a compiler see your code? It starts at the very bottom: your source code is nothing but a sequence of bytes in a file. The first job is to make sense of those bytes. That process is called tokenization (also called lexical analysis).

What Are Tokens?

A token is a meaningful unit — the smallest chunk of your program that carries semantic weight. Think of tokens as the "words" of a programming language.

Consider this C program:

Cint main() { return !getchar(); }

The tokenizer converts this byte string into a stream of tokens:

Token TypeExtra InfoSource Text
INT (keyword)—int
ID (identifier)"main"main
LPAREN—(
RPAREN—)
LBRACE—{
RETURN (keyword)—return
BANG (operator)—!
ID (identifier)"getchar"getchar
LPAREN—(
RPAREN—)
SEMICOLON—;
RBRACE—}

Notice what's discarded: whitespace, newlines, and the #include directive's content (which gets expanded into many more tokens first). What's kept: the token type, plus any extra information needed. For identifiers, we need the name. For integer constants, we need the value.

Tokens vs. Lexemes

A token is the abstract category: "this is an identifier" or "this is an integer constant." A lexeme is the token plus its extra information: "this is the identifier spelled main" or "this is the integer constant with value 42." Some languages also track source position (line number, column) in the lexeme for error reporting.

Greedy Tokenization

Tokenizers are designed to be fast. They read the source left to right and are greedy: at each point, they grab as many characters as possible that form a valid token. Once the next character can't extend the current token, the token ends.

This greedy rule can cause surprises. Consider this C expression:

Creturn a++ + ++b;

That works — the tokenizer sees a, then ++, then +, then ++, then b. But what about:

Creturn a+++++b;   /* no spaces! */

The greedy tokenizer sees a, then greedily grabs +++… wait, +++ isn't a valid token. So it backs up: ++ is valid. Token: a, ++. Then it sees +++b. It grabs ++. Token: ++. Then +b: token +, then b. Result: a ++ ++ + b. But you can't apply ++ to the result of ++ (it needs an lvalue, not a value). Compilation error.

The fix? Add a space: a++ + ++b. Whitespace is "just" a token separator, but sometimes it's essential.

Comments — More Trouble Than You'd Think

Comments are generally treated like whitespace — they separate tokens but carry no meaning to the compiler. But they have their own pitfalls.

Nesting

In C, comments /* ... */ do not nest. If you try to comment out a block of code that already contains a comment, the first */ inside ends the outer comment, and the rest becomes a syntax error. Supporting nested comments makes the tokenizer more complex (it needs a depth counter), but some languages do it.

Trigraphs — A Cautionary Tale

The professor told a story about a colleague who wrote an expressive comment containing a sequence that the C compiler secretly interpreted as a trigraph. Trigraphs were a feature (removed in C23) where sequences like ??/ meant backslash (\). Danish keyboards mapped certain ASCII characters to Danish letters, so Danish programmers couldn't type \ or # directly — trigraphs were the workaround.

In C, a backslash before a newline merges the two lines. So a comment that accidentally contained ??/ at the end of a line would cause the next line to be swallowed into the comment — silently removing code. The lesson: be very careful about comment syntax and edge cases in any language you use.

Reserved Words vs. Keywords

When the tokenizer sees the letters i, f, it needs to decide: is this the start of an identifier (like iffy) or is it the keyword if? It looks at the next character. If it's a letter, digit, or underscore, it's part of an identifier. Otherwise, it's the keyword if.

Reserved Words vs. Keywords

A reserved word is a keyword that cannot be used as an identifier. In C, if is reserved — you cannot have a variable named if. This makes the tokenizer's job simple but can cause trouble when extending the language. (If C++ adds a new keyword like class, old C code using class as a variable name breaks.)

A keyword (non-reserved) is recognized as special in certain syntactic positions, but can still be used as an identifier elsewhere. In PL/I, you can write IF IF(3) = IF(5) THEN ... where the first IF starts a conditional and IF(3) accesses an array named IF. This is more flexible but makes parsing harder.

Parse Trees — The Next Level Up

Once we have a stream of tokens, the next step is to organize them into a parse tree (also called a syntax tree). A parse tree reveals the structure of the program: which tokens combine with which, and in what order.

For example, the expression x + 1 becomes a tree where + is the root node, with x and 1 as its children. A full program is a tree of declarations (function definitions, variable definitions, etc.).

Parse tree for x + 1. Leaves are tokens; the internal node is the operator.

Parsing happens after tokenization. The tokenizer converts bytes into tokens; the parser converts tokens into a tree. We'll explore parsing in detail in the next lecture.

Putting It All Together

This lecture covered a lot of ground. Here's how everything connects:

Currying is the foundation of how functions work in OCaml — every function takes one argument, and multi-argument functions are chains of single-argument functions. This isn't just an academic curiosity: it enables partial application, which we used to elegantly define rev from apprev.

Pattern matching is how you safely decompose data in OCaml. It replaces accessor functions (hd, tl) with a mechanism that the compiler can verify: if all your patterns cover all cases, you'll never get a runtime match failure. This is the key to OCaml's promise of "if it compiles, it works."

Types are inferred automatically and serve as a powerful debugging tool. The professor's story about rev having type 'a list list -> 'a list instead of 'a list -> 'a list shows how wrong types instantly reveal wrong code — without running a single test.

Recursion with let rec is the standard way to process lists. The naive O(n²) reverse vs. the clever O(n) apprev version demonstrates a general technique: solve a harder, more general problem efficiently, then specialize it.

Finally, syntax and tokenization are the bridge from raw text to structured programs. Tokenizers are greedy and simple; their design choices (reserved words, comment nesting, whitespace handling) have real consequences for how you write and debug code.
