From stack frames and nested functions to heap allocation and garbage collection

Chapter 1

Closures: Functions That Remember

How nested functions access their surrounding environment — and how to build this yourself

Before we can talk about memory management on the heap, we need to finish a crucial idea about memory on the stack: what happens when a function needs to access variables that belong to another function — one that defined it. This idea, called a closure, shows up everywhere in modern programming and directly motivates why we eventually need heaps.

What Are Nested Functions?

In many languages (like OCaml, Scheme, Python, JavaScript), you can define a function inside another function. The inner function can refer to variables from the outer function. Here's the concept in pseudocode:

function f() {
    int a = 10;
    int b = 20;

    function g() {
        // g is NESTED inside f
        // g can refer to 'a' — a variable from f
        return a + 1;
    }

    return g();  // calls g, which uses f's variable 'a'
}

The function g is nested inside f. When g runs, it needs to read the variable a, but a doesn't belong to g — it belongs to f. We call a a non-local variable from g's perspective. The question is: how does g find a?

Remember, when functions run, their local variables live on the stack in stack frames (also called activation records). So a lives in f's stack frame. For g to access a, it needs to know where f's stack frame is.

Fat Pointers: The IP/EP Pair

Here's how most language implementations solve this. Instead of representing a function as just a single pointer to its machine code, they use two pointers:

  1. IP (Instruction Pointer) — points to the function's machine code (the instructions to execute)
  2. EP (Environment Pointer) — points to the definer's stack frame (the frame of the function that defined this function)

This two-pointer representation is called a fat pointer. Together, the IP and EP form everything you need to call the function: the code to run, and the environment in which to run it.

A fat pointer for function g: the IP points to g's code, the EP points back to f's stack frame so g can access variable a.

When g runs and needs to read a, it follows the EP to find f's stack frame, then looks at the right offset to find a.

This nesting can go multiple levels deep. If f defines g, and g defines h, then h's EP points to g's frame, and g's saved EP points to f's frame. To access a variable from f while inside h, the runtime follows two EP links — going up two levels of nesting.

Building Closures by Hand in C/C++

C and C++ do not have nested functions. Every function is defined at the top level. So there's no automatic EP — function pointers are just a single pointer to machine code.

But the concept of a closure is so useful that C/C++ programmers recreate it manually. Here's the recipe:

Step 1: Package the environment into a struct

Instead of relying on the language to give you access to surrounding variables, you explicitly put those variables into a structure:

struct Env {
    int a;
    int b;
};
// Instead of just having local variables a and b,
// we bundle them into a struct.

Step 2: Give the inner function an extra argument

The "nested" function takes an explicit pointer to that struct — this is the environment pointer, done manually:

int g(struct Env *env) {
    // Instead of just writing 'a', we write 'env->a'
    return env->a + 1;
}

void f() {
    struct Env env;
    env.a = 10;
    env.b = 20;

    // Call g, passing the environment explicitly
    int result = g(&env);
}

Now the "closure" is the combination of two things: the function pointer g (the IP) and the environment pointer &env (the EP). Together, they are exactly equivalent to the fat pointer the language would create automatically.

A closure = a function + an environment to run the function in. It's a function that has been "closed over" its surrounding variables — it carries with it everything it needs to execute, even if the original context is gone.

The tradeoff of manual closures

With automatic nested functions (like in OCaml or Scheme), closures happen for free — the compiler sets up the EP automatically. You get access to all of the definer's local variables.

With manual closures (like in C), you have more work to do, but you also get finer-grained control. You choose exactly which variables go into the environment struct. Maybe you only need a, not b — so you only put a in the struct. The closure is smaller and more efficient.

Another way to think about a closure: it's a tailored function — a specialization of a more general function. The general function g(env) can work with any environment. When you pair it with a specific environment, you get a specialized version that "knows" certain values.

Currying Is Just Closures

In languages like OCaml, currying is a common pattern: you write a function that returns another function, and the returned function refers to variables from the outer function's scope.

(* OCaml example *)
let add x = fun y -> x + y

(* add 3 returns a closure:
   - IP: the code for "fun y -> x + y"
   - EP: an environment where x = 3 *)
let add3 = add 3    (* add3 is a closure *)
let result = add3 5  (* returns 8 *)

When you call add 3, it returns a function. That returned function "remembers" that x = 3 because it carries a closure — the function code plus the environment containing x. Currying is closures in action.

Optimization note

Languages with nested functions are smart about optimization. If a function isn't actually nested (doesn't access non-local variables), the compiler won't bother creating an EP at all. The fat pointer collapses to a regular pointer. So you don't pay the cost unless you actually use the feature.

Chapter 2

Introducing the Heap

When the stack isn't enough — why programs need dynamically allocated memory

Why Do We Need a Heap?

So far, all the memory we've discussed (stack frames, local variables, function arguments) lives on the stack. The stack is wonderful for automatic memory management — when a function returns, its frame is popped, and the memory is reclaimed instantly.

But what happens when you create an object inside a function and want that object to survive after the function returns? For example:

// You call a function that creates a new linked list node
Node* createNode(int value) {
    Node* n = malloc(sizeof(Node));
    n->value = value;
    n->next = NULL;
    return n;  // The node must outlive this function!
}

If we put that node on the stack, it would be destroyed when createNode returns. That's useless — the whole point is that other parts of the program will use this node. We need memory that persists independently of function calls and returns.

That's what the heap is: a region of memory where objects can live for as long as they're needed, regardless of which functions are currently on the call stack.

There's a deeper connection to closures here. With features like Scheme's call/cc (call with current continuation), you can return into functions you've already exited from. This means stack frames themselves might need to live on the heap! Even without call/cc, continuation-passing style requires heap-like behavior. So closures are one of the motivations for having a heap in the first place.

Heap operations in different languages

Different languages provide different interfaces to the heap, but the underlying need is the same:

  • C: malloc(size) to allocate, free(ptr) to release
  • C++: new to allocate, delete to release
  • Java: new to allocate — but no free/delete (garbage collector handles it)
  • Python: objects are created implicitly — the runtime handles everything

All of these languages need a heap internally. The difference is in who is responsible for releasing memory: the programmer, or the system.

The Two Kinds of Stuff in a Heap

Think of the heap as a big arena of memory — like a large warehouse. Inside this warehouse, there are objects (things your program created and is using) and free space (empty space available for future objects).

The heap arena: green blocks are objects in use; gray gaps are free space.

The heap manager needs to track two things:

  1. Objects — In the simplest systems, the heap manager doesn't track these at all. The program keeps track of its own objects (via pointers). The heap manager trusts the program to know where its objects are.
  2. Free space — This is the heap manager's primary responsibility. When the program asks to allocate a new object, the heap manager needs to know where the free space is so it can carve out a piece.
The main job of the heap manager is to keep track of where things aren't. You hope the program keeps track of where things are. The heap manager's focus is the free space, because that's what it needs when the program asks for more memory.

Why Isn't the Heap Manager Part of the OS?

You might think: shouldn't the operating system handle heap management? Then every language could share the same manager.

In theory, yes. In practice, there are two big reasons this doesn't work well:

Reason 1: Speed. Every call to the operating system requires a context switch: the CPU has to save all the current process's state, switch to kernel mode, do the work, then switch back. This is expensive — hundreds or thousands of CPU cycles. Programs call malloc/new constantly, especially object-oriented programs. If every allocation required a system call, performance would be terrible.

Reason 2: Language-specific needs. Different languages have different heap requirements. A Scheme implementation might want a garbage-collected heap. A C implementation wants manual control. Putting everything in the OS would force a one-size-fits-all approach.

The result: heap management is done in user space, typically inside the language's runtime library (like the C library, libc). No system calls, no kernel involvement, no root privileges needed. The heap is just a big array of memory that we carve up ourselves.

Having per-language heap managers creates a problem for multi-language applications. If part of your program is in Scheme and part is in C++, they have separate heap managers that don't coordinate with each other. Each gets its own arena, and you may waste memory. This is a real-world pain point.

Chapter 3

Free Lists: Tracking Empty Space

The fundamental data structure behind memory allocation

The Free List: A Linked List of Available Blocks

We need a data structure to keep track of which parts of the heap are free. The simplest choice: a linked list where each node describes one contiguous block of free space.

Each node in the free list stores two pieces of information:

  1. The size of this free block (how many bytes are available)
  2. A next pointer to the next free block in the list
Each free-list node describes a free region in the heap. The linked list connects all free regions.

The Self-Hosting Trick: Store the Free List Inside the Free Space

There's an immediate problem: the free list is itself a data structure that grows and shrinks dynamically. It needs memory to live in. But where? If we put it on a separate heap, we'd need a free list for that heap, which needs its own heap, leading to infinite regression.

Imagine you're managing an empty warehouse. You need a notebook to track which shelves are empty. But the notebook needs to sit on a shelf too! You could buy a separate cabinet for the notebook… but then you need a second notebook to track the cabinet. The clever solution: just write your notes directly on the empty shelves. When a shelf gets used, erase the notes and use the shelf. The inventory system lives within the space it's tracking.

The elegant solution: store the free list nodes inside the free space itself. Each free block is large enough to hold at least two words (a size field and a next pointer). Since the space is free — nobody's using it — we can use the first few bytes of each free block to store the free list metadata.

This means the free list doesn't consume any extra memory. It lives inside memory that would otherwise be unused anyway. When a free block gets allocated (given to the program), the free list node that was sitting there simply gets overwritten by the object — that's fine, because we no longer need to track that block as free.

At the very start (before any allocations), the heap is entirely free. The free list has exactly one entry: a single node describing the entire arena. As objects are allocated and freed, the free list grows and shrinks, always living inside the unused portions of the heap.

First-Fit Allocation: How malloc Works

When a program calls malloc(n), requesting n bytes of memory, the simplest algorithm is first fit:

  1. Walk down the free list, examining each free block
  2. Find the first block whose size is ≥ n
  3. Split that block into two pieces:
    • A piece of size n — return this address to the caller
    • The leftover — keep this on the free list (just shrink the block's recorded size)

In practice, we carve off from the end of the free block, not the beginning. Why? Because if we carved from the beginning, we'd have to update the previous node's next pointer (since the block moved). By carving from the end, the block stays in the same place — it just has a smaller size. No pointer updates needed.

Edge case: the free block evaporates

Remember, each free block needs at least two words of space to store its metadata (size + next pointer). If after an allocation the leftover space is smaller than two words, we can't keep tracking it. The block evaporates: we give all of it to the caller (who gets a few extra bytes) and remove the entry from the free list entirely.

Edge case: no block is big enough

If we walk the entire free list and no single block is large enough for the requested allocation, malloc returns NULL — the allocation fails. This can happen even if there's plenty of free memory overall, if that memory is scattered in small pieces. This is the problem of external fragmentation, which we'll explore next.

Chapter 4

Fragmentation: The Enemy of Efficiency

Why free space can become unusable — and strategies to fight back

External Fragmentation

External fragmentation happens when there's enough total free memory to satisfy a request, but it's scattered across multiple small blocks instead of being in one contiguous piece.

// Free list: [1000] → [300] → [250]
// Total free: 1550 bytes
// Request: malloc(1200)
// Result: FAILS! No single block ≥ 1200
// Even though 1550 > 1200, it's split up

This is like having $15 in loose change scattered around your house, but the vending machine only takes bills. You have enough money; it's just in the wrong form.

External fragmentation is the more serious problem for most systems. It means your program can crash (malloc returns NULL) even when there's plenty of memory available. It wastes memory that exists but can't be used.

Internal Fragmentation

Internal fragmentation is wasted space inside an allocated object. This happens for two reasons:

1. Block evaporation. When a free block is too small to keep tracking (less than two words after splitting), we give the entire block to the caller. The caller gets more memory than they asked for — those extra bytes are wasted.

2. Alignment requirements. On x86-64 processors, the C memory allocator (malloc) always returns addresses that are multiples of 16. Why? Because some data types (like 128-bit SIMD values) require 16-byte alignment. Since malloc doesn't know what type you're storing, it always aligns to the strictest requirement. If you ask for 15 bytes, you'll get 16. That extra byte is internal fragmentation.

Internal fragmentation wastes a few bytes per object. External fragmentation can make large allocations impossible even when memory exists. Both are bad, but external fragmentation is usually the bigger concern.

Coalescing: Merging Adjacent Free Blocks

The most important technique for fighting external fragmentation is coalescing: whenever you free an object, check if the newly freed space is adjacent to an existing free block. If so, merge them into one larger block.

When free(p) is called, the implementation checks two things:

  1. Is the space just before p a free block? If yes, extend that block to include p's space.
  2. Is the space just after p a free block? If yes, extend to include that block too.

If both neighbors are free, a single free() call can merge three pieces (the predecessor free block, the freed object, and the successor free block) into one large block. This actually shrinks the free list by one node — a win for performance too, since shorter free lists mean faster searches.

Freeing O₂ and coalescing with both adjacent free blocks creates one large block (200 + 50 + 150 = 400 bytes).
Rule: No two free blocks should ever be adjacent. If they are, they should be coalesced. Adjacent free blocks mean unnecessary external fragmentation.

Object Headers: How free() Knows the Object Size

When you call free(p), the heap manager needs to know how big the object at p is. The simplest approach: each object has a hidden header — an extra word placed just before the object that stores its size. The pointer returned by malloc points just past this header.

// What malloc actually does internally:
//   [size | object data ...]
//          ^ pointer returned to user
// When free(p) is called, it looks at p[-1] to find the size

C also offers free_sized(p, n) where the caller specifies the size, avoiding the need to look up the hidden header. This can be faster for high-performance applications.

The Roving Pointer: Fixing First-Fit's Weakness

There's a hidden inefficiency in first-fit allocation. After many allocations and frees, the beginning of the free list fills up with tiny, useless blocks — the leftovers that were too small for any allocation. Every future allocation has to skip past all these tiny blocks to reach the first useful one.

The fix: use a circular free list with a roving pointer. Instead of always starting from the beginning, start from where the last successful allocation happened.

A circular free list. The roving pointer starts at the large block (8000), so small allocations don't have to skip tiny fragments.

This way, the next allocation typically succeeds immediately — the current block is probably big enough. As we chew through it, the pointer advances. Meanwhile, free() calls coalesce freed objects, growing the other blocks. By the time the pointer wraps around, those blocks should be big again.

Best-Fit Allocation

An alternative to first-fit is best-fit: instead of using the first block that's big enough, find the block that's closest in size to the request (but still ≥ n). This tends to minimize wasted space and reduce fragmentation. However, it requires searching the entire free list (or maintaining a sorted index), making it slower. In practice, best-fit is rarely used because the performance cost outweighs the fragmentation benefit.

Chapter 5

Quick Lists: A Speed Hack

When you know what you'll be allocating, you can beat the general-purpose allocator

The Idea

Suppose you're writing a Scheme interpreter in C. In Scheme, the most common operation is cons — creating pairs (two-word objects). Your program creates millions of these pairs. Each one is exactly 16 bytes (two pointers on x86-64).

The general-purpose malloc works, but it's doing a lot of unnecessary work for these tiny, fixed-size objects: walking free lists, checking sizes, worrying about coalescing. What if you maintained your own private free list just for pairs?

A quick list is a private, specialized free list for objects of a single fixed size. Since everything is the same size, there's no need to check sizes, no fragmentation to worry about, and allocation/deallocation is just a few instructions.

Code Walkthrough

Here's how you'd implement a quick list for Scheme pairs in C:

typedef struct pair {
    struct pair *head;
    struct pair *tail;
} pair;

// Our private free list — starts empty
static pair *my_free = NULL;

The fast allocator (cons)

pair* cons() {
    pair *p = my_free;

    if (p) {
        // FAST PATH: grab from our private list
        my_free = p->tail;  // remove from list
    } else {
        // SLOW PATH: our list is empty, ask malloc
        p = malloc(sizeof(*p));
    }
    return p;
}

The fast path is just 2-3 instructions: check if the list is non-empty, grab the first node, advance the pointer. No free list walking, no size checking, no coalescing. It's the fastest possible allocation.

The fast deallocator (my_free_pair)

void my_free_pair(pair *p) {
    // Just prepend to our private free list
    p->tail = my_free;
    my_free = p;
}

Freeing is also just 2 instructions: set the node's next pointer, update the list head. No coalescing, no searching.

The Tradeoff: Speed vs. Fragmentation

Quick lists are fast, but they come with a cost: the underlying malloc thinks all the objects in your quick list are still in use. From malloc's perspective, you allocated those objects and never freed them. This means:

  • The memory manager can't coalesce quick list objects with adjacent free blocks
  • You'll get more external fragmentation in the underlying heap
  • Memory is "hoarded" by your quick list even if it could be useful elsewhere

This is why quick lists are only worth using for objects that are extremely common in your program. If you only allocate pairs, and you allocate millions of them, the speed benefit far outweighs the fragmentation cost.

Some sophisticated memory allocators (like glibc's malloc) have quick lists built in. They maintain separate fast lists for common allocation sizes (16 bytes, 32 bytes, etc.) internally. If your allocator already does this, building your own quick list on top would be redundant — you'd have two layers of caching, which wastes memory without improving speed.

Chapter 6

What Can Go Wrong

The two fundamental bugs of manual memory management

If you've used malloc/free or new/delete, you know that manual memory management is error-prone. There are exactly two fundamental things that can go wrong, and understanding them deeply is crucial — because they motivate the invention of garbage collection.

Problem 1: Dangling Pointers

A dangling pointer is a pointer that still refers to memory that has already been freed. The object is gone (its space may be reused), but the pointer doesn't know that.

int *p = malloc(sizeof(int));
*p = 42;
free(p);         // The object is freed

// ... later ...
printf("%d", *p); // DISASTER: p is dangling!
                   // The memory at p might be:
                   //   - still 42 (by luck, undisturbed)
                   //   - some other value (reused by another malloc)
                   //   - cause a crash (segfault)

Why is this so dangerous?

  • Non-deterministic behavior: sometimes the old data is still there, so the bug seems to "work." Other times, the memory has been reused by a different part of your program, and two unrelated pieces of code are now writing to the same memory, corrupting each other's data.
  • Crashes: if the reused memory contains a NULL pointer or invalid data, dereferencing it causes a segfault.
  • Security vulnerabilities: use-after-free bugs are a major source of exploitable security holes.

Problem 2: Memory Leaks

A memory leak occurs when you allocate memory, stop using it, but never free it. The heap manager thinks the memory is still in use and won't reclaim it.

int *p = malloc(sizeof(int));
*p = 42;
// ... use p for a while ...

// From here on, the program never touches p again.
// But free(p) was never called!
// The memory sits there, reserved but unused, forever.

A single small leak might not matter. But if your program keeps leaking — allocating objects and forgetting to free them — memory consumption grows without bound. For long-running programs (servers, embedded systems, robots), this is fatal: eventually the program runs out of memory and crashes.

Which Problem Is Worse?

Dangling pointers are generally considered the worse problem because they cause crashes and unpredictable behavior. A memory leak makes your program gradually slower and fatter, but it keeps running. A dangling pointer can crash your braking system or your nuclear control rods. From a reliability standpoint, crashes are more dangerous than slowdowns.

However, some engineers care more about leaks because they cause gradual performance degradation that's hard to diagnose. Both are real problems. The question is: can we eliminate them?

Hiring "better programmers who never make mistakes" isn't realistic. We need a system-level solution that prevents these errors entirely. That's what garbage collection is for.

Chapter 7

Garbage Collection: Mark and Sweep

Letting the system free memory for you — no dangling pointers, ever

The Core Idea

The fundamental idea of garbage collection is simple: remove free/delete from the programmer's vocabulary entirely. The programmer allocates objects (with new or equivalent), and the system automatically reclaims objects that are no longer needed.

Languages like OCaml, Python, Java, Scheme, JavaScript, Go, and many others use garbage collection. It eliminates dangling pointers by construction — if you never free anything manually, you can never free something too early.

You could even simulate this in C with a macro:

#define free(p) ((void)0)  // free does nothing!

This eliminates dangling pointers, but your program now leaks like crazy — every allocation persists forever. We need something smarter: a system that automatically figures out which objects are no longer needed and reclaims them.

Prerequisites: Roots and Pointer Slots

For a garbage collector to know which objects are "in use," it needs two pieces of information:

1. Roots

Roots are pointers from outside the heap that point into the heap. These are the starting points — the objects that the program can directly reach. Roots include:

  • Local variables on the stack that hold pointers to heap objects
  • Global/static variables that point to heap objects
  • CPU registers containing pointers to heap objects

2. Pointer slots within objects

An object in the heap might contain pointers to other objects. The garbage collector needs to know which fields of each object are pointers (as opposed to integers, floating-point values, or other non-pointer data). These pointers create chains of reachability: if root R points to object A, and A contains a pointer to object B, then B is also "in use" — even though no root directly points to B.

Objects A, B, and C are reachable from roots — they're in use. Object D has no path from any root — it's garbage and can be freed.

The Mark and Sweep Algorithm

Mark and sweep is the simplest and most fundamental garbage collection algorithm. It works in two phases:

Setup: The Mark Bit

Every object has a hidden mark bit in its header (alongside the size field). This bit starts at 0 (unmarked). The mark bit typically shares a word with the size field since sizes are always even (so the lowest bit is available).

Phase 1: Mark

Starting from each root, recursively visit every reachable object and set its mark bit to 1.

function visit(p):
    if p.marked:        // Already visited? Stop.
        return            // (This handles cycles!)
    p.marked = true     // Mark this object
    for each pointer q in p:
        visit(q)          // Recursively visit objects it points to

// Mark phase:
for each root r:
    visit(r)

The mark bit serves double duty: it identifies objects in use, and it prevents infinite loops when the data has cycles (e.g., A → B → A). If an object is already marked, we skip it.

Phase 2: Sweep

Walk through every object in the heap (not just reachable ones). If an object is marked, clear the mark (for next time). If it's not marked, it's garbage — free it.

// Sweep phase:
for each object o in heap:
    if o.marked:
        o.marked = false  // Reset for next GC cycle
    else:
        free(o)            // Object is garbage — reclaim it

The free here does the same thing as the C free: put the block on the free list, coalesce adjacent blocks, etc. But it's safe because we've proven (by exhaustive marking) that no reachable pointer points to this object.

    visit(B): B already marked! Return. ← cycle broken!

Objects:  A[1]  B[1]  C[1]  D[1]  E[0]

Sweep phase. Walk through every object. A, B, C, D are marked → clear their marks (ready for next GC). E is NOT marked → free it!

Sweep:
  A[1] → marked, clear → A[0]  ✓ keep
  B[1] → marked, clear → B[0]  ✓ keep
  C[1] → marked, clear → C[0]  ✓ keep
  D[1] → marked, clear → D[0]  ✓ keep
  E[0] → NOT marked → FREE!  🗑️ garbage collected

Result: E's memory is reclaimed. All other objects survive.

Limitations of Mark and Sweep

1. It doesn't completely solve memory leaks

Mark and sweep reclaims objects that nothing points to. But what if a root (say, a static variable) still holds a pointer to an object that the program will never use again?

static int *p = malloc(1000000);
// p is used once, then never again.
// But p is a root! It still points to the object.
// The GC can't free it — it looks "in use."

Determining whether a program will use a variable in the future is equivalent to the halting problem — it's mathematically impossible to decide in general. So garbage collection can eliminate unreachable-object leaks but not all leaks.

2. It's not real-time

Mark and sweep must visit every reachable object in the system. If you have a billion objects, this takes a while. During that time, the program is paused — it can't do anything else. For a robot, a video game, or a stock trading system, a multi-second pause is catastrophic.

3. Performance overhead

Even ignoring real-time concerns, mark and sweep is expensive. It does work proportional to the total number of objects, not just the ones being freed. Running a full GC because memory is "a little bit low" can be painful.

4. Doesn't work natively with C/C++

Mark and sweep requires knowing where roots are and which object fields are pointers. Languages like C and C++ don't provide this information at runtime (for efficiency). This motivates conservative garbage collection, covered next.

For short-lived programs, none of this matters much — when a process exits, the OS reclaims all its memory. Garbage collection becomes critical for long-lived programs: servers, editors, interpreters, robots — anything that runs for hours or days and continuously allocates memory.

Chapter 8

Conservative Garbage Collection

Making GC work for C and C++ — when the compiler won't help you

The Problem: C Compilers Are Cheapskates

Mark and sweep needs to know: (1) where the roots are, and (2) which parts of each object are pointers. In languages like Java or OCaml, the compiler records this information. But C/C++ compilers don't — they optimize for speed and don't want to bloat the executable with metadata.

The memory manager only sees calls to malloc(n) and free(p). It knows how big each object is and where it was placed, but not:

  • Where the returned pointer was stored (which stack slot? which register?)
  • Which bytes in the object are used as pointers vs. integers vs. floats

The Conservative Approach: When in Doubt, Assume It's a Pointer

Conservative garbage collection works by using approximate knowledge:

Finding roots (approximately)

We don't know exactly which stack slots contain pointers, but we know three things:

  1. The stack — we know its boundaries (RSP register to base of stack). Roots could be anywhere in this region.
  2. Static/global storage — the OS tells us where global variables live.
  3. CPU registers — a pointer might be sitting in RAX or RSI without being stored to memory yet.

These regions form a "roots plus" set — every actual root is in there, plus a lot of non-pointer data.

The key trick

For every word in the roots-plus regions, we check: does this bit pattern happen to equal the address of any known heap object? We don't dereference the word (that could crash). We just compare its numeric value against our table of allocated objects.

  • If it matches an object address → treat it as a pointer. Mark that object as in use.
  • If it doesn't match → ignore it (it's probably an integer or float).

We do the same within objects: examine every word, and if it looks like a pointer to a known object, follow it recursively.

Conservative means the algorithm might make mistakes — but only in one direction. It might think a non-pointer is a pointer (causing a memory leak — the object won't be freed). It will never think a pointer is a non-pointer (which would free an object still in use — a dangling pointer). The mistakes are "conservative" — they err on the side of keeping too much, not too little.

Why this works in practice

If the heap lives in high memory addresses, it's unlikely that a random integer or float will happen to look like a valid heap pointer. The false-positive rate is low. Additionally, since the memory manager maintains a table of all allocated objects (it recorded them during malloc), checking whether a bit pattern is a valid object address is fast — it's just a table lookup.

Real-World Success Stories

Conservative GC has been successfully used in major, long-lived C/C++ programs:

  • Emacs — the text editor, originally written in C with manual memory management. As the codebase grew and more contributors joined over 50+ years, use-after-free crashes became a serious problem. Switching to conservative GC eliminated the crashes.
  • GCC — the GNU Compiler Collection. Same story: large C++ codebase, many contributors, memory bugs. Conservative GC fixed the reliability problems.

In both cases, the programs became slightly slower but dramatically more reliable. The slowdown was small enough that nobody cared.

Conservative GC is not a panacea. You can still leak large objects if a random integer happens to match their address. It works best when programs allocate many small objects (which is common). For programs with a few very large allocations, a false positive could waste significant memory.

Chapter 9

Reference Counting

A completely different approach — count how many pointers point at each object

The Idea

Reference counting takes a radically different approach from mark and sweep. Instead of periodically scanning all objects, it tracks usage continuously. Every object has a hidden reference count field that records how many pointers currently point to that object.

The rule is simple: when the reference count drops to zero, no one can reach the object, so free it immediately.

# Python example
p = "abc"     # "abc" created, refcount = 1 (p points to it)
q = p          # refcount becomes 2 (both p and q point to it)
p = "xyz"     # p now points elsewhere
               # "abc" refcount drops to 1 (only q remains)
q = None      # "abc" refcount drops to 0 → FREED immediately!

The Cost: Every Assignment Gets Expensive

With reference counting, a simple pointer assignment like r = p actually requires three memory updates:

  1. Decrement the reference count of whatever r used to point to (it's losing a reference)
  2. Update r itself to point to the new object
  3. Increment the reference count of the new object (it's gaining a reference)
A single assignment r = p requires updating two reference counts plus the pointer itself.

Advantages over mark and sweep

  • Incremental: objects are freed immediately when their count hits zero. No long GC pauses. No "stop the world."
  • Predictable: cleanup happens at known times (right after the last reference disappears).

Maximum reference count

The reference count field has limited size (often 1 byte or a few bytes). If an object gets 255 references (with a 1-byte counter), we treat 255 as "infinity" and never decrement below it. The object becomes permanently un-freeable — a small, rare memory leak that prevents integer overflow crashes.

The Cycle Problem

Reference counting has a fundamental flaw: it cannot handle cycles. If object A points to B and B points back to A, both have reference count ≥ 1 forever, even if nothing else in the program points to either of them.

# Python: creating a cycle
p = [100]       # list with refcount 1
p[0] = p        # the list now points to itself!
                  # refcount = 2 (p and the list's own slot)

p = [999]       # p now points to a new list
                  # old list's refcount drops to 1 (self-reference)
                  # It will NEVER reach 0 — it's leaked!

Could we detect cycles? In theory, yes, but checking for cycles on every assignment would make each assignment O(n) in the worst case (where n is the total number of objects), which is far too expensive.

The simple answer "don't create cycles" isn't always practical. Many data structures (doubly-linked lists, graphs, parent-child pointers in trees) naturally involve cycles.

Python's Hybrid Approach

The standard Python implementation (CPython) uses reference counting as its primary garbage collection mechanism. For most objects, this works well — objects are freed immediately when their last reference disappears.

But to handle cycles, CPython also has a mark-and-sweep garbage collector as a fallback. Periodically, when the reference counting system seems to be accumulating too many objects without freeing them, the mark-and-sweep collector kicks in and finds unreachable cycles.

This hybrid approach gives you the best of both worlds:

  • Reference counting handles the common case (non-cyclic objects) quickly and incrementally
  • Mark and sweep handles the uncommon case (cyclic garbage) as a safety net
This is a general lesson in systems design: real-world systems are hybrids. No single algorithm is best for everything. The most practical approach is often to combine a fast algorithm for the common case with a correct-but-slower algorithm for edge cases. You'll see this pattern everywhere: caches, networking protocols, database query planners, and more.
