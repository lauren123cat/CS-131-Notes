Prolog: Reverse, Unification, Debugging & Logic Theory

This lecture goes deep into how Prolog actually works under the hood — how it matches terms, passes arguments, and reasons about your code. We'll build up from a simple list-reversing predicate all the way to the foundations of formal logic itself, and see how Prolog fits into the bigger picture of computation and provability.

Setting the Stage

In the previous lecture, you saw the basics of Prolog: facts, rules, queries, and the append/3 predicate for concatenating lists. Now we're going to push further and understand Prolog at a deeper level.

Why does this matter?
Prolog is unlike every other programming language you've used. In Java or Python, you tell the computer how to solve a problem step by step. In Prolog, you describe what is true, and the computer figures out the steps. But this separation of "logic" from "control" has consequences — sometimes beautiful, sometimes dangerous. This lecture reveals both sides.

Here's the roadmap for today:

Part 1: We'll write a reverse predicate — first slowly, then fast — and see why "direction of use" matters in Prolog.

Part 2: We'll learn how to debug Prolog programs using the four-port model — a mental framework that's unique to logic programming.

Part 3: We'll understand unification — the engine that drives all of Prolog's computation — and see how it differs from pattern matching in other languages.

Part 4: We'll encounter a real bug caused by unification creating infinite data structures, and learn about the "occurs check" fix.

Part 5: We'll step back into theory — propositional logic and first-order logic — to understand the mathematical foundation that Prolog is built on, and why that foundation has hard limits.

Why Start with reverse?

The reverse/2 predicate (reversing a list) is historically used as the standard benchmark for Prolog systems. When researchers say a Prolog system can do "X logical inferences per second," they're typically measuring performance on reverse. It's also a great teaching example because we already saw how to do it (slowly and fast) in OCaml earlier in the course, so we can compare approaches across paradigms.

1. Naive reverse/2 — The Slow Way

Before we write any code, let's think about what "reversing a list" means. If I have the list [1, 2, 3], its reverse is [3, 2, 1]. If I have an empty list [], its reverse is also [].

Thinking in Cases

In Prolog, the first thing you do when writing a predicate is think about the cases. What are the different shapes the input could have? For lists, there are two fundamental cases:

Base case: The list is empty — [].

Recursive case: The list has at least one element — it looks like [X|L], where X is the first element (the "head") and L is everything else (the "tail").

Why put the base case first?
Prolog searches through clauses from top to bottom, depth-first. By putting the base case first, Prolog can quickly check "is this the simple case?" and move on. If the base case were last, Prolog would always try the recursive case first, which is less efficient.

The Base Case

The reverse of an empty list is an empty list. In Prolog:

reverse([], []).

This is a fact — it's always true, no conditions needed. Read it as: "The reverse of [] is []."

The Recursive Case

Now, suppose we have a list whose first element is X and the rest is L. Written in Prolog notation, that's [X|L]. How do we reverse this?

Think about it visually. If our list is:

[ X | L ] e.g. [ 1 | [2, 3] ]

We want the result to look like:

[ reverse of L ] ++ [ X ] e.g. [3, 2] ++ [1] = [3, 2, 1]

So the algorithm is: (1) reverse the tail L to get some reversed list RL, then (2) append the singleton list [X] to the end of RL.

reverse([X|L], R) :-
    reverse(L, RL),       % Reverse the tail L, get RL
    append(RL, [X], R).   % Append [X] to end of RL, get R

Read this as: "The reverse of [X|L] is R, if the reverse of L is RL, and appending [X] to RL gives R."

We've just written down the logic of what it means to reverse a list. We didn't specify a loop counter, an index variable, or any step-by-step instructions. That's the Prolog way.

Why This Is Slow: O(N²)

This definition is logically correct but computationally expensive. Here's why:

To reverse a list of length N, we first reverse a list of length N-1 (that's the recursive call). Then we call append, which has to walk all the way through the reversed list to add [X] at the end — that's O(N) work by itself.

So we do O(N) work for each of the N recursive calls: O(N) × O(N) = O(N²).

For a list of length 1,000,000, this would be catastrophically slow. We saw the exact same problem when we wrote a naive reverse in OCaml earlier in the course.

Quick check: Why does the naive reverse have O(N²) complexity?

2. Fast Reverse via revapp/3 — The Accumulator Trick

To fix the O(N²) problem, we use a classic technique (the same one we used in OCaml): solve a more general problem efficiently, then use it to solve the original problem.

The More General Problem: revapp/3

We define a new predicate revapp(A, B, C) that is true when:

Definition
revapp(A, B, C) is true if the reverse of A, concatenated with B, equals C.

Why is this "more general"? Because plain reverse(A, C) is just the special case where B is the empty list: revapp(A, [], C). The extra argument B acts as an accumulator — a place where we build up the reversed list as we go, so we never need to call append.

Base case

If A is empty, then "reverse of [] concatenated with B" is just B:

revapp([], B, B).

Recursive case

If A is [X|L], we want: reverse(L) ++ [X] ++ B. But that's exactly the same as: reverse(L) ++ [X|B]. And that's exactly revapp(L, [X|B], R)!

revapp([X|L], A, R) :-
    revapp(L, [X|A], R).  % Move X from input to accumulator

Each step just takes the head element off the input list and puts it at the front of the accumulator. No append call. Each step is O(1) work.

Using revapp to implement fast reverse

reverse(L, R) :- revapp(L, [], R).

Start with an empty accumulator. Each of the N elements gets moved once. Total: O(N).

3. The Catch: Using reverse "Backwards"

Here's where Prolog gets interesting — and dangerous. Because Prolog is declarative, you might think you can call reverse in any direction. For example:

% Normal use: "What is the reverse of [1,3,19]?"
?- reverse([1,3,19], R).
% R = [19,3,1].  ✓ Works perfectly.

% Backwards use: "What list, when reversed, gives [1,3,19]?"
?- reverse(L, [1,3,19]).
% L = [19,3,1].  ✓ Finds one answer...
% ...but if you ask for MORE answers: INFINITE LOOP!

What happens? Internally, revapp starts trying longer and longer candidate lists. It finds the correct one (length 3), but then keeps going — constructing lists of length 4, 5, 6, ... — none of which will ever match. It never stops.

Core Prolog Lesson
Separating logic from control does not mean you can ignore control. Code that runs fast in the "right" direction of use can be infinitely slow in the "wrong" direction. You must always think about how Prolog will execute your logic, not just what your logic says.

This is one of the fundamental tensions of Prolog programming. The logic is clean and beautiful, but the control — how Prolog searches through possibilities — can trip you up if you're not careful.

4. Debugging in Prolog

Every time you learn a new programming language, you need to learn how to debug it. Debugging in Prolog serves two purposes:

1. Learning: Understanding how Prolog actually executes — which is different from every other language you know.

2. Bug-fixing: Finding and fixing problems in your code.

A Simple Technique

The simplest debugging trick: iterate through all solutions and print them:

forall(G, (write(G), nl))

Here, G is whatever goal you want to investigate. This prints each solution on its own line. It's fine for small cases, but useless if execution loops (since it'll just print one answer and then hang forever).

The Four-Port Debugging Model

To truly understand Prolog's debugging, you need a mental model of how control flows. This is fundamentally different from languages you know.

In C or Java (the simple model)

When you call a function, there's one way in (call it) and one way out (it returns). That's a two-port model.

┌──────────┐ Call ──►│ function │──► Return └──────────┘

With exception handling (C++, Java, Python)

Now there's one way in and two ways out — the function can return normally, or it can throw an exception. That's a three-port model.

┌──────────┐ Call ──►│ function │──► Return │ │──► Throw (exception) └──────────┘

In Prolog: The Four-Port Model

Prolog has two ways in and two ways out. This is the crucial difference:

┌──────────┐ Call ──►│ │──► Exit (Success) │ Goal C │ Redo ──►│ │──► Fail (backtrack) └──────────┘

Let me explain each port with a concrete scenario. Imagine Prolog is executing goals A, B, C, D, E in sequence:

Call: The previous goal B succeeded. Now Prolog enters goal C to try to prove it. This is the normal "function call" you're used to.

Exit (Success): Goal C found a solution! Prolog moves on to try goal D. This is like a function returning.

Fail: Goal C couldn't find any (more) solutions. Prolog backtracks to goal B and asks B to find a different solution. This is like an exception, but it's normal in Prolog — failure is expected and handled gracefully.

Redo (Backtrack): Goal D (or something later) failed, and Prolog is coming back to goal C to ask: "Do you have another solution I haven't tried yet?" This is the port that doesn't exist in most other languages. It's the reason Prolog can explore multiple possibilities.

Why This Matters
When debugging Prolog, you need to understand that any goal can be entered twice (via Call or Redo) and exited twice (via Exit or Fail). If you only think in terms of "call and return" (the model from C/Java), you'll be confused by Prolog's behavior. The four-port model is your compass.
A goal C has been called and succeeded (Exit). Later, goal E fails and Prolog backtracks all the way to C. Through which port does Prolog re-enter C?

Using trace/0

The built-in predicate trace puts the Prolog interpreter into a special mode. Every time control passes through any port of any goal, it prints a line telling you what happened:

?- trace.
% Now every goal will print: Call, Exit, Fail, or Redo

?- reverse([1,2,3], R).
%   1  Call: reverse([1,2,3], _G1)
%   2  Call: revapp([1,2,3], [], _G1)
%   3  Call: revapp([2,3], [1], _G1)
%   4  Call: revapp([3], [2,1], _G1)
%   5  Call: revapp([], [3,2,1], _G1)
%   5  Exit: revapp([], [3,2,1], [3,2,1])
%   4  Exit: revapp([3], [2,1], [3,2,1])
%   ...and so on back up

For small programs (like reversing a list of length 3), trace is all you need. For larger programs, GNU Prolog and other systems support breakpoints, step-over, and other familiar debugging tools — but the four-port model is always the underlying mental model.

5. Unification — The Heart of Prolog

Unification is the single most important concept for understanding how Prolog works. It's the mechanism that drives all computation in Prolog — every time Prolog tries to match a goal against a clause, it's doing unification.

What Is Unification?

Unification is a process that takes two terms and tries to make them identical by finding appropriate values for their logical variables.

A unifier is a set of variable bindings (like "X = 5, Y = hello") that, when applied, makes the two terms the same.

Worked Example

Let's unify these two terms:

Term 1:  p(f(X), g(Y), 27)
Term 2:  p(f(h(Z)), g(19), Z)

Prolog walks through both terms simultaneously, position by position:

▸ Unification Walkthrough

Unification vs. Pattern Matching

This is a critical distinction that separates Prolog from every other language you've seen:

Pattern matching (OCaml, Python, Haskell, etc.) is one-way. You have a pattern with variables and data without variables. Variables only appear on one side. The data fills in the variables.

(* OCaml pattern matching — one way *)
match my_list with
| [] -> ...          (* pattern has no variables for data to fill *)
| x :: rest -> ...   (* x and rest are pattern variables *)

Unification (Prolog) is two-way. Both terms can contain variables. Variables on either side can get bound to structures on the other side, or even to each other.

% Prolog unification — two way
% Both sides can have variables:
?- f(X, 3) = f(7, Y).
% X = 7, Y = 3.  Variables on BOTH sides got bound!
The Big Difference
In OCaml, you can match a pattern against data. In Prolog, you can unify two patterns against each other. This two-way matching is what gives Prolog its unique power — and its unique complexity.

Another important case: two unbound variables can unify with each other. After Q = R where both Q and R are unbound, they become "linked" — whatever one gets bound to later, the other automatically becomes as well.

6. How Prolog Uses Unification: Logical Inference

Now that you understand unification as a mechanism, let's see how Prolog uses it to actually run your programs.

The Basic Step

When Prolog has a goal G it wants to prove, it looks through your program's clauses for one whose head H can unify with G. Here's what happens:

If the clause is a fact (just H. with no body):

Prolog unifies G with H, producing a unifier U. If successful, the goal is proved! The answer is G with the unifier applied (written G·U).

If the clause is a rule (H :- B, C, D.):

Prolog unifies G with H, producing unifier U. Then it replaces the goal G with new subgoals: B·U, C·U, D·U — the body of the rule, with the unifier applied to every part.

% Example rule:
grandparent(X, Z) :- parent(X, Y), parent(Y, Z).

% Goal:
?- grandparent(alice, Who).

% Step 1: Unify grandparent(alice, Who) with grandparent(X, Z)
% Unifier: {X = alice, Z = Who}

% Step 2: Replace goal with body, applying unifier:
%   parent(alice, Y), parent(Y, Who)
% Now Prolog must prove THESE two subgoals.
Facts vs Rules
Facts shrink Prolog's work — once you match, you're done. Rules grow the work — matching a rule replaces one goal with possibly many subgoals. That's why the Prolog interpreter "likes" facts: they simplify things immediately.

The Goal Stack Mental Model

A simple way to think about Prolog's runtime: it maintains a stack of goals that still need to be proved.

When Prolog matches a goal against a fact, the stack shrinks by 1 (that goal is done).

When Prolog matches a goal against a rule with N subgoals in the body, the stack shrinks by 1 (the original goal) but grows by N (the new subgoals). Net change: the stack grows by N-1.

This stack-based approach is part of why Prolog can be very efficient for certain computations — successful terms are just pushed onto the stack. There's no heap allocation or garbage collection needed in the simple case, unlike Python or Java which need garbage collectors.

Unification as Parameter Passing

Here's something surprising: unification handles all of Prolog's argument passing. There are no separate "input" and "output" parameters — it all depends on context.

Unbound variable → acts like a return slot:

?- reverse([1,2,3], R).
% R is unbound. The predicate "fills in" R with [3,2,1].
% R acted like a return value.

Bound variable → acts like a normal input:

?- reverse([1,2,3], [3,2,1]).
% Both arguments are bound. This just CHECKS if it's true.
% Like passing two inputs and getting a boolean answer.

Partially bound → acts as both:

% Suppose Q(X) binds X to s(Y) where Y is still unbound:
Q(X), P(X).
% Now P receives s(Y) — it has structure (input) but also
% a hole Y that P can fill in (output).

This flexibility is fundamentally different from any argument-passing mechanism in OCaml, Java, or Python. It's like passing a data structure with a pointer to where you want storage allocated.

7. The Occurs Check Problem

Now we'll see a real, dangerous bug that unification can cause — and it comes from Prolog choosing speed over logical correctness.

Setting Up: Peano Arithmetic

To demonstrate the problem, we'll implement arithmetic from scratch in Prolog, without using any built-in numbers. This is called Peano arithmetic, named after the logician Giuseppe Peano (~1900), who tried to build a complete logical foundation for arithmetic.

Representing Numbers

Since we're pretending we don't have actual integers, we represent numbers using just two building blocks:

z represents zero.

s(X) represents "the successor of X" — i.e., X + 1.

% Number representations:
z                 % = 0
s(z)              % = 1  (successor of 0)
s(s(z))           % = 2  (successor of successor of 0)
s(s(s(z)))        % = 3

Think of it as a tally system. The number "3" is represented by nesting the s function three layers deep around z.

Defining Addition

add(z, M, M).               % 0 + M = M
add(s(N), M, s(R)) :-       % (N+1) + M = (N+M) + 1
    add(N, M, R).

The base case says: adding zero to anything gives you that thing. The recursive case says: adding (N+1) to M is the same as adding N to M, then adding 1 to the result.

This is essentially the same structure as append — just specialized for our "tally mark" data structures instead of lists.

Subtraction, Multiplication, etc.

Here's the beautiful part — subtraction is just addition run backwards:

subtract(A, B, C) :- add(B, C, A).
% "A - B = C" means "B + C = A"

Multiplication is repeated addition. Division is multiplication run backwards. The logic just works.

Comparisons

le(z, _).               % 0 ≤ anything
le(s(M), s(N)) :-       % M+1 ≤ N+1  iff  M ≤ N
    le(M, N).

Now, here's where trouble starts. Let's define "less than":

% DANGEROUS! This fact will cause problems:
lt(M, s(M)).             % M < M+1

Logically, this is correct: any number is less than its own successor. But watch what happens when we use it the wrong way...

The Bug: Infinite Terms

Let's ask Prolog: "Is there any number that is less than itself?"

?- lt(I, I).     % Is there an I such that I < I?

Logically, the answer should be no — no number is less than itself. But watch what Prolog does:

▸ What happens when Prolog evaluates lt(I, I)
The Problem
Prolog claims to find a value I such that I < I. That value is essentially "infinity" — an infinite chain of s(s(s(s(...)))). This is logically wrong. Peano arithmetic doesn't have infinity. Prolog produced an answer that violates the rules of logic, all because it was trying to be efficient.

This isn't limited to Peano arithmetic. You can trigger the same bug with Prolog's built-in equality:

?- L = [_|L].
% Prolog says: yes!
% L is now a list whose tail is itself — an infinite list!

The = predicate in Prolog is just unification. It's defined internally as a single fact: eq(X, X). — anything equals itself. When you ask L = [_|L], unification happily binds L to a structure that contains itself, creating a cycle.

Why Does Standard Unification Allow This?

When unifying two terms, Prolog walks through them like trees. When it finds a variable on one side and a non-variable on the other, it simply stores a pointer. This is very fast — O(min(|T1|, |T2|)).

The problem arises when one term is inside the other. If Prolog needs to bind variable X to a term that contains X, it creates a circular pointer — a cycle. It doesn't check whether X appears inside the term, because checking would be expensive.

The Fix: unify_with_occurs_check/2

Prolog provides a built-in predicate that does the check:

unify_with_occurs_check(X, Y)

This works like X = Y, except: before binding any variable X to a term T, it walks through all of T to make sure X doesn't appear anywhere inside. If X does appear inside T, it fails instead of creating an infinite term.

The cost: O(max(|T1|, |T2|)) — significantly slower, because it has to scan entire subterms.

Fixing the lt predicate

% OLD (dangerous):
% lt(M, s(M)).

% NEW (safe):
lt(M, s(N)) :-
    unify_with_occurs_check(M, N).

Now lt(I, I) will correctly fail — no number is less than itself.

The General Rule

Anywhere you have repeated variables in a clause head (like the two M's in lt(M, s(M))), you risk creating infinite terms. You can fix this by replacing the implicit equality with an explicit unify_with_occurs_check.

Why Practical Programmers Don't Use It

The long name unify_with_occurs_check is intentional — it's a style hint that says "don't use this unless you have to." Here's the practical attitude:

Just don't create infinite terms. This is the same discipline as not creating circular data structures in Python. In Python, you can write d['key'] = d to create a circular dictionary, but you shouldn't — it'll cause infinite loops when you try to print or iterate over it. The practical solution is: just don't do that.

Similarly, in Prolog, you almost have to go out of your way to create infinite terms. If you write your predicates carefully and think about what kinds of values they'll receive, you'll naturally avoid the problem.

Related: ==/2 (Pointer Equality)

Prolog also has ==/2, which checks if two terms are literally identical without attempting any unification:

?- X == Y.       % NO — X and Y are different unbound variables
?- X = Y, X == Y. % YES — after unifying, they're the same variable

This is useful for checking variable identity without creating any bindings.

Why does ?- lt(I, I) succeed with the fact lt(M, s(M)) and produce an infinite term?

8. Propositional Logic

Now we shift gears from Prolog-the-language to the mathematical theory that Prolog is built on. Understanding this theory helps you see why Prolog works the way it does — and why it has the limitations it has.

Why learn logic theory?
Prolog is essentially a theorem prover. Your facts and rules are logical statements, and Prolog's execution is a form of logical proof. Understanding the theory tells you what kinds of things Prolog can and cannot prove, and why it sometimes gets stuck in infinite loops.

What Is a Proposition?

A proposition is a statement about the world that is either true or false. Not both. Not "maybe." Just one or the other.

Examples: "It is raining." / "There's a crash on the 405." / "2 + 2 = 4."

We abbreviate propositions with single letters: P, Q, R, etc. This lets us reason about the structure of arguments without getting distracted by what the propositions actually say.

Connectives: Combining Propositions

The power of propositional logic isn't just in individual propositions — it's in how you connect them. The connectors are called connectives.

NOT (Negation): ¬P

The simplest connective. "NOT P" is true when P is false, and false when P is true.

This is counterintuitive!

Look at the first row: when P is false and Q is false, P → Q is true. That feels wrong! If I say "if it's raining, then there's a crash on the 405," and right now it's not raining and there's no crash... my statement is considered true?

Yes. In formal logic, P → Q only promises something about the case when P is true. If P is false, the implication makes no claim — and "making no claim" is treated as "not being violated," which is true.

Think of it as a contract: "If you deliver the product (P), I will pay you (Q)." If you never deliver (P is false), I haven't broken my promise regardless of whether I pay or not.

The Surprise: Implication in C

Every connective we've seen has a direct C/C++ equivalent. What about implication?

Surprisingly, P → Q is equivalent to P <= Q when P and Q are boolean (0 or 1)! Check it: the only case where P <= Q is false is when P=1 and Q=0 — the same as implication.

Most programmers write it as the equivalent !P || Q instead, which is easier to read. A good C compiler will generate the same code for both.

Reverse Implication: P ← Q

"P if Q" — which is just Q → P written the other direction. In Prolog, this is written P :- Q. That's right — Prolog's rule syntax is reverse implication. The :- symbol literally means "if."

Tautologies

A tautology is a propositional formula that is true no matter what truth values you assign to its variables. It says nothing about the real world — it's true by its logical structure alone.

The word comes from ancient Greek and roughly means "saying the same thing twice" — i.e., being redundant.

Examples of Tautologies

Transitivity of implication: (P → Q) ∧ (Q → R) → (P → R)

"If P leads to Q and Q leads to R, then P leads to R." True no matter what P, Q, and R are.

Definition of biconditional: (P → Q) ∧ (Q → P) ↔ (P ↔ Q)

"P implies Q and Q implies P" is the same as "P if and only if Q."

P → P: Anything implies itself. This is the simplest tautology.

Tautologies in Prolog

Remember the rule loop :- loop. from last lecture? That's a tautology — it says "loop is true if loop is true." It adds no information and causes infinite loops. Never put a tautology in your Prolog program. It wastes time and resources.

9. First-Order Logic (Predicate Calculus)

Propositional logic is the foundation, but it has a fatal flaw. To see it, we need to visit ancient Athens...

The Death of Socrates

Socrates, the ancient Greek philosopher, was sentenced to death for "corrupting the youth of Athens" with his philosophical ideas. In the hours before his death (by drinking poison hemlock), he discussed a famous argument with his followers:

The Syllogism of Socrates

Premise 1: All men are mortal.

Premise 2: Socrates is a man.

Conclusion: Therefore, Socrates is mortal.

This argument is obviously valid. If you accept the two premises, the conclusion must follow. And yet, propositional logic cannot prove it.

Why not? In propositional logic, we'd label these as three separate propositions: P = "all men are mortal," Q = "Socrates is a man," R = "Socrates is mortal." The argument becomes "P and Q implies R." But that's not a tautology — it depends on what P, Q, and R actually mean. Propositional logic can't see the internal structure that connects them.

We need something more powerful.

What First-Order Logic Adds

First-order logic (also called the predicate calculus) extends propositional logic with two crucial features:

1. Predicates have arguments. Instead of bare propositions like P, we have predicates like man(X) and mortal(X) that can talk about specific things.

2. Quantifiers. We can say "for all" and "there exists" — making claims about entire collections of things.

The Two Quantifiers

Universal quantifier: ∀X — "For all X." Whatever follows is true no matter what X is.

Existential quantifier: ∃X — "There exists an X." At least one value of X makes the following true.

Solving the Socrates Problem

Now we can express the argument properly:

∀X. man(X) → mortal(X)     % All men are mortal
man(socrates)               % Socrates is a man
────────────────────────
mortal(socrates)            % Therefore, Socrates is mortal

The first line uses the universal quantifier ∀X to say: "For any X, if X is a man, then X is mortal." The variable X is what makes the connection that propositional logic couldn't see. We can substitute "socrates" for X, which gives us man(socrates) → mortal(socrates). Combined with the second line (man(socrates) is true), we conclude mortal(socrates).

This argument is a tautology in first-order logic — it's valid regardless of what "man," "mortal," and "socrates" actually mean.

Connection to Prolog

Notice how this looks just like Prolog! The first line is a Prolog rule: mortal(X) :- man(X). The second line is a Prolog fact: man(socrates). And the conclusion is a Prolog query: ?- mortal(socrates).

Prolog is essentially a theorem prover for a subset of first-order logic. That's the deep connection between the theory and the language.

A Useful Tautology in First-Order Logic

∀X. P(X)  ↔  ¬(∃Y. ¬P(Y))

"P is true for all X" is the same as "there is no Y for which P is false." In other words: something is universally true if and only if you can't find a counterexample. This kind of reasoning is at the heart of formal verification and software engineering proofs.

Why This Matters for Software Engineering

Properties we want to prove about programs — safety ("the system never enters a bad state"), liveness ("the system eventually responds"), timing guarantees ("the brakes respond within 100ms") — can be stated in first-order logic. Propositional logic is not expressive enough for most real-world correctness properties.

The Downside: Undecidability

With greater power comes a harsh tradeoff:

Propositional logic is decidable. Given any propositional formula, you can always determine whether it's a tautology by writing out the truth table (2n rows for n variables). It might be slow (exponential time), but it always terminates.

First-order logic is undecidable. This is Gödel's incompleteness theorem (1931): there exist statements in first-order logic that are true but cannot be proved — ever, by any algorithm. Since first-order logic can express arithmetic, and arithmetic is undecidable, no theorem prover can handle all first-order logic statements.

This is directly connected to Peano's original project. Peano wanted a complete logic for integer arithmetic — where every true statement about integers could be proved. Gödel showed this is impossible.

What This Means for Prolog

Since Prolog operates in the realm of first-order logic:

Prolog is very expressive — you can state complex relationships and Prolog can reason about them.

Prolog can loop forever on some queries — this is unavoidable, it's a consequence of undecidability.

Prolog is not always logically sound (e.g., the infinite term problem without occurs check).

You need to write your programs to stay in the "nice" subset where termination is guaranteed — just like any other language where you can accidentally write infinite loops.

Key Takeaways

The Big Picture

Prolog is a theorem prover for first-order logic. It uses unification — a two-way pattern matching mechanism — as its core computational step. This gives Prolog unique power (bidirectional computation, flexible argument passing) but also unique dangers (infinite terms, infinite loops, sensitivity to direction of use). Understanding the four-port debugging model, the occurs check problem, and the theoretical foundations of logic helps you use Prolog effectively while avoiding its pitfalls.

1. Naive reverse is O(N²); accumulator-based reverse is O(N) — same lesson as in OCaml. Solve a more general problem to get a more efficient solution.

2. Direction of use matters. Code that works perfectly in one direction can loop infinitely in the other. Separating logic from control doesn't mean you can ignore control.

3. The four-port model (Call, Exit, Fail, Redo) is essential for understanding Prolog execution. It has two ways in and two ways out, unlike the two-port model of C or three-port model of exception-handling languages.

4. Unification is two-way pattern matching. Both terms can have variables. This is fundamentally different from one-way pattern matching in OCaml/Python/Haskell.

5. Standard unification can create infinite terms. The occurs check (unify_with_occurs_check/2) prevents this but is slower. In practice, programmers avoid the problem by careful coding style.

6. Propositional logic is too weak to express quantified statements like "all men are mortal." First-order logic (predicate calculus) adds predicates with arguments and quantifiers, making it powerful enough — but also undecidable (Gödel's theorem).

7. Prolog is essentially a theorem prover for first-order logic. Its :- syntax is reverse implication. Its execution is proof search. Its limitations (possible infinite loops, occasional unsoundness) are inherent to the mathematical domain it operates in.
