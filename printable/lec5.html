Lecture 5: Grammars in Practice,
Compilers & Language Systems

How real grammars are written, simplified, and turned into working parsers — and how the code you write becomes a running program.

Why This Lecture Matters

In the previous lectures, you learned what grammars are — rules that describe which strings belong to a language. But knowing the rules is only the first step. In the real world, you need to:

1) Write grammars that are easy to read and don't have thousands of redundant rules.
2) Turn those grammars into parsers — programs that can check whether a piece of text follows the rules.
3) Feed parsed programs through a pipeline (compiler or interpreter) to actually run them.

This lecture connects grammars (the theory) to compilers and interpreters (the practice). By the end, you'll understand the full journey from the text you type in a file all the way to a running process on your machine.

Part 1 — Simplifying Complex Grammars

Quick Grammar Review

Before we dive in, let's make sure we're on the same page about grammars. A grammar is a set of rules that tells you how to build valid strings in a language. Each rule has two sides:

non-terminal  →  sequence of terminals and non-terminals
Key Terms

Terminal — A concrete symbol that actually appears in the final string. Think of it as a "leaf" that can't be expanded further. Examples: the character (, the keyword define, the digit 7.

Non-terminal — A placeholder that gets replaced by other things according to a rule. Think of it as a category. Examples: <expression>, <noun-phrase>, S.

Start symbol — The special non-terminal you begin with. Every valid string in the language can be derived by starting here and applying rules until only terminals remain.

For example, here's the grammar for a Scheme variable definition (Scheme is a programming language in the Lisp family). Scheme uses angle brackets around non-terminals and leaves terminals alone:

<var defn>  →  ( define <var> <expr> )
            |  ( define ( <var> <var>* ) <body> )
            |  ( define ( <var> <var>* . <var> ) <body> )

This says a variable definition can take three forms. Notice the parentheses ( and ) here are terminals — they're actual characters in Scheme code, not grouping symbols for the grammar notation.

The * after <var> means "zero or more of these." So the second form is: open paren, the word define, then an open paren, then one variable followed by zero or more additional variables, close paren, then a body, close paren.

Now, look at that grammar. Do you see the repetition? All three alternatives start with ( define. The second and third are very similar — the only difference is whether there's a dot followed by another variable. When grammars get big (hundreds or thousands of rules, like SQL), this kind of duplication makes them very hard to read. Can we do better?

Railroad Diagrams (Syntax Charts)

Definition — Railroad Diagram

A railroad diagram (also called a syntax chart) is a flowchart-style picture that shows how to build a valid string. You start at the left and follow the arrows to the right. Each path through the diagram produces a valid string. If you reach the end, what you've produced is syntactically valid.

The idea is simple: instead of writing out every alternative as a separate text rule, draw a picture where shared beginnings are merged together, and branches show you where the options diverge.

Here's how the Scheme variable definition looks as a railroad diagram:

                         ┌─── path 1 ──────────────────────────────┐
                         │                                         │
var defn ──→ ( ──→ define ──→ var ──→ expr ─────────────────────→ ) ──→ END
                         │                                         │
                         │  ┌─── path 2 ──────────────────────┐    │
                         └──→ ( ──→ var ──┬──→ ) ──→ body ──→ ) ──→ END
                                    ↑     │
                                    └─────┘   (one or more vars)
                                          │
                                          └──→ . ──→ var ──→ ) ──→ body ──→ ) ──→ END
                                               ↑─── path 3 ───┘
Railroad diagram for Scheme's <var defn>. The three paths correspond to the three grammar alternatives, but the shared ( define prefix appears only once.

Notice what happened: the three separate rules that all started with ( define now share a single beginning. The diagram branches only where the alternatives actually differ. This eliminates the duplication.

Key Insight — Why Railroad Diagrams Help

Reduced duplication: Shared prefixes are merged, so you don't repeat yourself.

Visual clarity: You can immediately see all the paths through a grammar rule at a glance. Many real-world SQL reference manuals use railroad diagrams for exactly this reason — they color different parts to show what's standard SQL versus what's vendor-specific extensions.

Blueprint for parsing: A railroad diagram is essentially a flowchart for a parser program. At each node, your program reads a token and decides which branch to take.

Railroad Diagrams as Parser Blueprints

Here's the really powerful idea. When you look at a railroad diagram, you're not just looking at a grammar — you're looking at a blueprint for writing a parser.

Think about it: a parser is a program that reads tokens one at a time and decides if they form a valid string. A railroad diagram tells you exactly how to do that:

At each node, read the next token

If the diagram says the next thing should be (, read the next token from the input. If it's (, keep going. If it's anything else, output a syntax error.

At each branch point, look ahead

When the diagram splits into multiple paths, peek at the next token to decide which path to take. For example, after ( define, if the next token is an identifier, go down the "path 1" branch. If it's another (, go down the "path 2/3" branch.

Loops become iteration

When the diagram has an arrow looping back, your parser uses a loop (like a while loop) to keep reading tokens until the loop condition ends.

Reaching the end means success

If you follow all the arrows and consume all the required tokens, the input is valid.

Problem — Ambiguous Branch Points

Sometimes a railroad diagram can have a branch point where two paths start with the same token. When that happens, looking at just the next token isn't enough to decide which way to go. This is a real problem for parser implementation.

For example, you could draw the Scheme grammar in a way where both path 2 and path 3 start with ( followed by var — and you'd need to look further ahead to distinguish them. The smarter version of the diagram (the one we drew above) avoids this by sharing the common beginning and only branching where they actually differ.

Compiler courses (like CS 132) deal with exactly this problem: how to build parsers that can handle tricky decision points.

In essence, a railroad diagram is a state machine. Each point on the diagram is a "state." You start in state 0 (the leftmost point), and at each state, you read a token and transition to the next state. If you can reach the final state, the input is valid. This is the same idea you may have seen in CS 35L with finite automata — just with a different visual representation.

Check Your Understanding

What is the main advantage of railroad diagrams over writing out every grammar rule as text?

  • They can express more languages than BNF can
  • They reduce duplication by merging shared prefixes and visually show all paths
  • They eliminate the need for non-terminal symbols entirely
  • They automatically generate parser code without any extra work
Railroad diagrams express exactly the same languages as BNF — they don't add expressive power. Their advantage is presentation: they merge shared structure so you see less duplication, and they visually lay out every valid path, making it easier to read complex grammars and to use them as blueprints for writing parsers.

Part 2 — Regular Languages vs Context-Free Languages

Railroad diagrams lead to an important question: can we always turn a grammar into a simple state machine (or equivalently, a regular expression)? The answer is sometimes yes, sometimes no — and understanding when each is the case is one of the most important ideas in this lecture.

From Grammar to Regular Expression: The RFC 5322 Example

Let's start with a grammar where things work out perfectly. This is from Internet RFC 5322, the standard that defines email message formats. Here's a simplified version of the grammar for a message ID:

msg-id         =  "<" dot-atom-text "@" id-right ">"
id-right       =  dot-atom-text / no-fold-literal
no-fold-literal = "[" *dtext "]"
dot-atom-text  =  1*atext *("." 1*atext)

atext          =  ALPHA / DIGIT / "!" / "#" / "$" / "%" / ...
dtext          =  %d33-90 / %d94-126    ; printable ASCII, no [ ] \

Let's trace through this grammar carefully. The start symbol is msg-id. It refers to dot-atom-text, which refers to atext. And atext is just a list of specific characters — it doesn't refer to any other non-terminal. Similarly, id-right goes to either dot-atom-text or no-fold-literal, and those eventually bottom out in atext or dtext, which are just character classes.

Key Observation

Notice what doesn't happen in this grammar: nothing refers back up. msg-id refers to dot-atom-text, and dot-atom-text refers to atext, and that's the end. You never have a message ID inside another message ID. The grammar always goes "downward" toward simpler and simpler things, never looping back up.

Because the grammar only goes downward, we can "flatten" it. We can replace every non-terminal with what it expands to, and we'll end up with a single railroad diagram that contains only terminal symbols — no non-terminals at all. And a railroad diagram with only terminals and no recursion is exactly what a regular expression is!

Building a Shell Script from the Grammar

Here's the really cool practical consequence. Since this grammar is equivalent to a regular expression, we can translate it directly into a shell script that validates email message IDs using grep:

# Build up regular expressions bottom-up

# atext: letters, digits, and certain special characters
atext='[A-Za-z0-9!#$%&'"'"'*+/=?^_`{|}~-]'

# dtext: printable ASCII chars except [ ] and \
dtext='[!-Z^-~]'

# dot-atom-text: one or more atext, then zero or more groups of (dot + atext)
dot_atom_text="$atext+(\.$atext+)*"

# msg-id: < dot-atom-text @ id-right >
msg_id="<$dot_atom_text@$dot_atom_text>"

# Use grep to find matching lines in a file
grep "$msg_id" file

What we did was work bottom-up: first define the simplest pieces (atext, dtext), then combine them into larger pieces (dot_atom_text), and finally combine everything into the full pattern (msg_id). Each step is just string substitution — we're plugging regex pieces into each other.

And that's it — we have a working parser for email message IDs! It's just a grep command. We started with a grammar, noticed it had no recursion, translated it to a regex, and put it in a shell script.

Where This Breaks Down: The XML Grammar

Now let's look at a grammar where this doesn't work. Here's a simplified version of XML 1.1:

element  ::=  EmptyElemTag  |  STag content ETag
content  ::=  CharData? ((element | Reference) CharData?)*

Do you see the problem? Let's trace the references:

element refers to content

An element can contain content between its start tag and end tag.

content refers back to element

Content can contain other elements inside it!

This is mutual recursion: element → content → element → content → ... It's like two functions calling each other. This means an XML element can contain other XML elements inside it, which can contain even more elements, nested as deep as you want. You can have a <div> inside a <div> inside a <div>, with no limit on depth.

And this is something a regular expression fundamentally cannot handle. A regex (or a finite state machine) has a fixed amount of memory — it has a fixed number of states. But to correctly match nested structures, you need to remember how deep you are. You need to count: "I've seen 5 opening tags, so I need to see 5 closing tags." A finite state machine can't count to an arbitrary number.

Definition — Regular Language

A regular language is one where you can validate any string by scanning left to right without needing to remember nested structures. You can always tell if a string is valid based on a fixed, finite amount of information about what you've seen so far. Regular expressions and finite state machines can recognize exactly this class of languages.

Examples: email message IDs (RFC 5322), the Scheme variable definition grammar above.

Definition — Context-Free Language

A context-free language is one where the validity of a string can depend on nested patterns — matching things that are arbitrarily far apart. To parse these, you need something more powerful than a finite state machine. The theoretical answer is: a finite state machine plus a stack (which gives you unlimited memory for tracking nesting depth).

Examples: XML, HTML, most programming languages (because of nested parentheses, nested if-statements, etc.).

Three Types of Recursion in Grammars

So recursion is the key. But here's the subtle point: not all recursion makes a grammar context-free. Some kinds of recursion are just fancy ways of writing repetition, and repetition is something regular expressions can handle. Let's look at the three types:

1. Left Recursion

S → S a
S → a

Here, S refers to itself, but always at the very beginning (the left side) of the right-hand side. What does this generate?

Start with S. Apply rule 2: we get a. Or apply rule 1 then rule 2: we get aa. Or apply rule 1 twice then rule 2: aaa. The pattern is: one or more a's. That's just a+ in regex — perfectly regular!

S ──→ a ──┬──→ END
          │
          └──→ a ──┘  (loop back)
Left recursion becomes a simple loop in a railroad diagram — equivalent to a+.

2. Right Recursion

S → a S
S → a

Now S refers to itself at the very end (the right side). What does this generate? Exactly the same thing: a, aa, aaa, ... It's still a+. Also perfectly regular.

3. Middle Recursion (the trouble-maker)

S → a S b
S → a b

Now S refers to itself in the middle — there's stuff both before it (a) and after it (b). What does this generate?

Apply rule 2: ab. Apply rule 1 then rule 2: aabb. Apply rule 1 twice then rule 2: aaabbb. The pattern is: n copies of a followed by n copies of b, written as anbn where n ≥ 1.

This is not regular. To know if a string like aaaabbbb is valid, you have to count the a's and then verify you see exactly the same number of b's. A finite state machine can't do that counting for arbitrary n.

The Rule of Thumb

Left recursion (the recursive call is at the start of the rule) → can be converted to a loop → regular.

Right recursion (the recursive call is at the end of the rule) → can be converted to a loop → regular.

Middle recursion (the recursive call has stuff on both sides) → requires matching/counting → context-free, not regular.

This is exactly what's happening with XML: element → STag content ETag has content in the middle, between the start tag and the end tag. That's middle recursion, which is why XML is context-free and can't be parsed with regex.

The theoretical way to handle context-free languages is to take a finite state machine and attach a stack to it. Every time you see an opening tag, push it onto the stack. Every time you see a closing tag, pop from the stack and verify it matches. The stack gives you unlimited memory for tracking nesting depth. This is called a pushdown automaton, and it's what real parsers for programming languages are based on.

Check Your Understanding

Consider the grammar: S → ( S ) and S → x. What kind of recursion is this, and is the language regular?

  • Left recursion — regular
  • Right recursion — regular
  • Middle recursion — NOT regular (context-free)
  • No recursion — regular
The recursive reference S appears in the middle of the right-hand side, with ( before it and ) after it. This is middle recursion, generating strings like x, (x), ((x)), (((x))) — matched parentheses nested to arbitrary depth. You need to count opening parens and match them with closing parens, which a finite state machine can't do. This is a context-free language.

Part 3 — ISO EBNF: A Standard Notation for Grammars

As you've seen, every community seems to invent its own notation for grammars. The RFC people have one, the XML people have another, textbooks use yet another. This creates a mess. So the International Organization for Standardization (ISO) — the same organization that standardizes everything from Java to electrical plugs — created a standard grammar notation called ISO EBNF (Extended Backus-Naur Form).

The Notation

Here are the key elements of ISO EBNF, presented in order from highest precedence (binds tightest) to lowest precedence (binds loosest):

SyntaxMeaningExample
"text" or 'text'Terminal symbol (literal text). Both quote styles work — a compromise between British and Continental European convention.'define', "("
[ ... ]Optional (0 or 1 times)[ '+' ] means the + sign can appear or not
{ ... }Repetition (0 or more times){ digit } means zero or more digits
( ... )Grouping (affects precedence)( 'a' | 'b' )
(* ... *)Comment (from Pascal tradition)(* this is a comment *)
*Repetition factor (integer * primary)3 * 'a' means aaa
-Exception / set difference: matches A but not Bletter - vowel
,Concatenation (in most BNF you just put things next to each other, but ISO EBNF uses an explicit comma)'a' , 'b' means ab
|Alternation (or)'a' | 'b'
=Rule definitiondigit = '0' | '1' | ... | '9' ;
;Rule terminatorEvery rule ends with a semicolon

One unusual thing: ISO EBNF allows spaces in identifiers. So syntax rule is a single non-terminal name, not two separate things.

The Exception Operator (-)

The - (exception) operator is unique and worth understanding. A - B means "anything that matches A, except things that also match B." It's a set difference.

Example — Identifiers in C

In C, an identifier is a letter or underscore followed by letters, digits, or underscores. But certain identifiers are reserved words (like if, while, for) and aren't valid as variable names. With the exception operator:

identifier = letter, { letter | digit | '_' }
           - 'if' - 'while' - 'for' - 'return' - ... ;

This elegantly says "anything that looks like an identifier, minus the reserved keywords."

Caveat: The language theorists are nervous about the exception operator because it can make grammars very hard to parse. For this reason, there's a restriction: the thing on the right side of - should be something simple enough to match with a regular expression.

A Grammar That Describes Itself

Here's where things get philosophical. We can write the grammar for ISO EBNF in ISO EBNF. That is, the rules that define what valid ISO EBNF looks like are themselves written in ISO EBNF:

syntax        = syntax rule , { syntax rule } ;

syntax rule   = meta id , '=' , defns list , ';' ;

defns list    = definition , { '|' , definition } ;

definition    = term , { ',' , term } ;

term          = factor , [ '-' , exception ] ;

exception     = factor ;

factor        = [ integer , '*' ] , primary ;

primary       = '(' , defns list , ')'
              | '[' , defns list , ']'
              | '{' , defns list , '}'
              | meta id
              | terminal string ;

Look at the second rule: syntax rule = meta id , '=' , defns list , ';' ;

This rule describes what a syntax rule looks like — and it is itself a syntax rule. It has four parts: a meta id (syntax rule), an equals sign (=), a definitions list (meta id , '=' , defns list , ';'), and a semicolon (;). The rule perfectly describes its own structure!

The Self-Reference Paradox

You might wonder: isn't defining something in terms of itself circular? If you need to already know ISO EBNF to read the grammar for ISO EBNF, haven't you explained nothing?

The ISO standards committee knew this. Their solution: the official definition of ISO EBNF is written in English, earlier in the document. The self-describing grammar is placed in an appendix, with a note saying: "If you understood the English explanation, this grammar should match it. If there's any conflict, trust the English." The grammar is a convenient formal summary for people who already understand the notation.

This pattern — using a language to describe itself — comes up repeatedly in computer science. It's like looking in a mirror. The formal grammar can be fed into a parser generator to automatically produce a parser for ISO EBNF, something you can't do with the English description.

Also notice: defns list contains definition, which contains term, which contains factor, which contains primary, and primary can contain defns list again (inside parentheses, brackets, or braces). This is middle recursion! ISO EBNF is itself a context-free language — you can nest groupings inside groupings inside groupings. So you couldn't parse ISO EBNF with just a regular expression.

Check Your Understanding

In ISO EBNF, what does expression - identifier mean?

  • Subtract the value of identifier from expression
  • Anything that matches expression, except if it also matches identifier
  • A sequence: expression followed by identifier
  • expression OR identifier
The - operator in ISO EBNF means "exception" or "set difference." A - B matches anything in the set of strings described by A, as long as it's not also in the set described by B. So expression - identifier matches any expression that isn't a bare identifier — it must be something more complex.

Part 4 — Trouble with Grammars

Grammars are, in a sense, a kind of programming language — you're writing rules that define behavior (what strings are valid). And just like any programming language, you can write buggy grammars. Some bugs are obvious, some are subtle. Let's go through the categories.

Trouble 1: Undefined Non-Terminals (Used but Not Defined)

This is like calling a function that doesn't exist in your code.

S → S a
S → T b    ← what is T? It's never defined!
S → c

The non-terminal T appears on the right-hand side of a rule (it's used), but no rule has T on the left-hand side (it's never defined). If you try to derive a sentence and you use rule 2, you get stuck with T in your string — and there's no rule to replace it with anything.

This means the rule S → T b can never produce a valid terminal string. It's a dead-end rule. You can simply remove it without changing the language:

(* Fixed: remove the useless rule *)
S → S a
S → c

Trouble 2: Unreachable Non-Terminals (Defined but Not Reachable)

This is like writing a function that nothing in your program ever calls.

S → S a
S → c
T → T b    ← T is defined, but no rule mentions T on its right side!

Here, T has a definition, but no rule reachable from the start symbol S ever produces T. Since you always start deriving from S, and S only goes to things involving S, a, and c, you can never reach T. The rule for T is dead code.

But it's slightly more subtle than just "defined but not used." Consider:

S → S a
S → T b
T → T c
S → c

Here, T is both defined (rule 3) and used (rule 2). But rule 3 says T → T c — the only way to expand T is to get another T plus a c. There's no "base case" for T. You'll generate Tc, then Tcc, then Tccc, but you'll always have a T left over. T can never produce a string of pure terminals.

So the real criteria are:

Two Necessary Properties

Reachable from the top: You must be able to get to the non-terminal starting from the start symbol, following rules.

Reachable to the bottom: The non-terminal must be able to eventually produce a string of pure terminal symbols (no non-terminals left over).

If a non-terminal fails either test, any rule involving it is useless and can be removed.

Trouble 3: Never-Ending Derivations

This is the problem we just saw with T → T c. Let's be explicit:

(* Trouble: T can never finish *)
S → S a
S → T b
T → T c     ← T always produces another T
S → c

If you apply rule 2, you get Tb. Then the only way to expand T is rule 3: Tcb. Again: Tccb. You're stuck in an infinite loop. T will never disappear from your string.

The fix: either remove the offending rules, or add a base case for T:

(* Option A: remove rules involving T *)
S → S a
S → c

(* Option B: add a base case for T *)
S → S a
S → T b
T → T c
T → d      ← NOW T can produce a terminal string!
S → c

Trouble 4: Extra Constraints the Grammar Can't Capture

Sometimes a grammar is technically correct — it describes a set of strings — but it's too generous. It allows strings that shouldn't be allowed, because there are extra rules that the grammar notation can't express.

The English Example

Consider this tiny grammar for English sentences:

S  → NP VP .
NP → N
NP → Adj NP
VP → V
VP → VP Adv

This generates noun phrases (NP) made of adjectives and nouns, and verb phrases (VP) made of verbs and adverbs. Perfectly fine — except it allows:

"blue dogs barks loudly."

This follows all the grammar rules perfectly: "blue dogs" is a valid NP (adjective + noun), "barks loudly" is a valid VP (verb + adverb). But it's wrong in English because "dogs" is plural and "barks" is singular. They don't agree in number.

Fixing It (and the Cost)

You can fix this by splitting every non-terminal into singular and plural versions:

S  → SingNP SingVP .
S  → PlurNP PlurVP .
SingNP → SingN
SingNP → Adj SingNP
PlurNP → PlurN
PlurNP → Adj PlurNP
SingVP → SingV
SingVP → SingVP Adv
PlurVP → PlurV
PlurVP → PlurVP Adv

We went from 5 rules to 10. The grammar doubled in size. And English only has two numbers (singular and plural). Many other languages have additional features that verbs and nouns must agree on:

FeatureValuesMultiplicative Factor
Numbersingular, plural×2
Gendermasculine, feminine, neuter×3
Casenominative, accusative, dative, ...×4+

With just number and gender, you'd need 2 × 3 = 6 versions of every rule. With 10 features, you'd need 1000+ rules where you started with 1. This combinatorial explosion makes grammars impractical for encoding all constraints. That's why real language processing systems handle things like subject-verb agreement outside the grammar, using separate type-checking or constraint systems.

The Same Problem in Programming Languages

You see the same issue in programming languages. For example:

cos("hello")   // syntactically valid function call, but type error!

The grammar says this is a valid function call: identifier, open paren, expression, close paren. But semantically, you can't take the cosine of a string. This type mismatch isn't caught by the grammar — it's caught by the type checker, which is a separate phase.

The Scylla and Charybdis of Grammars

(Scylla and Charybdis were two monsters from Greek mythology on opposite sides of a strait — avoid one and you hit the other.)

Too generous: The grammar allows invalid strings and you need extra machinery (type checking, constraint checking) to reject them.

Too detailed: The grammar tries to capture every constraint, making it explosively large and hard to read.

Real language designers try to find a sweet spot between these extremes. The grammar handles the structural/syntactic stuff, and separate mechanisms handle type checking and other constraints.

Trouble 5: Ambiguity and How Prolog Avoids It

An ambiguous grammar allows a string to be parsed in more than one way. This is a problem because the parse tree determines the meaning of the expression. If 3 + 4 * 5 can be parsed as either (3 + 4) * 5 or 3 + (4 * 5), you don't know what the expression means.

We've seen that you fix ambiguity with precedence (which operation binds tighter) and associativity (when two operators have the same precedence, do you group left or right). Most languages hard-code these rules. But Prolog takes a different approach: it lets you define your own operators with your own precedence and associativity.

Prolog's Operator Definitions

:- op(100, yf,  [!]).       (* factorial: suffix operator *)
:- op(200, xfy, [**]).      (* exponentiation: right associative *)
:- op(400, yfx, [*, /]).    (* multiply/divide: left associative *)
:- op(500, yfx, [+, -]).    (* add/subtract: left associative *)
:- op(700, xfx, [<, >, =<, >=, ==]).  (* comparisons: non-associative *)

Let's break down what each part means:

The number is the precedence level. Lower numbers mean higher precedence (do it first). So ! at 100 has the highest precedence (applied first), and comparisons at 700 have the lowest (applied last).

The letter code tells you the associativity. f represents the operator itself, and x/y represent operands:

CodeMeaningAssociativity
yfxy (can have equal precedence) on left, x (must have strictly higher precedence) on rightLeft associative — groups from the left: a - b - c = (a - b) - c
xfyx on left, y on rightRight associative — groups from the right: a ** b ** c = a ** (b ** c)
xfxx on both sidesNon-associative — chaining is a syntax error: a < b < c is illegal
yfy on left, nothing on rightSuffix unary operator: 5!
fyNothing on left, y on rightPrefix unary operator: -x

The key distinction between x and y: y means the operand can have the same precedence level as this operator (allowing chaining). x means the operand must have strictly higher precedence (preventing chaining at the same level).

Why This Eliminates Ambiguity

With this system, every operator has a unique numeric precedence and a definite associativity (left, right, or none). There's no way to write an ambiguous expression — the rules always determine exactly one parse. Even user-defined operators get the same treatment.

Compare this to C++, where you can overload existing operators (give them new meanings) but can't define new operators or change the built-in precedence rules. The C++ designers felt that letting users change precedence would be too confusing. Prolog's designers disagreed — they trusted programmers to design sensible notation.

Example — Non-associativity in Prolog vs C++

In Prolog: x < y < z is a syntax error because < is declared as xfx (non-associative).

In C++: x < y < z is legal because < is left-associative. It means (x < y) < z, which evaluates x < y to 0 or 1, then compares that 0 or 1 to z. This is almost never what the programmer intended!

Prolog's approach is arguably safer — it forces you to write (x < y) and (y < z) explicitly.

  • Prefix operators with precedence 400
yfx means: y (operand allowing same precedence) on the left, f (the operator) in the middle, x (operand requiring strictly higher precedence) on the right. The y on the left means you can chain: a * b * c groups as (a * b) * c because the left operand of the second * is allowed to be another multiplication. That's left associativity.

Part 5 — From Source Code to Running Program

Now let's zoom out. We've spent time on grammars and parsing — but parsing is just one step in turning your source code into a running program. Let's trace the full journey.

The Compiler Pipeline

A compiler is like a factory with multiple stations. Each station transforms the program into a slightly different representation, getting closer and closer to something the hardware can execute.

Input: Byte Sequence

Your source file is just bytes on disk. For example, int x = 3; is stored as the bytes 105 110 116 32 120 32 61 32 51 59 (ASCII values). The first thing the compiler does is decode these bytes into characters.

Character Sequence

The bytes get decoded (using ASCII, UTF-8, etc.) into a sequence of characters. Now you have letters, digits, spaces, and symbols. Unicode makes this interesting — characters can be encoded in multiple bytes, some characters look alike but mean different things, and the same character can sometimes have two different byte representations.

Lexical Analysis → Token Sequence

The character sequence gets broken into tokens. This strips away whitespace and comments, and groups characters into meaningful units. For int x = 3;, the tokens are: INT, ID(x), =, NUM(3), ;. There are a relatively small number of distinct token types — maybe 80-100 for a typical language. All identifiers are the same token type, all numbers are the same token type, etc.

Parsing → Parse Tree (or Reductions)

The parser takes the token sequence and checks it against the grammar rules, building a parse tree — a tree structure where each internal node represents a grammar rule that was applied. This enforces precedence, associativity, nesting, and all the structural rules of the language.

For efficiency, many compilers don't actually build the full tree in memory. Instead, they output a sequence of reductions — essentially a list of rule numbers: "apply rule 12, then rule 27, then rule 13." If you apply each rule to the leftmost unexpanded non-terminal, this sequence uniquely determines the parse tree.

Intermediate Code Generation

The compiler translates the parse tree into intermediate code — instructions for an abstract machine that doesn't correspond to any real hardware. This is a key design decision. The intermediate code is chosen to be convenient for the programming language, not for any specific processor.

Optimization (optional)

Optimization phases take the intermediate code and try to make it faster or use less memory. Examples include removing redundant calculations, unrolling loops, and eliminating dead code. This happens at the intermediate code level so it benefits all target architectures.

Code Generation → Assembly Language

The compiler generates assembly code targeted at a specific processor architecture (x86-64, ARM, RISC-V, etc.). This is now machine-specific.

Assembler → Object File (.o)

The assembler turns assembly language into an object file containing actual machine code (binary instructions). But the object file has "holes" — references to functions and variables that are defined in other files. The object file says, "I need the address of getchar — please fill it in."

Linker (ld) → Executable

The linker takes all the object files, combines them, and fills in all the holes. It resolves every reference to an external function or variable, replacing symbolic names with actual memory addresses. The result is an executable file with no unresolved references.

Kernel Loader → Running Process

When you run the executable, the operating system's loader reads it into memory, sets up the stack and heap, assigns a program counter, and starts executing. Your program is now a living process.

The Frontend / Backend Split

There's a beautiful modularity trick that almost all major compilers use. The intermediate code acts as a dividing line:

     FRONT ENDS                    BACK ENDS
  ┌──────────┐                  ┌──────────────┐
  │   C       │──┐          ┌──→│  x86-64      │
  └──────────┘   │          │   └──────────────┘
  ┌──────────┐   │   ┌───────┐  ┌──────────────┐
  │  C++     │──→│──→│ IR    │──→│  ARM         │
  └──────────┘   │   └───────┘  └──────────────┘
  ┌──────────┐   │          │   ┌──────────────┐
  │  Java    │──┘          └──→│  RISC-V       │
  └──────────┘                  └──────────────┘
IR = Intermediate Representation. Without this split, you'd need 3 × 3 = 9 complete compilers. With it, you need 3 front ends + 3 back ends = 6 half-compilers. The savings grow with more languages and architectures.

Each front end handles lexing, parsing, and intermediate code generation for one language. Each back end handles code generation and optimization for one hardware architecture. This is how GCC and Clang work — they support many languages and many architectures without writing a separate compiler for every combination.

Interpreters: A Different Approach

Compilers go through all those steps before your program runs. An interpreter takes a different approach: it combines the front-end steps (lexing, parsing, intermediate code generation) and then directly executes the intermediate code, without ever producing machine code.

CompilerInterpreter

Runtime speedFast (executes native machine code)Slower (2-10x) because intermediate code is simulated in software
DebuggingHarder (compiled code is far from source)Easier (intermediate code is close to source)

ExamplesC, C++, Rust, GoPython, Ruby, early JavaScript

JIT Compilation: The Best of Both Worlds

Modern systems often use a hybrid approach called Just-In-Time (JIT) compilation. The idea:

Start as an interpreter

Run the program using an interpreter for fast startup and easy debugging.

Profile during execution

While running, collect data about which functions get called frequently ("hot code"), what types of arguments they receive, which branches are taken, etc. This is called profiling.

Compile hot code to machine code

When a function has been called enough times, the JIT compiler kicks in and compiles it from intermediate code to native machine code — while the program is still running. From then on, calls to that function execute at near-native speed.

Example — The Java Virtual Machine (JVM)

Java compiles source code to bytecode (intermediate code for the JVM). The JVM starts by interpreting bytecode. As it runs, it profiles the code and identifies hot methods. For those methods, the JIT compiler generates optimized machine code.

The JIT can even do things traditional compilers can't: it can specialize methods based on runtime types. If a method is usually called with String arguments, the JIT can generate one version of machine code optimized for Strings. If it's later called with Thread arguments, it generates a different optimized version. This can sometimes make JIT-compiled code faster than statically compiled C++ for workloads with predictable patterns.

You'll find JIT compilers in: Firefox and Chrome (for JavaScript), the JVM (for Java/Kotlin/Scala), .NET (for C#), and even Emacs (for Emacs Lisp).

ModelStartup TimeLong-Term Speed
InterpreterVery fastSlow
CompilerSlowFast
JITFastFast (after profiling)

Dynamic Linking

In the classical model, the linker combines all object files into one big executable before you run it. Dynamic linking changes this: some of the linking happens while the program is running.

Definition — Static vs Dynamic Linking

Static linking: All code is combined at compile/link time into one executable. Result: big file, but completely self-contained. No surprises at runtime.

Dynamic linking: Libraries (shared objects / DLLs) are stored in separate files. They get loaded and linked when the program starts (or even later, on demand). The executable file is smaller because it doesn't include the library code.

Pros of Dynamic LinkingCons of Dynamic Linking
Smaller executables (library code isn't duplicated in every program)"DLL Hell" — your program worked with library v1.0 but breaks with v1.1
Easy to patch: fix a security hole in the C library, and all programs benefit immediately without recompilationRuntime failures if the library isn't found or has incompatible changes
Multiple programs can share one copy of a library in memoryType safety concerns — the dynamic linker doesn't do type checking, so mismatches can cause silent bugs

Python's from __future__ import something is a related idea — it says "this code uses a feature from a future version of Python; please make it available." It's the Python community's way of managing version compatibility, similar to the library version issues in dynamic linking.

Check Your Understanding

Why do compilers use an intermediate representation (IR) between the front end and back end?

  • To make the compiled program run faster
  • To enable modularity — any language front end can pair with any architecture back end
  • Because hardware can directly execute IR
  • To make the source code easier to read
The IR is the boundary between language-specific concerns (handled by front ends) and machine-specific concerns (handled by back ends). Instead of writing N × M compilers for N languages and M architectures, you write N front ends + M back ends = N + M half-compilers. The IR doesn't run faster by itself and isn't directly executable by hardware.

Part 6 — Types (Preview)

What Are Types?

The lecture ends with a brief introduction to types, which will be covered more fully in upcoming lectures. Here's the setup:

We've established that grammars handle the syntax (structure) of a language but can't capture all constraints (like type checking). Types are the main mechanism for capturing those additional constraints.

Two Definitions of Type

Concrete definition: A type tells you what a byte sequence represents. The bytes 01000001 could mean the integer 65, or the character 'A', or part of a floating-point number. The type tells you which interpretation is correct.

Abstract definition: A type is a category of data values together with the operations that make sense on those values. For example, the type "integer" includes values like 0, 1, -42, and operations like +, -, *, /. The type "string" includes values like "hello" and operations like concatenation and substring.

Types connect to our earlier discussion about grammars being "too generous." The grammar allows cos("hello") syntactically, but the type system rejects it because cos expects a numeric argument, not a string. Types are the extra layer of checking that sits on top of the grammar.

Summary — The Big Picture

How Everything Connects

Grammars define the valid structure of a language. They can be written in text (BNF, EBNF) or diagrams (railroad diagrams). Diagrams reduce duplication and serve as blueprints for parsers.

Regular vs Context-Free: If a grammar has no recursion (or only left/right recursion), it describes a regular language that can be handled with regex and finite state machines. If it has middle recursion (nesting), it's context-free and needs a parser with a stack.

ISO EBNF is a standardized grammar notation with features like the exception operator. It can even describe itself, illustrating the idea of self-referential formal systems.

Grammar troubles include undefined non-terminals, unreachable rules, never-ending derivations, extra constraints the grammar can't capture, and ambiguity. These mirror the kinds of bugs you can write in code — calling undefined functions, writing dead code, creating infinite loops, missing validation logic, and creating ambiguous control flow.

The compiler pipeline transforms source code through many stages: bytes → characters → tokens → parse tree → intermediate code → assembly → object files → executables → running processes. The intermediate representation enables modularity: front ends (one per language) pair with back ends (one per architecture).

Interpreters skip the back-end stages and directly execute intermediate code, trading runtime speed for fast startup and easy debugging. JIT compilers combine both approaches, using profiling to selectively compile hot code paths.

Dynamic linking defers part of the linking to runtime, enabling smaller executables and easier patching at the cost of potential version-mismatch issues.

Types fill the gap that grammars leave — they express constraints about data values and operations that can't be encoded in syntax rules without combinatorial explosion.

Key Definitions Cheat Sheet

Railroad Diagram — A flowchart for generating valid strings; each path = one valid form.

Regular Language — Validatable by scanning left-to-right with finite memory; no nesting.

Context-Free Language — May involve arbitrarily deep nesting; needs a stack to parse.

Left/Right Recursion — Recursion at the start or end of a rule; equivalent to repetition (regular).

Middle Recursion — Recursion with stuff on both sides; creates nesting (context-free).

ISO EBNF — International standard notation for grammars; uses , for concatenation, - for exception.

Lexical Analysis — Turning characters into tokens.

Parsing — Turning tokens into a parse tree by checking grammar rules.

Intermediate Code — Instructions for an abstract machine; the boundary between front end and back end.

Object File — Machine code with unresolved references ("holes").

Linker — Fills the holes in object files to produce an executable.

Loader — Puts an executable into memory and starts it running.

JIT Compiler — Interprets at first, then compiles hot code to machine code at runtime.

Profiling — Collecting runtime statistics (call counts, types, branches) to guide optimization.

Dynamic Linking — Loading and linking libraries at runtime instead of compile time.

Type — Concretely: what a byte sequence represents. Abstractly: a category of values + their operations.
