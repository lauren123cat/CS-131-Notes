Concurrency in Java

How multiple independent threads cooperate to solve problems — from hardware foundations to the Java Memory Model.

Section 01

The Big Picture: Why Concurrency?

Imagine you're cooking a big dinner. You could do everything one step at a time — chop vegetables, then boil water, then sauté onions, then set the table. But that's slow. A smarter approach is to do several things at once: while the water boils, you chop vegetables; while things cook, you set the table. That's the core idea behind concurrency — having multiple activities progress together rather than one-at-a-time.

Key Definition
Concurrency is how multiple independent threads of execution work together to solve a problem. It's not just about running things at the same time (that's parallelism). Concurrency is about coordinating multiple activities so they cooperate correctly.

In the world of computers, concurrency matters because modern machines have multiple CPU cores, and we want to use all of them — not let some sit idle. But getting multiple pieces of code to work together without stepping on each other's toes is one of the hardest problems in all of computer science. That's what this lecture is about.

Why This Matters
Every time you use a web browser, a phone app, or a video game, concurrency is happening behind the scenes. One thread handles your clicks, another fetches data from the network, another renders graphics. Without concurrency, your apps would freeze every time they waited for the network. Understanding how this works — and what can go wrong — is fundamental to writing reliable software.

This lecture covers Java's approach to concurrency specifically. Java was designed from the ground up with concurrency in mind, and it made specific choices about how threads communicate that shape everything we'll learn. Before we look at Java's tools, we need to understand the hardware landscape they were designed for.

Section 02

Two Kinds of Parallelism: SIMD vs MIMD

At the hardware level, there are two fundamentally different ways to do things in parallel. Understanding these two approaches is important because Java deliberately chose one of them, and that choice shapes the entire programming model.

SIMD: Single Instruction, Multiple Data

Imagine a drill sergeant shouting a single command: "Everyone, do ten push-ups!" Every soldier does the same exercise at the same time. That's SIMD. There's one instruction (the command) applied to many pieces of data (each soldier) simultaneously.

Key Definition
SIMD (Single Instruction Multiple Data): One instruction is applied simultaneously to many independent pieces of data. The data differs, but the instruction is the same — every "lane" follows the same control flow.

A concrete example: suppose you have a million numbers and you want to add 1 to each of them. With SIMD, you issue one "add 1" instruction and it gets applied to all million numbers at the same time. This is incredibly efficient for certain workloads — especially the kind of math used in graphics (which is why GPUs are built around SIMD). When you see a GPU running a machine learning model, it's largely doing SIMD: applying the same matrix multiplication operation across huge arrays of numbers simultaneously.

The limitation? SIMD only works when every piece of data needs the same operation. If you need different pieces of your program to do different things, SIMD can't handle that.

MIMD: Multiple Instruction, Multiple Data

Now imagine an office where each worker has their own task list. One person handles emails, another processes spreadsheets, a third manages the calendar. They're all working at the same time, but each is doing something different. That's MIMD.

Key Definition
MIMD (Multiple Instruction Multiple Data): Different processors or threads execute different instructions on different data. Each thread has its own instruction pointer, and their control flows can diverge — meaning each can make its own decisions about what to do next.

This is what CPUs do. Each CPU core can run a completely different program, or a different part of the same program. It's much more flexible than SIMD, but also less efficient for the kind of uniform math that SIMD excels at.

Real-World Analogy — SIMD vs MIMD

SIMD is like a factory assembly line where every worker does the exact same task on different items passing by. Super fast for that one task.

MIMD is like a restaurant kitchen where the chef chops vegetables, the sous chef makes sauce, and the dishwasher cleans plates. Everyone does something different, and they coordinate to get dinner served.

Why Java Chose MIMD

Java programs are about control, logic, coordination, and managing state — not uniform math on huge arrays. A web server needs one thread handling user login, another writing to a database, another talking to the network. These are all fundamentally different tasks. That's MIMD territory.

Java does have some SIMD support, but it's not the core of the language. From a programming languages perspective, the interesting part is the MIMD side — how Java lets you create multiple threads, each with their own instruction pointer, and coordinate them.

Connecting the Dots
Modern high-end machines (your phone, your laptop, supercomputers like El Capitan) actually use both SIMD and MIMD. The CPU cores handle the "smart" part — the control logic that decides what to do next. The GPU handles the "worker bee" part — crunching through massive amounts of identical math. Java's threading model handles the CPU/MIMD side of this picture. If you're into machine learning, the GPU/SIMD side is what frameworks like PyTorch manage — but someone still needs to tell the GPU what to compute, and that coordination is MIMD.

Section 03

How Threads Talk: Message Passing vs Shared Memory

Now we know Java uses MIMD — multiple threads each doing their own thing. But here's the critical question: when those threads need to cooperate (which is the whole point of concurrency), how do they communicate?

There are two fundamental approaches, and Java's choice here affects everything from how you write code to what kinds of bugs you'll encounter.

Approach 1: Message Passing

Think of how email works. When you want to tell a colleague something, you compose a complete message, address it, and send it. Your colleague receives the full message in their inbox. You don't share a desk — you each have your own workspace, and all communication happens through these discrete messages.

Key Definition
Message Passing: Threads communicate by sending complete messages through a channel. Each thread has its own private memory — they never directly touch each other's data.

This is extremely common on the internet. Think about how a web browser (client) talks to a web server: it constructs an HTTP request (maybe with JSON data), sends it across the network, and the server receives the full message and responds. Node.js, for example, is primarily about this model — sending messages between clients and servers.

Advantages: Since threads never share memory, there's no risk of one thread accidentally corrupting another's data. It's easier to reason about correctness. Disadvantage: Constructing, sending, and receiving messages has overhead. You have to serialize all the data (convert it to bytes), send it, then deserialize it on the other end.

Approach 2: Shared Memory

Now imagine two colleagues sharing the same whiteboard. Instead of writing each other messages, one person writes a number on the whiteboard, and the other person just walks over and reads it. Communication is instant — no message to construct, no delivery delay.

Key Definition
Shared Memory: Threads communicate by reading from and writing to the same memory locations. All threads can see the same objects. One thread says "I've stored XYZ at location ABC, go pick it up whenever."

This is Java's approach. When Java was developed in the 1990s, shared memory was simply faster — especially on machines where multiple CPU cores are attached to the same physical RAM. No need to construct messages, serialize data, or wait for delivery. Just write to a memory location, and the other thread can read it.

The Catch
Shared memory is faster, but it's also much more dangerous. If two threads write to the same memory location at the same time, the data can get corrupted. If one thread reads while another is halfway through writing, it might see garbage. These bugs — called race conditions — are among the hardest bugs to find in all of software engineering. We'll spend a large part of this lecture learning how Java tries to prevent them.
Why this analogy matters

Message Passing = passing notes in class. Safe (only you can read your note) but slow.

Shared Memory = a shared whiteboard. Fast (just look at it) but chaotic if two people write at once.

Java chose the whiteboard. The rest of this lecture is about the rules for using the whiteboard without chaos.

Right! When two threads share the same memory, they can both try to modify the same variable at the same time. This leads to race conditions — the results depend on which thread happens to run first. With message passing, each thread has private data, so this problem doesn't arise.

Section 04

Symmetric Multiprocessing (SMP)

Before we dive into Java's threading tools, there's one more piece of the hardware picture we need. Java assumes a specific kind of hardware setup called Symmetric Multiprocessing, and this assumption simplifies the programming model significantly.

Key Definition
Symmetric Multiprocessing (SMP) is a hardware architecture where multiple CPUs (or CPU cores) are connected to shared main memory (RAM), and all CPUs have roughly equal speed and equal access to that RAM.

Picture it like this: you have several equally-capable workers (the CPU cores) all sitting around the same table (the shared RAM). Any worker can reach any part of the table equally quickly. There's no "this corner is closer to me" — everyone has symmetric access.

Why does this matter for programming? Because it means you don't have to think about which CPU your thread runs on. Any thread can run on any CPU, and any thread can access any piece of memory at roughly the same speed. This simplifies programming enormously — you just write your threads and let the operating system figure out which CPU to run them on.

Connecting the Dots

In reality, this simple SMP model is starting to break down. Modern phones and laptops have "big" performance cores and "little" efficiency cores — they're not all equal. But even there, the key SMP assumption still holds: all cores can talk to all RAM at roughly the same speed. There are more exotic architectures where different CPUs have different memory speeds (called NUMA), but we won't worry about those.

Bottom line: Java's memory and thread model assumes SMP. This means when you write Java threading code, you can think of all threads as having equal access to all shared data.

Section 05

Processes, Threads, and Processors

We've established the hardware landscape. Now let's get precise about three terms that are easy to confuse: processes, threads, and processors. These are the building blocks of everything that follows.

What is a Process?

A process is a running program together with all the resources it needs. When you double-click an application on your computer, the operating system creates a process for it. That process includes:

Key Definition
Process: A running program plus all of its resources — its own private block of memory (RAM), file descriptors (tracking which files it has open), the current working directory, and everything else the program needs to operate.

The crucial property of processes is isolation. Different processes on the same machine cannot see each other's memory. This is a security feature — you don't want your web browser to be able to read your banking app's memory. If one process goes haywire and crashes, other processes are protected. You can just kill the bad process (kill 5931 on Linux) and everything else keeps running.

What is a Thread?

A thread is like a slimmed-down process. It's a unit of execution that lives inside a process.

Key Definition
Thread: A unit of execution within a process, consisting of its own instruction pointer (which line of code it's currently on), its own stack (for local variables and function calls), and its own stack pointer. But threads within the same process share the process's heap memory.

This is the key difference. Processes are isolated — they have separate memory. Threads within the same process share memory — they can all see and modify the same objects on the heap. This is exactly the shared memory model we discussed: Java threads communicate by reading and writing to the same shared heap.

Analogy — Process vs Thread

A process is like an apartment. Each apartment has its own kitchen, bathroom, and living space. Neighbors can't walk into your apartment.

Threads are like roommates in the same apartment. They each have their own toothbrush (instruction pointer, stack) but they share the kitchen and living room (heap memory). If one roommate makes a mess in the kitchen, the others have to deal with it.

This feeds directly into our MIMD model: each thread has its own instruction pointer, so they can each be executing different instructions (Multiple Instruction). And they work on shared data (Multiple Data). That's why your processors have to deal with lots of different instruction pointers — decoding different instructions for different threads all at once.

What is a Processor?

Key Definition
Processor: A physical execution unit — a CPU core. This is hardware, unlike processes and threads which are software concepts. It might be virtual (in a cloud environment) or real (a silicon chip in your computer).

Don't confuse "process" with "processor" — they're completely different things despite similar names. A process is a running program (software). A processor is a CPU core (hardware). Multiple threads from the same process can run on different processors simultaneously.

Trust and Threads

Since threads share memory, they have to trust each other. If one thread goes rogue — scribbling on memory it shouldn't, or going into an infinite loop — that's a bug in your program. Unlike processes, you can't easily kill one bad thread without affecting the others, because they all share the same memory space. The attitude is: if you have a bad thread, go fix your program and run it again.

Check Your Understanding

Correct! This is the fundamental property of threads — they share heap memory within the same process. Thread B can read and modify x directly. This is what makes shared-memory concurrency fast (no need to send messages) but also dangerous (both threads could modify x at the same time).

Each thread has its own stack (local variables), but the heap (where objects live) is shared. This shared access is both the strength and the danger of Java's threading model.
Section 06

Creating Threads in Java

Now we get to the practical part. Java represents threads as objects. There's a built-in class called Thread, and an object of type Thread represents a potential thread of execution. But here's an important subtlety: the object alone does nothing. Just creating a Thread object doesn't start any parallel execution. It's just some bytes sitting in memory. You need to explicitly tell it to start.

There are two ways to define what a thread should do when it runs. Let's learn both, because understanding why the second way is preferred teaches us something important about object-oriented design.

Method 1: Subclassing (Inheritance)

The first approach is to create a new class that extends Thread. Your new class inherits all the thread machinery (the ability to start, be scheduled by the OS, etc.) and you just tell it what to do by overriding a method called run().

Javaclass T extends Thread {
    public void run() {
        // Your code goes here.
        // When this thread runs, this method executes.
        System.out.println("Hello from a thread!");
    }
}

To actually run this thread:

JavaT x = new T();   // Create a Thread object (no parallel execution yet!)
x.start();          // NOW the OS creates a real thread and calls run()
Critical Distinction

start() and run() are very different!

start() asks the JVM to create a brand new OS thread (with its own instruction pointer and stack), and then calls run() inside that new thread. Your code now runs in parallel.

If you just call run() directly (without start()), it's just a normal method call — it runs in your current thread. No new thread is created. No parallelism. This is a very common beginner mistake!

Method 2: Delegation (Runnable interface)

The second approach separates "what code to run" from "being a thread." You create a class that implements the Runnable interface, then hand it to a Thread object.

Java// Runnable is a built-in interface with just one method:
interface Runnable {
    void run();
}

Runnable is remarkably simple — it's an interface with a single method, run(), that takes no arguments and returns nothing. A class that implements Runnable is just promising: "I have a run() method you can call."

Javaclass U implements Runnable {
    public void run() {
        // Your code goes here
        System.out.println("Hello from Runnable!");
    }
}

But U is not a thread — it has no start() method. It's just a regular object that happens to have a run() method. To run it in a thread, you hand it to a Thread object:

JavaU y = new U();            // Create the Runnable (no concurrency yet)
Thread t = new Thread(y);  // Create a Thread that delegates to y
                              // "When I start, ask y what to run"
t.start();                   // NOW the OS thread is created, y.run() is called

This is delegation: the Thread object t is responsible for thread creation and scheduling, but it delegates the question "what code should I actually run?" to the Runnable object y.

Section 07

Inheritance vs Delegation: Why Delegation Wins

At first glance, Method 1 (subclassing) seems simpler — it's two lines of code to start the thread instead of three. So why does almost everyone prefer Method 2 (delegation)?

The answer lies in Java's single inheritance rule. In Java, a class can only extend one other class. If your class T extends Thread, then T is permanently welded to the Thread class hierarchy. It can't extend anything else.

Analogy — The Hierarchy Problem

Imagine you're building a class hierarchy for a game. You have Character → Warrior. Now you want the Warrior to run in its own thread. With subclassing, you'd need Warrior extends Thread — but Warrior already extends Character! In Java, you can't do both. Your design is stuck.

With delegation, Warrior can extend Character AND implement Runnable. Java allows implementing multiple interfaces. The Warrior stays where it belongs in your class hierarchy, and you just hand it to a Thread when you want it to run concurrently.

Key Principle

Delegation separates concerns. The Thread class handles thread mechanics (OS thread creation, scheduling, lifecycle). The Runnable object handles the computation (what code to run). By keeping these separate, you maintain flexibility in your class hierarchy.

This inheritance-vs-delegation tradeoff isn't unique to threading — it comes up over and over in object-oriented programming. Delegation is generally preferred because it's more flexible.

Check Your Understanding

Exactly! Since Java has single inheritance, DatabaseConnection can't extend both Connection and Thread. But it CAN extend Connection AND implement Runnable — Java allows implementing multiple interfaces. Then you create a Thread and pass your DatabaseConnection to it: new Thread(myDbConn).start().
Remember: Java only allows single inheritance (one extends), but you can implement multiple interfaces. The solution is class DatabaseConnection extends Connection implements Runnable. Then wrap it in a Thread when needed: new Thread(myDbConn).start().
Section 08

The Thread Lifecycle

A Java Thread isn't just "running" or "not running." It goes through several distinct states during its lifetime. Understanding these states is crucial for reasoning about what your threads are doing — and for understanding the synchronization tools we'll learn next.

Why This Matters
When you debug multi-threaded code, you need to know whether a thread is actually executing, waiting for something, or finished. Each state has different implications for performance and correctness. The tools we learn later (wait, notify, synchronized) all cause state transitions.

Click any state in the diagram above to learn about it.

State by State

1. NEW

Immediately after you call new Thread(y), the thread object exists on the heap, but no OS thread has been created. There's no extra instruction pointer, no extra stack. The number of actual threads running hasn't changed. The thread is just a regular Java object at this point.

While in the NEW state, you can configure the thread — for example, setting its priority (t.setPriority(7)) or giving it a name (t.setName("worker-1")). These don't change the state.

2. RUNNABLE

When you call t.start(), two things happen: (1) the JVM asks the OS to create a real thread — allocating an instruction pointer, a stack, and scheduling metadata (on Linux, this would call pthread_create); (2) the run() method begins executing in that new thread.

The state is called RUNNABLE — not "RUNNING" — for an important reason. Suppose you create 100 threads on a machine with 16 cores. All 100 cannot physically run at the same time. The OS decides which ones get CPU time. So RUNNABLE means "eligible to run" — maybe it's running right now, maybe it's waiting for its turn on a CPU.

Key Advice from the Lecture
Never ask "is this thread currently running?" It doesn't matter. All that matters is: is it runnable? If it's runnable, it will get its turn eventually and make progress. Reason about your program in terms of what's runnable, not what's physically running at this exact nanosecond.

3. WAITING

A thread enters the WAITING state when it calls wait() on an object. It's frozen — waiting for some other thread to notify it that something has happened. While waiting, it uses zero CPU. This is important for efficiency: instead of spinning in a loop checking "is it ready yet?", the thread goes to sleep and gets woken up when needed.

4. TIMED_WAITING

Like WAITING, but with a time limit. Calling Thread.sleep(1000) puts the thread into TIMED_WAITING for 1000 milliseconds. You can also do a timed wait: "wait for this event, but give up after 2 seconds." This provides a guaranteed wake-up time.

5. BLOCKED

A thread enters the BLOCKED state when it's waiting for an external event — reading from a file, waiting for network data from a socket, etc. The difference from WAITING: WAITING is for internal events (another thread doing something), while BLOCKED is for external events (disk I/O, network, etc.).

6. TERMINATED

When the run() method returns (either by reaching the end or by returning explicitly), the thread enters the TERMINATED state. The OS thread is destroyed. The Thread object still exists (you might want to check data it stored before terminating), but it no longer corresponds to any running thread. This is permanent — you cannot restart a terminated thread. Calling start() again throws an exception. To run the same code again, create a new Thread.

Bonus: yield()

Thread.yield() is a hint to the OS: "I'm willing to pause right now if another runnable thread needs the CPU." The thread stays in RUNNABLE state — the OS can choose to ignore the suggestion entirely. It's useful when you know you're about to start an expensive new block of work (cache miss, context switch anyway), so you might as well let someone else go first.

Bonus: Garbage Collection

Like any Java object, if no references to the Thread object remain, it can be garbage collected. At that point, the object simply ceases to exist — there's no state anymore because there's no object.

Section 09

The Problem: Race Conditions

So far, threads sound great — free parallelism! But shared memory creates a devastating class of bugs called race conditions. These are one of the biggest disasters in conventional programming, and Java's original designers knew they had to address them.

Key Definition
Race condition (or "race"): A bug where the program's behavior depends on the unpredictable timing of how threads are scheduled. The result changes depending on which thread happens to run first — like two runners racing to modify the same data.

Here's a concrete example. Suppose two threads both want to increment a shared counter:

Java// Shared variable
int v = 0;

// Thread A does:        Thread B does:
v++;                  //   v++;

You'd expect v to end up as 2. But v++ is not atomic — it's actually three separate operations at the hardware level:

Step-through: How v++ Can Go Wrong

Setup: v = 0. Thread A and Thread B both execute v++. Each v++ is really three steps: (1) load v from memory into a register, (2) add 1 to the register, (3) store the register value back to v. Let's see what can go wrong...

Step 0 of 6

The result? Both threads tried to add 1, but v only ended up as 1 instead of 2. One increment was lost. And here's the terrifying part: this bug depends on exact timing. Run the program a million times, and it might work perfectly each time. Then on the million-and-first run, the threads happen to interleave differently, and you get a wrong answer.

Why This Is So Dangerous
Java was originally designed to run on embedded devices like toasters and refrigerators. Reliability was paramount. Race conditions are the enemy of reliability — your program can run correctly 999,999 times and then fail once, in a way that's nearly impossible to reproduce or debug. The early Java designers knew they needed a solution.
Section 10

The First Solution: synchronized

Java's first answer to race conditions is the synchronized keyword. It ensures that only one thread at a time can execute a particular piece of code on a particular object. Let's build up to it step by step.

The Concept: Locks

Every Java object — every single one — has a hidden lock field built into it. You never see this field in your code, but it's there in the object's internal representation. This lock can be in one of two states: unlocked (0) or locked (1).

What synchronized Does

When you declare a method as synchronized, Java automatically inserts lock/unlock code around the method body:

Java — What you writeclass C {
    int v = 0;

    synchronized int next() {
        return v++;
    }
}
Java — What actually happensclass C {
    int v = 0;

    int next() {
        lock(this);        // ← inserted automatically
        int n = v++;
        unlock(this);      // ← inserted automatically
        return n;
    }
}

When Thread A calls next() on some object obj, it first grabs the lock on obj. While Thread A holds the lock, if Thread B tries to call next() on the same object, it will spin — repeatedly trying to grab the lock in a tight loop until Thread A releases it.

Three Critical Facts About Locks

1. Locks are per object, not per class. Two different objects have two different locks. Threads can call a.next() and b.next() simultaneously with no conflict — they're locking different objects.

2. The lock/unlock is automatic. You never write it yourself. This prevents the extremely common C/C++ bug of forgetting to unlock. Java does it for you, even if an exception occurs.

3. Only one thread can hold an object's lock at a time. If two threads try to grab the same lock simultaneously, one wins (atomically, using special hardware instructions) and the other waits.

Java — Locks are per-objectC a = new C();
C b = new C();

// Thread 1 can call a.next() at the same time
// Thread 2 calls b.next() — no conflict! Different locks.

// But Thread 1 and Thread 2 CANNOT both call a.next()
// at the same time — same object, same lock.

The Performance Problem

Synchronized works. It prevents race conditions. But it has a serious performance cost. If you have 1000 threads all calling next() on the same object, only one core does useful work at a time. The other 999 are burning CPU cycles just spinning "Can I get this lock? Can I get this lock?" — a technique called spin locking.

This was acceptable for toasters and refrigerators. But for high-performance applications like web servers handling thousands of clients simultaneously? The performance is terrible. We need something better.

Connecting the Dots
The original Java class libraries used synchronized heavily — for example, the Hashtable class had all its methods synchronized. This was correct but slow. Modern Java provides better alternatives (like ConcurrentHashMap) that use more sophisticated techniques we'll learn next.
Section 11

Wait & Notify: Efficient Waiting

The problem with synchronized is that when a thread can't get a lock, it spins — it stays in the RUNNABLE state, burning CPU cycles doing nothing useful. What if instead, the thread could go to sleep and be woken up when it's needed? That's exactly what wait() and notify() provide.

Why This Matters
This is where the thread lifecycle states we learned become practical. The WAITING state exists precisely for this purpose — a thread that's waiting uses zero CPU. The operating system can give that CPU core to another thread that can actually get work done.

o.wait()

Any object o can have wait() called on it (you must hold the lock first — i.e., be inside a synchronized block for that object). Here's what happens:

Step 1: Release all locks held by this thread. This is crucial — if you held your locks while waiting, you'd still be bottlenecking other threads. You want them to make progress while you sleep.

Step 2: The thread enters the WAITING state. Its instruction pointer is frozen. The OS gives the CPU to someone else.

Step 3: Eventually, some other thread will signal that the object is "available" (we'll see how in a moment). When that happens, your thread wakes up, reacquires the lock, and resumes executing from where it left off.

There's also a timed version: o.wait(timeout) puts you in TIMED_WAITING — you'll wake up either when notified or when the timeout expires, whichever comes first.

o.notify() and o.notifyAll()

These are how you wake up threads that are waiting. Any thread that currently holds o's lock can call them.

o.notify() wakes up one thread waiting on o. The OS picks which one (typically FIFO — whoever's been waiting longest).

o.notifyAll() wakes up all threads waiting on o. They all try to reacquire the lock; one wins, the others block again. This is used when you want to manage the scheduling yourself rather than letting the OS pick.

Key Concept

What does "available" mean? The OS doesn't know. "Available" is whatever your program decides it means. notify() simply marks the object as available — it's a low-level signal with no built-in semantic meaning. Your program decides when to call notify() based on your application logic.

These are low-level primitives — building blocks. Most Java apps use higher-level tools built on top of them.

Section 12

Higher-Level Coordination Primitives

The wait()/notify() primitives are powerful but low-level. Over the years, people have identified common patterns for thread coordination and packaged them into reusable classes. Here are the four discussed in lecture, each solving a different coordination problem.

1. Semaphore: Controlling How Many

Key Definition
Semaphore: A synchronization object that maintains a counter (capacity) and controls how many threads can proceed concurrently. Each thread must acquire a permit before proceeding and release it when done.

Think of a parking lot with 10 spaces. Each car entering must acquire a space; if the lot is full, cars wait at the entrance. When a car leaves, it releases a space, and the next waiting car enters.

JavaSemaphore s = new Semaphore(10);  // 10 permits

s.acquire();      // Get a permit (WAITS if none available)
// ... do work with the resource ...
s.release();      // Return the permit

s.tryAcquire();   // Try to get a permit (returns false if unavailable,
                  //   NO waiting — stays RUNNABLE)
synchronized vs Semaphore

synchronized is about who — at most one thread in the critical section. Locks are binary (0 or 1), per-object, and enforce mutual exclusion.

Semaphore is about how many — up to N threads can proceed. Counting (0..N), shared/global, and controls resource access capacity.

2. Exchanger: Pairwise Handoff

Key Definition
Exchanger: A synchronization point where two threads meet and swap objects atomically. Neither thread proceeds until both have arrived. 
JavaExchanger<String> x = new Exchanger<>();

// Thread 1:                Thread 2:
v1 = x.exchange(v2);    //  v3 = x.exchange(v4);

// After both arrive:
// v1 == v4 (Thread 1 gets Thread 2's object)
// v3 == v2 (Thread 2 gets Thread 1's object)

Whichever thread arrives at the exchanger first will wait until the other arrives. Then they swap and both proceed. The exchange variable x is shared between both threads — it's the rendezvous point.

Since at most one thread owns each object at any time (they hand off ownership), you don't need synchronized — the exchanger handles the coordination. Good for producer/consumer patterns and pipeline stages.

3. CountDownLatch: One-Time Starting Gate

Key Definition
CountDownLatch: Initialized with a count N. Threads call await() to wait at the gate. A coordinator calls countDown() repeatedly. When the count reaches zero, all waiting threads are released simultaneously. Cannot be reused.

Think of a horse race. There are N slots in the starting gate. Horses arrive and wait. When all slots are filled, the gate opens and all horses run. The gate doesn't close again — it's a one-time event.

JavaCountDownLatch latch = new CountDownLatch(N);

// Worker threads:
latch.await();       // "I'm ready. Wake me when count hits 0."

// Coordinator thread:
latch.countDown();   // Reduces count by 1. When 0, all awaiters unblock.

4. CyclicBarrier: Reusable Checkpoint

Key Definition
CyclicBarrier: Makes threads wait until all N reach the barrier, then releases all simultaneously. Automatically resets for reuse — threads can hit the barrier again and again.
The Weather Simulation Example

Imagine simulating weather over the entire US. You split the map into 100 regions, each handled by its own thread. Each thread computes one minute of weather for its local region at full speed (no synchronization needed — they only look at local data).

After one minute of simulation, all threads hit the CyclicBarrier. When all 100 have arrived, a bit of update code runs to bleed information between neighboring regions (a cloud moving from one region into another). Then the barrier resets, and all threads compute the next minute.

This pattern — compute independently, synchronize at a checkpoint, share boundary data, repeat — is extremely common in scientific computing.

Section 13

Going Lower: volatile

We've gone higher in abstraction (from synchronized → wait/notify → semaphores, exchangers, etc.). Now let's go lower. The volatile keyword is simpler than synchronized but also trickier to use correctly. When used right, it's the highest-performance option.

Why Go Lower?
synchronized is reliable but slow — spinning threads waste CPU. wait()/notify() is more efficient but involves OS scheduling overhead. Sometimes you just need a single variable to be visible across threads without any locking at all. That's where volatile shines.

The Problem: Compiler Optimizations

Before understanding volatile, you need to understand why it's needed. Java compilers (and C/C++ compilers) aggressively optimize your code. They perform what are called "as-if" optimizations — they can rearrange, eliminate, or transform your code as long as the result behaves as if the original code ran.

Key Definition
As-if optimization: The compiler can change or reorder code as long as the program behaves the same in a single-threaded world. The output is "as if" the original code ran.

For example, the compiler sees this:

Before Optimizationint k = 3;
int j = v - v;  // always 0, no matter what v is
j++;            // j becomes 1
k--;            // k becomes 2

And transforms it into:

After Optimizationint k = 2;   // compiler computed the final value directly
int j = 1;   // compiler computed the final value directly

This is great for performance! The code runs faster, and a single thread can't tell the difference. But in a multi-threaded world, these optimizations can break things.

The Spin-Loop Bug

Here's a classic example of how optimization breaks multi-threaded code:

BROKEN — without volatileclass C {
    boolean busy;   // NOT volatile
}

void m() {
    while (busy)
        continue;   // spin until busy becomes false

    busy = true;     // enter critical section
    // ... do work ...
    busy = false;    // exit critical section
}

The intent: wait until no other thread is using the resource, then claim it. But the compiler looks at this loop and thinks: "Nothing inside the loop modifies busy. If busy is true when we enter, it'll be true forever. This is an infinite loop (or it's false and we skip it)." So it optimizes the busy checks away entirely! Thread A might set busy = false, but Thread B never re-reads it from memory — it's using a cached value that's permanently true. Thread B is stuck forever.

The Fix: volatile

FIXED — with volatileclass C {
    volatile boolean busy;  // forces real reads and writes!
}
Key Definition
volatile: Declaring a variable volatile tells the compiler and JVM: every time the source code reads this variable, generate a real load from memory. Every time it writes, generate a real store. Don't cache it in a register, don't optimize it away, don't reorder operations around it.

With volatile, each iteration of the while(busy) loop re-reads busy from actual memory. When Thread A sets busy = false, Thread B will eventually see the new value.

Volatile Is Not a Lock!

volatile ensures visibility (other threads see your writes) and prevents reordering. But it does NOT ensure mutual exclusion. Two threads can still read busy == false simultaneously and both set it to true — you've got a race condition again.

Making the spin-loop example actually thread-safe requires more sophisticated techniques (like compare-and-swap). But volatile solves the visibility problem, which is a necessary first step.

In high-performance Java libraries, you'll see volatile used alongside other techniques. It avoids the overhead of synchronized entirely — no locks, no spinning, no waiting. The tradeoff is that it's trickier to use correctly and provides weaker guarantees.

Section 14

The Java Memory Model (JMM)

When Java implementers tried to nail down the exact rules for how volatile and synchronized interact with compiler optimizations, they found that different compiler writers disagreed about what was allowed. To fix this, they created the Java Memory Model — a formal set of rules that every Java implementation must follow.

Key Definition
Java Memory Model (JMM): A formal specification that defines what memory operations are visible to which threads, in what order, and which compiler/CPU optimizations are allowed. It's a contract between programmers, the JVM, the compiler, and the CPU.
Why This Matters
Without a memory model, the same Java program could behave differently on different CPUs or with different compilers. The JMM ensures your multithreaded code is portable — if it works correctly on one Java implementation, it works on all of them.

The Minkowski Analogy

The professor draws a fascinating analogy to special relativity. In physics, there's a speed limit (the speed of light) that determines what you can and can't communicate. Events that are too far apart can't affect each other — they're in each other's "elsewhere."

The Relativity Analogy for Threads

In multi-threaded programming, your thread's past is everything it has done — you can see all of that reliably. Your future is everything it will do — you control that. But what other threads are doing right now is like the "elsewhere" in relativity — communication is fuzzy and unreliable unless you use proper synchronization.

Without synchronized or volatile, you simply cannot rely on seeing another thread's writes. They might be cached, reordered, or delayed. The JMM defines exactly where the "light cones" are — what's reliably visible and what's not.

Core Assumptions

The JMM assumes an abstract Java Virtual Machine with threads, objects, volatile variables, and synchronized methods. Each individual thread has sequential semantics — you can think of it executing statements one at a time. Within that thread, as-if optimizations (including reordering loads and stores) are allowed — as long as that single thread can't tell the difference.

The complexity arises when multiple threads interact. The JMM defines six operations that matter for cross-thread visibility:

OperationCategoryDescription
Normal LoadAReading an ordinary (non-volatile) variable
Normal StoreAWriting an ordinary (non-volatile) variable
Volatile LoadBReading a volatile variable (acts like "acquire")

Volatile StoreCWriting a volatile variable (acts like "release")

Notice how the operations are grouped into three categories: A (normal stuff), B (acquiring/observing — volatile load or entering a monitor), and C (releasing/publishing — volatile store or exiting a monitor). These categories are the key to the reordering rules.

The Grand Principle

Why are exactly these 6 cells "No" and 3 cells "Yes"? The underlying principle is about critical sections:

Making a critical section bigger is safe. If you move a normal operation into a critical section (from before the lock to inside it, or from after the unlock to inside it), you're just protecting more code than strictly necessary. Slower, perhaps, but still correct. That's why A→B (normal before acquire) and C→A (release before normal) allow swapping — the swap just moves code into the protected region.

Making a critical section smaller is dangerous. If you move a normal operation out of a critical section, that operation is no longer protected by the lock. Other threads could interfere. That's why A→C (normal before release) and B→A (acquire before normal) forbid swapping — the swap would move code out of the protected region.

Reordering synchronization operations themselves is dangerous. If two acquires, two releases, or an acquire-and-release get swapped, threads could observe shared state in the wrong order or release before they've finished their critical section. That's why all of B→B, B→C, C→B, and C→C are forbidden.

Bringing It All Together

This matrix is the core of the JMM. It tells compiler writers exactly what optimizations they can perform. For you as a programmer, the practical takeaway is:

If you don't use volatile or synchronized, the compiler can reorder your loads and stores freely (as-if rules apply). Other threads may see your writes in any order, or not at all.

If you use synchronized or volatile, you create ordering constraints that prevent the compiler from reordering operations across those boundaries. This is how you make inter-thread communication reliable.
