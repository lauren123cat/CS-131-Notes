Lecture 17

Memory Management — Generation-Based Copying, Rust Ownership, and OOP Concepts

Where We Are in the Big Picture

In previous lectures, you learned two ways a language runtime can automatically reclaim memory that your program no longer needs:

Mark and Sweep — the garbage collector walks all reachable objects ("marks" them), then scans every object in memory and frees the unmarked ones ("sweeps"). Simple, but it pauses your whole program and does work proportional to all objects — both live and dead.

Conservative GC — a variant that works even when the compiler doesn't tell the runtime exactly where every pointer is. It assumes anything that looks like a pointer might be one. Safe, but can leak memory (keeps objects alive that shouldn't be).

Today, we're going to learn three big topics that extend this story:

  1. Generation-Based Copying Collection — a smarter, faster GC used by Java and .NET that exploits the fact that most objects die young.
  2. Rust's Ownership Model — a radically different approach that eliminates GC entirely by making the compiler enforce memory safety rules at compile time.
  3. OOP Concepts — how object-oriented programming really works under the hood, and why "OO language" and "OO programming" are not the same thing.
Why This Matters
Every program you write allocates memory. How that memory gets reclaimed — and how fast — affects whether your app runs smoothly or freezes, whether your robot crashes into a wall, and whether your code has subtle bugs that corrupt data. These aren't theoretical concerns; they're the reason Java chose generation-based GC, the reason Rust was created, and the reason the Linux kernel is written in a particular style of C.

Generations: The Core Idea

Imagine you have a huge warehouse full of boxes. Some boxes have been sitting there for years — they contain important things. Other boxes arrived this morning — they might just be temporary packaging.

If you need to clean up the warehouse, which boxes should you check first? Probably the new ones — the stuff that just arrived is most likely to be throwaway packaging.

That's exactly how generation-based garbage collection works. Instead of treating all objects equally, we split memory into two regions:

┌──────────────────────────┐ ┌──────────────────────────┐ │ OLD (Tenured) │ │ NEW │ │ │ │ │ │ Objects that have been │ │ Objects recently │ │ around for a while. │ │ allocated with "new". │ │ Probably still needed. │ │ Often temporary. │ │ │ │ │ │ (Think: the furniture │ │ (Think: the packaging │ │ in your apartment) │ │ from today's delivery) │ └──────────────────────────┘ └──────────────────────────┘
Definition
Generations divide the heap into regions based on object age. In the simplest form, there are just two: old (also called "tenured") and new. More sophisticated implementations can have many generations (e.g., 10), but the principle is the same.

The "new" region is called the nursery (or sometimes "Eden" — as in the Garden of Eden, because everything here is freshly created). When you call new in Java, your object gets placed here.

Objects that survive enough garbage collections get "promoted" — they move from the nursery into the old generation. The idea is: if an object has been alive this long, it's probably going to keep being alive.

Real-World Usage
Java and .NET both use generation-based copying collectors as their default GC strategy. This isn't a textbook curiosity — it's the GC running on billions of devices right now.

Three Key Assumptions

Generation-based collection is built on three assumptions about how real programs behave. If these assumptions are wrong, the whole scheme falls apart. Fortunately, they're true for the vast majority of real-world programs.

Assumption 1: New Objects Are More Likely to Be Garbage

This sounds counterintuitive — you just created this object, why would you throw it away? But think about what code actually does:

// This creates a temporary StringBuilder object
String result = new StringBuilder()
    .append(firstName)
    .append(" ")
    .append(lastName)
    .toString();
// The StringBuilder is now garbage — we only needed the String

Programs constantly create objects for temporary purposes: packaging up data to pass to a function, building intermediate results, creating iterators. These objects live for a few microseconds and then nobody refers to them anymore. This pattern is so common it has a name: the "generational hypothesis" — most objects die young.

Assumption 2: New Objects Point to Old Ones (Not the Reverse)

OLD Generation NEW Generation ┌──────────────┐ ┌──────────────────┐ │ obj_A ──────│──┐ │ │ │ │ │ │ obj_X ───────────│──→ obj_A (OLD) │ obj_B ──────│──│──┐ │ │ │ │ │ │ │ obj_Y ───────────│──→ obj_B (OLD) │ │ │ │ │ │ └──────────────┘ │ │ │ obj_Z ───────────│──→ obj_A (OLD) │ │ └──────────────────┘ │ │ Common: ────────┘ └── arrows from NEW → OLD Rare: arrows from OLD → NEW

Why? When you create a new object, you initialize its fields to point to things that already exist — and those things are, by definition, older. It's possible for an old object to later be modified to point to a new one, but it's less common.

This assumption is important because it means: when we garbage-collect the nursery, we mostly only need to look at roots (stack variables, registers) that point into the nursery. We can ignore most of the old generation.

Assumption 3: There's One Contiguous Free Region in the Nursery

This is a design choice, not an observation about programs. Instead of maintaining a complex "free list" (a linked list of available memory chunks, like mark-and-sweep does), the nursery is organized so that all free space is in one contiguous block at the end:

Nursery ┌────────────────────────────────────────────┐ │ obj_1 │ obj_2 │ obj_3 │ FREE SPACE │ │ │ │ │ │ └────────────────────────────────────────────┘ ↑ ↑ HP LP (heap pointer) (limit pointer)

We track this free region with just two pointers:

  • HP (heap pointer) — points to the start of free space (where the next object will go)
  • LP (limit pointer) — points to the end of the nursery
Why This Makes new Blazingly Fast
When someone calls new MyObject(), the runtime just checks: is HP + size ≤ LP? If yes, place the object at HP and bump HP forward. That's it — roughly ~10 machine instructions. Compare this to mark-and-sweep's free list, where new might need to walk through dozens of free blocks looking for one that's big enough — potentially hundreds of instructions.
Analogy
Think of the nursery like a roll of paper towels. Allocating an object is like tearing off a sheet — you just rip from the current position, no searching needed. With a free list, it's more like finding an empty page in a used notebook — you have to flip through and check each page.

The Nursery & Making new Fast

Let's really understand why this design makes allocation so fast, because it's one of the main reasons real systems use generation-based collection.

How new Works Under Mark and Sweep

With traditional mark-and-sweep, free memory is scattered in a free list — a linked list of available chunks of different sizes:

Free List: ┌──────┐ ┌────┐ ┌──────────┐ ┌───┐ │ 24B │───→│ 8B │───→│ 128B │───→│16B│───→ NULL └──────┘ └────┘ └──────────┘ └───┘

When you call new MyObject() and it needs 32 bytes, the allocator has to walk this list: "24 bytes? Too small. 8 bytes? Too small. 128 bytes? That works!" Then it carves off 32 bytes from that 128-byte block. This is slow — potentially dozens or hundreds of instructions, and the free list can be long.

How new Works in a Generation-Based System

In the nursery, it's dramatically simpler. Both HP and LP can be kept in CPU registers (the fastest storage your processor has):

// Pseudocode for "new" with a nursery
if (HP + requested_size <= LP) {
    object_address = HP;    // object goes here
    HP += requested_size;   // bump the pointer
    return object_address;   // done!
} else {
    // Out of space — trigger garbage collection
    garbage_collect();
}

The fast path (no collection needed) is about 10 instructions: a load, an add, a compare, a branch, a store. This is roughly as fast as pushing something onto a stack.

What Happens When Space Runs Out?

When HP + size > LP, the nursery is full. Now we trigger garbage collection — but only of the nursery, not the entire heap. This is called a minor collection. We'll see exactly how it works in the next section.

Copy Collection in Action

Here's the key insight that makes generation-based collection different from mark-and-sweep: instead of marking live objects and then sweeping dead ones, we copy live objects to a new location and throw away everything else.

Step-by-Step Walkthrough

Step 1 of 5

What We Never Do

Notice what's missing from this algorithm: there is no sweep phase. We never look at the dead objects. We never walk through the old nursery finding unmarked objects and freeing them one by one. We just abandon the entire old nursery — all the garbage is thrown away in one shot by discarding the whole region.

No Reference Counts Needed Either
This algorithm doesn't need reference counts. We don't track how many pointers point to each object. We simply follow pointers from the roots, and anything we can reach gets copied. Anything we can't reach is garbage — we never even look at it.

When Does an Object Get "Tenured"?

If an object survives several minor collections (it keeps getting copied to new nurseries because it's still alive), eventually the runtime decides it's probably going to stick around. At that point, instead of copying it to a new nursery, the runtime promotes it to the old generation.

If the old generation eventually fills up with garbage too (rare, but it happens), the runtime has to do a major collection — a full garbage collection of everything, old and new. This is expensive, but with luck, it doesn't happen often.

Why This Beats Mark & Sweep

At first glance, copying sounds more expensive than mark-and-sweep — you're physically moving objects in memory! But there are two powerful reasons it's actually faster:

Reason 1: Work Is Proportional to Live Objects Only

Property Mark & Sweep Generation-Based Copying
Work during collection Proportional to all objects (live + dead) Proportional to live objects only
Dead objects Must sweep each one individually Never even looked at
If 90% of nursery is garbage Still scans all 100% Only copies the 10% that's live
Collection scope Entire heap Usually just the nursery

Since most new objects are garbage (Assumption 1), the number of live objects in the nursery at collection time is typically small. So the copying collector does very little work during each minor collection.

Reason 2: Cache Friendliness

This is the less obvious but equally important advantage, and it connects back to computer architecture (CSE 33).

Modern CPUs have caches — small, ultra-fast memory that sits between the CPU and main RAM. Accessing data in cache is roughly 100× faster than accessing data in RAM. Programs run fast when the data they need is already in cache.

BEFORE collection (nursery is scattered): ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐ │live │dead │dead │live │dead │dead │dead │live │ ← many cache lines └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘ AFTER copying to new nursery (compacted): ┌─────┬─────┬─────┬─────────────────────────────┐ │live │live │live │ FREE │ ← fewer cache lines └─────┴─────┴─────┴─────────────────────────────┘

After copying, all live objects are packed together tightly. They fit in fewer cache lines. When your program accesses these objects, they're more likely to already be in cache. Mark-and-sweep leaves objects scattered with dead space between them, wasting precious cache capacity.

The Trade-Off
Copying does add overhead — you're physically moving bytes around in memory. But the smaller working set (only live objects, no dead ones) and better cache behavior more than compensate in most workloads. And remember: the minor collection only covers the nursery, not the entire heap.

Edge Cases & Complications

Problem 1: Old-to-New Pointers

Remember Assumption 2: new objects point to old ones, not the reverse. But this isn't always true! Sometimes you modify an old object to point to a new one:

// oldList has been around for a while (in the old generation)
// newItem was just created (in the nursery)
oldList.add(newItem);  // Now old → new pointer exists!

If we only look at stack/register roots when collecting the nursery, we'll miss newItem — we'll think it's garbage and throw it away, even though oldList still refers to it!

Solution 1: Treat the old generation as additional roots. During a minor collection, scan the old generation for any pointers into the nursery. Those are roots too. This requires a linear scan of the old generation, which isn't free, but it works.

Solution 2: Use virtual memory tricks. Mark old-generation pages as read-only. When the program tries to modify an old object (creating an old→new pointer), the hardware triggers a page fault. A signal handler catches this, notes which page was modified ("this page is dirty"), and makes it writable again. During the next GC, we only need to scan "dirty" pages — not the entire old generation.

Hardware Support
Modern CPUs have "dirty bits" in their page tables that track which pages have been written to. The GC can read these bits directly without needing signal handlers, making this even more efficient.

Problem 2: The finalize() Problem

Java's Object class has a method called finalize():

public class Object {
    // Called by GC when object becomes garbage
    void finalize() { ... }
}

You can override this in your class to do cleanup — close a database connection, release a file handle, etc. But here's the problem: the copying collector never looks at dead objects! If we never see the dead objects, we can never call their finalize() methods.

Java's solution: objects with custom finalize() methods are placed in a separate arena that uses traditional mark-and-sweep. It's slower, but since very few classes override finalize(), it barely matters. In modern Java, finalize() is officially deprecated — you should use other cleanup mechanisms instead.

Problem 3: Incompatible with Conservative GC

Conservative GC works when you're not sure exactly where all the pointers are — you treat anything that looks like a pointer as one. That works for mark-and-sweep because objects don't move.

But copying moves objects, so you must update every pointer that references the moved object. If you updated a value that looks like a pointer but is actually, say, someone's bank account balance, you'd corrupt that data.

Critical Requirement
Generation-based copying collection requires that the runtime knows exactly where all pointers are. This is why it works for Java (the compiler tells the runtime where every pointer is) but not for C or C++ (compilers don't provide this information).

Multi-threading & GC

Java is multi-threaded. Multiple threads can call new at the same time. This creates a problem with our fast heap pointer scheme.

The Race Condition

// Thread A and Thread B both call new simultaneously
// Both read HP = 1000
// Thread A: HP = 1000 + 32 = 1032
// Thread B: HP = 1000 + 64 = 1064
// Both think they own memory starting at 1000!
// → DISASTER: two objects in the same memory

The naive fix is to put a lock (e.g., a spin lock) around the heap pointer. Grab the lock, bump HP, release the lock. But this defeats the purpose — our 10-instruction new becomes slower, and with 100 threads, there's heavy lock contention (threads waiting for each other).

The Solution: Per-Thread Nurseries

Give each thread its own private nursery (called a Thread-Local Allocation Buffer or TLAB). Each thread has its own HP and LP — no sharing, no locking, no contention.

Thread 1's Nursery: Thread 2's Nursery: Thread 3's Nursery: ┌──────┬───────┐ ┌────┬─────────┐ ┌──────────────┐ │objs │ FREE │ │obj │ FREE │ │ FREE │ └──────┴───────┘ └────┴─────────┘ └──────────────┘ ↑ ↑ ↑ ↑ ↑ ↑ HP₁ LP₁ HP₂ LP₂ HP₃ LP₃

Each thread can keep its HP and LP in CPU registers. Allocation is fast again — no locks needed because no other thread touches your nursery.

Complication
Threads can share objects between themselves. When Thread 1 sends an object reference to Thread 2, we now have cross-nursery pointers. This is rare but makes GC algorithms more complex — you need to handle the case where one thread's nursery contains pointers into another thread's nursery.

Real-Time Garbage Collection

Imagine you're programming a robot arm in a factory. The arm needs to move precisely, following a planned trajectory. If the program freezes for 500 milliseconds to do garbage collection, the arm might overshoot, collide with something, or drop what it's carrying.

Predictability Over Speed

Real-time systems care about predictability more than raw speed. They'd rather have an algorithm that's 20× slower but always finishes within a known time limit, versus one that's 20× faster on average but occasionally pauses for a long GC.

The Real-Time Guarantee
A real-time system needs an upper bound on how long any operation takes. If new always completes within 10,000 instructions, you can plan a time budget and guarantee your robot responds in time.

How Real-Time GC Works

Instead of collecting all garbage at once, every call to new does two things:

  1. Allocate the requested storage
  2. Do a little bit of garbage collection (say, 5000 instructions worth)

The GC is incremental — it does work in small chunks, interleaved with normal program execution. The program never freezes for a long time.

The Tricky Part: Mutation During Collection

The hard part is that between GC increments, the program keeps running and modifying objects. The partially-completed GC state might become invalid. For example:

  • GC marks object A as "live" and hasn't looked at object B yet
  • The program runs and makes A point to B, then removes all other references to B
  • GC resumes — it already processed A and won't revisit it, so it never discovers B through A
  • B gets incorrectly collected even though it's still alive!

Handling this correctly requires subtle algorithms that track what the program has changed since the last GC increment. It's complex, error-prone, and makes the overall GC slower — but that's acceptable because we get predictability.

An Extreme Trick: Disabling Cache

Some real-time developers flip a hardware bit that disables the CPU cache entirely. Everything goes straight to RAM, which is ~100× slower. Why would anyone do this? Because RAM access time is predictable. Cache hits are fast, but cache misses are slow and unpredictable. By always going to RAM, you get a reliable upper bound on timing. Once you've tested your robot with cache disabled and confirmed it meets timing requirements, you know it will always work.

Could Reference Counting Help?

Reference counting frees objects immediately when the count hits zero — no GC pauses at all! That sounds perfect for real-time. The downside: reference counting can't handle cycles (A → B → A, where both counts stay at 1 forever). If you can prove your system has no cycles, reference counting could be an excellent real-time solution.

Why Rust? A Different Approach to Memory

So far, every garbage collection strategy has a cost:

  • Mark & sweep — pauses the program, does work proportional to all objects
  • Generation-based copying — faster, but still pauses; requires knowing all pointer locations; incompatible with C/C++
  • Reference counting — slows down every pointer assignment; can't handle cycles
  • Real-time GC — predictable but slower; complex to implement correctly

Rust asks a radical question: what if we could manage memory without any runtime cost at all?

The answer: move the memory management work from runtime (when the program runs) to compile time (when the compiler checks your code). The Rust compiler enforces rules that guarantee memory safety — no garbage collector, no reference counting, no runtime overhead.

Rust's Promise
  • No garbage collector — no GC pauses, no runtime overhead
  • No dangling pointers — compiler prevents use-after-free
  • No data races — compiler prevents concurrent mutation bugs
  • No memory leaks (in the common case) — objects are freed deterministically
The trade-off: you have to write code that satisfies the compiler's rules, which can feel restrictive at first.

The foundation of Rust's approach is built on three concepts: immutability, ownership, and borrowing. Let's learn each one from scratch.

Immutability by Default

In most languages you've used, variables are mutable by default — you can change their value whenever you want:

// Java — mutable by default
int x = 5;
x = 10;  // perfectly fine

# Python — mutable by default
x = 5
x = 10  # perfectly fine

Rust flips this default. Variables are immutable by default:

let x = 5;
x = 10;  // ERROR! x is immutable

let mut y = 5;
y = 10;  // OK — we explicitly said "mut"

Why Does This Help Memory Management?

If objects are immutable, there's a powerful guarantee: an object can never be modified to point to something newer than itself. Think about it — when you create an immutable object, you fill in its fields at creation time. Those fields can only reference things that already exist, which are by definition older. No one can come along later and stuff a pointer to a newer object inside it.

This immediately solves the "old points to new" problem that generation-based collectors have to work around! With immutable objects, Assumption 2 (new → old only) is guaranteed, not just hoped for.

Connection to Functional Programming
This is the same principle behind functional languages like OCaml and Haskell — immutability simplifies reasoning about your program. Rust brings this benefit to a systems programming language.

Ownership: The Core Idea

This is the big one — the concept that makes Rust fundamentally different from every other language you've used.

Ownership Rule
Every value in Rust has exactly one owner at any time. When the owner goes out of scope, the value is dropped (freed).

Let's unpack what this means by contrasting it with what you know:

In Java / Python (No Ownership)

// Java — multiple "owners" (references), no single owner
String a = new String("hello");
String b = a;     // Both a and b refer to "hello"
String c = a;     // Now three references to "hello"
// When are we done? GC has to figure that out.

In Java, any number of variables can point to the same object. None of them is special. The garbage collector has to scan all references to figure out when the object is no longer needed.

In Rust (Single Ownership)

let a = String::from("hello");  // a owns "hello"
let b = a;    // ownership MOVES to b
                // a is now INVALID — using it is a compile error!
println!("{}", a);  // ERROR: a has been moved
println!("{}", b);  // OK: b is the owner

In Rust, when you assign one variable to another, ownership transfers. The old variable becomes invalid. There's always exactly one owner, so the runtime always knows exactly when to free the memory: when that one owner goes out of scope.

Analogy
Think of ownership like holding a physical book. You can hand the book to someone else (transfer ownership), but then you don't have it anymore. You can let someone look at it while you hold it (borrowing — we'll get to this), but there's always exactly one person "holding" the book. When that person leaves the room (goes out of scope), the book gets put back on the shelf (memory freed).

The drop Function

When an owner goes out of scope, Rust automatically calls drop on the value. This is similar to Java's deprecated finalize(), but better:

  • It's deterministic — you know exactly when it runs (at end of scope)
  • It's guaranteed — it always runs, no "maybe the GC will get to it eventually"
  • You can customize it by implementing the Drop trait for your type
fn example() {
    let s = String::from("hello");
    // use s...
} // ← s goes out of scope here, drop is called, memory freed

No garbage collector. No reference counting. The compiler inserts the deallocation call at exactly the right place.

Move Semantics

Let's be very precise about what happens with assignment in Rust, because it's different from any other language you've used.

Case 1: let x = y; — Declaring a New Variable

let y = String::from("hello");
let x = y;  // ownership transfers from y to x
            // y is now INVALID

After this assignment:

  • x is the new owner of the string "hello"
  • y is no longer valid — the compiler will reject any code that tries to use y
  • No data was copied — the pointer simply moved from y to x

Case 2: x = y; — Reassigning an Existing Variable

let mut x = String::from("world");
let y = String::from("hello");
x = y;  // TWO things happen:
        // 1. x's old value ("world") is DROPPED (freed)
        // 2. ownership of "hello" transfers from y to x
        // y is now INVALID

This is important: the old value of x is freed before the new value takes its place. There's no garbage collector that will "eventually" clean up "world" — it's freed immediately.

Move ≠ Copy
In Java, x = y creates a second reference to the same object. Both x and y remain valid. In Rust, x = y is a move — the value transfers, and y becomes invalid. This is the fundamental difference.

Copyable Types (The Copy Trait)

Not everything in Rust uses move semantics. Some types are so small and simple that it makes no sense to track ownership — just copy the bits.

The Copy Trait
Types that implement the Copy trait are duplicated on assignment rather than moved. After let x = y;, both x and y remain valid — each has their own independent copy of the data. 
let y: i32 = 42;
let x = y;      // COPIES the value (i32 is Copy)
println!("{}", y);  // OK! y is still valid — it's 42
println!("{}", x);  // OK! x is also 42

Which types are Copy?

  • All integer types (i32, u64, etc.)
  • Floating-point types (f32, f64)
  • Booleans (bool)
  • Characters (char)
  • Tuples of Copy types
  • Immutable references (&T)

Which types are not Copy?

  • String — it owns heap-allocated memory
  • Vec<T> — it owns heap-allocated memory
  • Any type that manages a resource (file handle, network connection)
  • Mutable references (&mut T)

The intuition: if a type lives entirely in a register or a few words on the stack and doesn't own any heap memory, it can be Copy. If it manages heap memory or external resources, it can't — because you'd have two owners of the same resource, and that's exactly what the ownership model prevents.

Borrowing & References

If every assignment moves ownership, how do you pass an object to a function without losing it? You'd hand over ownership, and the function would drop it when it's done — you'd never get it back!

fn print_length(s: String) {
    println!("Length: {}", s.len());
}  // s is dropped here!

let my_string = String::from("hello");
print_length(my_string);  // ownership moves INTO the function
println!("{}", my_string);  // ERROR: my_string was moved!

The solution is borrowing. The & operator creates a reference — a pointer to the value that doesn't take ownership:

fn print_length(s: &String) {  // takes a reference, not ownership
    println!("Length: {}", s.len());
}  // s (the reference) goes out of scope, but the original is unaffected

let my_string = String::from("hello");
print_length(&my_string);  // BORROWS — doesn't move ownership
println!("{}", my_string);  // OK! we still own it
Borrowing
&x creates a reference to x — a pointer that lets you read the value without owning it. The reference is borrowing the value. Key rules:
  • The original owner still owns the value
  • When the reference goes out of scope, the value is not dropped (the owner still has it)
  • The reference is read-only by default — you can look but not touch
Analogy
Borrowing is like lending a library book. The library (owner) lets you borrow it. You can read it, but you can't rip pages out or write in the margins. When you're done, the library still has the book. If you try to keep it past the due date (reference outliving the owner), the compiler catches that error.

The Dangling Pointer Problem — Solved

In C/C++, the & (address-of) operator is dangerous because nothing stops you from using a pointer after the value has been freed:

// C — dangling pointer disaster
int* bad_function() {
    int x = 42;
    return &x;  // returns pointer to stack variable!
}  // x is gone, pointer is now dangling

Rust prevents this at compile time. The rule is: a reference cannot outlive the value it refers to. The compiler tracks lifetimes and rejects code that would create dangling references.

Mutable References

Regular references (&x) are read-only. But sometimes you need to modify the value through a reference. For that, Rust has mutable references:

let mut x = 5;
let r = &mut x;  // mutable reference to x
*r = 10;          // modify x through the reference
println!("{}", x);  // prints 10

But there's a strict rule:

The Mutable Reference Rule
At any given time, you can have either:
  • One mutable reference (&mut x), OR
  • Any number of immutable references (&x)
But never both at the same time.

Why This Rule Exists: Preventing Data Races

This rule isn't just about memory management — it's about thread safety. A data race occurs when:

  1. Two threads access the same data
  2. At least one of them is writing
  3. There's no synchronization

Rust's mutable reference rule prevents this at compile time:

  • If someone has &mut x, nobody else can read or write to x — so no race is possible
  • If multiple threads have &x, they can all read — but nobody can write — so again, no race
let mut s = String::from("hello");

let r1 = &s;      // OK: first immutable borrow
let r2 = &s;      // OK: second immutable borrow
let r3 = &mut s;  // ERROR: can't borrow as mutable
                    // while immutable borrows exist

Lifetimes

References in Rust have lifetimes — the span of code during which the reference is valid. Most of the time, the compiler figures these out automatically (similar to how OCaml infers types). But sometimes you need to annotate them explicitly.

How Lifetime Inference Works

Originally, Rust inferred lifetimes from lexical scope — the reference was valid until the closing }. Modern Rust is smarter: it figures out the last point where the reference is actually used.

let mut x = 5;
let r = &x;         // r borrows x
println!("{}", r);   // last use of r
// r's lifetime ENDS here (not at the closing brace)
x = 10;              // OK! r is no longer active

The compiler does flow analysis — it traces which paths through the code could use the reference and finds the last possible use. This is more precise than simple scope-based analysis and lets you write more natural code.

When You Need Explicit Lifetimes

Sometimes the compiler can't figure out the lifetime on its own. The classic example: a function that returns one of two references it received:

// This WON'T compile — Rust doesn't know the return lifetime
fn longer(x: &str, y: &str) -> &str {
    if x.len() > y.len() { x } else { y }
}

The problem: x and y might have different lifetimes. The return value could be either x or y — the compiler doesn't know which until runtime. So what lifetime should the return type have?

You solve this with a lifetime parameter, written as 'a:

fn longer<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

This tells the compiler: "both inputs and the output share the same lifetime 'a." In practice, Rust computes 'a as the intersection (shorter) of the two input lifetimes. The result is guaranteed to be valid as long as both inputs are valid.

Lifetime Subtyping

A longer lifetime is a subtype of a shorter lifetime. This might seem backwards, but it makes sense: if a function expects a reference that lives for 10 seconds, it's safe to pass one that lives for 60 seconds — it will certainly be valid for the needed 10 seconds.

The 'static Lifetime

let s: &'static str = "I live forever";

'static means the reference is valid for the entire program duration. String literals have this lifetime because they're baked into the compiled binary. It's the longest possible lifetime — and therefore a subtype of every other lifetime.

How C/C++ Tries (and Fails) to Do What Rust Does

GCC has an extension that hints at Rust-style analysis. You can annotate a custom allocator function:

void *myalloc(size_t u)
    __attribute__((malloc(free, 1)));
// Tells GCC: this returns memory that should be freed with free()
// and the first argument is the size

With this annotation, GCC can detect some memory leaks:

{
    char* p = myalloc(27);
    // ... use p but never free it ...
}  // GCC warns: "memory leak"

But this is a warning, not an error. Your code still compiles. And it's easy to fool — store the pointer in a global variable and GCC loses track.

The Key Difference
Rust: Ownership rules are enforced. Violations are compile errors. Your program won't build if it has memory bugs.
C/C++: Annotations are optional hints. The compiler warns but still compiles. You can have memory bugs in production code.

The Cost of Rust's Strictness

Rust's approach isn't free of frustration. Sometimes you write perfectly safe code, but the compiler isn't smart enough to prove it's safe:

// Perfectly safe code that Rust might reject:
let mut v = vec![1, 2, 3, 4, 5];
let first = &v[0];    // borrow the first element
v.push(6);              // ERROR: can't modify v while it's borrowed
println!("{}", first);
// The push MIGHT reallocate the vector, invalidating first
// So Rust rejects it even though it might be fine this time

When the compiler is wrong and you know what you're doing, Rust provides an escape hatch: unsafe blocks. Inside unsafe, the compiler stops checking some rules. Your boss might not like it, but sometimes it's necessary.

The Rust Philosophy
No GC, no reference counts, no runtime overhead. Pointer assignments are as fast as they can be — no bumping counters, no updating reference lists. All the safety checking happens at compile time. You pay the cost in thinking harder about ownership, but your program runs as fast as C.

OO Language ≠ OO Programming

Here's a fact that seems obvious once stated but is worth being explicit about: using an object-oriented language does not mean your code is object-oriented, and vice versa.

OO Language, Non-OO Code

You can write perfectly procedural code in Java:

public class Main {
    static int add(int a, int b) { return a + b; }
    static void main(String[] args) {
        int result = add(3, 4);
        System.out.println(result);
    }
}
// No objects, no inheritance, no polymorphism
// This is C code wearing a Java costume

Non-OO Language, OO Code

You can write OO code in C! And this isn't theoretical — the Linux kernel does this.

// OO programming in C
struct ClassC {
    int v;
    int (*other_method)(struct ClassC* self, int arg);
};

int m(struct ClassC* self, int j) {
    // Access instance variable via self pointer
    return self->v + j;
}

// Calling a method:
self->other_method(self, 42);

The trick is simple: every "method" is just a function with an extra first argument — self — that points to the object. In C++, this self pointer exists too (called this), but the compiler inserts it for you automatically.

The Hidden Argument Pattern

OOP's self/this pointer is actually an instance of a much bigger pattern in programming languages: many "fancy" features are implemented as ordinary functions with an extra hidden argument.

Feature What You Write What Really Happens
OOP Methods obj.method(arg) method(obj, arg) — extra self argument
Continuations (Scheme) (call/cc ...) Every function gets an extra continuation argument (the return address)
Nested Functions inner_func(arg) inner_func(env_ptr, arg) — extra environment pointer

Python makes this explicit — you write self as the first parameter of every method:

class MyClass:
    def greet(self, name):
        return f"Hello, {name}. I am {self.id}"

# C++ hides the same thing:
# void greet(MyClass* this, string name) { ... }
A General Pattern
The hidden argument pattern is one of the most powerful ideas in PL design. Whenever you see a feature that seems "magical" — methods, closures, continuations — ask yourself: could this just be a function with an extra argument? Usually, the answer is yes.

OOP Disputes: What Even Is Object-Oriented?

People have had heated debates about what "object-oriented" really means. Here are two of the main disputes:

Dispute 1: Must Every Value Be an Object?

LanguageAnswerExplanation
PythonYesEven 42 is an object (it has methods like 42 .bit_length())
SchemeYesEverything is a value/object, even though Scheme isn't class-based
C++Noint values are just raw integers in registers, not objects
JavaNoPrimitive types (int, boolean, etc.) are not objects; only reference types are

The pattern: lower-level, performance-focused languages tend to say "no" (keeping primitives as raw values is faster). Higher-level, more abstract languages tend to say "yes" (uniformity is simpler to reason about).

Dispute 2: Do Classes Need to Exist?

Classes bundle together a lot of responsibilities: they group methods and data, they're the unit of inheritance, and they're the unit of instantiation (new). Some people think this is too many responsibilities in one construct.

Prototypes vs Classes

In class-based languages (Java, C++, Python), there are two ways to create an object:

  1. Instantiation: new MyClass() — create from a class blueprint
  2. Cloning: obj.clone() — copy an existing object

Prototype-based languages (like JavaScript) simplify this: there are no classes. Every object is created by cloning an existing object (a "prototype").

Class-Based (Java): Prototype-Based (JavaScript): ┌─────────────────────┐ ┌──────────────────────┐ │ class Dog { │ │ dogProto = { │ │ name: String │ │ name: "Fido", │ │ bark(): void │ │ bark: function() │ │ } │ │ } │ └─────────┬───────────┘ └──────────┬───────────┘ │ new Dog() │ clone ↓ ↓ ┌─────────────────────┐ ┌──────────────────────┐ │ fido: Dog │ │ myDog = { │ │ name = "Fido" │ │ ...dogProto, │ └─────────────────────┘ │ name: "Rex" │ └──────────────────────┘

The research language Self pioneered this approach. JavaScript brought it to mainstream use. Instead of defining a class as a template, you just create a "prototypical" object with all the right properties and methods, then clone it whenever you need a new instance.

Why Prototypes?
The argument for prototypes is simplicity: one mechanism (cloning) instead of two (instantiation + cloning). Classes bundle too many concepts together — data layout, method dispatch, inheritance, and instantiation. Prototypes separate these concerns. JavaScript eventually added class syntax, but under the hood it's still prototype-based.
