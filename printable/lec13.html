Part 1 · Prolog Theory

From Logic to Computation

Before diving in, let's understand where we are in the big picture. You've been writing Prolog programs — facts, rules, and queries. But why does Prolog work the way it does? Today we go beneath the surface and see that Prolog isn't just "a language with weird syntax." It's a direct implementation of mathematical logic.

🗺 The Big Picture

In most languages (Python, Java, C), you tell the computer how to do things step by step. In Prolog, you tell it what is true, and it figures out the rest using logical reasoning. Today we'll see exactly what kind of logical reasoning, why Prolog restricts itself to a specific subset of logic, and what tradeoffs that creates. Then we'll meet Scheme — a language with a radically different philosophy: "How simple can a language be while still being useful?"

Review: First-Order Logic

Prolog is grounded in first-order logic (also called predicate calculus). This is a formal system that lets you make precise statements about the world using variables, predicates, quantifiers (like "for all" and "there exists"), and logical connectives (and, or, not, implies).

For example, you can write a statement like:

For all X, P(X) is true  <=>  there does not exist a Y such that not P(Y)

This says: "P is true for everything" is the same as saying "there's nothing for which P is false." When you add axioms (basic assumptions — for example, rules about how integers work), you can express facts about arithmetic, data structures, relationships, and much more.

This naturally raises a question: if I write some statement in first-order logic, can I figure out whether it's always true (a tautology)? That's the central question that connects logic to computation — and it's exactly what Prolog tries to do.

Clausal Form: A Standard Format for Logic

Here's a key insight from mathematical logic: any statement in first-order logic can be mechanically converted into a standard format called clausal form (also called conjunctive normal form). Think of it like converting any number into decimal — no matter how weird the original representation, you can always get it into this standard shape.

What Does a Clause Look Like?

A clause has this general structure:

A₁, A₂, ..., Aₘ  =>  C₁ ; C₂ ; ... ; Cₙ

Let's break this down piece by piece:

The left-hand side (A₁ through Aₘ) is a list of antecedents — the "if" part. These are joined by AND (all of them must be true). Think of these as conditions that need to hold.

The right-hand side (C₁ through Cₙ) is a list of consequents — the "then" part. These are joined by OR (at least one of them must be true). Think of these as things you can conclude.

The numbers m and n can be any non-negative integer:

If m = 0 (no antecedents): The "if" part is empty. By convention, AND-ing over nothing gives you true. So the clause is saying "this conclusion is unconditionally true." (Just like summing zero numbers gives you 0 — the identity element for addition — AND-ing zero things gives you true, the identity element for AND.)

If n = 0 (no consequents): The "then" part is empty. By convention, OR-ing over nothing gives you false. So the clause says "if these conditions hold, then false" — which is equivalent to saying "these conditions cannot all be true."

Example Clause

For all X, for all Y:   P(X) and P(Y)  =>  P(X) or P(Y)

Here m = 2 (two antecedents: P(X) and P(Y)) and n = 2 (two consequents: P(X) or P(Y)). Is this a tautology? Yes — intuitively, if both P(X) and P(Y) are true, then at least one of them is definitely true. Not every clause is a tautology, though. The goal of a theorem prover is to convert what you want to prove into clauses and check whether they hold.

💡 Why does this matter for Prolog?

You can take any logical statement and convert it into clauses. If Prolog could handle all possible clauses, it could prove or disprove anything expressible in first-order logic. But Prolog deliberately restricts itself to a subset of clauses for practical reasons — which is what we'll see next.

Horn Clauses: Prolog's Sweet Spot

People have built general-purpose theorem provers that handle any kind of clause. The problem? They're less efficient, harder for programmers to understand, and more difficult to implement at practical speeds. Prolog takes a different approach: pick a well-chosen subset of logic that's powerful enough to be useful but restricted enough to be fast.

That subset is called Horn clauses, named after logician Alfred Horn, who identified this important special case even before modern computers existed.

🔑 The Horn Clause Restriction

A Horn clause is a clause where n ≤ 1 — you can have at most one consequent. Any number of antecedents is fine, but the right-hand side can have at most one thing.

This single restriction gives us exactly three types of Horn clauses. Here's the beautiful part: each type maps directly to something you already know in Prolog.

Type 1: Facts (n = 1, m = 0)

No antecedents, exactly one consequent. Since the "if" part is empty (which means "true"), this clause says something is unconditionally true.

mortal(socrates).          % Socrates is mortal. Period.
singleton([X]).             % For all X, [X] is a singleton list.

By convention, the universal quantifiers ("for all X") are left out — they're implied. Every fact in Prolog is a Type 1 Horn clause.

Type 2: Rules (n = 1, m > 0)

Exactly one consequent, one or more antecedents. This is the standard Prolog rule.

mortal(X) :- man(X), living(X).
% Read: X is mortal IF X is a man AND X is living.
% (m = 2 antecedents, n = 1 consequent)

Type 3: Goals / Queries (n = 0, m ≥ 0)

No consequents, one or more antecedents. This is the type that drives computation.

Remember: the OR of nothing is false. So a clause with no consequents logically means:

A₁, A₂, ..., Aₘ  =>  false

Which is equivalent to saying: NOT (A₁ AND A₂ AND ... AND Aₘ). In other words, you're claiming these things are false. When you type a query like ?- mortal(socrates)., you're telling Prolog "mortal(socrates) is false" — and Prolog tries to prove you wrong!

?- mortal(socrates).      % "I claim there is no mortal Socrates"
TypeConstraintProlog FormExample
Factn=1, m=0head.mortal(socrates).
Rulen=1, m>0head :- body.mortal(X) :- man(X).
Queryn=0, m≥0?- goal.?- mortal(socrates).
⚠️ Why No OR in Prolog Rules?

Because Prolog requires n ≤ 1 (at most one consequent), some first-order logic formulas cannot be expressed as Horn clauses. For example:

licensed(X) => dog(X) ; cat(X) ; pig(X)

This says "if X is licensed, then X is a dog OR a cat OR a pig." This has three consequents — it's not a Horn clause. If you tried to write this in Prolog with an OR on the left-hand side of the :-, Prolog would reject it. This is a deliberate design limitation: by giving up the ability to express every logical statement, Prolog gains efficiency and clarity.

Proof by Contradiction: How Prolog Actually Works

Now we get to the core of Prolog's execution model. Understanding this will change how you think about every Prolog query you write.

Prolog's execution model is proof by contradiction (also called refutation). Here's how it works, step by step:

You give Prolog facts and rules

These are Type 1 and Type 2 Horn clauses. Prolog assumes all of them are true. It stores them in a database and does nothing with them — it just remembers them.

You give Prolog a query

This is a Type 3 Horn clause. Prolog interprets this as your claim that something is false. When you write ?- mortal(Z)., you're saying "for all Z, Z is not mortal."

Prolog tries to find a contradiction

It takes your negative claim and tries to derive a contradiction from it using the facts and rules in its database. If it finds one, it reports the counterexample (the variable bindings that contradict your claim).

Prolog reports the result

If it finds a contradiction, it shows you the counterexample. If it can't find one, it says "no" — not that your claim is true, just that it couldn't disprove it.

Example Walkthrough

man(socrates).
living(socrates).
mortal(X) :- man(X), living(X).

?- mortal(Z).

The query ?- mortal(Z). asserts: "for all Z, Z is not mortal." Prolog looks at its database and finds Z = socrates as a counterexample — Socrates IS mortal (because he's a man and he's living). Contradiction found!

Prolog reports: Z = socrates

🔑 Key Distinction: What "yes" and "no" Really Mean

Prolog says "yes" = A counterexample was found. Your negative claim was disproved. The positive statement IS true (at least for this counterexample).

Prolog says "no" = It tried to find a counterexample and couldn't. This does NOT mean your claim is true — it just means Prolog couldn't disprove it.

Real-World Analogy

Think of the American jury system. The verdict is either "guilty" (we proved they did it) or "not guilty" (we couldn't prove they did it). "Not guilty" doesn't necessarily mean "innocent" — it could mean innocent, or it could just mean the evidence wasn't sufficient. Prolog's "no" is exactly like "not guilty."

Resolution: The Mechanical Proof Procedure

We now know what Prolog is doing (proof by contradiction), but how does it mechanically carry this out? The answer is a procedure called resolution.

The Resolution Step

Suppose we have two things:

1. A Horn clause (rule or fact) with head C and body A₁, ..., Aₘ

2. A current list of goals: G₁, G₂, ..., Gₚ

Resolution works as follows:

Compute a unifier U

Find a substitution of values for variables that makes C (the head of the clause) identical to G₁ (the first goal). This is the same unification process you've already seen in Prolog.

Replace the goal with the clause body

Take G₁ out of the goal list and replace it with A₁, ..., Aₘ (the body of the matched clause). Apply the unifier U to everything.

New goal list

The new goal list becomes: A₁·U, A₂·U, ..., Aₘ·U, G₂·U, ..., Gₚ·U

Prolog repeats this process until one of two things happens:

• Goal list is empty → contradiction found (success! Prolog reports the counterexample)

• No clause can unify with the current goal → backtrack (try another clause) or fail

✅ The Punchline

Resolution is a sound and complete proof procedure for Horn clauses. This means Prolog's computation is a direct implementation of logical reasoning. When Prolog gives you an answer, that answer genuinely follows from the logic you wrote down.

🗺 Connecting It All Together

So here's the full picture: (1) You write facts and rules — these are Horn clauses that Prolog assumes are true. (2) You write a query — Prolog treats this as a negative claim. (3) Prolog uses resolution to try to derive a contradiction from your claim. (4) If it succeeds, it shows you the counterexample. This chain — from first-order logic, to clausal form, to Horn clauses, to resolution — is why Prolog programs are "logical" in a very precise, mathematical sense.

Part 2 · Prolog Control

Order, Efficiency, and Cut

We've established that Prolog is grounded in logic. But logic alone isn't enough to write practical programs. Prolog is a marriage of two things: Logic (what the correct answers are) and Control (how and in what order to search for them). Now we explore the control side — and see where it can go wrong.

Order Matters — A Lot

Logically, the order of clauses doesn't change the set of correct answers. AND is commutative: A ∧ B is the same as B ∧ A. But practically, order matters enormously in Prolog. It changes which answer you find first, how fast you find it, and even whether your program terminates at all.

Example: member/2 with reversed clause order

% Standard (base case first — the usual way):
member(X, [X|_]).                    % base case first
member(X, [_|T]) :- member(X, T).   % recursive case second

% Reversed (base case last — unusual):
member(X, [_|T]) :- member(X, T).   % recursive case first
member(X, [X|_]).                    % base case second

These are logically equivalent — they define the same relationship. But they behave differently:

If you query ?- member(Z, [x, y, z]). with the standard definition, you get answers in order: Z=x, then Z=y, then Z=z.

With the reversed definition, Prolog recurses to the end of the list first, so you get Z=z, then Z=y, then Z=x.

Even worse, some orderings can cause infinite loops — the program never terminates, even though logically the answers exist.

Running Predicates "Backwards"

Here's an example where ordering creates a much bigger efficiency problem. Imagine Murphy Hall (the UCLA registrar's office) gives you a database of course prerequisites:

prereq(cs32, cs131).
prereq(cs131, cs132).
prereq(cs132, cs232).
% ...50,000 more entries for all departments...

You're asked to write a predicate that computes the transitive closure — whether one course is a prerequisite (possibly through a chain of intermediate courses) for another:

tcpr(A, Z) :- prereq(A, Z).                   % base case: direct prereq
tcpr(A, Z) :- tcpr(A, B), prereq(B, Z).        % recursive: chain through B

Forward query: ?- tcpr(cs32, X). works great. A is bound to cs32, so tcpr(cs32, B) in the recursive case efficiently finds immediate prerequisites, and chains forward.

Backward query: ?- tcpr(X, cs232). is a disaster. In the recursive case, the subgoal tcpr(X, B) has X unbound. Prolog has to search all 50,000 prerequisite entries trying to find matches. Incredibly slow!

Could you fix the backward query by swapping the two subgoals in the recursive case?

tcpr(A, Z) :- prereq(B, Z), tcpr(A, B).  % swapped subgoals

Yes — this fixes the backward query. But now the forward query becomes slow instead!

⚠️ The Takeaway

There is no single ordering that is efficient in both directions. Prolog programmers must consciously choose clause and subgoal ordering to match the intended direction of query. Logic is symmetric; control is not. This is a fundamental tension in logic programming.

The Problem of Unnecessary Backtracking

Here's a more subtle efficiency problem. Imagine you're writing a predicate with this structure:

p(X, Y, Z) :-
    ... generate candidate values for Q and X ...,
    member(Q, X),             % fast membership check
    ... expensive_test(Q, B) ... .   % slow, costly test

The idea: the first part generates candidates. member(Q, X) is a quick filter. expensive_test is the real verification (maybe it talks to an external database — very slow).

The problem emerges when we trace what happens. Suppose Q = a and X = [a, b, c, d, e]:

member(a, [a,b,c,d,e]) succeeds immediately

The first clause of member matches: a is the head of the list. Fast!

expensive_test(a, B) runs and... fails

The costly test takes a long time and determines this candidate doesn't work.

Prolog backtracks into member

member has another way to succeed: try the recursive case. It recurses through the list and eventually finds 'a' again (as the first element of the remaining list).

expensive_test(a, B) runs AGAIN with the same value

Complete waste of time! It already failed with Q=a, and it'll fail again.

This happens 5 times (once for each element in the list where member can find 'a' through different recursive paths). The expensive test runs 5 times with the same value — making the code 5 times slower than necessary.

What we need is a way to make member succeed at most once for a given value. That's where cut comes in.

The Cut Predicate (!)

Cut (!) is a meta-logical control predicate. "Meta-logical" means it's not expressible in pure first-order logic — it's an extra tool that Prolog provides for controlling how the search works. The exclamation point is deliberate: it's short to type, and the "!" warns you there's some danger involved.

How Cut Works

To understand cut, you need to think about Prolog's proof tree — the tree of all possible ways to prove a goal.

1. When Prolog encounters !, it succeeds immediately (just like true).

2. If Prolog later backtracks into the !, the entire calling predicate immediately fails — no other clauses for that predicate are tried.

3. Conceptually, cut picks up a pair of scissors and prunes all remaining alternatives from the proof tree for the predicate containing the cut.

membercheck/2: Cut in Practice

membercheck(X, [X|_]) :- !.
membercheck(X, [_|T]) :- membercheck(X, T).

Let's trace through different scenarios:

If X matches the head of the list: The first clause fires. Cut executes (succeeds). membercheck succeeds exactly once. If the caller backtracks, they hit the cut, and membercheck fails immediately — the second clause is never tried.

If X does NOT match the head: The first clause fails before reaching the cut. So the second clause is tried normally, and we recurse.

✅ Rule of Thumb

If every clause in a predicate (except the last) ends with !, the predicate will succeed at most once. This can dramatically improve efficiency for "checking" predicates — situations where you only care whether something is true, not about finding multiple ways to prove it.

Now we can fix the inefficiency from earlier: just change member(Q, X) to membercheck(Q, X) in our predicate. The membership check succeeds once, the expensive test runs at most once per candidate value, and our program runs at the correct speed.

The Danger of Cut

If cut is so useful, why doesn't Prolog use it everywhere by default? Because cut breaks the purely logical interpretation of Prolog programs. Once you use cut, you can no longer rely on the guarantee that Prolog's answers are logically correct.

Example: Cut Causes a Wrong Answer

?- membercheck(X, [a,b,c]), X = b.
% Expected: X = b (because b IS in the list)
% Actual:   no

Let's trace why this happens:

Goal: membercheck(X, [a,b,c]), X = b.
Clause 1 matches: X = a, list head = a. ✓ Cut fires.
Now try: a = b → FAILS (a ≠ b)
Backtrack. But we hit the cut!
membercheck immediately fails. No more clauses tried.
Prolog reports: no
But logically, X = b IS a valid solution!

Here's what happened: membercheck bound X to 'a' (the first element). Cut fired. Then X = b failed because X is already 'a'. Prolog tried to backtrack, but hit the cut — so membercheck failed entirely. The logical answer (X = b) exists, but Prolog can't find it.

⚠️ Critical Warning

Once you use cut, you can no longer rely on pure logical correctness. Cut can cause Prolog to answer "no" even when a logical solution exists. Use it carefully and deliberately — only when you understand exactly which solutions you're cutting away and why that's acceptable for your use case.

Negation-as-Failure (\+)

There's something Prolog programmers often want but can't get from pure logic: the ability to say "this is NOT true" inside a rule body. Remember, the only place "not" appears in Prolog is implicitly in queries (the ?- prompt). You can't use NOT inside your rules.

For example, the registrar wants a predicate npr(A, B) that succeeds when A is not an immediate prerequisite for B. With pure Prolog, there's no way to write this.

The workaround is negation-as-failure, written \+. Here's its source code — it's built from cut:

\+(P) :- call(P), !, fail.
\+(_).

Let's walk through how this works:

Try to prove P

The first clause calls P. If P succeeds...

If P succeeds: cut-fail

We execute cut (preventing backtracking) and then fail. So \+(P) fails. "I proved P is true, so P is NOT 'not true' — therefore \+(P) should fail."

If P fails: fall through to second clause

If P can't be proven, we never reach the cut. We fall through to the second clause \+(_). which always succeeds. So \+(P) succeeds. "I couldn't prove P, so I'll say P is 'not true'."

The Variable Instantiation Problem

Here's where negation-as-failure gets dangerous:

?- X = 3, \+(X = 4).     % succeeds, X = 3 ✓
?- \+(X = 4), X = 3.     % FAILS ✗

The first query works: X is bound to 3, then we check "is 3 ≠ 4?" — yes, it succeeds.

The second query fails: \+(X = 4) is called with X unbound. Prolog tries to prove X = 4 — and it succeeds (by binding X to 4)! So \+ fails. The entire query fails.

We just swapped two conjuncts (A AND B became B AND A) and got a different answer. AND is supposed to be commutative! But once \+ is involved, it's no longer commutative.

✅ Rule of Thumb

Only use \+ when its argument is fully ground (no unbound variables). If you use \+ with unbound variables, you will get incorrect behavior. The negation-as-failure operator can never instantiate (bind) variables — the only way it succeeds is by not doing anything.

The Closed World Assumption

There's a deep philosophical issue lurking behind negation-as-failure. When we asked the registrar for prerequisites and got a list of facts, can we conclude that dance101 is NOT a prerequisite for cs131? Logically, no. The list only tells us what IS true — it says nothing about what's false. Maybe the registrar forgot to include it!

But in practice, we make an assumption:

🔑 The Closed World Assumption (CWA)

Anything not stated in the database is false. If a fact is not present, it is assumed to be untrue.

This assumption is built into Prolog's negation-as-failure. When \+(P) succeeds because P can't be proven, Prolog is essentially saying "P is false" — which is only valid if the database is complete.

When the CWA is valid: When your database is authoritative and complete. The registrar's official prerequisite list really is the complete list — if a prerequisite isn't listed, it doesn't exist.

When the CWA is dangerous: When your database is partial or open-ended. If you're building a criminal case database, "not in the guilty list" does NOT mean "innocent" — it just means "not convicted." The database doesn't have complete information about the world.

🗺 Connecting to Databases

This is the same problem as the NULL problem in relational databases. In SQL, a NULL value means "unknown" — not "false." Database designers constantly grapple with the question: does the absence of a record mean the thing is false, or just that we don't have the information? Prolog's CWA is the same issue in a different guise.

Part 3 · Introduction to Scheme

The Simplest Useful Language

We're shifting gears now. After Prolog (logic programming) and OCaml (typed functional programming), we're meeting Scheme — a language that asks: "How much can we throw away and still have something useful?"

Why Study Scheme After OCaml?

You might think: Scheme is simpler than OCaml — why bother studying a simpler language? There are two compelling reasons:

Reason 1: Minimalism as a design exercise. Scheme represents a serious attempt by computer science experts to design the simplest possible language that is still fully general and useful. It's easy to make a simple language that can't do anything. It's easy to make a powerful language that's complex (look at C++ or JavaScript). The hard part is having simplicity and generality and usefulness all at once. What can be stripped away while retaining power?

Reason 2: Extra power from simplicity. Scheme's simplicity enables techniques that are technically possible in other languages but so cumbersome that nobody actually does them. In particular, Scheme makes it trivial to write programs that generate and run other programs. In C++, you could write a program that generates C++ source code, invokes the compiler, dynamically links the resulting object file, and runs it. But nobody does this because it's incredibly painful. In Scheme, you just build a list and evaluate it. This is why Lisp/Scheme dominated symbolic AI research for decades.

🗺 Where Scheme Fits

So far in this course: OCaml = typed functional programming with pattern matching. Prolog = logic programming via proof by contradiction. Scheme = untyped functional programming with extreme simplicity and code-as-data. Each language makes different tradeoffs, and studying all of them helps you understand the design space of programming languages.

The Eight Core Properties of Scheme

Scheme is defined by eight properties. Many languages have some of them, but any language with all eight is essentially Scheme. Let's go through them one at a time.

Property 1: Dynamic Allocation with Garbage Collection

All objects in Scheme live on the heap (dynamic allocation). You never explicitly free memory — a garbage collector automatically reclaims objects when they're no longer needed. You don't have to worry about dangling pointers, double-frees, or memory leaks from forgetting to free something.

This isn't unique to Scheme — Python, Java, OCaml, and most modern languages do this too. But Scheme was one of the earliest languages to make this a core requirement.

Property 2: Dynamic (Latent) Type Checking

Types belong to objects (values), not to variables or source code expressions. Scheme uses two terms to distinguish approaches:

Latent types (what Scheme uses): Type information travels with the data at runtime. A variable doesn't have a type — it can hold any kind of value. You only find out if there's a type error when the program actually runs and tries to do something invalid (like adding a string to a number).

Manifest types (static typing): Type information is visible in the source code. You can look at a variable declaration and know what type it is. Java, OCaml, and Rust work this way.

In this respect, Scheme is like Python — variables are just names that point to values, and values carry their own types.

Property 3: Static (Lexical) Scoping

Every variable reference in Scheme can be resolved to its declaration by reading the source code alone — no runtime information needed. If you see a variable x used inside a function, you can figure out where x was declared just by looking at the surrounding code.

This is in contrast with dynamic scoping, which is what traditional Lisp uses. With dynamic scoping, to find a variable's value, you look at the function that called you, then the function that called that, all the way up the call stack. You can't determine which variable you're referring to until runtime, because it depends on who called you.

Familiar Example

You've actually already encountered dynamic scoping — in Bash! Environment variables like $PATH are dynamically scoped. A shell script inherits whatever variables the calling shell has set. If script A sets PATH=/usr/bin and then calls script B, script B sees that PATH value. If script C sets PATH=/opt/bin and calls script B, script B sees a different PATH. The value depends on who called you, not on where you were defined.

Dynamic scoping is widely considered a design mistake. It's confusing to read code (you can't tell what a variable refers to without knowing the runtime call stack), it's confusing for compilers (harder to optimize), and it's less efficient (runtime lookup required). Even Common Lisp and Emacs Lisp, which are dynamically scoped by default, provide opt-in static scoping because dynamic scoping is such a pain.

Scheme says: "No. Static scoping only. If you want to pass data from a caller, pass it as an argument."

Property 4: Call by Value

When you call a function in Scheme, the caller evaluates all arguments first, then passes copies of the resulting values to the function. The function cannot modify the caller's variables.

This is the same as C, Python, and OCaml. The only common counterexample you've likely seen is C++, which supports call by reference (where the function can modify the caller's variables directly). Scheme says no — call by value only. Keep it simple.

Property 5: Procedures as First-Class Objects

Functions (called "procedures" in Scheme) are values just like any other — you can store them in variables, pass them as arguments to other functions, return them from functions, and put them in lists. This is what "first-class" means: functions are full citizens of the language, not second-class constructs.

Python does this too (functions are objects). But Scheme is stricter about it: in Java, primitive types like int and float are not objects — they're something different. In Scheme, everything is an object. There are no exceptions, no "primitive types that aren't really objects." In this sense, Scheme is more uniformly object-oriented than Java is, even though Scheme doesn't have classes.

Property 6: Homoiconicity — Code as Data

This is Scheme's most distinctive property — the one that sets it apart from almost every other language.

Scheme has the simplest possible syntax, based entirely on parenthesized lists. And here's the key insight: every valid Scheme program can be directly represented as a Scheme data structure (specifically, a list). The syntax for code and the syntax for data are the same thing.

This means writing Scheme programs that generate and execute other Scheme programs is trivial. You just build a list (which represents a program), and evaluate it. That's it.

Compare this to C++: if a C++ program wanted to generate another C++ program at runtime, it would need to: generate source text as a string, write it to a file, invoke the C++ compiler on that file, dynamically link the resulting object file back into the running program, and then call the compiled function. Technically possible, but incredibly cumbersome. Nobody does it.

In Scheme? You just build a list and call eval on it. Done.

💡 Historical Connection

This property is why Lisp/Scheme dominated symbolic AI research for decades. Genetic algorithms (programs that modify themselves), meta-programming, and code generation are all natural in Scheme. Even JSX in JavaScript (mixing HTML and JavaScript) got its idea from Scheme/Lisp's code-as-data paradigm.

Property 7: High-Level Arithmetic

Scheme treats numbers as mathematical objects, not as machine words. This has several consequences:

Arbitrary-precision integers: No overflow. Numbers just keep growing as needed. Once you go past what fits in a 64-bit integer, Scheme automatically switches to multi-precision arithmetic. You'll never see incorrect results from integer overflow.

Exact rational arithmetic: The value 1/3 is stored as the exact fraction 1/3, with no rounding error. No floating-point approximation. You can do arithmetic with fractions and get exact results.

Sane float printing: When you print a floating-point number, Scheme prints exactly enough digits so that reading it back in reproduces the exact same bit pattern. You'll never see 0.09999999997 when you meant 0.1. Even though internally 0.1 is an infinitely repeating binary fraction that gets rounded, Scheme prints it as 0.1 because that's the shortest decimal that round-trips perfectly.

Complex numbers: Built-in as a first-class numeric type. You can write 3+4i directly.

Property 8: Tail Recursion Optimization (TRO) is Required

A tail call is a function call that is the very last action of a function — after the call returns, the caller has nothing left to do except return the same value. For example:

(define (f x)
  ... compute y ...
  (g y))     ; this is a tail call — f's last action

Tail Recursion Optimization (TRO) means: before making the tail call, the system reclaims the calling function's stack frame. The new function reuses that space. No extra stack growth.

Here's why this matters. In C/C++, TRO is an optional optimization — compilers like GCC and Clang do it with -O2, but they're not required to. In those languages, it's no big deal because programs tend to be iterative (loops), not deeply recursive.

In Scheme, TRO is mandated by the language specification. Every conforming Scheme implementation MUST do it. This is crucial because Scheme encourages recursion as the primary way to express iteration. With TRO guaranteed, a properly tail-recursive function uses constant stack space regardless of recursion depth — you could recurse a million times deep and use only one stack frame.

✅ For Your Assignment

Write recursive functions (not while loops) and structure them to be tail-recursive. You will not blow the stack. Scheme has loops, but you should use recursion — once you have recursion with TRO, you don't need loops.

Scheme Syntax

Scheme's syntax is remarkably compact. The designers deliberately traded syntactic convenience for uniformity and homoiconicity. Let's learn it piece by piece.

Comments

; Everything from a semicolon to end of line is a comment
; Just like // in C++ or # in Python

Identifiers

Scheme identifiers are unusually permissive. Valid characters include letters, digits, and a huge set of special characters:

+ - . ! # $ % & * / : < = > ? @ ^ _ ~

Most of the characters you'd think of as "operators" or "punctuation" in other languages are just valid parts of identifiers in Scheme! This means there's very little room for punctuation — which is intentional.

Restriction: Identifiers cannot begin with a digit, +, -, or . (because those would look like numbers). Exception: The three identifiers +, -, and ... are valid (they're special cases that aren't followed by digits).

Identifiers serve three roles: variables (names of runtime values), symbols (data, not evaluated), and keywords (built-in syntactic forms like let, if, define).

Importantly, keywords are not reserved words — you can rebind them:

(let ((+ 3)) (* + +))  ; + now means 3; result is 9

This rebinds + to the number 3 within the let scope, then multiplies it by itself. Scheme doesn't care that + is normally the addition function — it's just an identifier.

Lists

(a b c)        ; proper list of 3 elements
()             ; empty list
(a b . c)      ; improper list: tail is symbol c, not ()
(a . b)        ; a pair (cons cell) with car=a, cdr=b

Lists are written with parentheses, elements separated by spaces — no commas needed. An improper list (where the tail isn't a list) uses dot notation with spaces around the dot.

Booleans

#t    ; true
#f    ; false
⚠️ Important

#f is the ONLY false value in Scheme. Everything else — including 0, () (empty list), and "" (empty string) — counts as true. This differs from C (where 0 is false), Python (where empty containers are false), and traditional Lisp (where the empty list is also false). In Scheme, only #f is false. Period.

Strings and Characters

"hello, world"    ; string literal (double quotes)
#\a               ; character literal for 'a'
#\space           ; character literal for space

Numbers

42       ; exact integer
3.14     ; inexact (floating-point)
1/3      ; exact rational (stored as the fraction 1/3)
3+4i     ; complex number

Quoting: The Bridge Between Code and Data

This is one of the most important concepts in Scheme, and it directly relates to homoiconicity (Property 6). Quoting is how you tell Scheme "treat this as data, not as code."

The Quote Operator (')

A single quote (apostrophe) in front of an expression prevents evaluation — it returns the expression as raw data:

'(+ 1 2)      ; => the list (+ 1 2), NOT the number 3
'x             ; => the symbol x (data, not the variable x)
''x            ; => the list (quote x)

Without the quote, (+ 1 2) would be a function call: call the + function with arguments 1 and 2, producing 3. With the quote, it's just a list containing three elements: the symbol +, the number 1, and the number 2.

Technically, 'expr is syntactic sugar for (quote expr). They mean the same thing. Quoting is essential for the code-as-data paradigm: to get a piece of code as a data object, you quote it.

Quasiquoting: Templates with Holes

Regular quoting is all-or-nothing — everything inside the quote becomes data. But what if you want most of an expression to be data, but you want to compute and insert some parts? That's what quasiquoting does.

Quasiquote uses a backtick (`) instead of a regular quote. Inside a quasiquoted expression, you can use a comma (,) to "unquote" — to say "evaluate this part and insert the result."

(let ((x 42)) `(a ,x c))   ; => (a 42 c)

Here, `(a ,x c) creates a template: a and c are literal data, but ,x means "evaluate x and insert its value." Since x is 42, the result is the list (a 42 c).

Think of quasiquoting as a template system: the backtick creates the template, and commas mark the "holes" where computed values get inserted.

💡 Connection to JavaScript

If you've used JSX in React, you've seen this idea: JSX lets you write HTML-like templates, and curly braces {} let you drop back into JavaScript to compute values. Quasiquoting in Scheme is the same concept — and in fact, the JavaScript/JSX designers got this idea from Scheme and Lisp.

Function Calls and the Parentheses Rule

Here's how you call a function in Scheme:

(f e₁ e₂ ... eₙ)

The first element (f) must evaluate to a procedure/function. The remaining elements are arguments, evaluated left to right. n can be 0 (calling a function with no arguments).

The Critical Parentheses Rule

In most languages (C, Python, JavaScript), parentheses are often optional or used for clarity. If you can't remember operator precedence, just add more parentheses — no harm done.

In Scheme, every parenthesis has semantic meaning. You cannot add extra parentheses "for clarity" — doing so changes what your program means.

(cos x)     ; ✓ CORRECT: calls cos with argument x

((cos) x)   ; ✗ WRONG: calls cos with NO arguments,
             ;   then tries to call the RESULT as a function with argument x

The second form, ((cos) x), doesn't mean "cosine of x with extra parentheses." It means: first, call cos with zero arguments (which would be an error, since cos needs exactly one argument). If that somehow returned a function, then call that function with x.

Adding a parenthesis = adding a function call. Removing a parenthesis = removing a function call. This is a direct consequence of Scheme's uniform syntax: everything is either data or a function call, and parentheses are how you distinguish between them.

⚠️ Remember This

In Scheme, extra parentheses change meaning. (x) means "call x as a function." x means "the value of variable x." They are completely different things. If you're coming from C/Python/Java, you need to break the habit of adding parentheses for clarity.
