🗺 The Big Picture

This lecture covers three intertwined themes:

1. Parsing — How do we take a flat string of code and figure out its structure? This is the core problem your homework is solving.

2. Hardware Realities — The world's fastest computer shows us that how you program something matters as much as how powerful the machine is. This motivates why programming language design choices matter.

3. Java — A language designed to solve real engineering problems (portability, reliability, safety). Understanding why Java was built the way it was explains every design decision in the language.

These three topics connect: parsing is how we turn text into structure, hardware shows us the constraints we program under, and Java is one answer to the question "how should we design a language given those constraints?"

Why Parsing Matters

Before a computer can run your code, it has to understand what your code means. But code starts as just a long string of characters — letters, symbols, spaces. Parsing is the process of taking that flat string and figuring out its structure.

Think of it like reading a sentence in English. The words "time flies like an arrow" could mean different things depending on how you group the words. Is "flies" a verb (time moves like an arrow) or a noun (there are flies called "time flies" that enjoy an arrow)? Parsing is about resolving exactly this kind of structural ambiguity — but for programming languages.

In your course, you're not just writing a parser — you're writing a parser generator: a function that takes a grammar (the rules of a language) and automatically produces a parser. That's a level above: you're building the tool that builds parsers.

Finite State Machines

Definition

Finite State Machine (FSM): A simple computational model that has a fixed number of states, and no memory other than knowing which state it's currently in. It reads input one symbol at a time and transitions between states based on what it reads.

Imagine you're a security guard at a door. You have a short checklist: "If someone says the first password word, move to state 2. If in state 2 and they say the second word, let them in. Otherwise, go back to the start." That's a finite state machine — you only need to remember where you are in the checklist, nothing else.

Key properties of a finite state machine:

— No stack: it can't remember a growing list of things
— No recursion: it can't call itself and remember where to come back to
— No counting: it can't track "how many times has X happened" beyond its fixed states

Connection

Regular expressions can be parsed by finite state machines. Each part of a regex pattern can be checked by walking through the input left to right. You never need to "remember" nested structure — you just check one symbol at a time.

Context-Free Grammars & The Stack

Regular expressions are powerful, but they have a hard limit: they can't handle nesting. And nesting is everywhere in programming languages — think of nested parentheses ((())), nested if-statements, nested function calls.

Definition

Context-Free Grammar (CFG): A set of rules that describe a language which can have nesting. To parse a CFG, you need a finite state machine plus a stack — extra memory that can grow as deeply as the nesting goes.

Why do we need a stack?

Consider matching parentheses. When you see (, you need to remember "I'm inside one level of parens." When you see another (, you're now two levels deep. When you see ), you go back one level. This "remembering depth" requires memory that grows — that's what the stack does.

Input:   ( ( a + b ) * c )

Stack after each (
─────────────────
See first ( → push → stack: [(]
See second ( → push → stack: [(, (]
See first ) → pop → stack: [(]
See second ) → pop → stack: []  ✓ matched!
The stack tracks nesting depth. A FSM alone can't do this for arbitrary depth.
Key Formula

CFG = Finite State Machine + Stack

This is the fundamental equation of parsing theory. Regular expressions need only a FSM. Context-free languages (which describe most programming languages) need that stack on top.

But we don't need to literally build a FSM + Stack

Here's the practical insight: programming languages already have a stack built in. Every time you call a function, the language pushes state onto the call stack. Every time you return, it pops. So instead of manually implementing a stack data structure, you can use recursive functions — and the programming language's own call stack does the work for you.

This is exactly what you're doing in your homework: writing recursive functions that parse a grammar, where the recursion naturally tracks nesting depth through the call stack.

The Ambiguity Problem in Parsing

When a parser reads input left to right, at any point it has seen some tokens and hasn't seen the rest. The challenge is: based on what you've seen so far, which grammar rule should you apply next?

Sometimes the answer is obvious. But often it's not — multiple rules could plausibly apply, and you won't know which is correct until you've read more input.

A Real Example

Suppose you're parsing code and you've just read everything up to an open parenthesis (. What comes next? There are several possibilities:

(* Possibility 1: start of a sub-expression *)
( x + y )

(* Possibility 2: a type cast *)
(int) x

(* Possibility 3: a tuple *)
(3, 6)

All three start with (. The parser has read the same tokens so far, but needs to do completely different things depending on which case it is. This is the ambiguity problem.

Lookahead

Definition

Lookahead: A technique where the parser peeks at the next token(s) without consuming them to decide which grammar rule to apply.

For the parenthesis example above: if after ( the parser peeks ahead and sees a type name like int, it can assume it's a cast. If it sees a variable or number, it's probably an expression.

Lookahead is covered extensively in follow-on courses (like CS 132). In this course, we use a different — lazier but simpler — approach.

Backtracking Parsers

Definition

Backtracking: Instead of trying to predict the right rule upfront, just try one. If it works, great. If it fails, undo everything and try the next option.

Think of it like navigating a maze. At each fork, you pick a path. If you hit a dead end, you walk back to the fork and try a different path. Eventually you find the exit (or exhaust all options).

Important Limitation

Backtracking only works for unambiguous grammars — grammars where each input has exactly one valid parse tree. If a grammar is ambiguous (multiple valid parse trees for the same input), a backtracking parser might find one valid parse but miss that there are others. It can never be sure it found the "right" one.

The Three Problems a Backtracking Parser Must Solve

Listed from easiest to hardest:

Problem 1 — Recursive Grammar: Your grammar has rules that refer to themselves. Solution: write recursive functions. The grammar's recursion maps directly to function recursion. This is the easy one.

Problem 2 — Disjunction (OR): A non-terminal has multiple production rules, and you can't tell which to use from the current input. Solution: try the first option; if it fails, backtrack and try the second; and so on.

(* E can be parsed as A B C  or  as D E F *)
(* We don't know which — try one, backtrack if needed *)
E -> A B C
E -> D E F

Problem 3 — Concatenation: This one sounds easy but is actually the trickiest. When you need to parse F followed by A, you parse F first, then try A on the remaining input. But if A fails, you can't just give up — you need to go back to F and ask "do you have a different way of matching?" If F matches differently (consuming more or fewer tokens), maybe A will succeed on the new remainder.

(* Parse F, then A on whatever's left *)
E -> F A

(* If F succeeds but A fails on the remainder: *)
(*   → backtrack to F, ask for a different match *)
(*   → retry A on the new remainder *)
(*   → repeat until both succeed or F runs out of options *)
(* With three parts, backtracking gets deeper *)
E -> A B C

(* If A,B work but C fails → backtrack to B *)
(* If no B alternatives left → backtrack to A *)

Matchers & Acceptors

Now we get to the actual machinery that makes backtracking work. The key insight is to split the parser into two cooperating parts: matchers and acceptors.

Definition

Matcher: A function that tries to match some initial prefix of the input. It takes an acceptor and an input fragment (a list of tokens), and returns either the unmatched suffix (if parsing succeeds) or None (if it fails).

Type signature: matcher : acceptor → fragment → fragment option

Definition

Acceptor: A function that receives the remaining suffix after a matcher has matched some prefix, and decides whether this particular parse is acceptable. It returns the suffix if it likes the parse, or None if it doesn't.

Think of a matcher as a chef who prepares a dish (matches a prefix), and the acceptor as a food critic who tastes what's left over. The chef says "I've handled the appetizer, here's the rest of the meal for you to judge." The critic either approves or says "no good, try a different appetizer."

Here's how the input gets divided:

Input fragment: [t₁, t₂, t₃, t₄, t₅, t₆, t₇]
               ├── matched prefix ──┤├── unmatched suffix ──┤
                 (matcher handles)    (passed to acceptor)

If acceptor says "None" → matcher tries a different split

The Simplest Matchers

(* Match the empty prefix — always succeeds *)
(* The suffix is the entire input, unchanged *)
let match_empty accept frag = accept frag

(* Match the entire input — suffix must be empty *)
let match_all accept _ = accept []

(* Never match anything — always fails *)
let match_nothing _ _ = None

Notice the difference between match_empty and match_nothing:

— match_empty matches the empty prefix (zero tokens) and passes everything to the acceptor. It's like a grammar rule that produces ε (the empty string). The acceptor still gets a chance to accept or reject.

— match_nothing never even tries. It immediately returns None no matter what. It's like a grammar with no production rules at all — total failure.

The Simplest Acceptors

(* Always reject — no matter what suffix, say "no" *)
let accept_nothing _ = None

(* Always accept — no matter what suffix, say "yes" *)
let accept_all frag = Some frag

Building a Matcher for Disjunction (OR)

Now let's build something real. Suppose our grammar says:

E -> A B C
E -> D E F

This means E can be parsed as either A B C or D E F. We need a matcher that tries the first option, and if it fails, falls back to the second.

We'll assume we already have a function make_matcher that can build matchers for any rule. (This is the recursive trust: we're building the system, and we trust that the other pieces will work once we finish.)

Step 1 of 4 
(* Build matchers for each alternative *)
let m1 = make_matcher (A, B, C)
and m2 = make_matcher (D, E, F)
in

(* Return a matcher for the disjunction *)
fun accept frag ->
    match m1 accept frag with
    (* m1 failed → try m2 *)
    | None -> m2 accept frag
    (* m1 succeeded → use its result *)
    | x -> x

The logic is simple: give the input to m1. If m1 returns None (failure), try m2 with the same input and acceptor. If m1 returns something (success), use that result — we don't even bother trying m2.

Building a Matcher for Concatenation

Now the hard part. Suppose our grammar says:

E -> A B

We need to parse A first, then B on whatever A didn't consume, and then pass the remaining suffix to our acceptor.

Input: [t₁ t₂ t₃ t₄ t₅ t₆ t₇]
       |← A →|← B →|← suffix →|

The acceptor checks the suffix.
But we don't know where A ends and B begins!
The split point between A and B is unknown — that's the challenge.

The Trick: Turn B + Acceptor into an Acceptor for A

Instead of trying every possible split, we use a clever functional programming trick. We bundle up "match B, then check with the acceptor" into a single acceptor that we hand to A.

When A finishes matching some prefix, it calls this bundled acceptor. The bundled acceptor then tries to match B on whatever A left behind. If B succeeds, it passes the remaining suffix to the original acceptor. If B fails, A's matcher will backtrack and try a different prefix.

let m1 = make_matcher A
and m2 = make_matcher B
in

(* Return a matcher for the concatenation *)
fun accept ->
    (* Call m1, giving it an acceptor that runs m2 *)
    m1 (fun frag1 -> m2 accept frag1)

Let's unpack this line by line:

— fun accept → : our concatenation matcher takes an acceptor

— m1 (fun frag1 → m2 accept frag1) : we call m1 (matcher for A) and pass it a custom acceptor

— The custom acceptor (fun frag1 → m2 accept frag1) says: "when A is done and gives me the leftover frag1, try matching B on frag1, and pass B's leftover to the original accept."

Why This Works for Backtracking

If B fails (returns None), then the acceptor we gave to m1 returns None. From m1's perspective, the acceptor rejected that particular parse of A. So m1 automatically backtracks and tries a different way to match A — consuming more or fewer tokens. This back-and-forth continues until either both A and B succeed, or m1 exhausts all possibilities.

This is like a curried optimization. Notice we wrote fun accept → m1 (fun frag1 → m2 accept frag1) and dropped the frag argument. We don't need to explicitly pass the fragment because m1 already expects one — it's curried. We just build the function and m1 will apply it to whatever fragment it receives.

Left Recursion vs Right Recursion

There's one critical problem this approach can't handle.

Definition

Left Recursion: A grammar rule where the non-terminal being defined appears as the leftmost symbol on the right-hand side.

"E keeps expanding left — infinite loop!"
E -> E C

Expansion:
E
E C
(E C) C
((E C) C) C
...

To parse E, we first need to parse E (since it's the leftmost symbol). To parse that E, we again need to parse E. The input never gets consumed, so we never reach a base case. Infinite loop.

Definition

Right Recursion: A grammar rule where the non-terminal appears after at least one token on the right-hand side.

"E expands right — works fine!"
E -> A E

Expansion:
E
A E
A (A E)
A (A (A E))
(* Each A consumes input → eventually reach base case *)

Here, each recursive step must first parse an A, which consumes at least one token. The input shrinks each time, so eventually we reach the end and hit a base case (like E → ε). Terminates.

Homework Note

The spec says you don't need to handle left-recursive grammars. But you should understand why they're problematic. The fix is to transform left-recursive rules into right-recursive ones (covered in CS 132), or to use a different parsing technique entirely.

Check Your Understanding — Parsing

Question 1

What's the fundamental difference between what a finite state machine can parse versus what a context-free grammar parser can parse?

  • FSMs are faster; CFG parsers are slower but handle the same languages
  • FSMs can handle nesting; CFG parsers cannot
  • CFG parsers use a stack to handle arbitrary nesting depth, which FSMs cannot do
  • There is no difference; they're equivalent in power
Correct! CFGs need a stack to track nesting depth (like matching parentheses). FSMs have no extra memory, so they can't count arbitrarily deep nesting. That's why CFG = FSM + stack.
Not quite. The key difference is the stack. CFG parsers have one, FSMs don't. This stack lets CFG parsers handle arbitrarily deep nesting (like nested parentheses), which FSMs fundamentally cannot do because they have no growing memory.
Question 2

Why does E → E C cause an infinite loop in a backtracking parser, but E → C E does not?

  • E → E C is left-recursive: parsing E immediately requires parsing E again, with no input consumed. E → C E first parses C (consuming input), so the recursion eventually terminates.
  • E → E C creates ambiguity, while E → C E does not
  • E → E C requires two stacks, but our parser only has one
  • There is no difference; both cause infinite loops
Exactly! In E → E C, we try to parse E by first parsing E — no input is consumed, so we loop forever. In E → C E, we must parse C first (consuming at least one token), so the input shrinks with each recursion and we eventually hit a base case.
The key is input consumption. E → E C tries to parse E as the very first thing, consuming zero tokens, leading to infinite recursion. E → C E parses C first, which eats at least one token, guaranteeing progress toward the base case.
Question 3

In the concatenation matcher for E → A B, we pass (fun frag1 → m2 accept frag1) as the acceptor for m1. Why?

  • It bundles "match B then check with the real acceptor" into a single acceptor for A, enabling automatic backtracking if B fails on A's leftover
  • It makes m1 run faster by giving it less work to do
  • It prevents m1 from consuming too many tokens
  • It converts m2 from a matcher into an acceptor permanently
Right! By wrapping m2 + the real acceptor into one acceptor for m1, we create a chain: A matches → leftover goes to B → B's leftover goes to the real acceptor. If any step fails, it propagates back, and m1 tries a different match for A.
The point of this pattern is to create a chain of responsibility. We wrap "match B, then check with the real acceptor" into a single acceptor for A. If B fails on A's remainder, the wrapped acceptor returns None, causing A to backtrack and try a different match. This is the heart of how concatenation + backtracking work together.

El Capitan: World's Fastest Computer

Why Are We Talking About This?

A computer's performance depends heavily on what problem you give it, not just how powerful the hardware is. This matters for programming language design because there is no single architecture that's best for everything — and the language you choose determines how well your code maps onto the hardware.

Also, the supercomputers of today become the everyday devices of tomorrow. Understanding these machines gives you a glimpse of the hardware you'll be programming in your career.

El Capitan is a supercomputer at Lawrence Livermore National Laboratory. It has 11,316 compute nodes. Each node contains one AMD CPU (24 cores) and four AMD MI300A GPUs — each GPU has 228 compute units with matrix cores (for heavy matrix/tensor math) and 64 stream processors (for processing long sequences of numbers).

The result is massive parallelism: millions of cores working simultaneously. It has enormous memory and is designed for heavy computation like nuclear weapon simulations.

FLOP Benchmarks

El Capitan's theoretical peak is 2.8 EFLOPS (exa-floating-point-operations per second). That's 2.8 × 10¹⁸ operations every second. For reference, a typical laptop CPU does roughly in the GigaFLOPS range — about a billion times slower.

But "theoretical" means everything running perfectly. In practice:

BenchmarkPerformanceWhat It Tests
HPL (High Performance Linpack)~1.8 EFLOPSDense linear algebra — lots of matrix multiplies. GPUs love this.
HPCG (Conjugate Gradient)~0.017 EFLOPSMemory-heavy, lots of data movement. GPUs struggle with this.

The same machine shows up to a 100× performance difference depending on the workload. The bottleneck is almost always data movement — getting numbers to the processors that need them — not the computation itself.

The Lesson

Raw computing power means little if you can't feed data to it efficiently. This is why programming languages, compilers, and how you structure your code matter so much — they determine how well your computation maps onto the hardware.

Energy Constraints

El Capitan requires 30–35 megawatts to run — roughly the power consumption of a small city (35,000 households). Connecting it to the power grid was a significant engineering challenge.

This makes energy efficiency (work per unit of energy) extremely valuable. The Green500 ranking measures performance per watt:

ComputerGFLOPS/wattNote
Kairos (University of Toulouse)73Most energy-efficient in the world
El Capitan (Livermore)59Fastest in the world, but less efficient per watt

A slower, cheaper machine can be more efficient per unit of energy. Efficiency ≠ raw speed. Different architectures win under different constraints.

The Tradeoff: Specialization vs Generality

El Capitan's specialized hardware (GPUs, tensor cores, vector units, Fortran-optimized pipelines) is extremely fast for the right kind of work — dense matrix operations, for instance. But this specialization has costs: it's hard to program, less portable, and loses performance on workloads it wasn't designed for.

General-purpose systems are flexible, portable, and easier to program, but sacrifice peak performance.

Why This Matters for Us

Programming languages make the same tradeoff. A language can give you fine-grained hardware control (like C/C++) or high-level abstractions (like Java or Python). The choice affects performance, safety, portability, and how easy the code is to write. There is no perfect language — just different tradeoffs.

This is precisely why Java was designed the way it was: it chose portability, safety, and ease of programming over maximum performance.

Why Java Was Created

To understand Java, you need to understand the problem it was designed to solve. Let's go back to the early 1990s.

Sun Microsystems and the Network

Sun Microsystems was a company that sold networked workstations and servers. Their marketing slogan was "the network is the computer" — the idea being that computation should be distributed across many machines, not trapped in a single workstation.

Everything ran on Solaris (a Unix variant, essentially an ancestor of Linux). Programs were written in C/C++. This worked great inside data centers where every machine used the same hardware (Sun's SPARC chips).

The Ambition: Network Everything

Sun wanted to expand beyond workstations and servers to everyday devices — toasters, refrigerators, phones. (Yes, this sounded ridiculous in 1993. It's called the Internet of Things now.) But this ambition crashed into several hard problems:

Problem 1 — Portability

Different devices use different CPU architectures (SPARC, x86, ARM). C/C++ executables are compiled for a specific architecture — a program built for SPARC won't run on ARM. Sun needed executables that work on any CPU.

Problem 2 — Reliability / Safety

C/C++ programs are prone to crashes: buffer overflows, dangling pointers, memory leaks, core dumps. On a workstation, you reboot. On a toaster, flames shoot up. You cannot have your toaster crash. Sun needed a language that prevents these categories of bugs.

Problem 3 — Software Updates

Networked devices need remote updates. But in the early '90s, home internet was ~15 kilobits/sec. C++ executables are huge, platform-specific blobs. Sun needed small, portable, downloadable executables.

SmallTalk & Oak

Sun's engineers drove down the freeway to Xerox PARC in Palo Alto — one of the greatest computer science research labs in history (they invented Ethernet, the mouse-driven GUI workstation, and much more).

SmallTalk

Xerox PARC had built a language called SmallTalk that solved many of Sun's problems:

— Object-oriented design
— Garbage collection: no manual memory management → no dangling pointers or memory leaks
— Bytecode interpreter: programs compile to abstract "bytecodes" for a virtual machine, not for a specific CPU → portable and small executables

But SmallTalk had drawbacks:

— Unusual syntax: completely unlike C or anything else, making it hard to learn
— Dynamic typing: type errors only caught at runtime, not at compile time → less reliable
— Single-threaded: not designed for multi-processor systems

Oak (→ Java)

Sun's engineers took SmallTalk's good ideas and fixed the bad ones:

SmallTalk ProblemOak/Java Solution
Unusual syntaxC-like syntax (familiar to millions of programmers)
Dynamic typingStatic typing (catch bugs at compile time)
Single-threadedBuilt-in multithreading support

They kept the good parts: garbage collection, bytecode + virtual machine, object-oriented programming. The result: safer than C++, more predictable and scalable than SmallTalk.

Hot Java: From the Web to Servers

The toaster project was about to be cancelled — Sun couldn't see how to make money on it. In a weekend, the engineers pivoted: they combined their Java technology with the brand-new concept of the web browser.

The dominant browser at the time was Mosaic (written in C++, famous for crashing). Sun built Hot Java — a browser written in Java that was more reliable (garbage collection, no pointer bugs) but slower (bytecode interpretation overhead).

Hot Java introduced Java applets: small Java bytecode programs downloaded from websites and run inside the browser. This was revolutionary — portable code that ran on any machine with a Java interpreter.

But Hot Java didn't win the browser war. Mosaic evolved into Netscape and then Internet Explorer, which used native C++ plugins instead. Hot Java faded.

However, Java found its true home on servers. Server-side Java thrived because:

— Multithreading scaled well on multi-processor servers
— Memory safety meant fewer crashes in production
— Portability meant the same code ran on any server hardware
— It was far less painful than multi-threaded C++ programming

Java's Type System

Definition

Static Typing: The type of every variable is known at compile time, before the program runs. The compiler checks that you're not doing impossible things (like adding a string to an integer) before you ever run the code.

Java chose static typing for reliability — catch bugs early, before they reach the toaster (or the server). This was a deliberate departure from SmallTalk's dynamic typing.

Java has two major categories of types, and understanding the distinction is essential:

Primitive Types

Definition

Primitive Types: Built-in, low-level types that store raw values directly (not objects). They have fixed, platform-independent sizes — the same on every machine.

Why fixed sizes? Portability. A long is always 64 bits, whether you're on an x86 laptop, an ARM phone, or even an IBM mainframe. No surprises. (This is unlike C, where long can be 32 bits on some platforms and 64 on others.)

TypeSizeWhat It Stores
byte8 bitsSmall integers (-128 to 127)
short16 bitsMedium-small integers
int32 bitsStandard integers
long64 bitsLarge integers
char16 bitsA single Unicode character
booleanJVM-definedtrue or false
float32-bit IEEEFloating-point number
double64-bit IEEEDouble-precision floating-point

These types are not objects. They're stored directly as raw bits. This is important for performance — no overhead of object headers or garbage collection for simple numbers.

Reference Types

Definition

Reference Types: Internally implemented as a pointer to a chunk of memory that holds the actual object data. You never see or manipulate the pointer directly — the Java Virtual Machine manages all memory automatically.

In C/C++, you work with raw pointers: you can do arithmetic on them, cast them to other types, and generally shoot yourself in the foot. Java calls them "references" (never "pointers") precisely because you can't do any of that. The JVM handles all pointer operations behind the scenes.

When you assign a reference variable to another:

MyClass a = new MyClass();
MyClass b = a;  // copies the reference (pointer), NOT the object

Both a and b now point to the same object in memory. No copying of the object's data occurs.

Arrays

Arrays are reference types. When you write int[] arr = new int[10];, the variable arr holds a reference (pointer) to a block of memory containing 10 ints. The array's size is fixed once allocated — you can't grow or shrink it. But the size is not part of the type; int[] is the type regardless of how many elements it has.

Strings

Strings are reference types that are immutable — once created, a String's contents cannot be changed. Any operation that appears to modify a string actually creates a new one.

Single Inheritance

Definition

Single Inheritance: Each Java class has exactly one parent class (superclass). Every class ultimately extends Object, forming a tree-shaped hierarchy.

Object
├── String
├── Thread
├── Integer
├── T[]  (arrays of any type)
└── MyClass
    └── MySubclass
Java's class hierarchy is always a tree — each class has exactly one parent.

Why not multiple inheritance?

Multiple inheritance means a class can have two or more parents. C++ supports this. The advantage: you inherit methods and data from both parents, making your class "richer."

The problem: what if both parents define a method with the same name? Which one do you get? C++ has complex rules for resolving this, but they're confusing, error-prone, and tricky to implement efficiently.

Java's designers said: "Multiple inheritance is too complicated. The confusion it causes isn't worth the power it provides. Single inheritance keeps things simple."

But then how do you get behavior from multiple sources? That's what interfaces are for.

Interfaces

Definition

Interface: A contract that declares what methods a class must have, without providing any implementation. It specifies method signatures only — no code.

Think of inheritance as receiving an inheritance (wealth) from a parent — you get actual code, for free. An interface is more like taking on a debt — you get an obligation to implement certain methods, but no help actually writing them. You have to do the work yourself. 
// An interface declares methods without implementing them
interface I {
    int m(float x);   // only the signature!
    int n(String s);  // no body, no code
}

You can't instantiate an interface

Since an interface has no implementation, you can't create an object from it:

// This does NOT work — interfaces have no implementation!
I obj = new I();  // ✗ Compile error

But you can declare a variable of an interface type:

I v;                   // Declares v as "something that implements I"
v = new C();           // ✓ OK if class C implements I
v = null;              // ✓ null is always allowed

The key rule: one parent, many interfaces

In Java, a class can extend only one class, but can implement any number of interfaces:

class C extends P implements I, J, K {
    // Must provide all methods from I, J, and K
    // Inherits code from P
}

Implementing Interfaces

A class satisfies an interface by providing implementations of all its methods — either directly or through inheritance from a parent class.

Direct implementation

class C implements I {
    int m(float x) {
        return 3;
    }
    int n(String s) {
        return s.length();
    }
}

Satisfying via inherited methods

// Parent class P provides method n
class P {
    int n(String s) {
        return s.length();
    }
}

// C extends P and implements I
// C defines m itself, inherits n from P
class C extends P implements I {
    int m(float x) {
        return 3;
    }
    // n is inherited from P — satisfies I's requirement!
}

Extra methods and interface visibility

A class can have methods beyond what the interface requires. But if you access the object through an interface variable, you can only see the interface's methods:

class C extends P implements I {
    int m(float x) { return 3; }
    // inherits n from P
    int q() { return 0; }  // extra method, not in I
}

I v = new C();
v.m(1.0f);    // ✓ m is in interface I
v.n("hi");     // ✓ n is in interface I
v.q();         // ✗ COMPILE ERROR — q is not in I

The variable v has type I, so the compiler only knows about methods declared in I. Even though the actual object is a C (which has q), the compiler plays it safe.

Interface obligations flow downward

If a superclass implements an interface, all its subclasses automatically implement it too:

interface I {
    void m();
}

class A implements I {
    public void m() {
        System.out.println("A");
    }
}

class Achild extends A {
    // inherits m() from A
    // therefore also implements I!
}

I x = new Achild();  // ✓ works because Achild implements I via A
Direction Matters

Interface obligations flow downward (parent → child), not upward. A superclass must fulfill its own interface — it can't rely on a future subclass to provide missing methods, because the superclass can be used on its own.

Abstract Classes

Definition

Abstract Class: A class that is intentionally incomplete — a hybrid between a regular class and an interface. It can have implemented methods (like a class) and unimplemented methods (like an interface). You cannot create objects from it directly; subclasses must fill in the blanks.

Why would you want this? Sometimes you have shared behavior that multiple subclasses should inherit (like a class), but also some behavior that each subclass must define differently (like an interface). Abstract classes give you both.

interface I {
    void speak();
}

// Abstract class: has some code, but not all
abstract class Animal implements I {
    void sleep() {
        System.out.println("zzz");  // ← implemented!
    }
    // speak() from I is NOT implemented here
    // That's okay because Animal is abstract
}

// Cannot instantiate an abstract class!
Animal a = new Animal();  // ✗ Compile error

// Subclass fills in the missing method
class Dog extends Animal {
    public void speak() {
        System.out.println("woof");  // ← now implemented!
    }
}

// This works because Dog is complete
Animal a = new Dog();
a.speak();  // "woof" — from Dog
a.sleep();  // "zzz"  — from Animal
Comparison: Interface vs Abstract Class vs Class

Interface:       All methods unimplemented  | Can't instantiate | Multiple allowed
Abstract Class: Some methods implemented   | Can't instantiate | Single parent only
Concrete Class: All methods implemented    | Can instantiate   | Single parent only

Question 4

Java's primitive types have fixed, platform-independent sizes (e.g., int is always 32 bits). Why was this design choice made?

  • To make Java code run faster than C++
  • Because 32 bits is the optimal size for all CPUs
  • To ensure programs behave identically on every machine (portability)
  • Because Java doesn't support 64-bit values
Correct! Fixed sizes ensure that a Java program produces the same results whether it runs on ARM, x86, SPARC, or any other architecture. This was essential for Sun's vision of portable code that works everywhere — from servers to toasters.
Think about Sun's original goal: code that works on any CPU. If int were 32 bits on one machine and 16 bits on another, the same program could produce different results. Fixed sizes guarantee identical behavior everywhere — that's portability.
Question 5

Given this code, which call will cause a compile error?

interface I { void m(); void n(); }
class C implements I {
    public void m() { }
    public void n() { }
    public void q() { }
}
I v = new C();
  • v.m();
  • v.n();
  • v.q();
  • None — all three work
Right! v has type I, and interface I only declares m() and n(). The compiler doesn't know that the actual object has q() — it only sees what I promises. To call q(), you'd need v to be of type C.
Since v is declared as type I, you can only call methods that interface I declares: m() and n(). Method q() exists on the actual C object, but the compiler only sees the interface type, not the runtime type. v.q() fails at compile time.
Question 6

Why did Java choose to exclude multiple inheritance (having more than one parent class)?

  • It's impossible to implement in a programming language
  • It creates ambiguity when two parents define the same method, and the complexity wasn't worth the benefit
  • Java's garbage collector can't handle objects with two parents
  • Single inheritance runs faster on all hardware
Exactly! The "diamond problem" — two parents defining the same method — creates ambiguity about which implementation to use. C++ has complex rules for this; Java's designers decided the simpler approach (single inheritance + interfaces) was more reliable and easier to understand.
Multiple inheritance is possible (C++ and Python have it). The issue is the diamond problem: if two parents both define method m(), which does the child get? The resolution rules are complex and confusing. Java chose simplicity: one parent for code (single inheritance), multiple interfaces for behavioral contracts.
Wrapping Up

Parsing: We learned that parsing is about figuring out the structure of flat text. Finite state machines handle regular expressions, but programming languages need the extra power of a stack (context-free grammars). Backtracking parsers try options and undo failures. The matcher/acceptor pattern elegantly handles disjunction and concatenation with backtracking — but left-recursive grammars will cause infinite loops.

Hardware: El Capitan showed us that a machine's effective speed depends enormously on the workload, and that there's always a tradeoff between specialization and generality.

Java: Was born from the practical need for portable, reliable, safe code on diverse hardware. Every design choice — static typing, garbage collection, bytecode, single inheritance, interfaces — traces back to solving specific engineering problems that C++ couldn't solve for networked toasters and servers.
