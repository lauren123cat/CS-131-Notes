Part I

The Java Memory Model Flaw

A subtle bug that tripped up Java's own designers — and what it teaches us about multi-threaded programming.

Quick Recap: What Is the JMM?

Before we get to the flaw, let's make sure you understand the context. When you write a Java program, the code you write isn't necessarily executed in the exact order you wrote it. The Java Memory Model (JMM) is a set of rules that defines how the Java Virtual Machine (JVM), the compiler, and the CPU are allowed to reorder your operations for speed.

Key Definition

Java Memory Model (JMM): The rules that define when one thread is guaranteed to see writes made by another thread. It governs what reorderings are legal.

Why does reordering happen? Because modern CPUs and compilers can execute things out of order to go faster. For example, if you write:

x = 1;
y = 2;JAVA

The JVM might actually execute y = 2 first, then x = 1. In a single-threaded program, this doesn't matter — you'll never notice. But in a multi-threaded program, another thread might see y change before x, which could cause bugs.

The JMM categorizes operations into three types, each with different rules about how freely they can be reordered:

Operation Type Reordering Freedom Visibility Guarantee What It Means
normal read/write Lots of freedom No guarantee Thread A writes, Thread B might not see it
volatile load/store More restricted Immediate guarantee Thread A writes, Thread B sees it immediately
enter/exit monitor Heavily restricted Strong guarantee Thread B sees everything A did before releasing the lock

The rules seemed complete: don't move things across volatile boundaries, don't move things outside critical sections. But then someone found a problem…

The Constructor Flaw: Seeing Garbage Memory

Why This Matters

This isn't just a Java trivia fact. It teaches a fundamental lesson: in multi-threaded programs, you have to think very carefully about when objects become visible to other threads. This same issue appears in any concurrent programming environment.

Here's the scenario. Imagine you have a string and you call substring() on it:

String s1 = "user_temp";
String s2 = s1.substring(4);  // creates a new String pointing to "_temp"JAVA

What does substring() actually do? It invokes a constructor — it creates a brand-new String object. Internally, that constructor does something like this:

// Pseudocode for what happens inside substring()
String s = new String();     // (1) Allocate fresh memory
s.base = s1.base;            // (2) Point to same underlying char array
s.offset = 4;               // (3) Start from the 4th character
return s;                    // (4) Hand it back to whoever askedPSEUDOCODE

Normally, when you create a new object in Java, the process follows these steps:

Allocate memory — Reserve a chunk of bytes for the new object.
Initialize to defaults — Set all fields to their zero values (0, false, null).
Run the constructor — Fill in the actual values (like base and offset).
Return the reference — Now other code can use the object.

The important thing: by the time anyone else sees this object, all four steps are done. The object is fully initialized. That's the promise.

The Optimization That Breaks Things

Here's where it gets interesting. Constructors have a very common optimization: since you're about to fill in all the fields anyway (step 3), why bother zeroing them first (step 2)? Just skip the zeroing, run the constructor, and everything ends up correct.

This is perfectly fine in single-threaded code — nobody sees the object until the constructor returns. But what if a constructor does something very unusual?

class Bad {
    static Object G;       // A global variable, visible to ALL threads

    Bad() {
        G = this;           // ← DANGER: publishes object BEFORE it's ready!
        value = 42;
    }
    int value;
}JAVA
⚠️ The Problem

The constructor stores this into a global variable G — meaning other threads can now access this object. But the constructor hasn't finished yet! The field value hasn't been set to 42.

Now imagine this timeline with two threads:

/* ─── THREAD A ─── */
// Constructor starts. Optimization: skip zeroing!
// value = garbage (whatever bytes were in memory)
G = this;      // Object is now visible to the whole program!
value = 42;    // ...but this hasn't happened yet

/* ─── THREAD B ─── */
Bad obj = G;           // Thread B grabs the object from G
System.out.println(obj.value);  // Reads GARBAGE, not 42!TIMELINE

Thread B accesses obj.value and sees random garbage memory — not 0 (the default), not 42 (the intended value), but whatever bytes happened to be sitting in that memory location. This completely violates Java's safety guarantees. In Java, you should never be able to observe uninitialized memory.

🏗️
Analogy: The Unfinished House

Imagine you're building a house (the object). Normally, you finish building, then hand the keys to the buyer (return from constructor). But here, you posted the address on a public bulletin board (global variable) while the walls are still being put up. A visitor could walk in and find exposed wires and no floor — that's the "garbage memory" other threads see.

✦ Check Your Understanding

Why can Thread B see garbage instead of 0 or 42 for value?

  • Because Java doesn't guarantee field values in constructors
  • Because the global variable G hasn't been properly synchronized
  • Because the optimizer skipped zero-initialization, and the constructor published this before setting the field
  • Because Thread B is reading from a cached copy of memory
Correct! The optimizer skipped step 2 (zeroing all fields) as an optimization. Then the constructor published this into a global variable before step 3 (setting value = 42). So Thread B sees the object with its fields still containing whatever was previously in that memory — raw garbage bytes.

The Fix: Two Additional JMM Rules

The Java designers realized they needed extra rules to prevent this scenario. The fix consists of two rules — one for the optimizer, one for the programmer:

Rule 1 — For the Optimizer

Stores to an object's fields (like zeroing or initializing them) cannot be delayed until after the constructor returns. The optimizer must ensure that by the time the constructor finishes, all field initialization is visible.

In other words: the optimizer must follow the sequence — allocate → set defaults → run constructor → return reference.

Rule 2 — For the Programmer

A constructor must not store this anywhere that is visible to another thread. You cannot "publish" the object you're constructing before the constructor finishes.

When both rules are followed together, the skip-zeroing optimization becomes safe again: since no other thread can see the object until the constructor returns, it doesn't matter if we skip zeroing — by the time anyone sees it, all fields have been properly initialized.

Design Lesson

The professor calls this "a design bug in Java." The fact that you need a special rule like this suggests the language didn't quite get multi-threading right from the start. Storing this into a global variable during construction was always bad style — but it was legal Java code. These extra rules had to be added after the fact, which is messy.

✦ Check Your Understanding

Which of these is allowed under the additional JMM rules?

  • A constructor that assigns this to a static field
  • A constructor that only sets instance fields and then returns
  • A constructor that passes this to another thread's method
  • A constructor that stores this in a collection visible to other threads
Correct! The only safe option is a constructor that just sets its own fields and returns. All other options "publish" this to places visible to other threads, which violates Rule 2.
Part II

How Programs Go Wrong

A broader look at errors — what they are, how to classify them, and why this matters for programming language design.

Why Study Errors?

We're taught to write correct programs. But in reality, most software has mistakes. A huge part of software engineering is dealing with imperfect code. Understanding the types of errors helps us pick the right language features and strategies to minimize damage.

Error → Fault → Failure

The first classification comes from general engineering (applies to mechanical engineering, electrical engineering, etc.). It distinguishes between three stages of how things go wrong:

Definition: Error

A mistake in the programmer's head. You misunderstood the requirements, or had a wrong mental model. Example: you thought you needed to write a square root function, but the spec actually called for a cube root.

An error exists even before you write any code. It's a conceptual mistake.

Definition: Fault

A bug sitting in the code. It's latent — it exists whether or not the program has been run. You could theoretically find it just by reading the source code carefully.

Example: if (x = 5) instead of if (x == 5) — the assignment instead of comparison is a fault.

Definition: Failure

The observed, incorrect behavior. The program doesn't work, and the user notices. This is the bottom line — failures are what we ultimately care about preventing.

These three stages form a chain, but the links aren't guaranteed:

Error → may lead to → Fault → may lead to → Failure

Not every error leads to a fault. Maybe you made a conceptual mistake but the code you wrote happens to be correct anyway. Or maybe the error was in a comment — wrong comment, right code.

Not every fault leads to a failure. If nobody ever exercises the buggy code path, the user never sees a problem. But every fault came from some earlier error, and every failure came from some earlier fault.

Why the Distinction Matters

When someone says a language will "make your programs bug-free" (which is always an exaggeration), you should ask: does this feature reduce errors (helps you think better), faults (catches bugs before they run), or failures (prevents crashes at runtime)? Different features target different levels.

✦ Check Your Understanding

A programmer accidentally puts a wrong formula in a comment. The code itself is correct. What is this?

  • An error (but not a fault or failure)
  • A fault (because it's in the source code)
  • A failure (because it could mislead future developers)
  • None of the above — comments don't count
Correct! The programmer made a mistake (error) — their comment doesn't match the code. But since the code itself is fine, there's no fault in the executable program, and no failure will occur.

Scheme's Four Categories of Errors

The Scheme programming language (which you'll learn about later in this course) classifies errors in a different way. This classification is more focused on programming languages specifically, rather than engineering in general. It asks: when something goes wrong, what kind of "wrong" is it?

1. Implementation Restrictions

Definition

Your program is perfectly correct, but the system running it can't handle it. The limitation is in the implementation (the compiler, runtime, or hardware), not in your code.

The classic example is memory exhaustion. Your program needs 60GB of RAM but your machine only has 32GB. Is that a bug in your program? Not really — if you had a bigger machine, it would work fine. But from the user's perspective, the program still crashed.

🍳
Analogy

You have a perfect recipe, but your kitchen doesn't have enough burners or the right sized pots. The recipe isn't wrong — the kitchen is just too small.

Every language that does dynamic memory allocation (OCaml, Python, Java, etc.) can hit this. You'll get an "out of memory" message, but it's really a limitation of the machine, not a bug in the logic.

2. Unspecified Behavior

Definition

Your program does something where the language standard says multiple outcomes are all valid. The implementation gets to choose which one happens. A portable program must work correctly regardless of which choice the implementation makes.

Here's the example from class. In OCaml, consider this expression:

(* Two floating point 1.0 values *)
# 1.0 == 1.0;;OCAML

The == operator in OCaml tests reference equality (pointer comparison) — are these two things stored at the same memory address?

The answer depends on the implementation:

Implementation A

Creates two separate memory locations for the two 1.0 values. They have different addresses. Result: false

Implementation B

Notices both constants are 1.0 and coalesces them — stores only one copy. Both point to the same address. Result: true

Both implementations are correct! The language standard allows either behavior. If your program assumes the answer is true, it's not portable — it will break on Implementation A. A well-written program should work regardless of which answer it gets.

An important detail: the implementation will almost never give you an error message for relying on unspecified behavior. It's just too expensive to detect. Your program will silently produce different results on different systems.

3. Undefined Behavior

⚠️ Definition

The program can do literally anything. There are no guarantees from the language about what will happen. The program might crash, produce garbage, corrupt memory, or appear to work fine. The implementation has complete freedom.

This is much stronger than unspecified. With unspecified behavior, you get one of a defined set of outcomes and computation continues. With undefined behavior, all bets are off.

The classic example from class — integer division by zero in C++:

print("x");
int i = i/0;    // integer division by 0 = UNDEFINED BEHAVIOR
print("y");C++

You might expect the output to be x followed by a crash. But because the behavior is undefined, the compiler is allowed to reorder things:

// The compiler might transform this to:
i = i/0;          // moved first!
print("x");
print("y");

// Result: program crashes BEFORE printing anything
// Or it might even optimize away the entire program!POSSIBLE TRANSFORMATION
Critical Insight

Undefined behavior can affect code that comes before it in the source. If the compiler can prove that your code will eventually hit undefined behavior, it can do anything with the entire program — including removing code that appears earlier. This is because the C/C++ standard says if undefined behavior occurs, the entire program's behavior is undefined, not just the statement that caused it.

Another example — using a freed pointer in C:

char* p = malloc(10);
strcpy(p, "abc");      // copy string to allocated memory
free(p);                // memory is released back to the system

char* q = p;            // UNDEFINED! Can't even LOOK at a freed pointer!C

In C, you're not just forbidden from dereferencing a freed pointer — you can't even read the pointer's value. You can't test p == NULL. It's undefined. Why? Because C is designed to run on "paranoid machines" that have special hardware registers for pointers with built-in checking. Even loading a freed pointer value into such a register could trigger a hardware fault.

✦ Check Your Understanding

What's the key difference between unspecified and undefined behavior?

  • Unspecified means the compiler warns you; undefined means it doesn't
  • Unspecified is for runtime errors; undefined is for compile-time errors
  • Unspecified gives you one of several valid outcomes; undefined means absolutely anything can happen
  • They mean the same thing — just different terminology
Correct! With unspecified behavior, the standard says "pick from these valid options" (e.g., true or false for the OCaml example) and computation continues. With undefined behavior, the standard says nothing — the program can crash, give random output, corrupt memory, or even break code that appears before the undefined operation.

4. An Error Is Signaled

Definition

A signal is how the operating system tells a program about certain runtime events. When an error is "signaled," the system actively notifies the program that something went wrong.

Examples include SIGSEGV (segmentation fault — invalid memory access), SIGINT (interruption, like pressing Ctrl+C), and SIGFPE (floating-point exception). These are the system's way of saying "something bad happened, here's what."

Part III

Five Ways to Address Errors

Languages have very different philosophies about errors. These five approaches represent fundamentally different design decisions.

Why This Matters

Every programming language makes a choice about how to handle errors. C/C++ allows undefined behavior because they prioritize performance and low-level control. Java tries to eliminate undefined behavior by throwing exceptions and enforcing memory safety. Understanding these approaches helps you understand why languages are designed the way they are.

Approach 1: Static Checking

Definition

Static checking means the compiler examines your program before it runs and guarantees that certain kinds of errors cannot happen. The most common form is static type checking.

In languages like OCaml, Java, Rust, and Haskell, the compiler checks your types at compile time. If you try to assign a string to an integer variable, the compiler rejects your program entirely — it won't even produce an executable.

// Java: this won't compile!
int x = "hello";   // ERROR: incompatible typesJAVA

Languages like Python and JavaScript use dynamic type checking instead — types are checked at runtime, not compile time. This means type errors only show up when you actually run the code.

But it's not always black-and-white. Java mostly uses static type checking, but it also has dynamic type checking in some cases — specifically, when you use casts:

Object o = "hello";       // declared as Object
String s = (String) o;    // runtime check: is o really a String?
// If o were NOT a String → ClassCastException at runtimeJAVA

So why do Python and JavaScript use dynamic checking? The professor identified several reasons:

Why Static Checking (Java, OCaml)

More reliable — once the compiler accepts your program, you know certain bugs are impossible. Catches type errors before anyone ever runs the code.

Why Dynamic Checking (Python, JS)

More flexible — You can write code faster, don't have to figure out types upfront. Code can work on more kinds of data. Better for quick scripts and prototyping.

A concrete example of flexibility: In Java, List<String> is not a subtype of List<Object>, even though String is a subtype of Object. Java's type checker is cautious — it worries about what you could do with a List<Object> (like adding an Integer to it). In Python, you'd just pass the list and it would work, because Python only cares about what you actually do.

Approach 2: Preconditions

Definition

Preconditions are formal requirements that must be true before a function runs. They are the caller's responsibility. If the precondition is false, the behavior is undefined.

Let's look at the professor's detailed example — integer division:

int divide(int x, int y)
  // Precondition 1: y != 0
  // Precondition 2: NOT (x == INT_MIN AND y == -1)
{
    return x / y;
}C

Why two preconditions? There are exactly two ways integer division can go wrong:

1. Division by zero — Obvious: x / 0 is mathematically undefined.

2. Overflow — This one is subtle! On a 32-bit system, integers range from −2,147,483,648 (INT_MIN) to 2,147,483,647 (INT_MAX). If you divide INT_MIN by -1, the mathematical result is 2,147,483,648 — but that's one more than INT_MAX. It can't be represented! In C/C++, this is undefined behavior. On x86-64, your program will crash.

When you have preconditions, the caller is responsible for ensuring they're met. In debugging mode, you can add runtime checks for them:

int divide(int x, int y) {
    if (y == 0) { error(); }            // runtime check
    if (x == INT_MIN && y == -1) { error(); }
    return x / y;
}C — WITH RUNTIME CHECKS

These checks slow down execution, but help you catch violations during testing. In production, you can remove them and rely on the caller to get it right (better performance).

Preconditions vs. Static Checking

Preconditions typically depend on runtime values (like whether y is zero), so the compiler can't verify them statically. This makes them more flexible than static checking but less guaranteed — you're trusting the caller to do the right thing.

Approach 3: Total Definitions

Definition

A total definition makes a function return a value for every possible input, even invalid ones. Instead of crashing or being undefined, you get a "special" answer that represents the error.

The best real-world example is IEEE floating-point arithmetic:

1.0 / 0.0    → Infinity
0.0 / 0.0    → NaN        // "Not a Number"
-1.0 / 0.0   → -InfinityIEEE FLOATING POINT

There are no errors. Every division gives you some answer. Even dividing by zero! The "errors" have been migrated into the values — you get Infinity or NaN instead of crashing, and your computation keeps going.

For integer division, you could define a total version: dividing by zero returns INT_MAX or INT_MIN depending on the sign. It's not mathematically correct, but it keeps the program running.

The Trade-off

With preconditions, it's the caller's job to check before calling the function. With total definitions, it's the caller's job to check after — they need to look at the result and handle special values like NaN. The error hasn't disappeared; it's just been moved to a different place.

Approach 4: Fatal Errors

Definition

A fatal error stops the program immediately. Unlike undefined behavior (where anything can happen), a fatal error guarantees that the program terminates — it won't continue and cause more damage.

// C:
abort();   // crash immediately, core dump on Linux

// Java:
System.exit(100);   // exit with status code 100EXAMPLES

This is safer than undefined behavior because at least the program stops instead of silently corrupting data. But it has a major scalability problem:

The Scalability Problem

Imagine a million-line program. One small buggy corner calls abort(). Now your entire program crashes because of one bad section. For a server that needs to run 24/7, or an embedded system in a toaster, this is unacceptable. You want the rest of the program to keep running.

This problem motivates the fifth and final approach…

Approach 5: Exceptions

Definition

Exceptions let a program signal that something went wrong without killing the entire program. The part that detected the error stops, but control transfers to a different part of the program that handles the recovery.

Think of it this way: with fatal errors, when you encounter a problem, you yell "FIRE!" and the whole building evacuates. With exceptions, you call the fire department — the fire is contained, and only the affected area is dealt with.

✦ Check Your Understanding

IEEE floating-point returning NaN for 0.0/0.0 is an example of which approach?

  • Static checking — the compiler catches it
  • Preconditions — the caller should prevent it
  • Total definition — every input gets a result
  • Exception — the error is propagated
Correct! IEEE floating-point uses the total definition approach. Every possible division returns some value: a number, Infinity, or NaN. There are no errors that stop or redirect computation — the error is embedded in the value itself.
Part IV

Exception Handling In Depth

How try/catch/finally actually works — from syntax to stack unwinding to Java's checked exception system.

Java's Try / Catch / Finally

Java uses three blocks to separate normal code from error-handling code:

try {
    // Code that might fail
    read(b);
    write(b);
}
catch (IOException e) {
    // What to do if an I/O error occurs
    System.out.println(e);
}
finally {
    // ALWAYS runs, no matter what happened above
    close(file);
}JAVA

Let's break down each piece and understand why it exists:

The Try Block

This contains your normal, happy-path code — what should happen if everything goes well. You read a buffer, then write a buffer. Simple. But in the back of your mind, you know reading and writing can fail — the flash drive might be disconnected, there might be a disk error, etc.

The Throw Statement (Inside read/write)

Somewhere deep inside the read() or write() implementation — possibly many method calls deep — a piece of code detects a problem and creates an exception:

throw new IOException("Flash drive disconnected");JAVA

The thrower creates a new exception object and throws it. This immediately stops normal execution at that point. No more code in the try block runs. Control jumps directly to the matching catch block.

The thrower could be many layers deep in the call stack: read() calls readInternal() which calls deviceRead() which calls ioCheck()… and ioCheck() throws. Control jumps all the way back up past all those methods to your catch block.

The Catch Block

This handles the exception. The variable e is a reference to the exception object that was thrown, so you can inspect it to find out what went wrong. In this example, we just print it — not the most useful recovery, but it demonstrates the idea.

The Finally Block

This runs no matter what. Let's understand why it exists with a specific scenario:

What if the try block throws an exception that your catch block doesn't handle? Say a NullPointerException instead of an IOException. In that case:

The try block throws NullPointerException
Java looks at your catch: it only catches IOException — no match!
The catch block is skipped entirely
The finally block still runs (closes the file!)
The exception propagates up to the caller to find a handler

If you had put close(file) after the entire try/catch (without finally), it would be skipped for any uncaught exception. The file would remain open, leaking a resource. That's why finally exists — to guarantee cleanup code always runs.

Here are all the possible execution paths:

/* Case 1: No error */
try → read → write → finally → continue

/* Case 2: read() throws IOException */
try → read → THROW → catch → finally → continue

/* Case 3: write() throws IOException */
try → read → write → THROW → catch → finally → continue

/* Case 4: Uncaught exception (e.g., NullPointerException) */
try → THROW → skip catch → finally → propagate to callerEXECUTION PATHS

Exceptions vs. Return-Value Error Handling (C-Style)

Languages without exceptions (like C) handle errors by having functions return a status value:

// C-style: error handling with return values
failureObject f = read(b);         // returns null if OK, error object if not
if (f == null) {
    f = write(b);                  // only write if read succeeded
}
if (f != null) {
    printf("Error: %s\n", f);    // handle the error
}C-STYLE
Exception Handling Pros

Cleaner separation: Normal code is clearly in the try block. Error handling is clearly in catch. You can read the happy path without wading through error checks.

Exception Handling Cons

Hidden control flow: At any point in the try block, execution might jump to catch. This makes the code harder to reason about and harder for the compiler to optimize.

Return-Value Pros

Explicit flow: You can see exactly what happens. The compiler can optimize aggressively because there are no hidden jumps.

Return-Value Cons

Code clutter: The normal logic and error logic are interleaved. The "real" algorithm is obscured by all the error checks.

Under the Hood: Stack-Based Implementation

Why This Matters

Understanding how exceptions are implemented helps you understand their performance characteristics and what happens when things go wrong with exception handling itself.

In a stack-based language like Java or C++, every function call pushes a stack frame onto the call stack. The stack grows downward in memory:

High addresses
┌─────────────────┐
│   main's frame   │  main calls f
├─────────────────┤
│    f's frame     │  f has a TRY block; f calls g
│    [exception    │
│     records: E₁] │
├─────────────────┤
│    g's frame     │  g calls h
├─────────────────┤
│    h's frame     │  h has a FINALLY block; h calls i
│    [finally: F₁] │
├─────────────────┤
│    i's frame     │  i does: throw IOException ← HERE!
└─────────────────┘
Low addressesCALL STACK

When i throws an exception, here's what has to happen:

Check i's frame — Does i have a catch that matches IOException? No. Does it have a finally? No. Pop i's frame off the stack.
Check h's frame — Does h have a matching catch? No. Does it have a finally? Yes! Run h's finally block. Then pop h's frame.
Check g's frame — Does g have a matching catch? No. No finally either. Pop g's frame.
Check f's frame — Does f have a matching catch? Yes! f catches IOException. Deliver the exception to f's catch handler.

Key observations from this process:

It's a loop, not a single instruction. The throw operation has to iterate through stack frames, checking each one. This makes throwing exceptions relatively slow compared to normal function returns.

Every frame needs exception metadata. Each frame must store information about what exceptions it catches and what finally blocks it has. This takes space.

If no handler is found: the search continues all the way to main. If main doesn't catch it either, it's a fatal error — the program terminates.

Historical Note: Code on the Stack

Old C++ implementations would stuff actual machine code (usually just 1-2 jump instructions) directly onto the stack to handle exception matching faster. This was a security nightmare — if a buffer overflow let an attacker write to the stack, they could inject executable code. This is one reason modern systems mark the stack as non-executable.

Java's Checked Exceptions

Why This Exists

One of the problems with exceptions is that if nobody catches a thrown exception, it becomes a fatal error. The Java designers wanted to prevent this with a compile-time guarantee that every exception will be caught.

Definition

Checked exceptions require that the set of exceptions a method can throw is part of the method's signature. The compiler statically checks that every checked exception is either caught or declared.

The rule is simple:

// This method DECLARES that it might throw IOException
int m(float x) throws IOException {
    // ... code that might throw IOException ...
}JAVA

If a method throws an exception (or calls a method that throws one), and doesn't catch it, it must declare throws in its signature.

Any caller of m now has two choices:

// Option 1: Catch it
void caller1() {
    try {
        m(3.0f);
    } catch (IOException e) {
        // handle it
    }
}

// Option 2: Propagate it (declare it in YOUR signature too)
void caller2() throws IOException {
    m(3.0f);   // if it throws, our caller handles it
}JAVA

This chain of declarations ensures that the compiler can trace every exception from where it's thrown to where it's caught. The main method must catch everything — it can't declare throws for checked exceptions (or if it does, the JVM handles it as a fatal error).

Java's Exception Hierarchy

All throwable things in Java form a hierarchy:

Throwable                    ← the root: anything that can be thrown
├── Error                   ← JVM/system problems (usually DON'T catch)
│   ├── OutOfMemoryError
│   └── StackOverflowError
└── Exception               ← recoverable problems (DO catch)
    ├── IOException
    └── RuntimeException    ← unchecked exceptions
        ├── NullPointerException
        ├── ArithmeticException
        └── IndexOutOfBoundsExceptionHIERARCHY

Throwable is the root — anything you can put in a throw statement.

Error is for JVM-level problems. These map to Scheme's "implementation restrictions" — your program might be fine, but the JVM can't run it (out of memory, stack overflow). You generally don't try to catch these.

Exception is for recoverable problems. When your code detects something wrong, this is what you throw. If you catch Exception, you catch nearly everything recoverable.

⚠️ Anti-Pattern: Silent Swallowing

This is bad practice (but common in real code):

catch (Exception e) { }   // catches everything, does NOTHING

This "swallows" the error silently. The program continues as if nothing went wrong, which can lead to corrupted state and bugs that are incredibly hard to track down. The exception carried useful information about what went wrong — and you threw it away.

✦ Check Your Understanding

Why does the finally block exist? Why not just put cleanup code after the try/catch?

  • It's just syntactic sugar — there's no functional difference
  • Code after try/catch is skipped for uncaught exceptions; finally always runs
  • Finally runs before catch, giving you a chance to clean up first
  • Finally is required by Java — you can't have try/catch without it
Correct! If the try block throws an exception your catch doesn't handle, the code after the try/catch is skipped entirely as Java searches up the call stack for a handler. But the finally block runs regardless — even for uncaught exceptions. This guarantees resources (like open files) are cleaned up.
Part V

Java's Object Class: The Foundation

Looking at the "back side of the castle" — the methods every Java object inherits, and what they teach us about Java's design.

Why Study Object?

Every single class in Java — String, ArrayList, your own classes — extends Object. The methods defined in Object are available on every object. Understanding them reveals the fundamental operations Java considers essential for all objects.

The Constructor: public Object()

Every class in Java implicitly extends Object:

// When you write:
class Person { ... }

// Java actually sees:
class Person extends Object { ... }JAVA

Every constructor eventually calls Object():

public Person() {
    super();   // ← calls Object's constructor
    // ... rest of Person's initialization
}JAVA

But why would you ever create a plain Object?

Object o = new Object();  // a "vanilla" object with no special propertiesJAVA

The professor explains: a plain Object is the blandest object possible. It's useful for testing — if you have code that claims to work on any object, pass it a plain Object and see if it handles it correctly. For example, testing that casts throw exceptions properly:

Object o = new Object();
String s = (String) o;      // → ClassCastException! (o is NOT a String)JAVA

equals() and hashCode()

equals()

public boolean equals(Object o);JAVA

Tests whether two objects are logically equal. The default implementation just does reference comparison (pointer comparison with ==) — are they the same object in memory?

Object a = new Object();
Object b = new Object();
a.equals(b);   // false — different objects in memory
a.equals(a);   // true — same objectJAVA

You can override this in subclasses. For example, String overrides equals() to compare the actual characters rather than the memory addresses. Two different String objects with the same text will return true.

hashCode()

public int hashCode();JAVA

Produces an integer "hash" of the object, used by hash table data structures (like HashMap). The default implementation is derived from the object's memory address.

The Critical Consistency Rule

⚠️ Must-Know Rule

If you override equals(), you must also override hashCode(). They must be consistent:

If o1.equals(o2) is true → then o1.hashCode() == o2.hashCode() must be true

The reverse is not required: two unequal objects can have the same hash code (that's just a hash collision).

Why? Because hash tables use hashCode() to decide which bucket to put an object in, and equals() to determine if two keys in the same bucket are the same. If equal objects hash to different buckets, you'll never find them!

What happens if you violate this? No compile-time error. No runtime error message. Your program just silently gives wrong answers whenever it uses those objects as keys in a hash table. These bugs are notoriously hard to find.

toString()

public String toString();JAVA

Returns a printable string representation. This is what System.out.println() calls internally on any object:

System.out.println(someObject);
// is actually:
System.out.println(someObject.toString());JAVA

The default implementation returns something like ClassName@1a2b3c4d (the class name and a hex number). Override it to make debugging easier:

class Person {
    String name;

    public String toString() {
        return "Person: " + name;   // e.g., "Person: Alice"
    }
}JAVA

getClass() — Final and Type-Safe

public final Class getClass();JAVA

Returns a Class object representing the runtime type of the object. Two very important things about this method:

1. It Is final

The keyword final means subclasses cannot override this method. This is the opposite of abstract:

KeywordMeaning
abstractParent doesn't supply an implementation; child must override it
finalParent supplies the implementation; child cannot override it

Why is getClass() final? Objects must not be able to lie about their type. If an object could override getClass() and pretend to be a String when it's not, it would break debugging, security, and type safety. The JVM guarantees this method tells the truth.

Under the hood, getClass() is extremely efficient — it's essentially a single pointer dereference (load the type information from the object header). One instruction.

2. It Returns the Runtime Type (Not the Declared Type)

Object o = "hello";      // declared type: Object; runtime type: String
Class c = o.getClass();
System.out.println(c);   // prints: class java.lang.StringJAVA

Even though o is declared as Object, it's actually a String at runtime. getClass() tells you the real type.

Generic Type Erasure

There's a limitation. Java uses type erasure for generics — generic type parameters exist only at compile time and are erased at runtime:

List<String> a = new ArrayList<>();
List<Integer> b = new ArrayList<>();

// At runtime, Java sees both as just "ArrayList"
a.getClass() == b.getClass()   // TRUE! Same class.

// getClass() returns ArrayList, not ArrayList<String>
System.out.println(a.getClass());   // class java.util.ArrayListJAVA

Why? Generics were added to Java after the language was created. For backward compatibility with older code, the JVM doesn't need to know about generic type parameters at runtime. So getClass() can only return the raw class (e.g., ArrayList), not the full generic type (e.g., ArrayList<String>).

finalize() — Deprecated

protected void finalize() throws ThrowableJAVA

This is the opposite of a constructor: it runs just before an object is garbage collected (removed from memory). The idea was that you could use it for cleanup — closing file descriptors, releasing external resources, etc.

It has been deprecated (officially discouraged). Here's why:

Performance — The garbage collector has to check every object for a finalize() method before collecting it. If it has one, the GC must run it first, adding significant overhead.
Unpredictable timing — Garbage collection happens whenever the JVM decides. Your finalizer might run immediately, 10 seconds later, or never. You can't rely on it for time-sensitive cleanup.
Object resurrection — A finalize method could store this in a global variable (GlobalStore.obj = this), preventing the object from being collected! This creates bizarre bugs and security issues.

clone() — Copy an Object

protected Object clone() throws CloneNotSupportedExceptionJAVA

Creates a copy of an object — same data, different memory location. You can modify the copy without affecting the original.

Person p1 = new Person("Alice");
Person p2 = (Person) p1.clone();  // p1 and p2 are different objects
// p1 != p2, but they contain the same dataJAVA

Important details:

Cloning is opt-in. By default, clone() throws CloneNotSupportedException. To allow cloning, your class must implement the Cloneable interface:

class Person implements Cloneable { ... }JAVA

Cloning can be dangerous. The default clone() does a shallow copy — it copies the object's fields, but if those fields are references to other objects, both the original and the clone will point to the same inner objects. Modifying the inner object through one will affect the other.

Thinking Question (from the professor)

Why would some objects sometimes want to be cloned and sometimes not? And why is this method at the very top of the hierarchy?

Think about objects that represent unique resources (like a database connection or a lock). Cloning them would create two references to the same underlying resource, which could cause conflicts. By putting clone() in Object and making it opt-in, Java lets every class make this decision for itself.

✦ Check Your Understanding

Why is getClass() declared as final?

  • Because it's too simple to be worth overriding
  • Because it accesses private JVM internals
  • So objects cannot lie about their type — no "imposter" objects
  • Because the return type (Class) is also final
Correct! If getClass() could be overridden, a subclass could return a fake Class object, pretending to be a type it's not. This would break debugging, type safety, and the instanceof check. Making it final ensures the JVM always reports the true runtime type, efficiently (just one pointer dereference).
Lecture 10 — Wrap-Up

Here's how everything connects:

The JMM flaw showed us that even language designers make mistakes — the error/fault/failure chain applies to them too. The flaw was an error in the JMM design, which became a fault in any program that published this during construction, which became a failure when threads saw garbage memory.

The five approaches to errors are fundamentally about trade-offs: reliability vs. flexibility, performance vs. safety. Java's design philosophy is heavily weighted toward safety (static checking, checked exceptions, no undefined behavior), while C/C++ prioritize performance (undefined behavior, preconditions, manual everything).

Java's Object class embodies these design choices: equals/hashCode have a consistency requirement you must follow (precondition-style), getClass is final for safety, finalize was deprecated because it traded too much performance for too little benefit, and clone is opt-in to prevent dangerous copies.
