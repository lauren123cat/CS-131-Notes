Lecture 1: Why Languages Matter

An introduction to programming languages — and why the way we write code shapes the way we think.

This first lecture sets up the entire philosophy of the course. Before we write a single line of OCaml, Prolog, or Java, the professor wants you to understand a deeper question: why do different programming languages exist, and why should you care?

The answer will change how you approach every software problem for the rest of your career.

Chapter 1Why Study Programming Languages?

You already know how to code. You've written C++, some Python, maybe JavaScript. So why spend an entire course studying programming languages themselves as a subject?

Here's the core insight: technology changes constantly. The language that's popular today might be obsolete in ten years. But if you understand the principles behind programming languages — why they're designed the way they are, what trade-offs they make, what computational models they embody — then you can adapt to any new language or paradigm that comes along. You won't just be a user of tools; you'll understand why the tools work.

The professor's one-line summary:

Studying programming languages affects how we build software efficiently and the decisions we make in the process — whether it involves choosing the right language, or using the structure of one language inside another.

Let's break this down into five specific, concrete reasons.

The Five Reasons

🏗️

1. Build & Design Better Software

Learning PL concepts gives you techniques you can apply in any language. Google invented MapReduce by borrowing ideas from functional programming languages — and it made their search engines dramatically faster.

🔧

2. Create Domain-Specific Languages

General-purpose languages (Python, Java) do everything. DSLs are specialized for one thing and do it really well — like SQL for databases, regex for pattern matching, or CSS for styling. Understanding PL lets you create your own DSLs.

🧠

3. Learn New Computational Models

Different languages embody different ways of thinking about computation — imperative, functional, logic-based. Each model opens new approaches to solving problems. This circles back to reason #1.

⚡

4. Speed Up Language Learning

The more languages you know, the easier it is to learn the next one. Someone who only knows Python will struggle to pick up Rust. But if you know the underlying concepts (type systems, memory models, paradigms), every new language clicks faster.

🎯

5. Choose the Right Language

Knowing pros and cons of different languages lets you pick the best tool for the job. This is exactly what the Knuth vs. McIlroy story (coming soon!) is about — the right language for a problem can turn pages of code into a single line.

Real-world example — AI and MCP: The professor notes that modern AI tools support popular languages like Python and Java but leave out C and C++ because they're "too hard" for AI to work with reliably. This is a design decision driven by language characteristics — the complexity and low-level nature of C/C++ makes them harder to auto-generate safely. Understanding why helps you predict what tools can and can't do.

Chapter 2The Sapir-Whorf Hypothesis

Before we get into specific languages or code, the professor introduces a powerful idea from linguistics — the study of natural human languages — that directly applies to programming.

The Sapir-Whorf Hypothesis: The language we use, to some extent, determines how we view the world and how we think.

This idea comes from linguists Edward Sapir and Benjamin Lee Whorf, who studied how different natural languages shape their speakers' perceptions. The key insight is that language isn't just a way to express pre-existing thoughts — language actually shapes what thoughts are possible.

A Classic Example: Snow

The Inuit people have approximately seven distinct words for "snow" — different words for falling snow, packed snow, drifting snow, and so on. English speakers see snow as basically one thing (we modify it with adjectives: "wet snow," "powdery snow"). But Inuit speakers perceive these as fundamentally different things, because their language treats them as different things.

Their language gives them mental categories that English speakers don't naturally have.

What Does This Have to Do with Programming?

The Sapir-Whorf Hypothesis applies to programming languages in exactly the same way. The programming language you think in shapes the solutions you imagine.

If you only know C++, every problem looks like it needs classes, pointers, and manual memory management. If you know a functional language like OCaml, you start seeing problems in terms of recursive transformations and pattern matching. If you know shell scripting, you start seeing problems as pipelines of small tools. Each language gives you different mental tools.

The professor also makes an important theoretical point:

The structural diversity of natural languages is essentially limitless and recursive. There is no one general rule of syntax that all languages follow.

This is also true for programming languages. There's no single "right" way to structure a language. Different languages make radically different choices about syntax, type systems, memory models, and execution models — and each choice leads to different strengths and weaknesses.

Connection to the course: This is why you'll be learning OCaml, Prolog, and Java in this class — not just one language. Each one embodies a fundamentally different computational model (functional, logic-based, and object-oriented). By learning all three, you're expanding your mental vocabulary for thinking about computation — just like learning a new natural language gives you new ways to think about the world.

Chapter 3Choice of Notation Matters

The Sapir-Whorf idea leads to a very practical principle that the professor emphasizes as one of the most important takeaways of the entire course:

Choice of notation matters. If you have a problem that fits well in one language's design space, you probably won't do as well trying to solve it in a different language. And the converse is also true.

This isn't just academic theory. The professor tells a story: a former student came back and said Microsoft used the word-counting problem (which we'll see shortly) as an interview question — but forced candidates to solve it in C++. The professor's reaction: "Those Microsoft interviewers totally missed the point."

The point isn't to solve every problem in one language. The point is to choose the right tool for the job. A problem that takes 4 pages of carefully optimized Pascal can sometimes be solved in a single line of shell script — not because one programmer is better, but because one language is better suited to that particular problem.

Connection: This principle ties everything together. Reason #5 for studying PL (choose a good language) and the Sapir-Whorf Hypothesis both lead here. The rest of this lecture will give you a dramatic concrete example of this principle in action: the Knuth vs. McIlroy story.

Chapter 4Donald Knuth & the Birth of Literate Programming

To understand the famous Knuth vs. McIlroy story, you need some background on who these people are and what they were trying to do.

Who Is Donald Knuth?

Donald Knuth is one of the most important computer scientists in history. He won the Turing Award (the "Nobel Prize of computing") and is best known for writing The Art of Computer Programming, a multi-volume masterwork on algorithms and data structures. He's also the creator of TeX, the typesetting system that is still used today to write virtually all mathematics and computer science papers.

Knuth created TeX because he was unhappy with how the publishers were typesetting his books. Rather than just complain, he spent years building an entire typesetting system from scratch. The source code for TeX was written in Pascal and was, at the time, one of the most carefully crafted programs ever written.

What Is Literate Programming?

While writing TeX, Knuth invented a new approach to programming called literate programming. The core idea is revolutionary:

Literate programming is a method where programs are written as plain-language explanations with embedded code snippets — making them both readable (as a document) and executable (as a program). The program and its documentation live in a single source file.

Here's how it worked concretely. Knuth wrote a single source file — let's call it something like tex.tangled — that contained both the Pascal source code for TeX and the English text of the book explaining it, all intermingled together.

How the Single-Source File Worked

Pascal uses curly braces { } for comments. Knuth exploited this: inside Pascal comments, he would write the text of his book. To the Pascal compiler, these looked like ordinary comments and were ignored. But a separate simple program (let's call it "untangle") could read the same file and extract just the text portions to produce the book.

How the tangled file lookedPascal
begin { This section handles line breaking. When TeX encounters a paragraph, it needs to find optimal break points... } procedure lineBreak(text: string); { The algorithm uses dynamic programming to minimize the total "badness" of all lines... } var breakPoints: array of integer; ... end

The brilliant part: when Knuth wanted to write a comment about some piece of code, the comment was right there next to the code. He didn't have to say "go look at file X, 100 kilobytes away." Everything was in one place.

From this single file, two outputs could be generated:

One source file → two outputs: a readable book and a runnable program

Modern equivalent: If you've ever used Jupyter Notebooks, you've seen a descendant of literate programming. Jupyter notebooks interleave code cells and Markdown text cells in a single document, so you can read the explanation and run the code in one place. Knuth invented the concept decades earlier.

Chapter 5The Concordance Problem

Knuth wanted to publish a paper demonstrating literate programming in the Communications of the ACM (the ACM is the Association for Computing Machinery — the premier professional organization for computer scientists). The editor in charge of software papers was Douglas McIlroy, a manager at Bell Labs (where Unix was invented).

McIlroy suggested that Knuth demonstrate literate programming by solving a specific problem:

The Concordance Problem: Given a text input, produce a list of all the words in the text (considering only alphabetic characters A-Z, a-z), count how many times each word appears, and output the list sorted by frequency in descending order.

Let's make sure you understand the problem completely with a small example:

ExampleInput → Output
Input: four score and score years four and plus Step 1: Extract words (only A-Za-z characters count) → four, score, and, score, years, four, and, plus Step 2: Count each word → and: 2, four: 2, score: 2, plus: 1, years: 1 Step 3: Sort by count, descending Output: 2 and 2 four 2 score 1 plus 1 years

Simple to understand, right? But the way you solve it depends dramatically on what language you use. And that's exactly the point of this whole story.

Chapter 6Knuth's Solution: Elegant Pascal

Knuth solved the concordance problem in Pascal, using literate programming, and published it in the ACM paper. Remember: Knuth is a Turing Award winner — his solution was both elegant and efficient.

The Data Structure: Hash Tries

To understand why Knuth's solution was impressive (and also why it was complex), you need to know about the data structure he used.

📚 Background: What is hashing? ▼

Hashing is a technique for quickly looking up data. You take a piece of data (like a word), run it through a hash function that converts it into a number, and use that number as an index into an array. This lets you check "have I seen this word before?" in approximately constant time — O(1) — rather than searching through a list.

You've probably used hash tables (also called hash maps or dictionaries) in languages like Python (dict) or C++ (unordered_map). They're one of the most fundamental data structures in CS.

📚 Background: What is a trie? ▼

A trie (pronounced "try") is a tree-like data structure specifically designed for storing strings. Each node in the tree represents a character, and paths from the root to leaves spell out words. For example, storing "cat" and "car" would share the nodes for 'c' and 'a', then branch at the third character.

Tries are extremely fast for string operations because you only need to look at each character once — there's no comparison of entire strings like in a binary search tree.

Knuth combined both techniques into a hash trie: a trie where each node uses a hash table to store its children. This gave him blazing-fast performance — the program could process enormous texts very efficiently on a sequential machine.

The result: an elegant, highly optimized Pascal program. The paper was about four pages long (two-column, small print), containing both the code and the literate-programming commentary explaining it. It was, by all accounts, a beautifully crafted piece of work.

The professor says: "I remember reading that paper and being impressed by the idea and being impressed by the code. And then at the end, McIlroy dropped his bomb."

Chapter 7McIlroy's Bomb: One Line of Shell

Douglas McIlroy wrote an afterword to Knuth's paper. He praised Knuth's work and agreed that literate programming was a great idea. But then he said he would have solved the problem differently.

Who Is Douglas McIlroy?

McIlroy was a manager at Bell Labs and is famous for inventing one single symbol in the Unix shell: the pipe operator |. The pipe takes the output of one command and feeds it as input to the next command. It's a simple idea with enormous power — it lets you chain small, specialized tools together to solve complex problems.

McIlroy's One-Line Solution

Instead of building an elaborate data structure in Pascal, McIlroy just piped together existing Unix utilities:

McIlroy's complete solutionShell
tr -cs 'A-Za-z' '[\n]*' | sort | uniq -c | sort -nr

That's it. One line. Let's break it down piece by piece, because understanding each piece is crucial to understanding why choice of notation matters.

Step-by-Step Breakdown

Step 1: tr -cs 'A-Za-z' '[\n]*' — Split into words ▼

tr stands for "translate" or "transliterate." It's a Unix utility that replaces characters.

The flags mean:

-c means complement: instead of matching the specified characters, match everything that is not in the specified set. So -c 'A-Za-z' means "match every character that is NOT a letter."

-s means squeeze: if you produce multiple consecutive copies of the replacement character, collapse them into one.

The replacement is '[\n]*', which means "replace with a newline."

Net effect: every non-letter character (spaces, punctuation, numbers) gets replaced with a single newline. So the input text gets split into one word per line.

Example
Input: four score and score years four and plus After tr -cs 'A-Za-z' '[\n]*': four score and score years four and plus
Step 2: | sort — Sort alphabetically ▼

The pipe | takes the output of tr and feeds it into sort. The sort command arranges lines in alphabetical order. This groups identical words together — which is essential for the next step.

After sort
and and four four plus score score years
Step 3: | uniq -c — Count duplicates ▼

uniq looks at adjacent lines and removes duplicates. But with the -c flag, it doesn't just remove them — it counts how many copies of each line there were and outputs that count.

This is why we needed sort first: uniq only detects adjacent duplicates. If the identical words aren't next to each other, uniq won't merge them.

After uniq -c
2 and 2 four 1 plus 2 score 1 years
Step 4: | sort -nr — Sort by count, descending ▼

We sort again, but this time with special flags:

-n means numeric sort: sort by the numerical value of the first field, not alphabetically. (Without this, "9" would sort after "10" because '9' > '1' alphabetically.)

-r means reverse: highest counts first.

Final output
2 and 2 four 2 score 1 plus 1 years

Done! The most frequent words are at the top.

Knuth vs. McIlroy: Which Is Better?

This is a trick question — and that's the whole point. Neither solution "wins" in all categories:

🏛️ Knuth's Pascal + Hash Tries

  • Faster execution — the hash trie is a carefully optimized data structure that runs rings around the shell pipeline
  • Single-processor performance — designed to be blazingly efficient on one machine
  • Complex to write — took 4 pages of dense two-column code + commentary
  • Hard to understand — even the professor admits hash tries are tough
  • Demonstrates literate programming

🐚 McIlroy's Shell Pipeline

  • Faster to write — literally one line
  • Easy to understand — each step does one clear thing
  • Slower execution — spawns multiple processes, sorts the whole file, etc.
  • Composable — builds on existing tools rather than reinventing
  • Demonstrates the Unix philosophy: small tools chained together

The lesson: The "best" solution depends on what you're optimizing for. If you need raw speed, use a low-level language with a custom data structure. If you need to write something quickly and clearly, use the right high-level tool. Choice of notation matters.

The professor's verdict: "I knew which solution I liked. I like the short one because I could understand it."

The Microsoft Interview Story: The professor recounts that a former student reported Microsoft used this exact problem as an interview question — but required solving it in C++, forbidding other languages. The professor's take: "Those Microsoft interviewers totally missed the point." The whole lesson of the Knuth/McIlroy paper is that choice of notation matters — forcing one language defeats the purpose entirely.

Chapter 8Introduction to OCaml

Now we shift gears from philosophy to practice. The rest of the course will use three main languages — OCaml, Prolog, and Java — each representing a fundamentally different computational model. We start with OCaml.

Why OCaml? OCaml (Objective Caml) is a functional programming language in the ML family. The course textbook uses ML as one of its three example languages, and OCaml is a close variant. The differences between ML and OCaml are minor enough that you can read about ML and apply it to OCaml. You'll use OCaml to study programming language syntax — the first major topic of the course.

What Is Functional Programming?

If you've only used C++ and Python, you've primarily used imperative programming: you write sequences of statements that change the state of variables. "Set x to 5. Then increment x. Then if x > 6, do this..."

Functional programming is a different paradigm. In functional programming:

You write expressions that evaluate to values, rather than statements that modify state. Functions are "first-class citizens" — they can be passed as arguments, returned from other functions, and stored in variables, just like numbers or strings. You work with immutable data — instead of changing a value, you create a new one. Computation happens through function composition and recursion rather than loops and mutation.

This might sound strange right now, but it'll make more sense as we see examples. The Sapir-Whorf connection: learning functional programming will give you a new way of thinking about problems that you can use even when you go back to writing C++ or Python.

The OCaml REPL

To use OCaml on the course's Linux servers, you would SSH in and type ocaml. This gives you a REPL — a Read-Eval-Print Loop:

1. It reads an expression you type.
2. It evaluates (executes) that expression.
3. It prints the result.
4. It loops back to step 1, ready for your next input.

If you've used Python's interactive interpreter, it's the same concept. You type code, and immediately see the result.

Important syntax note: In OCaml's REPL, you must end every expression with ;; (two semicolons). This tells OCaml "I'm done with this expression, evaluate it now." This is a quirk of the REPL — in OCaml source files, the rules are slightly different, but for now, always type ;; at the end.

Values, Names, and Types

Let's start with the simplest possible OCaml expression: arithmetic.

OCaml REPLOCaml
# 2+2;; - : int = 4

The output - : int = 4 looks cryptic at first, but it has three parts, and understanding them is essential:

Every result in OCaml has three components:

name : type = value

In this case: - (no name) : int (the type is integer) = 4 (the value is 4)

Name (-): The dash means this result doesn't have a name. We just typed an expression; we didn't store it anywhere. It's an anonymous result — like typing 2+2 in a calculator.

Type (int): This is where OCaml differs from Python. OCaml tells you the type of every result. Python would just say 4 and let you figure out it's an integer. OCaml explicitly says "this is an int." This is part of OCaml's static type system — the language checks types at compile time, not at runtime.

Value (4): The actual computed result.

Giving Names to Values with let

To store a result with a name, use the let keyword:

OCaml REPLOCaml
# let n = 37 * 37;; val n : int = 1369

Now the output says val n : int = 1369. Let's decode it:

val means "this is a named value" (instead of - for anonymous).
n is the name we gave it.
int is the type.
1369 is the computed value (37 × 37).

Key difference from C++/Python: In OCaml, let n = 37 * 37 creates a binding, not a variable assignment. By default, n cannot be changed later — it's immutable. This is a fundamental concept in functional programming: values don't change once they're created. (There are ways to have mutable state in OCaml, but immutability is the default.)

Lists in OCaml

Lists are one of the most important data structures in functional programming. In OCaml, lists use square brackets with semicolons to separate elements (not commas, like Python):

OCaml REPLOCaml
# let l = [3; 5; 17; -12];; val l : int list = [3; 5; 17; -12]

The output val l : int list = [3; 5; 17; -12] tells us:

l is the name. The type is int list — a list of integers. The value is the list [3; 5; 17; -12].

Type Constructors: Reading Types Right-to-Left

Notice something interesting about the type: it says int list, not list of int. In OCaml, type constructors follow the data type. You read it as "an int list" — the list part is the type constructor, and int is the element type. This is the opposite of how you might say it in English ("list of integers") or in Java (List<Integer>).

Lists Must Be Homogeneous

Unlike Python, where a list can contain mixed types ([1, "hello", 3.14]), OCaml lists require all elements to be the same type:

Type error exampleOCaml
# let n = [3; 5; "ABC"; 12];; Error: this constant has type string but an expression was expected of type int

OCaml saw that the first elements were integers, so it expected the whole list to be int list. When it hit the string "ABC", it raised a type error. This is OCaml's type system protecting you: it catches mismatches before your program runs, not during.

Why is this useful? In Python, you might accidentally put a string in a list of numbers and not notice until the program crashes at runtime. OCaml catches this immediately. This is one of the big advantages of a strong, static type system — you find bugs earlier.

The Empty List and Polymorphic Types

OCaml REPLOCaml
# let nt = [];; val nt : 'a list = []

When you create an empty list, OCaml can't know what type of elements it will contain — there aren't any elements yet! So it gives it a polymorphic type: 'a list.

The 'a (pronounced "alpha" or "tick-a") is a type variable — a placeholder that means "any type." An 'a list is "a list of anything." Once you use this list in a context that requires a specific type (like appending an integer to it), OCaml will figure out what 'a should be.

String Lists and Concatenation

OCaml REPLOCaml
# let n = ["ax"; "by"];; val n : string list = ["ax"; "by"] # let m = [""];; val m : string list = [""] # n @ m;; - : string list = ["ax"; "by"; ""]

The @ operator concatenates (joins) two lists together. Here, n @ m joins ["ax"; "by"] and [""] to produce ["ax"; "by"; ""]. Notice that [""] is a list containing one element: the empty string. It's not the same as the empty list [].
